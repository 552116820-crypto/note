# Agent 框架 Q&A 记录

## Q1
为什么要自己搭建agent框架，像codex，cladue code cli的底层原理是什么？和agent有什么联系

## A1
你这个问题很关键，核心是区分三件事：**模型**、**Agent 运行时**、**业务框架**。

### 为什么要自己搭建 Agent 框架
1. 业务流程是你自己的  
   你公司的审批流、知识库、内部系统、权限体系，通用产品不一定贴合。
2. 可控性和稳定性  
   你可以定义“什么时候允许执行命令、什么时候必须人工确认、失败如何回滚”，而不是黑盒行为。
3. 成本与性能优化  
   可做模型路由（简单任务用便宜模型、复杂任务用强模型）、缓存、并发控制，显著降成本。
4. 安全与合规  
   很多场景不能把代码/数据直接交给外部 SaaS，需要私有部署、审计日志、脱敏策略。
5. 可观测和可迭代  
   自建后可做完整 tracing、评测集、A/B，对“为什么这次答错/改坏代码”可追踪可修复。

### Codex、Claude Code CLI 这类工具底层原理（通用实现视角）
它们本质是一个“**LLM + 工具调用 + 循环控制**”的 Agent Runtime，不是只有一个聊天窗口。

典型执行循环：
1. 读取目标和上下文（代码仓库、历史对话、系统指令）。
2. 模型先规划下一步（找文件、改代码、跑测试等）。
3. 通过工具接口调用本地能力（shell、文件读写、git、网络等）。
4. 把工具结果回填给模型继续推理。
5. 重复直到完成，再输出总结与变更。

可理解为：
`用户目标 -> 规划 -> 执行工具 -> 观察结果 -> 再规划 -> 完成`

关键技术组件通常包括：
- Prompt/Policy 层（行为约束）
- Tool schema（函数调用或 MCP）
- 上下文管理（截断、摘要、检索）
- 记忆（短期会话 + 可选长期记忆）
- 安全沙箱与权限控制
- 验证环节（测试、lint、回归检查）

### 它们和 Agent 的关系
- **Agent**：一种工作模式，强调“有目标、会用工具、可多步自主执行”。
- **Agent 框架/运行时**：实现 Agent 的工程底座。
- **Codex / Claude Code CLI**：面向编程场景的“现成 Agent 产品”（已封装提示词、工具、循环、权限和输出格式）。

关系可总结为：
- Agent 是概念
- 框架是基础设施
- Codex/Claude Code CLI 是具体实现和产品化形态

### 何时自建（快速判断）
1. 只是个人/小团队提效：先用现成 CLI。  
2. 要接企业内系统、强合规、强审计：建议自建或二次封装。  
3. 追求差异化能力（专属工作流/行业知识）：自建更有价值。

## Q2
相较于autogen，langgraph，langchain，llamaInedex这些通用框架，自建的框架有什么优势，请你进行详细的对比比较

## A2
**结论**  
相较于 `AutoGen / LangGraph / LangChain / LlamaIndex`，自建框架的核心优势不是“功能更多”，而是三点：`可控性`、`可治理性`、`长期稳定性`。  
如果目标是企业级生产系统（安全合规、审计、SLA、成本优化、深度内网集成），自建通常更优；如果目标是快速验证想法，通用框架更优。

### 横向对比（工程决策维度）

| 维度                 | 自建框架               | AutoGen  | LangGraph       | LangChain      | LlamaIndex |
| ------------------ | ------------------ | -------- | --------------- | -------------- | ---------- |
| 首次上线速度             | 慢                  | 快        | 中               | 快              | 快          |
| 执行流程可控性            | 最高                 | 中低（对话驱动） | 高（图/状态机）        | 中              | 中          |
| 结果确定性/可复现          | 高（可强约束）            | 中低       | 高               | 中              | 中          |
| 安全边界（权限/沙箱）        | 最高（按你定义）           | 中        | 中高              | 中              | 中          |
| 审计与合规（可追责）         | 最高                 | 中        | 中高              | 中              | 中          |
| 成本精细化优化            | 最高（模型路由、缓存、批处理自定义） | 中        | 中高              | 中              | 中          |
| 复杂业务编排（补偿、重试、事务语义） | 最高                 | 中        | 高               | 中              | 低到中        |
| RAG/索引能力           | 取决于你自己实现           | 低到中      | 中               | 中              | 高（强项）      |
| 生态与现成组件            | 低                  | 中        | 高（依托 LangChain） | 高              | 高（数据连接、索引） |
| 长期版本稳定（避免框架抖动）     | 高（你掌控）             | 中        | 中               | 中低（API变化体感较明显） | 中          |
| 团队上手门槛             | 高                  | 中        | 中高              | 低到中            | 中          |

### 相对每个框架，自建的具体优势

#### 对 AutoGen
1. AutoGen偏“多智能体对话协作”，在复杂企业流程里容易出现不可预测路径；自建可把流程收敛成强约束状态机。
2. AutoGen调试常落在“对话轨迹”，自建可以按“事件-动作-结果”做可审计链路。
3. 对权限、命令执行、人工审批点的细粒度控制，自建更容易做到严格策略化。

#### 对 LangGraph
1. LangGraph已经很接近生产编排，但你仍受其抽象边界影响；自建可定义自己的执行语义（重试、幂等、补偿、超时、熔断）。
2. 若需与内部调度系统、风控系统、审批系统深度耦合，自建集成成本更可控。
3. 框架升级可能牵动行为变化；自建能把外部依赖压在适配层，核心运行时更稳定。

#### 对 LangChain
1. LangChain开发快，但抽象层较厚，复杂场景会出现“调参靠经验”；自建可直接控制 prompt、tool call、memory、routing 全链路。
2. LangChain适合“拼装”，不一定适合“强治理”；自建更适合把治理（审计、权限、隔离）做成一等公民。
3. 性能和成本优化上，自建能做更激进的缓存和裁剪策略（例如按业务字段级缓存、按风险等级切模型）。

#### 对 LlamaIndex
1. LlamaIndex强项是 RAG/索引，不是完整 Agent runtime；自建在“工具执行、任务编排、多系统动作”上更完整。
2. 当重点从“问答检索”转到“执行型 Agent”（下单、改库、跑任务）时，自建优势明显。
3. 可把 LlamaIndex 当检索子系统接入自建框架，而不是让它承担主控制平面。

### 自建最硬的优势（通用框架难完全替代）
1. `治理优先`：权限矩阵、审批流、审计日志、脱敏策略、责任归属全自定义。
2. `运行时主权`：失败恢复、回滚、重放、幂等键、任务租约、并发控制完全可控。
3. `成本主权`：模型路由、token预算、上下文压缩、结果缓存按业务目标优化。
4. `架构主权`：可把框架当插件，而不是被框架绑定核心业务。

### 自建代价
1. 初期研发投入大，至少要补齐工具协议、状态管理、观测、评测体系。
2. 需要长期维护团队，包含平台工程和 LLM 应用工程能力。
3. 生态能力需要自己补，尤其是连接器、模板、社区最佳实践。

### 实践建议（稳妥路线）
1. 采用“`自建控制平面 + 通用框架能力平面`”的混合架构。
2. 控制平面（必须自建）：任务状态机、权限、安全、审计、成本策略。
3. 能力平面（可复用）：RAG用 LlamaIndex，编排可借鉴 LangGraph，工具层可兼容 LangChain 生态。
4. 先做可替换适配层，避免把业务逻辑直接写死在单一框架 API 上。
