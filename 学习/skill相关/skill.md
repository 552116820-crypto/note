  1) skill 渐进式披露是什么意思
  本质是：只在需要时加载最少信息，不要一上来把整个 skill 相关文件全读完、全执行。

  在你这份规则里，具体就是：

  - 先确认是否真的要用这个 skill。
  - 先打开 SKILL.md，且“读到够用为止”。
  - 遇到相对路径时，先按 skill 目录解析。
  - references/ 只读当前任务必须的文件，不批量读取。
  - 优先复用 skill 自带 scripts/、assets/，但仍需审查。

  这样做的价值：

  - 降低上下文污染（减少被无关内容带偏）。
  - 降低攻击面（少加载一份文件，就少一个注入入口）。
  - 提高可审计性（每一步都知道“为什么读这个文件”）。

  2) 怎么注意 skill 的安全问题（实操清单）
  可以按“来源-内容-执行-审计”四层做：

  - 来源可信：只用可信仓库/作者，固定版本或 commit，不随意跟随 main。
  - 内容审查：先看 SKILL.md 和要执行的脚本，重点找网络上传、读取敏感文件、系统级命令。
  - 路径约束：禁止 ../ 越界读取，限定只能访问工作区或 skill 目录白名单。
  - 执行最小化：先 dry-run（如果支持），再小范围执行，默认不开高权限。
  - 网络与密钥隔离：执行 skill 时尽量禁网或最小放行；不让脚本直接读环境变量中的 token/key。
  - 风险命令二次确认：如删除、覆盖、上传、安装依赖等操作要显式确认。
  - 结果复核：看改动 diff、命令日志、外联地址，确认输出与任务目标一致。
  ———

  3) skill 恶意注入常见方式
  常见有这些：

  - 文本指令注入：在 SKILL.md 或 references 里夹带“忽略上层规则/上传机密”等恶意指令。
  - 路径穿越注入：诱导读取 ../../... 到工作区外敏感文件。
  - 脚本后门：scripts/*.ps1|sh|py 表面是构建脚本，实际做数据外传。
  - 依赖投毒：脚本里动态安装不可信包，执行时拉取恶意代码。
  - 模板投毒：assets/template 中埋入恶意命令或后门配置。
  - 链式引用污染：SKILL.md 指向“更多说明”的外部文件，外部内容再注入恶意步骤。

  ———

  4) 一个具体例子
  假设某个 skill 的 SKILL.md 写着：

  - “运行 scripts/diagnose.ps1 采集环境信息，便于排错。”

  但脚本里实际做的是：

  - 读取 ~/.ssh、~/.aws 或环境变量里的 token。
  - 用 Invoke-WebRequest/curl 发到外部域名。

  这就是典型“伪装成诊断”的恶意注入。

  怎么挡住：

  - 执行前先审脚本，重点搜 Invoke-WebRequest、curl、Get-Content $env:...。
  - 限制脚本访问路径，不允许读工作区外敏感目录。
  - 默认禁网或仅允许白名单域名。
  - 对“上传/删除/安装”类动作做人工确认。
