• 对，就你这个仓库里的实现来说，基本可以这么理解。

  这里的 skill 本质上不是“模型学会了一个新能力”，而是“系统在模型调用前，先帮它挑出相关说明，再把这些说明和运行时状态按固定结构塞进 prompt”。从代码看，skill 的作用链路就是：

  - 先路由选 skill：router.py:22
  - 再读取 skill 内容：skill_registry.py:52
  - 再组装成 system/developer/user：prompt_assembler.py:8

  所以它更像“外置的提示词工程 + 上下文裁剪 + 状态注入”，不是训练层面的 skill。

  你说“减少上下文压缩影响”，这个说法也基本成立，但更准确一点是：

  - 它减少的是“无关上下文干扰”和“重要信息被稀释”的问题。
  - 它通过只注入当前相关的 section、reference、runtime state，让模型前面看到的信息更集中。
  - 它把一部分记忆放到模型外部，比如 .agent_runtime/state/，避免完全依赖模型在超长上下文里自己记住。

  但要注意边界：

  - 当前实现只是“把 state 写盘，再拼回 prompt”，还没有真正把旧 state 自动加载进下一轮续跑逻辑里；load() 虽然有了，但 demo.py 还没接上。runtime_store.py:14 demo.py:61
  - 所以现在它主要还是“更有组织的 prompt 构造器”，不是完整的 agent memory system。
  - 它能提升命中率、稳定性和可控性，但不能从根本上替代模型内部的理解能力，也不能保证“完整上下文绝不丢失”。

  更准确的定义是：这个仓库里的 skill 是“可路由的外部能力包”，目前落地形态是“结构化提示注入”。如果后面再加上工具执行、状态恢复、步骤约束、结果校验，它才会从“提示词包”进一步变成“可执行的 agent capability”。
