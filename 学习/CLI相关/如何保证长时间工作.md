# 最后两轮对话原文

## 第 1 轮

### Q

你是如何进行长时间的工作的，比如我要你做一个项目，你可能会做几个小时，你是怎么保证长时间工作能正常运行

### A

长时间工作靠的不是“把所有内容一直记在脑子里”，而是把关键信息持续外部化，然后反复校验。

更具体地说，我通常靠这几层来保证长时间任务还能稳定推进：

1. 把工作拆成小阶段  
一个几小时的项目，我不会当成一整块连续操作去做，而是拆成明确阶段，比如：
- 先读代码和确认约束
- 再改一小块
- 然后跑验证
- 再进入下一块

这样即使中间上下文被压缩、会话变长，损失的也只是局部细节，不会把整个任务弄乱。

2. 把状态写进“工作区”，而不是只留在上下文里  
真正可靠的长期记忆，不是对话本身，而是这些持久化产物：
- 代码改动
- 提交差异
- TODO 文档
- 临时说明文件
- 测试结果
- 日志输出
- 配置文件和脚本

所以长任务里，我会尽量让“当前进展”体现在文件系统里。这样即使会话被压缩、重启，或者我要重新读取上下文，也可以从项目本身恢复状态，而不是靠回忆。

3. 在关键节点做“可恢复检查点”  
我通常会在每个阶段结束时确认几件事：
- 已经改了什么
- 为什么这么改
- 哪些已验证
- 哪些还没做
- 下一步是什么

如果任务很长，最稳妥的做法是把这些写成短说明，或者直接体现在代码结构、测试、文档里。这样我后面继续工作时，只要重新读这些检查点，就能接着做。

4. 反复重新读取，而不是假设自己一直没忘  
长时间工作时，可靠性来自“重新获取上下文”，不是“永远记住上下文”。  
例如我会反复：
- 重新看相关文件
- 搜索调用链
- 检查最新 diff
- 看测试是否通过
- 核对之前结论有没有被后续改动推翻

这其实和人工程师做大型任务一样。好的方式不是死记，而是随时能重新定位事实。

5. 用验证来兜底，而不是只靠推理  
长任务最容易出问题的地方，是前面改动看起来合理，后面却发现破坏了别的部分。  
所以我会尽量用这些手段压住风险：
- 运行测试
- 静态检查
- 构建验证
- 查看运行日志
- 对照需求逐项核对

也就是说，我不是靠“我觉得没问题”，而是靠“我验证过没问题”。

**和上下文窗口的关系**
这和你前面问的上下文机制直接相关：  
我并不是把几个小时的全部对话原文一直塞在上下文里工作。上下文是有限的，长会话里旧内容可能被压缩、筛选、总结。

所以要长期稳定，核心不是扩大记忆，而是把重要状态转移到外部：
- 项目文件
- 结构化笔记
- 计划
- 测试结果
- 变更记录

**最实际的一句话**
如果你让我做一个要持续几小时的项目，最可靠的方法不是“让我一直记住所有对话”，而是我一边做，一边把进展沉淀成代码、文档、检查点和验证结果。这样即使上下文变短，工作也不会失控。

- 怎样降低上下文丢失风险

## 第 2 轮

### A

下面这个模板适合让 AI 连续做几小时的项目，核心目标只有两个：任务可分段推进，进度可恢复。

**协作模板**
```markdown
# 项目任务单

## 1. 目标
我要你在当前工作区完成这个项目：

- 项目名称：
- 最终目标：
- 交付物：
- 截止标准：

## 2. 范围
这次要做的内容：

- [ ] 功能 A
- [ ] 功能 B
- [ ] 功能 C

这次不要做的内容：

- 不做：
- 不改：
- 不处理：

## 3. 约束
必须遵守这些约束：

- 不要破坏现有接口
- 优先复用现有实现
- 需要兼容：
- 只能修改这些目录：
- 不要动这些文件：
- 需要保留日志 / 注释 / 文档风格：

## 4. 验收标准
完成后至少满足：

- [ ] 能正常构建
- [ ] 相关测试通过
- [ ] 关键路径可运行
- [ ] 没有明显回归
- [ ] 给出变更说明
- [ ] 给出未完成项和风险

## 5. 工作方式
按下面方式持续工作，直到任务完成或遇到明确阻塞：

1. 先读代码和现状，不要先假设实现方式。
2. 先给出简短计划，再开始改。
3. 每完成一个阶段，就把进展写入一个项目内的进度文件。
4. 每次改动后，优先运行最相关的验证，不要只凭推断。
5. 如果上下文变长，优先依赖进度文件、TODO、测试结果和代码现状继续工作。
6. 除非有真实阻塞，否则不要频繁停下来问我。
7. 如果发现需求有歧义，先做最保守、最容易回滚的实现，并把假设写清楚。

## 6. 进度文件要求
请在仓库里维护这些文件：

- `WORKLOG.md`：记录已完成内容、当前结论、下一步
- `TODO.md`：记录待办、阻塞项、优先级
- `DECISIONS.md`：记录关键技术决策和原因
- `VERIFY.md`：记录测试、构建、验证结果

如果项目较小，至少维护一个 `WORKLOG.md`。

如果遇到阻塞，按这个格式汇报：

一句话说明问题

## 9. 最终输出要求
任务结束时，请给我：

- 做了什么
- 为什么这么做
- 哪些文件改了
- 怎么验证的
- 还有什么风险或未完成项
```

**最实用的用法**
你下次可以直接这样对我说：

```markdown
按这个任务单方式持续工作：

目标：给现有项目增加用户权限控制和审计日志
范围：后端接口、数据库迁移、后台页面
不做：短信通知、第三方登录
约束：不要改现有 API 路径；优先复用现有 auth 模块
验收：相关测试通过；管理员能分配角色；操作有审计记录

额外要求：
- 维护 `WORKLOG.md`
- 每完成一个阶段就更新检查点
- 除非真实阻塞，否则继续往下做
```

**为什么这个模板有效**
因为长时间任务真正可靠的不是“对话记忆”，而是这四样东西：

- 明确范围，避免任务无限膨胀
- 阶段检查点，避免做到一半失去状态
- 文件化进展，避免依赖会话记忆
- 持续验证，避免错误累积到最后才暴露

**如果你要把稳定性再提高一层**
可以再加一条规则：

```markdown
每完成一个里程碑，就把“当前状态总结 + 下一步 + 风险”写入 `WORKLOG.md`，后续继续工作时优先读取这个文件恢复上下文。
```
