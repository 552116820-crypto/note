不是的，这是个很常见的误解。来解释真实的机制：

---

## 上下文是怎么工作的

**LLM 没有"记忆"，它每次都是从零开始读完整个对话历史。**

每一轮对话，发给模型的内容是：

```
系统提示 + 第1轮问答 + 第2轮问答 + ... + 第N轮问答 + 当前问题
```

**全部原文，不做任何筛选或压缩。** 这就是为什么上下文会越用越少——每轮都在累积。

---

## 上下文容量的计算方式

```
已用 token = 系统提示 + 所有历史消息(原文) + 工具调用结果 + 当前输入
```

Claude Code 里还包括：

- CLAUDE.md 内容
- 读取过的文件内容（`cat`、`read` 的结果）
- bash / grep / 搜索的输出
- 工具调用记录

这些**全部塞进同一个上下文窗口**，所以读一个大文件就能吃掉大量 token。

---

## 那压缩是什么时候发生的？

压缩**不是每轮自动发生**，而是在两种情况触发：

1. **手动 `/compact`**：你主动触发
2. **自动 auto-compact**：上下文接近 95% 时系统强制触发

触发后，Claude 把整段对话历史**归纳成一份摘要**，用摘要替换掉原始对话，上下文就"腾出来了"。这是一次性有损压缩，不是每轮都在做的持续处理。

---

## 为什么会有 Cache（缓存）？

你可能见过 `cache_read_input_tokens` 这个字段。这是 **Prompt Caching** 机制：

对于每次请求中**重复出现的前缀**（比如系统提示、CLAUDE.md），Anthropic 服务器会缓存它的 KV 计算结果，下次请求直接复用，**不重新计算但仍然占用上下文空间**。它能降低延迟和成本，但不会减少 token 占用量。

---

## 一句话总结

> 上下文就像一张纸，每轮对话都往上写，写满了才压缩成摘要换一张新纸。不是每轮在筛选——而是一直在积累，直到装不下为止。
