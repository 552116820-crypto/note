## Claude Code CLI 的"剩余百分比"是什么？

**是的，它就是可用上下文余量**——即当前会话的上下文窗口还剩多少空间可用。

---

### 三个概念的区别

**1. 上下文窗口（Context Window）**

这是模型在单次会话中能"看到"的最大信息量，相当于 AI 的工作记忆上限。上下文窗口是 AI 的工作记忆，模型在任意时刻只能处理有限数量的信息，这个上限以 token 为单位衡量——大致相当于词语，但标点和代码符号也计入。 Claude Code 通常使用的模型上下文窗口为 **200,000 tokens**。

**2. 剩余百分比（remaining_percentage）**

这个百分比来自 `context_window` 对象，其中 `used_percentage` 是基于当前 API 调用中的输入 token 计算出来的（包括 `input_tokens`、`cache_creation_input_tokens`、`cache_read_input_tokens`），不包含输出 token。`remaining_percentage` 就是 `100 - used_percentage`。

简单说：**剩余% = 这一轮 API 调用实际占用了多少上下文后，还剩下多少空间。**

**3. 消耗总 Token（total_input_tokens / total_output_tokens）**

累计总量（`total_input_tokens`、`total_output_tokens`）是整个会话所有轮次的 token 总和，适合追踪整体消耗；而当前用量（`current_usage`）才是最近一次 API 调用的真实状态，用于计算准确的上下文百分比。

---

### 关键区别汇总

|概念|含义|用途|
|---|---|---|
|上下文窗口大小|200k token 的总容量上限|固定参考值|
|剩余百分比|当前这轮调用还剩多少上下文空间|判断是否需要压缩|
|消耗总 Token|整个会话累计消耗（计费参考）|追踪费用/用量|

---

### 实际影响

低于 70% 是理想状态。超过这个阈值，Claude Code 可能开始遗忘会话开头讨论的细节。当超过 80% 时，建议开启新会话或使用 `/compact` 命令压缩上下文。

当上下文使用率接近 95% 时，系统会自动触发压缩（auto-compact），在保留关键信息的前提下允许会话继续进行。

你可以随时在 Claude Code 中输入 `/context` 命令，获取详细的上下文分解，包括系统提示、工具定义、Memory 文件、对话消息各自占用的 token 数。

## 用完了会发生什么？

Claude Code 会自动触发 auto-compact：先清除旧的工具输出（文件读取、grep 结果、bash 输出等），再把整段对话历史压缩成一份结构化摘要（完成了什么、正在做什么、修改了哪些文件），然后以这份摘要作为新的上下文基线继续运行。

所以**是的，压缩后百分比会重置**，会话可以继续。

---

## 压缩是有损的——这是关键

压缩不是无损备份，它是**有损摘要**。你的请求和关键代码片段会被保留，但对话早期的详细指令可能丢失。

摘要本质上会损失部分信息——Claude 擅长提炼要点，但某些细节会被压缩或省略。

---

## 大型项目会变差很多吗？

**取决于压缩发生在什么时机。**

在实现达到逻辑断点时压缩 → 问题很少；在复杂实现中途压缩 → 有质量下降风险。如果在工作中途触发压缩，当时持有的实现细节和上下文可能在摘要中丢失，导致表现出"忘记了"的行为。

开发者描述了这种模式：你深入进行复杂重构，进展顺利，然后突然 Claude Code 开始挣扎——响应变得泛泛，之前的决策被遗忘，代码质量明显下降。

---

## 怎么应对大型项目？

**核心思路：主动管理，而不是被动等待自动压缩。**

**1. 用 CLAUDE.md 固化关键信息** CLAUDE.md 是 Claude Code 在每次会话开始时读取、并在每次压缩中保留的特殊文件，是唯一能在整个会话中可靠存活的指令存放处。把编码约定、项目结构、关键文件路径、常用命令写进去，保持在 200 行以内。

**2. 在逻辑断点手动 `/compact`** 建议在完成一个任务时运行 `/compact`，避免不相关任务堆积在同一个对话里。 还可以加指令：`/compact Preserve all file paths, error messages, and the list of modified files`

**3. 不相关任务用 `/clear` 开新会话** 当你从实现一个功能切换到调试完全不相关的问题时，运行 `/clear`。第一个任务的上下文对第二个任务来说是纯噪音，重新开始能给你干净的 160k+ token，而不是被污染的 80k。

**4. 用子代理隔离复杂任务** 子代理有自己独立的新鲜上下文，完全与主对话隔离，工作结束后只返回摘要，不会撑大主上下文。这也是为什么子代理对长会话很有帮助。

---

**总结**：大型项目本身不会"因为超出上下文就坏掉"，但如果在关键时刻被迫自动压缩，确实会变差。主动在任务断点手动压缩 + 善用 CLAUDE.md，可以让效果保持稳定。
