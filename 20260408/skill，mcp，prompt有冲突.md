❯ 当skill，mcp，prompt有冲突时，你会怎么做，有没有个优先级

● 好问题。以下是冲突时的实际优先级，从高到低：

  优先级排序

  1. 安全约束（最高） — 内置的安全规则永远不会被任何来源覆盖
  2. 用户在对话中的直接指令 — 你当前说的话优先级最高
  3. System prompt / 系统提示 — Claude Code 的基础行为定义
  4. CLAUDE.md 文件 — 项目级配置指令
  5. Skill 展开后的提示 — 技能被触发后展开为一段 prompt，指导行为
  6. MCP 工具 — 只是可调用的工具，本身不携带行为指令

  具体冲突场景

  Skill vs 用户 Prompt： 用户的直接指令优先。如果你说 "不要用 TypeScript"，即使 skill 模板里建议用
  TypeScript，也会遵从你的要求。

  MCP vs Skill： 它们冲突较少，因为本质不同 — MCP 是"能力"（tools），Skill
  是"行为指导"（prompts）。如果两者都能完成同一任务，一般优先使用更专业/匹配的那个。

  Prompt（CLAUDE.md）vs Skill： Skill 被显式触发时，其指令在该任务范围内生效，但不会覆盖 CLAUDE.md
  中的通用规则。两者互补而非互斥。

  MCP vs 内置工具： 内置工具（Read/Edit/Grep 等）优先于 MCP 提供的类似功能，因为内置工具体验更好、更可控。

  核心原则

  - 越接近用户意图的来源，优先级越高
  - 安全约束不可协商
  - 具体指令 > 通用指令（比如针对某个文件的指令 > 全局规则）
