• 不是。更准确地说：

  图片不是简单先 OCR 成文本。
  GPT、Gemini、Claude 这类多模态模型可以把图片作为视觉输入处理。内部当然会把图片转成模型能处理的表示，比如 image
  tokens / embeddings，但这不是“先识别成文字再喂给文本模型”。所以它能看物体、布局、图表、颜色、空间关系，不只是读图里的
  字。OpenAI 文档明确说模型可以处理 image inputs，Gemini 也说模型是 multimodal 并支持图像理解。

  文件通常会被解析。
  PDF、Word、Excel、CSV 这类“文件”本身不是模型天然理解的语义单位。平台一般会做一层预处理：提取文本、分页、切块、建立检索
  索引，或者把 PDF 页面渲染成图片再让视觉模型看。Claude 的 PDF 文档就提到 PDF 支持依赖视觉能力，并且某些模式会退回到纯文
  本提取；Claude Files API 也说明不支持为 document block 的格式要先转成纯文本。

  视频可以上传，但不是所有模型/入口都支持。
  Gemini API 明确支持视频输入，可以上传视频文件、inline video data，甚至 YouTube URL；其文档还说明默认会按约 1 FPS 抽
  帧，并处理音频。也就是说视频通常会被拆成帧、音频、时间戳等 token 化输入，不是“完整 mp4 原封不动进 Transformer”。

  对 GPT/OpenAI 来说，当前官方 API 文档明确列的输入模态是 text、image、audio/voice，另有 Sora/video generation 是生成视
  频，不等同于把视频上传给 GPT 做视频理解。Claude API 的文件/视觉文档主要列图片、PDF、文本等，没看到通用 video content
  block；通常要先抽帧、转字幕、转音频文本再给模型。

  一句话：图片已经能被多模态模型直接“看”；文件多数是解析后进入模型；视频有些模型能直接接收，但背后仍会抽帧/处理音频，不
  是所有 GPT/Claude/Gemini 入口都一样。

• 大体上是的，但要把“处理方式”分成几层：

  1. 图片：多模态原生输入
  不是先转文字，而是把图片编码成视觉 token / embedding，再和文本 token 一起进模型。模型能看图像内容、布局、颜色、图表、w
  文字等。

  2. 文档：通常先解析，再进模型
  PDF、Word、Excel、CSV 这类文件一般会被平台处理成：文本抽取、页图像、表格结构、分块、向量检索结果等。
  所以文档不是“原始文件直接被模型读懂”，而是经过解析管线后交给模型。

  3. 视频：通常抽帧 + 音频处理 + 时间信息
  支持视频的模型，比如 Gemini，会把视频采样成帧，同时处理音频和时间戳。不是把 mp4 原封不动塞进模型。OpenAI/Claude 的通用
  聊天 API 对“视频理解”的支持则没 Gemini 那么直接，常见做法还是抽帧、转字幕、提音频后输入。

  4. 音频：转成音频 token 或先转文字
  有些模型原生处理音频，有些系统会先 ASR 转文字。实时语音模型一般更接近原生音频输入。

  5. 大文件/知识库：RAG 或工具调用
  如果文件太大，通常不会全部塞进上下文，而是切块、索引、检索，把相关片段给模型。

  所以结论是：主流多模态模型并不只是“全都转成文本”了；图片、音频、视频可以有原生多模态表示。但文件、长视频、大型资料库通
  常仍依赖解析、抽帧、切块、检索这些工程管线。

  更准确的一句话：现在的大模型能处理多模态输入，但它们处理的不是原始文件本身，而是被编码/token 化后的多模态表示。
