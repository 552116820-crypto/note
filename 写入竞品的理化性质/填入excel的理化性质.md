## 你的判断，我基本同意，但理由要拧一下

**1. 入口用 command —— 对。** Skill 的定位是「模型靠 description 自动触发」；command 是「你手动触发」。你每次"手上有新产品要对标"是一个你说了算的明确时刻，不该让模型猜。入口必然是 command。

**2. 重活用 agents —— 对，但理由不是"agent 管执行"这种笼统说法。** 真正理由是 **fan-out + 上下文隔离**：6 个品牌，每个要翻官网、产品目录、PDF 数据表，全是网页噪音。串在主线程里做，做到第 6 家时上下文已被前 5 家的原始网页喂脏，质量下滑。一个品牌派一个 agent,各自独立上下文里翻、只回 5 行结构化结果、还能并行——这才是该上 agent 的点。

**3. "skill 根本不合适" —— 结论同意，说法太狠。** Skill 不只是"一种触发方式",它是个**知识包**(能捆带资源文件、模板、脚本，渐进式加载)。它在这里不合适的真正原因是:你**用不上它的特性**——不需要模型自动触发，方法论也小到不需要渐进式加载。是"用不上牛刀",不是"牛刀错"。

## 最终结构(3 个文件)

**① `commands/competitor-xref.md` —— 入口命令**
你触发它、丢一个产品进去。职责:读 reference → 解析源产品(品类/分类/VG)→ 并行派 6 个 `competitor-scout`(一个品牌一个)→ 汇总 → 渲染矩阵表 → 写 CSV + 一份 markdown 摘要(列出来源链接和低置信度格子)。
参数:产品规格,比如 `HDRS 系列 / HDRS32,46,68 / HFDU 抗燃液压液`,或直接丢一份数据表路径让它自己解析。

**② `agents/competitor-scout.md` —— 单品牌调研 agent**
工具受限为 `WebSearch / WebFetch / Read`。角色固定:给定**一个**品牌 + 源产品规格,去该品牌目录找各 VG 等效型号,按严格 schema 回 `品牌 / 各VG对应型号 / 数据表URL / 置信度 / 备注`。找不到就回"同系列无对应"。6 个实例并行跑,各自独立上下文,主线程不被网页噪音污染。

**③ `competitor-xref-method.md` —— 方法论 reference**
command 和 agent 都引用同一份。内含:
- 固定品牌清单:福斯 / 壳牌 / 美孚 / 道达尔 / 嘉实多 / 奎克好富顿
- 判定标准:按流体分类(ISO 6743/4,如 HFDU)→ ISO VG 黏度等级 → 应用工况 三级匹配
- 输出表 schema + "同系列无对应"写法 + 低置信度标记规则
- 检索式模板:`[brand] fire resistant hydraulic fluid HFDU [VG]` 等

## 几个落地细节

- **品牌清单写死在 reference**,以后换/加竞品只改这一处,command 和 agent 自动同步。
- **每格强制带数据表 URL + 置信度**(防 LLM 编型号),低置信度在 CSV 里单列标出,你人工复核。
- CSV 用 UTF-8 BOM,Excel 双击即开;以后要真 `.xlsx` 带样式,再加个 python 转换脚本即可,不影响主流程。
