## 行为时间轴 · 是什么 & 为什么这样设计

> **行为时间轴**：把一家公司的可见动作（工商变更、资质、专利、投资、诉讼、政策提名、合规事件……）按时间排列，每条配一句 AI 写的"是什么意思"，再附一段爬取的最新动态。

为什么这样组织：

1. **左列时间是骨架** —— 让人一眼看出"哪几年密集发生事"。元丰的密度峰值在 2024-08 到 2025-12，这是真正值得追问的窗口。
2. **中列事件是事实** —— 必须能溯源到工商/政府/法院/专利局，否则不放。
3. **右列 AI 一句话是解读** —— 把"股东变更"翻译成"实控人换血"，把"经营范围变更"翻译成"业务从单一型钢扩到综合贸易"，让非财经背景的人也能秒读。
4. **末尾的水位线** —— 行为时间轴最大的风险是"AI 看似什么都知道，其实拼接了错信源"。所以专门留一节标出哪些是确证、哪些待核、同名公司怎么排除的——这是这类产品能否被尽调/风控真正用起来的关键。

如果你打算把这个做成产品页面，建议把"右列 AI 一句话"和"末尾的水位线"作为可点开/可收起的两层——一线用户看时间+一句话就够，做尽调的人再展开看信源核验。

## 主张

## 7 个非显而易见的坑

1. **同名消歧会指数级爆炸**。我做元丰那条时差点就把"正丰/国丰/福丰"的事件混进去了——百万规模里这种"X丰钢铁"会成片出现。**USCC（统一社会信用代码）必须是主键**，公司名只是别名。任何只用名字 join 的步骤都是定时炸弹。

2. **搜索引擎不是数据源，是兜底**。Tavily/Jina 这类适合做"长尾事件 + 实时新闻"，不适合做骨架。骨架要走结构化源：企查查/天眼查 API（工商+诉讼+处罚）、裁判文书网、国家知识产权局、信用中国、地方生态环境局公示页。**先把 90% 的常规事件用结构化源批量拉满，搜索只用来补长尾和"最新动态"那一栏**。

3. **存事件，不要存 Markdown**。时间轴在数据库里应该是 `event(uscc, ts, type, payload_json, source_url, confidence)` 这种规范化的事件表。"AI 一句话"和最终的 Markdown 是渲染层的产物，可缓存可重渲染。这样换 prompt、换模型、换前端样式不用重跑 100w。

4. **LLM 调用要走杠杆**。把事件类型枚举化（股东变更/法人变更/经营范围变更/行政处罚/专利/投资/政策提名……），每类预写模板句式，LLM 只对**长尾事件**和**整段时间轴的总结**做生成。算账：
   - 全 LLM：100w × 20 事件 ≈ 2000w 次调用
   - 模板 + LLM 整体归纳：100w × 1~2 次 ≈ 100~200w 次
   差 10~20 倍。

5. **热度分层，不要均匀刷新**。100w 客户不可能都日更。按使用频次 / 风险评分分 A/B/C：A 类订阅日更，B 类周更，C 类按需触发（用户点开才补"最新动态"那一段）。

6. **可信度水印不是装饰**。刚才 demo 末尾那段"信息核验提示"在单条里看像花活，在 100w 规模下是核心产品差异点——用户用你的时间轴做尽调/风控，需要知道哪些字段是工商口径硬证据、哪些是 AI 拼接。建议每个事件挂 `source_count / source_diversity / contradicts_other_source` 三个字段，前端按颜色分级显示。

7. **冷启动数据债会复利**。如果你 V1 就直接灌 100w，错误率 1% 也意味着 1w 条错误流到下游。**建议先选 1000 家分布合理的标杆（含同名陷阱、跨省迁移、已注销、个体户等边界情况）跑通管道，再放量**。

## 最小架构

```
实体层(USCC 主键, 别名/历史名)
   ↓
事件层(标准化 schema, 多来源 join)
   ↓
摘要层(模板句 + LLM 整体归纳, 全部带 source_url)
   ↓
服务层(JSON API, 前端再渲染 Markdown)
   ↑
信源监听层(订阅工商变更/裁判文书/处罚公示 → 增量进事件层)
```

## 你会遇到的 4 类输入

| 类型 | 占比（经验值） | 处理 |
|---|---|---|
| L1 标准化后精确命中 | 60~70% | 直接拿 USCC |

1. **采购数据，别靠 API 现查**。100w 次企查查/天眼查 API 调用按单价算几十万人民币；买一份企业库全量 dump 本地建索引一次性几万到十几万，且后续可复用。这是为数不多"买比租划算"的场景。

→ 100w 量级靠人手点页面不现实；爬虫面临反爬 + 法务风险，不推荐。要么升级到 API，要么从他们那里采购一份"全量企业风险数据 dump"（百万级客户场景他们都有报价方案）。

**3. 全部走免费政府公开源**
→ 国家企业信用信息公示系统、信用中国、裁判文书网、生态环境局公示、应急管理局通报。覆盖、稳定性、字段统一性都比商业 API 差一档。**制造业有个利好**：产能置换、排污许可、安全处罚这些公开度比较高，比互联网/服务业的公开度好得多。

## 一个常被忽略的捷径

你公司既然已经花钱从企查查/企知道**拉出客户名单**，**问采购/IT 一下当初签的是哪种合同**——很多企业级合同里"导出名单 + API 调用额度 + 数据 dump"是打包的，可能你不知道而已。一句话问出来，能省你一个项目周期。

如果只是"我同事手动导了张表给我"，那这事卡在数据采购、不在技术。建议先把这条厘清，再决定走 1/2/3 哪条路——三条路对应的工程方案、成本量级、上线时间完全不同。

## 正确分工

```
骨架(90% 事件) ← 结构化源（企查查 API / 政府公开库）
长尾 + 实时动态(10%) ← LLM + 搜索 ← 你擅长的这一路
```

具体说：

- **"工商/股权/处罚/司法/招投标/专利"** → 必须结构化源，LLM 不沾
- **"行业政策提名/最新新闻/舆情/AI 一句话解读"** → 这才是 LLM + 搜索的主场

元丰那条 demo 里最后那段"AI 爬取的最新动态"就是这一类——它是骨架时间轴**之外的附加价值**，不是骨架本身。

## 一句话

把 LLM + 搜索当成**时间轴的"加分项"**，而不是"主路径"。骨架数据必须采购或拉公开库，省不掉。
