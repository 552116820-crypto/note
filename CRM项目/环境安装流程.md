• 先按这个顺序做，目标是把后端服务先跑起来。

  1. 先把项目放到 WSL Linux 文件系统
  你现在项目在 /mnt/e/...，建议先移动到类似 ~/projects/crm_manage_system，不然 Docker bind mount 性能和权限都容易出问题。

  mkdir -p ~/projects
  cp -r /mnt/e/crm_system_dev/crm_manage_system_2.0_full-dev-3a3022f1d0335a18abd1929f58ab8b1952124575 ~/projects/crm_manage_system
  cd ~/projects/crm_manage_system

  2. 确认 Docker Desktop 已装好并打开了 WSL Integration
  Windows 里先装 Docker Desktop，然后打开：

  - Settings > General > Use WSL 2 based engine
  - Settings > Resources > WSL Integration，把你当前 Ubuntu 打开

  在 WSL 里确认：

  docker version
  docker compose version

  3. 准备环境文件
  项目里现在有两套模板：

  - ThinkPHP 应用环境：.env.docker.example
  - Compose 环境：.compose.env.example

  执行：

  cp .env.docker.example .env
  cp .compose.env.example .compose.env

  如果你本地要连 NCC SQL Server，再改 .env 里的 [NCC_DATABASE]。
  如果暂时不调 NCC，先保持空着也可以，只是相关功能别点。

  4. 构建并启动容器

  docker compose --env-file .compose.env build php
  docker compose --env-file .compose.env up -d

  5. 安装 PHP 依赖

  docker compose --env-file .compose.env exec php composer install

  6. 检查关键扩展

  docker compose --env-file .compose.env exec php php -m | grep -E 'redis|imagick|sqlsrv|pdo_sqlsrv'

  7. 如果你有数据库备份，导入 MySQL
  这是关键。没有库结构和数据，项目服务就算启动了，大部分页面也不能正常用。

  先进 MySQL：

  docker compose --env-file .compose.env exec mysql mysql -uroot -p

  如果你有 sql 文件，导入类似：

  docker compose --env-file .compose.env exec -T mysql mysql -uroot -proot123 crm < /path/to/your_dump.sql

  8. 打开项目
  后端入口会在：

  http://localhost:8080

  Nginx 配置在 default.conf:1，Compose 在 compose.yaml:1。

  9. 如果还要跑 VueAdmin 前端

  docker compose --env-file .compose.env run --rm node14 npm install
  docker compose --env-file .compose.env run --rm node14 npm run dev

  10. 常用排查命令

  docker compose --env-file .compose.env ps
  docker compose --env-file .compose.env logs -f php
  docker compose --env-file .compose.env logs -f nginx
  docker compose --env-file .compose.env logs -f mysql
  docker compose --env-file .compose.env exec php php think

• 如果你是想“停掉项目容器”，用这个：

  docker compose --env-file .compose.env down

  如果你只是想“临时停止”但保留容器：

  docker compose --env-file .compose.env stop

  如果你是在看日志，退出日志界面直接按：

  Ctrl+C

  如果你是进了容器 shell 里，退出容器终端用：

  exit

  或者按：

  Ctrl+D

  补一句区分：

  - stop：停容器，数据卷还在
  - down：删容器和网络，但卷默认还在
  - down -v：连数据卷一起删，这个别随便用，会删 MySQL/Redis 数据


• 不是 .compose.env 的问题，真正卡住的是镜像构建。

  我复现到的失败原因有 3 个：

  - docker/php/Dockerfile:5 用的是 php:7.3-fpm-buster，Buster 已经 EOL，原来的 buster-updates 源失效，apt-get update 会报错。
  - docker/php/Dockerfile:34 里 gd 扩展配置参数写成了 --with-freetype --with-jpeg，这套写法不适用于 PHP 7.3，所以会报 unrecognized options.
  - docker/beanstalkd/Dockerfile:3 构建时偶发 DNS 解析失败，导致 apt 找不到 beanstalkd 包。

  我已经把这两个 Dockerfile 修掉了：Buster 改成走 archive 源，给 apt 加了重试，并把 GD 参数改成 PHP 7.3 兼容写法。之后重新执行 docker compose --env-file .compose.env up -d 已经成功，当前容器都在运行，Nginx 暴
  露在 8080，MySQL 在 3307，Redis 在 6379，Beanstalkd 在 11300。

  补一句长期建议：PHP 7.3 + Debian Buster 都太老了，这次是兼容性修补，后面最好安排升级基础镜像。
