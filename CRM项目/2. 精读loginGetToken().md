  3. 组装 admin_info 198-226

  这里开始把登录人上下文打包成一个大对象：

  - 基础账号信息：id、username、type、role_id
  - 权限和菜单：permissions、menu_list、menu_path
  - 数据范围：unit_departments_code_list、department_relations
  - 业绩范围：warzone_performance、system_performance、department_performance
  - 员工信息：real_name、people_name、employee_code、userID
  - 部门信息：department_code、department_name
  - 联系方式及验证状态：mobile、email、各类 verification 标记

  这个 admin_info 很关键。你可以把它理解成：

  admin_info = {
      "who_am_i": ...,
      "what_can_i_do": ...,
      "what_menu_can_i_see": ...,
      "what_data_scope_can_i_access": ...
  }

  也就是说，这个项目登录成功后，不只是回一个 token，而是回一个“完整运行上下文”。

  6. JwtService 到底做了什么

  看 JwtService.php:25。

  createToken() 的逻辑很简单：

  - iat：签发时间
  - nbf：生效时间
  - exp：过期时间
  - data：自定义数据

  真正放进去的是：

  [
    'token_validity_time' => $login_time,
    'adminToken' => $randomString
  ]

  所以这个 JWT 本质上只是一个“门票”，真正的登录上下文还是服务端表里的那条 crm_admin_token 记录。

  这就是为什么：

  - 退出登录时删数据库记录就能失效
  - 中间件里还要再去查 AdminTokenModel

  这是“JWT + 服务端会话表”的混合方案，不是纯无状态 JWT。

  7. 用 Python 思维重写一遍

  你可以把 loginGetToken() 暂时理解成这样：

  def login_get_token(admin, login_safe_set):
      if not admin.employee_code:
          return fail("管理员员工编号资料缺失")

      employee = find_employee(admin.employee_code)
      if not employee or employee.deleted:
          return fail("员工信息异常")

      department_record = find_employee_department(admin.employee_code)
      if not department_record:
          return fail("员工部门信息不存在")

      department = find_department(department_record.dept_code)
      if not department:
          return fail("部门信息不存在")

      permissions, menu, scope = build_permission_context(admin)

      admin_info = {
          "id": admin.id,
          "username": admin.username,
          "permissions": permissions,
          "menu_list": menu,
          "scope": scope,
          "employee_code": employee.employee_code,
          "department_code": department.code,
          ...
      expire_at = now + timeout
      admin_token = random_string()

      delete_old_tokens(admin.id)
      delete_expired_tokens()

      save_admin_token(
          admin_id=admin.id,
          token=admin_token,
          admin_info=json.dumps(admin_info),
          login_time=expire_at,
      )

      admin.is_lock = 0
      admin.last_login = now
      admin.save()

      jwt = create_jwt({
          "token_validity_time": expire_at,
          "adminToken": admin_token,
      })

      return success({"token": jwt, "admin_info": admin_info})

  8. 你现在应该记住的结论

  - 这个项目的登录成功，不是“用户名密码对了”这么简单。
  - 登录过程强依赖员工、部门、角色、菜单、数据范围这些组织数据。
  - 前端拿到的不是轻量 token，而是一个很重的 admin_info。
  - 服务端会删除旧 token，所以这是单端登录模型。
  - 中间件后续校验时，不只是验 JWT 签名，还会反查数据库里的 adminToken。

  9. 这一段里最值得你警惕的点

  - AdminApiBase.php:95 部门为空判断变量写错了。
  - admin_info 存成 JSON 快照，意味着权限或组织关系改了以后，未必实时生效，往往要重新登录。
  - randomString = md5(time + uniqid) 可用，但不是现代高强度随机方案。
  - 登录态耦合业务字段很多，后续重写时不能只迁 token，要迁整套上下文模型。
