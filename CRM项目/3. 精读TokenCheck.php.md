• 现在看 TokenCheck.php:1。它是“登录成功之后，每个受保护接口进控制器前的门卫”。

  先把它放回整条链路里：

  - route.php 里，登录接口之外的大部分后台接口都挂了这个中间件
  - 前端在 request.js 里把 token 放进请求头 X-Token
  - 登录时 AdminApiBase.php#L228 生成了两层登录态：
      - 一层是返回给前端的 JWT
      - 一层是服务端表 crm_admin_token 里的 adminToken + admin_info

  1. 文件头 1-11

  - 6-8：TokenCheck.php#L6 导入 3 个关键依赖：
      - JwtService：验 JWT
      - RbacPermissionModel：查权限表
      - AdminTokenModel：查服务端登录记录
  - 9：Log 被导入了，但这个文件里没用到，是个无效 import。

  2. 入口方法 20-25

  - 20：TokenCheck.php#L20 handle($request, \Closure $next)
    这和 Django/FastAPI middleware 很像：
      - 先看请求
      - 通过就 next
      - 不通过就直接返回响应
  - 22：从请求头取 X-TOKEN
  - 23-25：如果没有 token，直接返回：
      - code = 50008
      - message = 授权验证接口,token不能为空

  这里很关键：它不是返回 HTTP 401，而是返回业务码 JSON。前端会在 request.js 里按 50008 做重新登录处理。

  3. 验 JWT 26-29

  - 26：实例化 JwtService
  - 27：checkToken($x_token) 校验 JWT 是否合法、是否过期
  - 29：如果 status == true，说明 JWT 本身合法，才能继续

  你要记住，这里只是在验“门票真不真”。

  4. 从 JWT 里取 adminToken 30-32

  - 30：TokenCheck.php#L30
     $request->adminToken = $result['decrypt_data']['data']->adminToken;

  这句非常关键。它说明：

  - JWT 里真正存的是 adminToken
  - 中间件把这个 adminToken 挂回 request
  - 后面的控制器基类就能从 request 里拿到它

  这和 AdminApiBase.php 构造函数正好对上：基类会读 $this->request->adminToken。

  5. 查服务端登录会话 32-35

  - 32：按 token = adminToken 去 crm_admin_token 表查 admin_info
  - 34-35：如果查到了，就把 JSON 解码成数组 $adminInfo

  这说明这个项目的认证不是纯无状态 JWT，而是：

  - JWT 负责证明“这个 token 是服务器签发的”
  - crm_admin_token 负责证明“这次登录会话现在还有效”

  如果数据库里这条记录没了，就算 JWT 还没过期，也会被判失效。

  6. 权限判断 37-48

  - 37：只对 type == 0 的账号做权限拦截
  - 38-39：取当前请求的控制器名和方法名
  - 41：拼成 Controller/Action
  - 44：去权限表里查有没有这条路由配置
  - 46-47：如果这条路由受管控，但当前账号的 permissions 里没有对应权限 ID，就拒绝访问

  这里你必须记住 3 件事。

  第一，这个项目的接口权限粒度是“控制器方法级”。

  第二，permissions 不是实时查的，是登录时在 AdminApiBase.php#L198 组装进 admin_info 的快照。

  第三，也是最重要的一点：

  - 只有“权限表里存在的路由”才会检查权限
  - 如果权限表里查不到这条路由，代码会默认放行

  也就是这句逻辑的真实含义：

  if (!empty($permission) && !in_array(...)) {
      return no_permission
  }

  不是“没权限就拒绝”，而是“只有被配置为受控的路由才拦截”。
  这是一种“默认放行，按表收口”的模型。

  7. 已登出和验签失败 50-55

  - 50-52：如果 admin_info_json 查不到，返回：
      - code = 50012
      - message = 账号已被登出！

  这通常有几种情况：

  - 用户主动退出了
  - 同账号在别处登录，把旧 token 顶掉了
  - 定时清理把过期会话删掉了
  - 53-55：如果 JWT 验签没过，返回：
      - code = 50014
      - message = 授权验证未通过

  这对应：

  - token 被篡改
  - token 过期
  - JWT key 不匹配

  8. 放行到控制器 57

  - 57：TokenCheck.php#L57
    return $next($request);

  到这里才真正进入控制器。

  也就是说，这个中间件通过后，控制器默认可以认为：

  - 请求里有合法 token
  - 服务端会话还存在
  - 如果路由受控，那权限也已经过了

  9. 用 Python 思维重写这段

  你可以把它当成这样：

  def handle(request, next):
      x_token = request.headers.get("X-TOKEN")
      if not x_token:
          return json_error(code=50008, message="token不能为空")

      result = jwt_service.check_token(x_token)
      if not result["status"]:
          return json_error(code=50014, message="授权验证未通过")

      admin_token = result["decrypt_data"]["data"].adminToken
      request.adminToken = admin_token

      admin_info_json = AdminTokenModel.where(token=admin_token).value("admin_info")
      if not admin_info_json:
          return json_error(code=50012, message="账号已被登出")

      admin_info = json.loads(admin_info_json)

      if admin_info["type"] == 0:
          route = f"{request.controller()}/{request.action()}"
          permission = Permission.where(route_name=route, is_deleted=0).find()
          if permission and permission.id not in admin_info["permissions"]:
              return json_error(code=50000, message=f"【{permission.name}】暂无权限")

      return next(request)

  10. 这一段最值得你掌握的项目结论

  - 这个系统的认证是“双层”的：
      - JWT
      - 数据库会话记录
  - 这个系统的授权是“路由级”的：
      - Controller/Action
  - 权限不是实时查，而是登录快照
  - 路由不在权限表里，默认可能放行
  - type == 0 和 type == 1/2 的权限行为不同
    从代码看，1/2 类型在这里没有走同样的接口权限拦截
