## 结论：能，而且地基已经铺好了

核心抽象 `AgentRoute`（`config.py:207`）就是为"一个 bot 接多个 agent"设计的——每个 route 可以指向**完全不同**的 `base_url / app_id / secret_key / agent_id`，也就是不同智能体、甚至不同租户。配套设施也都在：

- `AgentChatClient` 按 `route.name` 懒加载缓存（`message_handler.py:549`），多 agent 各持独立 Session;
- 会话 ID 按路由隔离——`build_conversation_key()` 的 key 里带了 `route_name`（`message_handler.py:418`），切 agent 不会串上下文。

所以"一个机器人桥接多个智能体"不是要不要重写架构的问题，是**把已有的路由能力放开给私聊**的问题。

## 现状：多 agent 路由现在只认群聊

目前两条路由入口都绑死在群聊上：

- `DEEPEXI_GROUP_ROUTES_JSON` —— 按群 `chatid` 整群切 agent;
- `DEEPEXI_GROUP_COMMAND_ROUTES_JSON` —— 群内按命令前缀切 agent。

而私聊被**显式短路**了，`config.py:682` 的 `resolve_agent_route_for_prompt()`：

```python
route = self.resolve_agent_route(...)          # 单聊这里永远返回 default
if chat_type != "group" or not chat_id:
    return AgentRouteSelection(route=route, prompt=prompt)  # 直接返回，不进命令路由
```

所以私聊现在恒等于"一个机器人 = 一个默认智能体"。

## 一个平台层面的硬约束要先讲清

企微的"智能机器人"在用户那侧就是**一个固定头像/入口**，没法在企微原生 UI 里摆出"多个 agent 供选择"。所以多智能体的区分只能由我们在**消息内容层**做——就像群聊命令路由现在做的那样。这决定了私聊只能在下面三种形态里选：

| 形态 | 交互 | 改动量 | 代价 |
|---|---|---|---|
| **A 命令前缀**（无状态） | `/法务 这份合同…`、`/研发 报错…`，不带前缀走默认 | 最小，基本是放开那个 `if` + 加一份私聊命令路由配置 | 用户每条要打前缀 |
| **B 会话粘性**（有状态） | `/切换 法务`，之后都走法务直到再切；配 `enter_chat` 欢迎语列菜单 | 中等，要加"用户→选中 route"的粘性存储（仿 `conversation_store`） | 要管状态，用户可能忘了在哪个 agent |
| **C 自动路由**（分类器） | 后台判断该归哪个 agent，用户无感 | 大 | 多一跳延迟；你压测里长回答已 ~50s、尾部贴近 120s 超时，再加一跳风险高 |

## 推荐

**A 起步**：改动最小、行为可预测、会话天然隔离，私聊里立刻就能"一个机器人桥接多个智能体"。如果之后想要"像在多个 bot 间切换"的体验，再叠加 **B** 的粘性切换。**C 暂不建议**——延迟预算这块你自己压测的数据就不支持。

## 配置长这样

新增一个环境变量 `DEEPEXI_SINGLE_COMMAND_ROUTES_JSON`，键是命令前缀，值复用现有 route 覆盖语法（字符串 = 只换 `agent_id`；对象 = 覆盖任意字段，可指向另一套滴普实例）：

```json
DEEPEXI_SINGLE_COMMAND_ROUTES_JSON={
  "/法务": "law-agent-id",
  "/研发": {"agent_id": "dev-agent-id", "app_id": "...", "secret_key": "..."},
  "/hr":  "hr-agent-id"
}
```

用户侧体验：

- 私聊发 `/法务 这份合同有什么风险` → 走法务 agent，前缀被剥掉，传给滴普的 prompt 是"这份合同有什么风险";
- 私聊发 `你好`（无前缀）→ 走默认 agent;
- 法务/研发/HR 各自上下文独立，互不串（会话 key 已带 `route_name`）。

## 拍板前留意这几个点

1. **无前缀的兜底**：走默认 agent（推荐），还是回一句"请选 /法务 /研发 …"。
2. **每条都要带前缀**：这是无状态的代价，也正是 B（粘性切换）要消除的;如果你们的人会连续追问同一个专家，B 的体验差距会很明显。
3. **可发现性**：建议把可用前缀列进 `PASSIVE_WELCOME_MESSAGE`（`enter_chat` 时推送），否则用户不知道有哪些智能体。
4. **前缀格式**：现有 `_strip_command_prefix()` 要求前缀后跟空格或结尾，所以 `/法务 xxx` 最稳，`法务：xxx` 这种要额外改。

## 核心转变:从"前缀"换成"描述清单"

前缀方案的死结是——前缀既要你**手写**、又要用户**记住**。换成大模型分诊后,这两件事都消失:

- 每个 agent 只需要一行**用途描述**(`id + 名称 + 一句话场景`),这份描述**只喂给大模型读,用户永远不用记**。
- 加一个新 agent = 加一行描述,几十上百个都行,没有"教用户记前缀"的成本。

所以"智能体太多"不再是问题——它从"用户的记忆负担"变成了"模型的检索输入"。

## 分诊器:复用现有 client 调一个"路由 agent"

不用引入任何新东西。在滴普上建一个**分诊 agent**,系统提示词里塞进全部候选的"名称+描述",让它读用户的自然语言、输出推荐候选(结构化,比如返回编号+名称+理由)。在本服务里它就是再加一个 `AgentRoute`,用现成的 `AgentChatClient` 调一次(blocking 即可,不用流式,因为只要一小段候选列表)。

```
用户问题 + 全部 agent 描述  ──►  分诊 agent  ──►  top-3 候选
```

## 交互流程(模型选 + 用户确认)

```
用户: 我想咨询一下报销标准
bot:  帮你找到几个可能的助手，回数字选择：
      1. 财务·报销政策（差旅/发票/额度）
      2. HR·福利制度
      3. 行政·办公采购
      回复数字，或换个说法重新描述。
用户: 1
bot:  已接入「财务·报销政策」。
      （随即把原问题"我想咨询报销标准"直接转给它，开始回答）
      ……财务 agent 的回答……
```

两个体验关键点:

1. **选完直接把原 query 转过去**,用户不用重打一遍。
2. **选中即粘性**——复用上一轮讲的 B 的 `RouteSelectionStore`,之后连续对话都在财务 agent,直到 `/退出` 或 `/重选`。

所以这套是 **C(模型路由) + B(粘性) 的组合**:模型做召回排序,用户一键确认,然后粘住。

## agent 清单从哪来(一个要你确认的事实)

- **理想**:滴普平台若有"列出我的 agent"的 openapi(你 memory 里有 knowledge.amer.cn 的 openapi,说明平台有 openapi 体系),就**动态拉清单**,零维护、新建的 agent 自动出现。
- **退路**:平台不提供就**集中配一份描述清单**(JSON/文件)。即便这样,也只维护"描述"这一项,比"前缀+用户教学"轻得多。

## 落地点(在 B 的基础上加)

| 部分 | 改动 |
|---|---|
| 分诊 agent | 新增一个 `AgentRoute` 指向滴普路由 agent;清单动态拉或静态配 |
| `message_handler` | 加分诊状态机:无粘性时→调分诊器→列候选→存 pending;下一条收数字→写粘性+转发原 query |
| pending 待选表 | `key=userid` 存"候选列表+原问题",仿 `conversation_store` 加 TTL |

| `agent_chat`/`wecom_ws` | 零改动 |

## 三个要拍板的点

1. **延迟**:分诊多一跳,但它只返回简短候选、且只在"切换时"发生(不是每条消息),通常远快于一次完整回答。你压测担心的长回答超时是回答那一跳的事,和分诊无关。
2. **高置信免确认**:可配置——分诊器只给出一个明显候选时直接接入、不让选;多个候选才列菜单。省一次往返。
3. **pending 过期**:用户收到候选后没选、隔了很久发别的,要把旧 pending 丢掉,把新消息当新分诊起点。

一句话:**不要前缀,改成"一句话描述喂给分诊 agent + 用户回数字确认 + 粘性",模型扛检索、用户只点选**。要做的话先告诉我 agent 清单是动态拉还是我给你设计静态配置格式。
