
### Turn 3  *(2026-06-10 06:23)*

**User:**

滴普智能体桥接企微机器人，还是企微机器人桥接滴普智能体

**Assistant:**

严格说都不是——**这个项目本身是个独立的中间桥接服务，两头都是它的"对端"**。但如果要选一个说法来描述，"企微机器人桥接到滴普智能体"更贴近实际：企微机器人只是个壳/渠道，滴普智能体才是大脑，本项目做的事就是给这个壳接上大脑。

消息链路上谁连谁、谁推谁是这样的：

```
用户 ─@机器人─▶ 企业微信 ─WebSocket推送─▶ 本项目（桥接服务） ─HTTP streaming─▶ 滴普 Agent
用户 ◀─气泡/附件─ 企业微信 ◀─stream.id刷新─ 本项目（桥接服务） ◀─SSE answer增量─ 滴普 Agent
```

两个关键点：

1. **桥接服务对两边都是主动方（客户端）**：对企微是主动建 WebSocket 长连接并订阅（`aibot_subscribe`），企微把群里/单聊的消息推过来；对滴普是主动发 `/chat/chat-messages` 的 streaming 请求。企微和滴普互相完全不知道对方存在，全靠中间这个服务转译。
2. **代码命名也是"企微为入口"的视角**：`wecom_ws.py` 收消息在前，`agent_chat.py` 转发滴普在后，项目名 wecom-deepexi-aibot 同理。

唯一的例外是上次说的文档 MCP 那条链路——那是**滴普 Agent 直连企微的文档 MCP 端点**（streamable-http），不经过桥接服务：

```
滴普 Agent ──MCP工具调用──▶ 企微文档/智能表格（qyapi.weixin.qq.com/mcp/robot-doc）
```

所以接上 MCP 之后的完整图景是：消息走"企微 → 桥 → 滴普"，写文档走"滴普 → 企微"直连，一来一回正好闭环。对外介绍这个系统时，比较准确的一句话是："滴普智能体通过桥接服务接入企微机器人渠道，并直连企微文档 MCP 实现资料录入。"
