## 一、本质区别:你要从"出站长连接"切到"入站 HTTP 服务"

| 维度 | 你现在(长连接) | 要做的(webhook/短连接) |
|---|---|---|
| 连接方向 | 容器拨出 `wss://openws…` | 企微 POST 进你的**公网 URL** |
| 进程形态 | WS 客户端(无监听端口) | **HTTP 服务器**(FastAPI/aiohttp),要映射端口 |
| 流式刷新 | 你主动 push(`aibot_respond_msg`) | **企微回调轮询**,你在每次 HTTP 响应里返回快照(101463 对照表) |
| 凭证 | Bot ID + Secret | **Token + EncodingAESKey**(+回调URL),与长连接的 Secret 不是一套 |
| 加解密 | 仅下载媒体用 per-file aeskey | **每个回调收发都要 WXBizMsgCrypt 加解密** |
| 图文混排 | ✗ 不支持 msg_item | ✓ **finish=true 时 msg_item 内嵌图片**(这是切过去唯一的功能收益) |
| 流式总时限 | 10 分钟 | **6 分钟**(从用户发消息算起,100719) |

> ⚠️ 企微后台「API 模式」**只能二选一**:长连接 / 设置接收消息回调地址。切到回调模式后,原长连接立即失效,反之亦然(LangBot 文档明确)。所以这是替换,不是并存。

## 三、必须实现的两个 HTTP 端点

### 端点 A:`GET /callback` —— URL 有效性验证(只在配置/保存时触发)

企微带 `msg_signature` `timestamp` `nonce` `echostr` 四个 query 参数过来:

1. 用 `Token` 校验签名:`sha1(sort(token, timestamp, nonce, echostr))` 是否等于 `msg_signature`;
2. 用 `EncodingAESKey` 解密 `echostr`,拿到明文 `echostr`;
3. **原样明文返回** `echostr`(HTTP 200 body)。

### 端点 B:`POST /callback` —— 接收消息 + 被动回复(主链路)

1. query 仍带 `msg_signature/timestamp/nonce`,**body 是加密密文**;
2. 验签 → 解密 → 得到明文 JSON 回调体;
3. 按 `msgid` **排重**(网络原因会重复回调,100719 明确);
4. 分发处理(见第五节);
5. 把回复**加密+签名**后写进 HTTP 响应体返回(被动回复也要加密)。

## 四、加解密(WXBizMsgCrypt,强烈建议用官方库,别手写)

算法本身(90968 已核实):
- `AESKey = Base64_Decode(EncodingAESKey + "=")` → 32 字节;`EncodingAESKey` 固定 43 字符。
- **AES-256-CBC**,PKCS#7 填充,**IV = AESKey 前 16 字节**。
- 明文打包:`random(16B) + msg_len(4B,网络字节序) + msg + receiveid`,整体 AES 加密再 Base64。
- 签名:`msg_signature = sha1(sort(token, timestamp, nonce, msg_encrypt))`,16 进制小写。
- `receiveid`:企业内部场景为 `corpid`(企业 ID)。

> 智能机器人有专门的「回调和回复的加解密方案」页,算法同上但回调体是 **JSON**(不是老应用的 XML)。**直接用官方加解密库**(C++/Python/PHP/Java/Go/C# 都有),只把里面的明文按 JSON 解析即可,自己实现 IV/填充极易翻车。

## 六、回复机制(核心,与长连接差别最大的地方)

### 6.1 流式回复 = "被动应答 + 企微轮询"

- **第 1 次回调**(用户消息)的 HTTP 响应里,返回首帧 `stream{id, finish:false, content}`,**必须带 `stream.id`**(你自定义的唯一 id)。
- 之后企微**不断 POST `msgtype:"stream"` 的"流式消息刷新"回调**(只带 `stream.id`),你在**每个响应里返回当前完整快照**:

```json
// 企微来的刷新轮询
{ "msgtype": "stream", "stream": { "id": "STREAMID" }, "msgid": "...", "aibotid": "..." }

// 你的响应体(加密前)
{ "msgtype": "stream",
  "stream": { "id": "STREAMID", "finish": false, "content": "<到目前为止的完整正文>" } }
```

- `content` 是**全量快照不是增量**(101031:"第一次回'1',第二次回'123',展示'123'"),≤ **20480 字节**,支持 markdown,`<think></think>` 会渲染成思考过程。
- 收尾:某次刷新响应里置 `finish:true`,可同时挂 `msg_item` 图片(见 6.2)。**6 分钟内必须 finish**,否则自动结束。

### 6.2 图片内嵌(切 webhook 的唯一功能红利)

```json
{ "msgtype": "stream",
  "stream": {
    "id": "STREAMID", "finish": true,
    "content": "最终完整文本",
    "msg_item": [
      { "msgtype": "image", "image": { "base64": "<图片base64>", "md5": "<原图md5>" } }
    ]
  } }
```

- 仅 `finish=true` 这一帧能带 `msg_item`;`image` 用 **base64 + md5(base64前原图的md5)**,单图 ≤10M,JPG/PNG,**最多 10 张**(101031)。
- 这正是你长连接做不到、只能 finish 后补发独立 image 消息的那件事 —— webhook 下可以**图文同框**。

### 6.3 `response_url` 主动回复(缓解轮询延迟的逃生口)

每个回调给的 `response_url` 是临时 URL,可由你**主动 POST** 推送回复("主动回复消息"),不必干等企微下一次轮询。SSE 来了新增量就推一帧,体验更接近长连接。但它是补充手段,基线机制仍是 6.1 的轮询。

## 七、桥接滴普 SSE 的架构(轮询模型下的关键设计)

长连接是"SSE 推→你推",方向天然对齐;webhook 是"SSE 推→企微来拉",**阻抗失配**,必须加一层按 `stream.id` 的服务端状态:

```
收到用户消息回调:
  stream_id = gen_id()
  state[stream_id] = { buf: "", done: false, images: [] }
  异步启动: 滴普 streaming 请求 → 每个 delta: state[stream_id].buf += delta
                                 → 结束: state[stream_id].done = True; 抽取图片
  HTTP 响应: stream{id: stream_id, finish:false, content:"正在思考…"}

收到"流式消息刷新"回调(stream.id):
  s = state[stream.id]
  if s.done:
     HTTP 响应: stream{id, finish:true, content: fit(s.buf,20480), msg_item: s.images[:10]}
     del state[stream.id]
  else:
     HTTP 响应: stream{id, finish:false, content: fit(s.buf,20480)}
```

要点:
- `state` 要能跨多个 HTTP 请求存活(单进程内存即可;多副本则需 Redis,因为轮询可能打到不同实例)。
- `safe_stream_display` / `fit_stream_content` / `extract_image_refs` / 滴普累计→增量归一化 **全部可复用**,只是触发点从"主动 push 节流"改成"被轮询时取快照"。
- 滴普慢于 6 分钟的尾部要兜底(直接 finish + 提示),否则气泡被企微强制结束。

## 九、限制与坑(动手前必看)

- **一次回复只有一个气泡**:智能机器人对"用户一条消息"原则上只回一个气泡(AstrBot 社区文档)。要发多条/额外文件,得再配「消息推送(群机器人)webhook」绕过 —— 跟你现在长连接里"超长拆多条文本""图片独立发"的策略要重新对齐。
- **频率限制**:同一会话 30 条/分、1000 条/小时,24 小时内可回(101463)。轮询快照刷新频繁,注意别自己把配额刷爆,该节流还得节流。
- **6 分钟硬超时**:长回答(你压测基线 T≈50s 还好,但有尾部 81s 的案例)要确保 6 分钟内 finish。
- **排重**:`msgid` 必须去重,否则重复回调会重复触发滴普请求。
- **多副本一致性**:轮询会落到不同实例,`stream.id` 状态必须共享(Redis)。
- **安全**:公网入站多了攻击面,验签做严、限流、WAF。

## 不是。回调 URL 填的是**你自己这个桥接服务**的公网地址

企微要回调的,是你 `wecom-deepexi-aibot` 服务**新暴露的那个 HTTP 端点**,比如 `https://你的域名/wecom/callback`。三方关系是这样的:

```
                    回调(加密,企微JSON)              POST /chat/chat-messages(滴普格式)
   企业微信  ──────────────────────────►  你的桥接服务  ──────────────────────────►  滴普
            ◄──────────────────────────  (本项目)      ◄──────────────────────────  knowledge.amer.cn
                被动回复(加密,stream结构)                  SSE 增量 answer

   ▲ 这里填的 URL = 你服务的地址          ▲ DEEPEXI_BASE_URL 配在你服务内部,企微不经手
```

**你的服务永远夹在中间当翻译**,两边说的根本不是一种话。

## 为什么滴普地址不能直接填给企微

就算填了也立刻失败,因为滴普完全接不住企微的协议:

1. **加解密**:企微发的是用你 `Token`/`EncodingAESKey` 加密+签名的密文。滴普不认识 WXBizMsgCrypt,既验不了签也解不了密。
2. **URL 验证握手**:保存回调时企微发 GET `echostr` 要求原样解密返回,滴普不会响应这个握手,后台直接报"请求 URL 失败"。
3. **报文格式对不上**:企微回调是 `{msgtype, stream, from, response_url…}`;滴普要的是它自己的 `{prompt, streaming…}`。字段完全两回事。
4. **回复格式对不上**:企微要你回 `stream{id, finish, content, msg_item}` 这种加密结构;滴普吐的是 SSE 流。中间这层转换,只能你来做。

## 落到你项目上:两个地址各归各位

| 地址 | 是什么 | 切 webhook 后怎么变 |
|---|---|---|
| **滴普地址** `DEEPEXI_BASE_URL` | 你**主动调出去**的上游 | **完全不变**,还是 `http://knowledge.amer.cn/api/openapi`,还是 `agent_chat.py` 那套 SSE 调用 |
| **企微回调 URL** | 企微**回调进来**找你的入口 | **新增**,= 你服务对公网暴露的 HTTPS 端点(现在长连接模式压根没有这个) |

关键认知:**你这个项目本来就是干"中间翻译"这件事的**。今天它通过长连接**拨出去**连企微;切 webhook 只是把"企微这一侧"从"出站 WS"换成"入站 HTTP 回调"——**滴普那一侧一个字都不用改**。所以企微回调 URL 永远指向你自己,滴普地址永远是你内部配置里往外调的目标,两者不会、也不能合并成一个。
