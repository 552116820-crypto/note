## 为什么不是 ab/wrk

读码确认这个桥接**不是 HTTP 服务器**，而是企微 WebSocket 客户端（每个 bot 只允许一条连接，另开连接会把线上踢下线）。所以采用注入式压测：脚本导入项目自己的 `MessageHandler`，用记录时间戳的 fake WS 客户端替代真实连接，按阶梯并发注入合成消息（msgid 用 uuid 绕过去重），统计三个指标——调度延迟、首响（首条真实流式内容）、完整耗时，外加吞吐和 p50/p95。

## 本地验证结果

- mock 模式数字与设定节奏完全吻合：单条 ≈0.48s 时，并发 1 串行 4 条用 1.95s、并发 4 并行只用 0.50s；
- 排队拐点可被正确测出：把线程池压到 2 再打并发 4，首响从 0.39s 涨到 0.88s（正好多排一轮），和理论值一致；
- 上游报错不会再被误计为成功（finish=True 但内容是"AI 服务暂时繁忙"等兜底文案的样本会标记为失败）。

过程中发现一个**本地环境坑**（已存入记忆）：你的 `~/.bashrc` 会自动 `source .env`，bash 把 JSON 值里的双引号吃掉了，导致本地直跑任何该项目代码都会报 `DEEPEXI_GROUP_ROUTES_JSON must be valid JSON`，且 shell 代理会把 127.0.0.1 请求劫持成 502。脚本内部已自带 loopback NO_PROXY 兜底；线上容器走 compose env_file 解析，不受影响。

## 服务器上怎么跑

脚本不在镜像里（Dockerfile 只 COPY 了包目录），用挂载起一次性容器即可，**线上容器全程不受影响**。先把 `scripts/loadtest.py` 弄到服务器（git 提交后 pull，或直接 scp），然后：

```bash
# ① mock：测桥接自身并发极限（零成本，约 2 分钟）
docker compose run --rm --no-deps -v "$(pwd)/scripts:/app/scripts" \
  wecom_deepexi_aibot python scripts/loadtest.py --upstream mock

# ② real：真实端到端延迟（默认仅 15 次调用，提示词是"请只回复：收到"省 token）
docker compose run --rm --no-deps -v "$(pwd)/scripts:/app/scripts" \
  wecom_deepexi_aibot python scripts/loadtest.py --upstream real
```

每次跑完终端出汇总表，原始样本 JSONL 落在 `scripts/loadtest_results/`（已加 .gitignore）。常用参数：`--levels 1,4,8,16,32`、`--per-level 20`、`--workers 16`（预演调大 `AGENT_STREAM_MAX_WORKERS` 后的效果）、`--verbose`。

解读时记住两点：有效并发上限 ≈ 线程池的 8，超过就排队（首响线性抬升）；real 模式的 429/500 会被服务自动重试，表现为长尾延迟而非失败。

## Mock 轮：桥接自身的账

mock 设定单条 2.45s，实测数字和理论值严丝合缝，这正是有价值的地方：

- **桥接开销 ≈ 10ms**。并发 1 时完整耗时 2.46s vs 理论 2.45s，差值就是去重、路由、SSE 解析、节流刷新的全部成本，可以忽略。
- **事件循环远未饱和**。disp 一直是 0.1ms，32 路并发注入也没抬头——asyncio 这一层至少还有两个数量级的余量，内存锁（去重/会话表）也无压力。
- **拐点精确出现在 8**。并发 ≤8 时首响/完整与单条完全一致（无排队）；并发 16 时首响 1.20→3.66s、完整 2.46→4.91s，多出来的正好是一个完整单条耗时（排了一轮队）；并发 32 时 p95 到 6.12s/7.38s，即排了两轮。这是教科书式的线程池饱和曲线，证明瓶颈唯一且明确：`AGENT_STREAM_MAX_WORKERS=8`。
- 吞吐表格里 8/16/32 档都是 2.70 msg/s，略低于理论稳态 8÷2.46≈3.25 msg/s，是因为 20 条消息分 3 波（8+8+4），最后一波只有半载。消息量更大时会逼近 3.25。

## Real 轮：真实端到端的账

- 最短问答（"请只回复：收到"）：首响 p50 1.6~2.5s，完整 p50 2.4~3.2s，max 3.9s。滴普在 ≤5 路并发下没有任何限流/重试迹象（没有出现 retriable 告警，零失败）。
- 三档之间 ttfr 2.54 vs 1.83 的波动不用解读——每档只有 5 个样本，p95 基本没有统计意义，这轮的价值是拿到量级和确认链路健康。
- 一个口径提醒：**real 的"并发 8"那档实际只测到了并发 5**，因为每档只注入 5 条消息，信号量根本没满。想要真实的 8 路数据点需要 `--levels 8 --per-level 16`（16 次调用）。

## 合起来：容量怎么估

真实容量由一个公式决定：**可持续吞吐 ≈ 8 ÷ 单条平均完整耗时**。

| 场景 | 单条耗时 | 可持续吞吐 | 含义 |
|---|---|---|---|
| 短问答（本次实测） | ~2.5s | ~3.2 条/s ≈ 190 条/min | 上限很高 |
| 典型业务问答（估 15~30s 长回答） | 15~30s | 0.27~0.53 条/s ≈ 16~32 条/min | **同时最多 8 人在等回答**，第 9 人排队，且每排一轮多等一个完整回答时长 |

注意 mock/real 测的都是 2.5s 量级的短回答，而生产里用户问题会让 agent 流式输出几十秒——worker 线程被占用的时间是**整个回答的时长**，所以真正决定体验的是"同时有几个人在等长回答"。

## 建议的下一步

1. **如果预期同时活跃提问超过 8 人**，把 `.env` 里 `AGENT_STREAM_MAX_WORKERS` 调到 16 或 32 再重启线上容器——线程几乎全程阻塞在网络 I/O 上，调大很便宜。可先用 harness 预演验证拐点确实右移：`--upstream mock --workers 16 --levels 8,16,32`，预期拐点出现在 16。调大前值得和滴普侧确认他们的 QPS/并发配额，否则瓶颈只是从你这边挪到对方限流。
2. **JSONL 写入失败**是因为服务器上 `scripts/` 归 root、容器内是 appuser。跑之前执行一次即可：
   ```bash
   mkdir -p scripts/loadtest_results && chmod 777 scripts/loadtest_results
   ```
3. 可选补测（各消耗一点配额）：`--upstream real --levels 8 --per-level 16` 拿真实的满 8 路数据；以及用一条真实业务问题 `--prompt "..."` 测长回答的单条耗时，把上面表格第二行从估算变成实测。

**8 路 —— 但要分两个口径说：**

1. **不排队的最大并发 = 8**。这是 `AGENT_STREAM_MAX_WORKERS=8`（线程池）决定的硬上限：同时最多 8 条消息在向滴普请求回答。实测并发 ≤8 时延迟与单条完全一致（首响 1.2s、完整 2.46s，零排队）。

所以实际容量取决于你能接受用户多等多久，按本次实测的规律换算：

| 瞬时并发 | 用户额外等待 |
|---|---|
| ≤8 | 0 |
| 9~16 | +1 个回答时长 |
| 17~24 | +2 个回答时长 |
| 25~32（实测上限） | +3 个回答时长 |

短问答（~2.5s）多等一轮无感；但生产里长回答 15~30s 的话，第 17 个人就要干等半分钟到一分钟才见到第一个字。

如果要把"不排队上限"从 8 提上去：`.env` 里调大 `AGENT_STREAM_MAX_WORKERS`（比如 16/32）重启容器即可，线程全程阻塞在网络 I/O 上，开销很小；真正的天花板会变成滴普侧的并发/QPS 配额，调大前建议先和平台确认，否则只是把排队挪到了对方的限流上。

**Assistant:**

p50 就是**第 50 百分位数**（中位数）：把这一轮所有请求的耗时从快到慢排个序，正中间那个值。「首响 p50 = 3.3s」意思是**一半请求比 3.3s 快，一半比它慢**——可以读作"典型用户的体验"。

同理：

- **p95**：95% 的请求比这个值快，只有最慢的 5% 超过它——读作"运气差的用户有多惨"，反映长尾；
- **max**：整轮里最慢的那一条。

用你 32 路那轮的实际数据串起来读：完整耗时 p50 54.6s、p95 64.7s、max 68.3s ——「典型用户约 55 秒拿到完整回答，最慢的 5% 要等 65 秒上下，极端那条 68 秒」。三个数挨得近，说明体验很均匀；而 48 路那轮 p50 70s、max 134s 拉开了一倍，就是过载后体验开始两极分化的信号。

压测里不用「平均值」而用百分位，是因为平均数会被极端值带偏：99 条 3 秒 + 1 条 300 秒，平均 6 秒看着还行，但 p99 会诚实地告诉你有人等了 5 分钟。
