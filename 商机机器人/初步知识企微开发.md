## 一、先建立心智模型(半天，最值)

企微开发的本质就一句话：**你的服务器拿 `access_token` 调 HTTP 接口；企微把用户消息加密 POST 回你的服务器**。三个概念立住就通了：

1. **凭证链**：`corpid`(企业) + `secret`(应用钥匙) → 换 `access_token`(2h 有效、必须缓存) → 所有接口都带它。
2. **可见范围 = 权限**：应用能给谁发消息、调谁的信息，由应用的"可见范围"决定，不在范围内直接报错。
3. **两个方向**（决定难度）：
   - **主动**(你→企微)：发消息、查数据。就是个 HTTP POST，**简单**。
   - **被动**(企微→你)：用户提问、点按钮。需要**公网 HTTPS 回调 + 加解密**，这是新手的墙。

📖 必读两页：[基本概念/90665](https://developer.work.weixin.qq.com/document/path/90665) · [开发前必读/90664](https://developer.work.weixin.qq.com/document/path/90664)

## ① 可见范围 —— 设给销售部就行

**结论：新建独立项目，但放成 `amer_sales_assist` 的兄弟目录（比如 `E:\anzai-bot`），别嵌在它里面。**

## 为什么不塞进 `E:\amer_sales_assist`

| | amer_sales_assist | 安仔 bot |
|---|---|---|
| 干的事 | 把数据**算出来、写进库** | 把库里结果**送到销售手机** |
| 运行方式 | 跑批 / 贴着 8.138 MySQL | 定时推送，后期还要**公网 HTTPS 回调** |
| 密钥 | DB 账号 | 企微 corpid/secret/AESKey |
| 变更节奏 | 跟模型、数据走 | 跟企微平台、UI 走 |

两边唯一的连接点是**数据库本身**——`amer_sales_assist` 这个库就是它俩之间的"接口/合同"。bot 只**读库**拿算好的商机，**不去 import churn 项目的代码**。把这条缝画清楚，将来谁都能单独改、单独部署。

> 嵌套的坏处很具体：以后 bot 要挂到一台公网服务器上跑回调，你会被迫把整个 churn 分析、DB 写入逻辑一起拖过去；而且 `amer_sales_assist` 这名字本来指那个**库**，再把"读这个库的程序"塞进同名目录，概念就糊了。

## 什么时候放子目录也行

如果 v1 你就想要**一个定时脚本**、同一个人维护、还想复用 `~/.my.cnf` 那套 DB 连接——那 `E:\amer_sales_assist\anzai_bot\` 起步也没错。**唯一要求**：bot 代码全部自包含在这个子目录里，别跟 churn 代码交叉引用，将来一剪就能独立出去。

## 推荐骨架（独立 / 子目录都长这样）

```
anzai-bot/
  config/      # 企微凭证 + DB连接（secrets 记得 gitignore）
  wecom/       # 企微API封装：token缓存、发消息、拼卡片
  data/        # 查 amer_sales_assist 的查询层（只读）
  cards/       # 6种商机卡的拼装
  jobs/        # 定时任务：每天9:48 生成+推送日报  ← v1 核心
  callback/    # （M4再加）回调服务：验签/解密/路由
  main.py
```

v1 只会用到 `config / wecom / data / cards / jobs` 五个，`callback/` 空着等以后。
