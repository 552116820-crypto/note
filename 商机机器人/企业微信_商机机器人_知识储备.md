# 企业微信开发 · 商机机器人知识储备

> 面向「安仔商机机器人」——企业**内部自建应用**，把流失预警/商机日报以卡片形式推给销售。
> 架构 = **推送优先**（M0–M3 不需要公网回调、不碰加解密）。
> 本文是官方开放文档（developer.work.weixin.qq.com）的提炼，已结合本项目场景标注。最后整理：2026-06-30。

---

## 0. 一张图看懂：商机机器人要走的链路

```
你的后端(churn库/amer_sales_assist)
   │  ① 用 corpid + 应用secret 换 access_token（缓存2h）
   │  ② 组装消息体（text / markdown / 模板卡片）
   ▼
POST qyapi.weixin.qq.com/cgi-bin/message/send?access_token=xxx
   │  touser = 业务员的 userid（用 | 分隔）
   ▼
企业微信服务器  ──►  销售的「企业微信」收到应用消息/卡片
```

整条链路**只出不进**：你的服务器主动发 HTTPS 请求，企业微信不需要回调你。
所以 M0–M3 **不需要公网 IP、不需要域名备案做回调、不需要 AES 加解密**。
（需要回调的能力——卡片按钮点击、用户回消息——是 M4+ 的事，见第 8 节。）

## 1. 五个核心名词（建应用时在后台拿）

| 名词 | 是什么 | 哪里拿 |
|---|---|---|
| **corpid** | 企业唯一ID | 管理后台 → 我的企业 → 企业信息（需管理员） |
| **agentid** | 应用唯一ID（整型） | 应用管理 → 你的自建应用页 |
| **secret** | 应用密钥（**严禁泄漏/进前端**） | 应用管理 → 你的自建应用页 |
| **userid** | 成员账号（推送收件人就靠它） | 通讯录 → 点成员 → 详情页 |
| **access_token** | 调所有接口的临时票据 | 由 corpid+secret 换取，见第 3 节 |

> **可见范围 = 权限范围**（关键概念）。应用配置的可见范围既决定哪些销售能在工作台看到这个应用，**也决定你能给哪些成员调接口/发消息**。可配置成员/部门/标签；配部门则其所有子部门+成员都在范围内。
> → 本项目：可见范围设为**销售部**，超出范围的 userid 会进 `invaliduser`、发不出去。

## 2. M0：建应用 + 配置（一次性，纯后台操作，无代码）

1. 管理后台 → **应用管理 → 应用 → 创建应用**：填 logo、名称、可见范围（选**销售部**）。
2. 记下 **agentid**、**secret**；企业信息里记下 **corpid**。
3. ⚠️ **配置「企业可信IP」**（2022-06 后新建自建应用的硬门槛）：
   - 应用详情页 → **企业可信IP** → 填**你后端服务器的公网出口IP**。
   - 没配的话，调任何接口都报 **`errcode 60020`（访问IP不在白名单）**，配完**约1分钟生效**。
   - 不能填第三方服务商IP。配可信IP前可能需要先设「可信域名」或「接收消息服务器URL」。
   - 找自己出口IP的土办法：服务器上 `curl` 一次 gettoken，错误信息会回显被拒的IP。
4. **接口调用许可 = 不用买**（见下方专栏，本项目最大的成本问号已解决）。

> ### 📌 专栏：接口调用许可（基础账号/互通账号）到底要不要钱？
> **结论：你这种「企业内部自建应用」不需要购买，不受影响。**
> - 「接口调用许可」（基础账号¥/互通账号¥）是**服务商（第三方应用、代开发应用）**的付费许可体系——服务商下单、付费、把激活码绑定到成员。整套文档都挂在「第三方应用开发」树下，用的是 `provider_access_token`。
> - 官方 FAQ 原文：「自建应用**代开发**也需要接入接口调用许可吗？答：需要的。**代开发应用与第三方应用**都需要接入。」——只点名代开发+第三方，**不含企业自己开发自用的自建应用**。
> - 发消息返回里的 `unlicenseduser`（没有基础接口许可的成员）只在走代开发/第三方通道时才非空。你用企业自己的 secret 给可见范围内的销售推送，正常为空。
> - **最终验证（建议做）**：M1 跑通后看一眼 `message/send` 返回里 `unlicenseduser` 是否为空，空即确认不受许可影响。

## 3. M1：换 access_token + 发出第一条消息

### 3.1 获取 access_token
```
GET https://qyapi.weixin.qq.com/cgi-bin/gettoken?corpid=ID&corpsecret=SECRET
```
返回：
```json
{ "errcode":0, "errmsg":"ok", "access_token":"xxxx", "expires_in":7200 }
```
**必须遵守的 4 条**：
- **缓存它**，有效期 7200 秒（2h）。**不要每次发消息都重新换**，会被频率拦截。
- access_token 最长 512 字节，**每个应用独立**（按应用分开缓存）。
- **绝不返回给前端**，只在后端持有，所有请求由后端发起。
- 企业微信可能提前让它失效 → 代码要实现「失效就重新换」的兜底（典型触发：返回 `40014/42001`）。

> 工程实现建议：起一个带锁的单例缓存（key=agentid），过期前 5 分钟主动刷新；失败重试时先强制刷 token 再重发一次。

### 3.2 发消息（统一入口）
```
POST https://qyapi.weixin.qq.com/cgi-bin/message/send?access_token=ACCESS_TOKEN
```
最小可用文本消息：
```json
{
  "touser": "ZhangSan|LiSi",
  "msgtype": "text",
  "agentid": 1000002,
  "text": { "content": "今日有 3 个流失客户待跟进，点开查看" },
  "enable_duplicate_check": 1,
  "duplicate_check_interval": 1800
}
```
- `touser`/`toparty`/`totag` 三选一不能全空；`touser` 用 `|` 分隔，**最多1000个**；`@all` 发给全可见范围。
- `enable_duplicate_check=1`：相同内容在 `duplicate_check_interval` 秒（默认1800，最大4小时）内不重复下发——**防抖很有用**，避免重试/补发把销售刷屏。

### 3.3 返回码体系（必须按它做容错）
```json
{ "errcode":0, "errmsg":"ok",
  "invaliduser":"userid1",      // 不在可见范围/不存在
  "unlicenseduser":"userid3",   // 没有基础接口许可（你这场景应为空）
  "msgid":"xxxx",               // 用于撤回
  "response_code":"xyz" }       // 仅交互型卡片返回
```
- 部分收件人无效 → 仍会发给有效的人，无效的列在 `invaliduser/invalidparty/invalidtag`。**最常见原因：收件人不在应用可见范围内。**
- **全部**收件人无效 → 整个调用失败，`errcode 81013`。
- 监控要点：把每次返回的 `invaliduser` 落日志，定期对账「业务员离职/转岗导致 userid 失效」。

## 4. M2：把消息做成卡片（三选一，按"要不要回调"决定能用哪种）

| 形态 | msgtype | 要回调吗 | 适合 | 关键限制 |
|---|---|---|---|---|
| **markdown** | `markdown` | 否 ✅ | 日报正文、富文本列表 | ≤2048字节；微工作台不支持；语法是子集 |
| **文本卡片** | `textcard` | 否 ✅ | 带标题+跳转的通知 | title≤128、desc≤512(支持HTML三色)、**url必填** |
| **模板卡片·文本通知型** | `template_card` (`text_notice`) | 否 ✅ | 商机日报主推荐 | `card_action`整卡跳转**必填**；**别填 action_menu** |
| 模板卡片·图文展示型 | `template_card` (`news_notice`) | 否 ✅ | 带大图的卡片 | 同上 + 需要图片 |
| 模板卡片·按钮/投票/多选型 | `template_card` | **是 ❌** | M4+ 再做 | 无回调接口的应用**根本发不出去** |

### 4.1 markdown（最省事，推荐 M2 起步）
```json
{
  "touser":"ZhangSan", "msgtype":"markdown", "agentid":1000002,
  "markdown": { "content":
    "**📋 今日商机日报**\n>客户：<font color=\"warning\">XX工业</font>\n>状态：<font color=\"comment\">90天未下单</font>\n>建议：本周内回访\n[查看详情](https://你的页面)" }
}
```
支持的 markdown 子集：
- 标题 `# ~ ######`（# 后**要空格**）、加粗 `**x**`、链接 `[文字](url)`、行内代码 `` `x` ``（不能跨行）、引用 `> x`
- **三种内置字体色**：`<font color="info">绿</font>`、`<font color="comment">灰</font>`、`<font color="warning">橙红</font>`

### 4.2 模板卡片·文本通知型（视觉最规整，商机日报首选）
结构字段（都可选，按需取用）：
- `source`：来源（icon 72×72 + desc≤20字，desc_color 0灰/1黑/2红/3绿）
- `main_title`：一级标题≤36字 + 辅助说明≤44字
- `emphasis_content`：**关键数据**（title≤14字 + desc≤22字）——适合突出"今日 5 个流失客户"这种数字
- `quote_area`：引用样式块
- `sub_title_text`：二级文本≤160字
- `horizontal_content_list`：二级"标题:值"列表（≤6项；type 1=跳url / 2=下载附件 / 3=跳成员详情）
- `card_action`：**整卡点击跳转，text_notice 必填**（type 1=url / 2=小程序）

> ⚠️ **两个坑**：
> 1. 即使是"纯推送"的文本通知型，`card_action` 也**必填**——还没有详情页时，先指向一个占位 URL（如企业官网/一个静态说明页），别留空。
> 2. 一旦填了 `action_menu`（右上角"接受/不再推送"菜单），卡片就升级成"需回调"型，**没配回调接口会直接发送失败**。M0–M3 千万别填。

### 4.3 id 转译（让卡片显示人名而非 userid）
开 `enable_id_trans:1`，在支持的字段里写模板：
- `$userName=USERID$` → 姓名、`$userAlias=USERID$` → 别名、`$departmentName=DEPT_ID$` → 部门名
- template_card 支持转译的字段：`source.desc`、`main_title.title/desc`、`sub_title_text`、`horizontal_content_list.value`

## 5. M3：精准投递 —— 拿到销售的 userid + 按业务员隔离商机

推送的本质是「`touser` 填对 userid」。两条路：

**路线A（推荐 M0/M1 先用）：手工维护映射表。**
销售就那么些人、相对稳定。在你的库里建一张 `业务员 ↔ wecom_userid` 映射表，手动填一次即可。完全绕开通讯录接口权限，最快跑通。

**路线B：接口批量拉取。**
- `POST /cgi-bin/user/list_id`（获取成员ID列表）→ 返回 `dept_user:[{userid, department}]`，分页 `cursor`+`limit`(1~10000)。
  ⚠️ **仅支持用"通讯录同步secret"调用**，不是应用自己的 secret——要去管理后台「安全与管理→管理工具→通讯录同步」拿那个 secret。
- 拿姓名/部门名：`读取成员`(user/get)，企业自建应用可取（注意 2022 后新建应用的**敏感信息**——手机/邮箱/头像——默认不返回，需成员登录自主授权；但**推送只用 userid，不受影响**）。

**按业务员隔离商机**（你的两层权限的"后端层"）：
这层不靠企业微信做，而是**在你后端做**——查 churn 库时按 `业务员=当前userid` 过滤，只把属于该销售的商机塞进发给他的那条消息。企业微信只负责"把这条消息送到这个 userid"。

## 6. 频率与配额（排期推送时必看）

- **每应用每天**：≤ 账号上限数 × **200 人次/天**。（发给1000人算1000人次；若企业上限500人，则该应用每天上限 100000 人次。）
- **每应用对同一个成员**：≤ **30 次/分钟**、**1000 次/小时**，超出部分**直接丢弃不下发**。
- **避开整点**：大量企业在每小时 **0分/30分** 触发推送易拥塞投递延迟，**错峰**（如 9:07 而非 9:00）。
- gettoken 不能频繁调（要缓存）。

> 商机日报是低频（每人每天1~几条），离配额很远；唯一要小心的是**重试风暴**——失败重试务必配合 `enable_duplicate_check` 和退避，别在1分钟内对同一人重发>30次。

## 7. 错误码速查（按出现概率排序）

| errcode | 含义 | 处理 |
|---|---|---|
| **60020** | 访问IP不在可信IP白名单 | 把服务器出口IP加进应用「企业可信IP」，等1分钟 |
| **81013** | 全部收件人无效 | 检查 userid 是否都在可见范围；多半是 touser 写错/全离职 |
| **60111 / 60011** | userid 不存在 / 无权限 | 成员离职、被删、不在可见范围 |
| **40014 / 42001** | access_token 无效/过期 | 触发重新 gettoken 后重发 |
| **45009** | 接口调用超频 | 退避重试，检查是否对单人>30次/分 |
| **48002** | 应用无该接口权限 | 检查应用是否启用、可见范围 |
| **701099** | 成员无有效接口许可 | 你这场景不该出现；若出现说明走了代开发通道，复核 secret 来源 |
| 40056 | agentid 不合法 | 核对 agentid |

完整表：全局错误码 `developer.work.weixin.qq.com/document/path/90313`

## 8. M4+ 的"墙"：什么时候你不得不上回调 + 加解密

下面这些能力**必须**有公网可达的回调 URL，并实现企业微信的 AES-256-CBC 加解密（Token + EncodingAESKey），跨过去才有：

- **卡片交互**：按钮交互型/投票/多选卡片，或文本卡片右上角 `action_menu` 的点击 → 用户点了之后企业微信回调你，你再调「更新模板卡片消息」改卡片。
- **用户主动发消息给应用 / @机器人**：接收消息与事件。
- **通讯录变更通知**：成员/部门变动主动推给你（否则只能定时全量拉）。

回调要点（先了解，不实现）：
- 后台填 URL + Token + EncodingAESKey；企业微信先用 GET `echostr` 验证 URL 归属。
- 消息体 AES-256-CBC 加密；官方有「加解密库下载」和各语言示例。
- `response_code`（交互卡片返回，**72小时有效、只能用一次**）是更新卡片的钥匙。

> 判断标准：**只要销售只是"看"卡片、点卡片跳到你的H5/小程序——不需要回调**（跳转是浏览器行为，不回调企业微信）。**只有"在企业微信里点按钮就地改卡片状态/收集投票"才需要回调。** 商机机器人 M0–M3 完全可以靠"卡片 + 跳转到详情页"满足，把回调推迟到确有按钮交互需求时。
