• 按 2026年3月17日 查到的官方信息，你这边如果是 公司 H20 服务器部署，我还是建议先上 Qwen3.5-27B，硬件上也更稳。

  硬件结论

  - H20 是 Hopper GH100, 96GB 显存, 支持 MIG。如果你拿到的是 整卡 96GB，这两个模型都能考虑；如果只是 MIG/vGPU 切片，就别直接上这两个 BF16 大模型。
  - Qwen3.5-27B 官方 BF16 权重仓库总大小是 55.6GB。
  - Qwen3.5-35B-A3B 官方 BF16 权重仓库总大小是 71.9GB。
  - 两个模型默认上下文都是 262,144 tokens，但官方给的 vLLM/SGLang 262K 启动命令，都是 8 卡 tensor parallel = 8。这基本说明：256K 不是单张 H20 的实际目标配置。
  - 官方还专门提供了 --language-model-only，文本场景下可以跳过视觉编码器，把显存留给 KV cache。

  实战怎么配
  下面这部分是我根据官方权重大小和官方部署方式做的工程推断，不是官方硬门槛：

  - 1x H20 96GB + 128GB RAM + 100GB 以上空闲磁盘：27B 单卡部署比较稳，建议先跑 32K 或 64K context。
  - 1x H20 96GB + 128GB 到 256GB RAM + 140GB 以上空闲磁盘：35B-A3B 也能试，但余量更小，建议先跑 16K 或 32K context，并发别太高。
  - 如果你们是 8x H20/HGX，那两者都没压力，262K context 才比较有意义。
  - 如果只是做 文本助手 / 知识库 / 编码助手，优先开 --language-model-only。

  为什么还是更推荐 27B

  - 官方 benchmark 里，27B 在通用和代码上整体更强：MMLU-Pro 86.1 > 85.3、GPQA Diamond 85.5 > 84.2、SWE-bench Verified 72.4 > 69.2。
  - 35B-A3B 更适合 agent / tool use / 搜索型 workflow，不是“参数更大就全面更强”。
