## 先看清:这篇笔记其实只讲了"一件事"

它看着像三个硬核话题摞在一起,其实是 **同一个问题、同一个解法,在两个尺度上各讲一遍**。

- **要解决的问题**:某个 key 的缓存没了(过期/没算过),这时候一堆请求**同时**涌进来要这个 key。
- **要避免的坏结果**:让这一堆请求**每一个都去跑那个昂贵的计算**(笔记里叫 `compute()`,可能是查数据库、调模型、跑一段重逻辑)。100 个请求就冷算 100 次,数据库被打爆——这叫**缓存击穿 / 惊群**。
- **唯一的核心思路**:只让**一个**去算,其余的**等它算完、然后共享它的结果**。这个模式有个名字,叫 **singleflight**(一次飞行)。

整篇笔记剩下的全部内容,都只是在回答一个问题:**"怎么保证只放一个进去算"**。而这件事在两个尺度上要用不同的工具:

| 尺度 | 谁在争 | 用什么锁 | 对应 |
|------|--------|----------|------|
| 一个进程内 | 多个**线程** | **striped lock**(条带锁) | Turn 1 + 2 |
| 多个进程/多台服务器之间 | 多个**进程** | **Redis 分布式锁** | Turn 3 |

> 🔑 **记住这一句,后面所有"为什么不上更严格的方案"都会变得自洽:**
> 这套代码每次取舍都选「便宜且够用」而不是「绝对正确」,因为**这里万一搞错,代价只是"多算几次"——是浪费,不是数据错误**。所以它宁可偶尔多算,也不愿意把请求卡死。

这就是骨架。下面我们填血肉。

## 第 1 层:锁是什么 → 三种粒度 → 什么是 striped lock

### 锁最朴素的作用

一句话:**让一段代码"同一时刻只允许一个线程在里面跑",其他想进的线程排队等。** 进去的那段代码区域叫"临界区"。

为什么需要?因为两个线程同时干同一件事会互相踩。但在**这篇笔记的语境里**,锁不是为了防数据踩坏,而是为了**串行化**——"别让两个线程同时去跑那个昂贵的 `compute()`,一个就够了"。抓住这个目的,后面就顺了。

### 三种粒度:用"电话亭"理解

想象一间办公室,大家要打电话(= 要算某个 key)。电话亭就是锁,**一个亭子同一时刻只能进一个人**。

**方案 A:全公司就一个电话亭(全局一把锁)**
你打给妈妈、我打给客户,八竿子打不着,可还是得排同一个亭子的队。
→ 简单,但**不相关的请求互相堵**,并发度极差。

**方案 B:每个号码建一个专属亭子(per-key 锁)**
打给妈妈就有个"妈妈亭",打给客户有个"客户亭",互不干扰,并行度拉满。
→ **但号码是无限的**,亭子越建越多,办公室被塞爆(内存爆炸)。更麻烦的是:一个号码很久没人打了,**谁来拆这个亭子?** 这就是笔记里说的那套 per-key 锁字典的**内存泄漏**——锁建了不回收,越堆越多。

**方案 C:只建固定 64 个亭子,按号码分配(striped lock ⭐)**
要打电话时,拿你的号码算一下 `hash(号码) % 64`,得数是几就用几号亭。

```
lock = locks[hash(key) % N]   # N=亭子总数,固定不变
with lock:
    ...                        # 只锁这一个亭子,不影响别的亭子
```

- **大部分时候**,你和我号码不同 → 分到不同亭子 → 照样并行。✅
- **亭子永远就 64 个**,从不新建、从不拆除 → 没有内存泄漏、不用清理。✅
- **唯一的代价**:有时你我两个**毫不相关**的号码,恰好都算出"3 号亭",于是被迫排队等对方——明明该并行的却堵了。这叫 **假争用 / false contention**。

`striped` 直译"条带":你可以想象那 64 个锁是 64 条平行的带子,每个 key 被分到其中一条上。

### 一句话总结第 1 层

> **Striped lock = 用"固定 N 把锁"换来"接近 per-key 的并行度",代价是偶尔的假争用。** N 是个旋钮:N 越大,撞车(假争用)概率越低,但占的内存越多。笔记里的代码用了 `N = 2048`,撞车概率低到可以忽略。

这正好回答了笔记 **Turn 1** 的那张表——你现在回头看那张"全局锁 / per-key / striped"对比表,应该就是大白话了。

# 第 2 层:`InProcessTTLCache` 逐行拆

先说它是个啥:**一个住在你 Python 进程内存里的"带过期时间的缓存表",而且当多个线程同时来要同一个没算过的 key 时,它保证只算一次。** 就这两件事。下面一块块拆。

## 2.1 它到底存了什么?(第 28 行)

```python
self._data: OrderedDict[str, tuple[float, Any]] = OrderedDict()
```

逐个词翻译:

- `OrderedDict` = 一个**会记住插入顺序**的字典。普通 dict 也记顺序,但 OrderedDict 多了个本事:能把某个元素**挪到末尾**(`move_to_end`)、能**从最前面弹出**(`popitem(last=False)`)。这两个本事专门为后面的 LRU 淘汰服务,先记住。
- 它的形状是 `键 → (过期时刻, 值)`。比如 `"lead:v3:abc" → (1234.5, [客户列表])`。
- 那个 `float` 是**过期时刻**,不是"还能活多久",是一个绝对的时间点。怎么来的?看 `set` 里 `time.monotonic() + self._ttl` —— **现在这一刻 + 存活秒数 = 到这个点就算过期**。

> 🕐 小白补丁:`time.monotonic()` 是一个"只会往前走、绝不回拨"的秒表。为什么不用平时的 `time.time()`(墙上时钟)?因为墙上时钟会被系统校时、夏令时往回跳,一跳你的"过期判断"就乱了。monotonic 永不回头,专门用来算"过了多久 / 到点没"。

## 2.2 两把锁,各管各的(第 29、35 行)—— 这是全篇的关键

这个类里有**两套完全不同的锁**,搞混它俩是看不懂这份代码的头号原因。

```python
self._lock = threading.Lock()                                    # 第29行:就 1 把
self._stripes = tuple(threading.Lock() for _ in range(2048))     # 第35行:2048 把
```

打个比方,把这个缓存想象成一个**办公室**:

| | 比作 | 守护谁 | 持有多久 |
|---|---|---|---|
| `self._lock` | 白板前的"**写字权**",一次只一人能上去写 | 那张 `_data` 白板(读它/写它/擦它) | **极短**(扫一眼、写一行就走) |
| `self._stripes[i]` | 办公室里的 **2048 个电话亭**(接第1层的比方) | "打电话"这个**昂贵动作**(跑 `compute()`) | **可能很长**(算多久占多久) |

**为什么非要分两把?** 假设只用白板那把锁来守整个流程:那么 T1 线程为了算 key A 占着锁打了 3 秒电话,这 3 秒里 **T2 连读一下 key B 都做不到**(白板被锁死了)—— 整个缓存退化成"全公司排一个队",废了。

分开之后:

- 读写白板(`get`/`set`)只占 `self._lock` **几微秒**,谁都不怎么挡谁。
- 慢动作 `compute()` 由 `self._stripes` 守,而它有 2048 个,**不同 key 在不同电话亭里同时打,互不干扰**。

这就是笔记里"两套锁,职责分明"那句话的实锤。记牢这张表,下面三个方法你会发现它们各自只碰其中一把。

## 2.3 `get()` 逐行 —— 读缓存(第 40–51 行)

```python
def get(self, key):
    now = time.monotonic()
    with self._lock:                  # ① 上"白板写字权"(其实读也要锁,防别人同时改)
        item = self._data.get(key)
        if item is None:
            return None               # ② 白板上没这条 → 没命中
        expires_at, value = item
        if expires_at < now:          # ③ 过期了吗?(过期时刻 早于 现在)
            self._data.pop(key, None) # ④ 过期就擦掉
            return None               #    → 当作没命中
        self._data.move_to_end(key)   # ⑤ 没过期:把它挪到末尾 = 标记"刚用过"
        return value                  # ⑥ 命中,返回值
```

几个小白点:

- `with self._lock:` —— 这是 Python 的"自动加解锁"写法。进入 `with` 块自动**加锁**,离开(哪怕中途报错抛异常)自动**解锁**。等价于 `lock.acquire()` ... `try/finally: lock.release()`,但不会忘记解锁。
- 注意它**全程只碰 `self._lock` 这把白板锁,从不碰电话亭锁**。所以 `get` 很快。
- ⑤ 这步 `move_to_end` 是 LRU 的一半:每次用到就把它推到队尾,于是"**队尾=最近用过,队首=最久没碰**"。这个排序下一节淘汰时要用。

## 2.4 `set()` 逐行 + LRU 淘汰(第 53–58 行)

```python
def set(self, key, value):
    with self._lock:                                       # 同样只占白板锁
        self._data[key] = (time.monotonic() + self._ttl, value)  # 存:(过期时刻, 值)
        self._data.move_to_end(key)                        # 标记"刚用过"→ 推到队尾
        while len(self._data) > self._max_keys:            # 白板写满了?(超过上限,默认256)
            self._data.popitem(last=False)  # 从队首弹出 = 扔掉最久没碰的那条
```

**LRU = Least Recently Used,最近最少使用**。缓存满了得扔东西,扔谁?扔"最久没人碰的"。靠的就是上面那套排序:

- 每次 `get`/`set` 都 `move_to_end` → 热的在队尾。
- 满了就 `popitem(last=False)` → 从队首拿,正好是最冷的。
- 整个过程都是 O(1),很快。

**第 58 行那句注释是整个"为什么用 striped"的题眼:**

```python
self._data.popitem(last=False)  # striped 锁池固定大小，无需随淘汰回收
```

意思是:我在这儿**扔掉一个 key**的时候,**完全不用去管它的锁**。因为锁不是这个 key 私有的,而是 2048 个公用电话亭里的一个,扔 key 跟拆电话亭没关系。—— 这句话现在你可能还没完全体会到爽点,等 2.7 节讲完"旧设计为什么漏内存"你就懂它在得意什么了。

## 2.5 `get_or_compute()` —— singleflight 的心脏(第 60–70 行)⭐

这是**整个第 2 层最重要的 10 行**,笔记 Turn 2 全在讲它。

```python
def get_or_compute(self, key, compute):
    cached = self.get(key)              # 1️⃣ 快路径:先不上电话亭锁,瞄一眼白板
    if cached is not None:
        return cached                   #    命中就直接走,连锁都不碰
    with self._key_lock(key):           # 2️⃣ 没命中:抢这个 key 对应的那个电话亭
        cached = self.get(key)          # 3️⃣ 双重检查:进了亭子再瞄一眼白板
        if cached is not None:
            return cached               #    别人趁我抢锁时已经算好了 → 直接吃
        value = compute()               # 4️⃣ 全 key 只有我一个线程能跑到这,冷算
        self.set(key, value)            #    算完写白板
        return value
```

(`compute` 是调用方塞进来的一个函数,"该怎么算这个 key"。`self._key_lock(key)` 就是第 37 行那个 `self._stripes[hash(key) % 2048]` —— 把 key 哈希一下,对应到 2048 个亭子里的某一个。)

### 为什么要 `get` 两次?用三个线程的"慢动作回放"讲透

这是新手第一次读必然卡住的地方。设想 key `"X"` 没缓存,**三个线程 T1/T2/T3 几乎同一瞬间**都来要它:

| 时刻 | T1 | T2 | T3 | 白板上的 X |
|---|---|---|---|---|
| ① 快路径 `get` | 没命中 | 没命中 | 没命中 | 空 |
| ② 抢电话亭锁 | **抢到✅** | 被挡⛔等着 | 被挡⛔等着 | 空 |
| ③ T1 双检 `get` | 还是没→继续 | (还在等) | (还在等) | 空 |
| ④ T1 跑 `compute()` | 算3秒…写白板 | (还在等) | (还在等) | 空→**X=结果** |
| T1 放锁 | 返回✅ | | | X=结果 |
| ② T2 拿到锁 | | **抢到✅** | 还在等 | X=结果 |
| ③ T2 双检 `get` | | **命中!**🎯 直接返回,**不算** | 还在等 | X=结果 |
| T2 放锁,轮到 T3 | | | 同理双检命中,**不算** | X=结果 |

**结果:`compute()` 全程只跑了 1 次(T1),** T2/T3 等了一小会儿、然后白吃缓存。这就是 **singleflight(同一 key 只让一次"冷算飞行")**,也是"防缓存击穿"的本体。

现在回答"为什么两次 `get`":

- **第 1 次(快路径,不上锁)**:纯优化。绝大多数请求其实是命中的,何必为了命中还排队抢锁?先白瞄一眼,命中就溜,省掉锁开销。
- **第 2 次(双检,锁内)**:正确性的关键。"我排队抢到锁了,但**在我排队那段时间里,前面那个人(T1)是不是已经替我算好了?** 是的话我就别傻乎乎再算一遍。" —— **把这一行删了,T2/T3 拿到锁就会各自再 `compute()` 一遍,singleflight 当场失效。**

> 一句话记忆:**第一次 get 是为了"快"(命中就别上锁),第二次 get 是为了"省"(别人算过我就别重算)。**

## 2.6 `_cacheable` 变体 —— "只缓存健康结果"(第 72–85 行)

跟 `get_or_compute` 几乎一模一样,**唯一区别**:`compute()` 这次返回的是 `(值, 能不能缓存)` 两个东西:

```python
value, cacheable = compute()
if cacheable:            # 只有"健康"的结果才写白板
    self.set(key, value)
return value             # 但不管健不健康,都把值返回给当前这个调用方
```

**为什么要这个?** 想象 `compute()` 内部要调下游服务,结果下游挂了,它只能返回一个"降级占位结果"。这种垃圾结果你**不想让它在缓存里赖 300 秒**毒害后面所有人。于是:这次的调用方先拿去用(`return value`),但**不写进缓存**(`cacheable=False`)。下一个请求来,白板还是空的,会重新尝试算 —— 给下游一个"恢复了就立刻缓存好结果"的机会。

> ⚠️ 埋个伏笔:正因为"不健康就不写白板",会带来一个副作用 —— 等在外面的线程**读不到结果**(白板一直空),只能等持锁人放锁后**自己再串行算一遍**。这个行为在进程内版无伤大雅,但到 Redis 版会变成一条需要专门写进文档的"已知行为",下一层细说。

## 2.7 回收旧账:为什么是 striped 池,而不是 per-key 锁字典

现在你看完了 `set` 会扔 key、`compute` 可能抛异常、`_cacheable` 可能不写缓存,可以来真正体会 30–34 行那段大注释了。

**旧设计(被替换掉的)**:一个字典 `{key: 这个key专属的Lock}`。来个新 key 就 new 一把锁塞进去。

问题是:**这把锁啥时候删?** 一个 key 会从系统里消失有三条路:

1. **TTL 过期**(`get` 把它 pop 了);
2. **`compute()` 抛异常**(key 没算成、没进缓存,但锁可能已经建了);
3. **结果不可缓存**(`_cacheable` 那条,`cacheable=False`)。

这三条路上,key 没了,**但它在锁字典里那把锁还赖着**。你想清理?得"锁住锁字典 → 确认没人正在用这把锁 → 删掉",又绕又容易出竞态,非常难写对。结果就是**锁字典随着 key 不停换新而只增不减 —— 慢性内存泄漏**。

**Striped 池的解法**:开机时一次性造好 2048 把锁,**此后永不新建、永不销毁**。任何 key 来,`hash(key) % 2048` 对号入座用其中一把。于是:

- 扔 key?不碰锁。`compute` 抛异常?不碰锁。结果不缓存?不碰锁。
- **内存永远封顶在 2048 把锁**,跟你见过多少个不同 key 彻底解耦。泄漏的根被刨了。

**代价**就是第 1 层讲的**假争用**:俩不相关的 key 偶尔哈希到同一个亭子,被迫互等一下。但 2048 个槽,撞上的概率极低;而且就算撞上,也只是"白等一会儿",**绝不会算错**(singleflight 语义"同 key 只算一次"丝毫不破)。

这就是注释里"代价:不同 key 撞同一槽偶发假争用…2048 槽下概率极低,且不影响 singleflight 语义"的全部含义。

## 2.8 顺带一句:为什么不会死锁

笔记提了一句"加锁顺序始终是 stripe → data,所以不死锁",给你补上原理:

**死锁**长这样:T1 占着锁 A 想要锁 B,T2 占着锁 B 想要锁 A,俩人互相等对方松手,**永远卡死**。破解的标准办法是:**所有人永远按同一个顺序拿锁**(比如永远先 A 后 B),这样就不可能转成环。

这份代码里:`get_or_compute` 是先拿**电话亭锁**(2️⃣),再在里面调 `get/set` 拿**白板锁** —— 永远是 `亭子 → 白板` 这个方向。而 `get/set` 自己只拿白板锁、从不反过来拿亭子锁。**方向永远一致,成不了环,所以天然无死锁。**
