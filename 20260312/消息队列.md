• 消息队列最常用在这几类场景：

  1. 异步处理
  主流程先快速返回，慢操作丢到队列里慢慢做。

  例子：

  - 注册后发邮件、发短信
  - 下单后生成发票、通知仓库
  - 上传图片后生成缩略图

  好处：

  - 用户响应更快
  - 慢任务不阻塞接口

  2. 解耦
  让系统之间不要直接强绑定。

  例子：

  - 订单系统下单后，不直接调用积分系统、短信系统、物流系统
  - 而是发一个“订单已创建”的消息
  - 谁需要就去消费

  好处：

  - 一个系统改动，不容易把别的系统一起拖垮
  - 扩展新功能更容易

  3. 削峰填谷
  瞬时请求太多时，先放进队列，后面慢慢处理。

  例子：

  - 秒杀
  - 大批量导入数据
  - 节日群发消息

  好处：

  - 防止数据库或下游服务被瞬间打爆
  - 系统更稳

  4. 重试和容错
  任务失败后，可以稍后重试，不必当场报错。

  例子：

  - 调第三方支付、物流、短信接口时对方临时超时
  - 先重试几次，再人工处理

  好处：

  - 提高成功率
  - 避免一次网络抖动就失败

  5. 延迟任务
  不是马上执行，而是过一段时间再执行。

  例子：

  - 30 分钟未支付自动取消订单
  - 10 分钟后检查回调结果
  - 定时发送通知

  6. 广播/事件驱动
  一个事件发生后，多个系统都想知道。

  例子：

  - 用户注册成功后
  - 风控系统、营销系统、积分系统都要处理

  这时一条消息可以被多个消费者使用。

  什么时候特别适合

  - 接口里有明显的慢操作
  - 系统之间依赖太多，想拆开
  - 有高峰流量，想平滑处理
  - 任务允许“稍后完成”，不要求立刻完成

  什么时候不一定要用

  - 系统很小，直接调用更简单
  - 任务必须立刻成功，不能接受异步
  - 团队暂时没有能力处理重试、幂等、消息积压这些问题

  一句话：
  消息队列本质上是把“现在立刻做”改成“先记下来，稍后处理”，用来提升响应速度、系统解耦和抗压能力。




• 截至 2026-03-12，如果按工程实践来分，主流消息队列/消息平台大致是这些。下面“主流”的归类是我基于官方产品定位和当前常见选型做的工程判断，不是某个统一官方排名。

  开源自建里最常见

  - RabbitMQ：通用消息中间件，适合业务队列、任务分发、路由、多协议接入。微服务里很常见。
  - Kafka：更偏事件流平台，适合日志、埋点、数据管道、流处理、大吞吐事件总线。
  - RocketMQ：在国内和交易链路场景里很常见，事务消息、延迟消息、顺序消息是强项。
  - Pulsar：消息队列 + 流平台一体化，强项是多租户、跨地域复制、分层存储，适合更大规模平台化场景。
  - NATS + JetStream：云原生、低延迟、服务间通信和边缘场景很强，运维相对轻。
  - ActiveMQ Artemis：在传统 Java/JMS 企业系统里仍然常见，但新项目里热度通常不如前几种。

  云上托管里最常见

  - Amazon SQS：AWS 上最常见的托管队列，简单稳妥，适合任务队列和服务解耦。
  - Google Cloud Pub/Sub：GCP 上常见，偏事件分发和流式集成。
  - Azure Service Bus：Azure 体系里常见，适合企业应用集成、队列和发布订阅。

  轻量方案

  - Redis Streams：很多团队会用，但它更像 Redis 里的流/队列能力，不是专门的独立消息中间件。
  - Beanstalkd：简单、快、轻，但现在更像轻量后台任务队列，不算当下一线主流选型。

  如果你现在做选型，通常可以这么落：

  - 业务异步任务、订单后处理、邮件短信：RabbitMQ 或云上的 SQS/Service Bus
  - 埋点、日志、实时数据管道、事件总线：Kafka
  - 国内业务、事务消息、延迟/顺序要求明显：RocketMQ
  - 云原生低延迟服务通信：NATS
  - 需要更强平台化能力和跨地域能力：Pulsar
  - 只是轻量后台任务：Beanstalkd 或 Redis Streams
