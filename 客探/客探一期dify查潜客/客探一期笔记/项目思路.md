• 下面给你一套「可落地的产品+技术思路」，按优先级从业务闭环→技术实现→测试落地。尽量短、能直接拿去开评审。

  整体判断

  - 这是典型“销售语音助手 + CRM 数据对接 + RAG 检索”的产品，关键不是模型多强，而是流程闭环、权限正确、数据同步可靠。
  - 建议先做 MVP：查客户 + 录拜访 + 录日志 三条闭环，先跑真实数据与真实销售场景。

  ———

  1) MVP 范围建议（第一阶段）

  - 功能：查客户、录拜访、录日志（先不做“拜访总结/待办”，除非业务强需）
  - 入口：移动端 CRM 悬浮按钮（浮层展开）
  - 交互：语音 → 转写 → 结构化提取 → 用户确认 →写回 CRM
  - 目标：把“录入/查询”从 3 分钟降到 1 分钟以内

  ———

  2) 关键业务闭环（必须保证）

  - 查客户：客户名称→多轮消歧→展示客户卡片→跳转 CRM 客户详情页
  - 录拜访：语音→提取客户名&内容→编辑确认→写回拜访记录
  - 录日志：语音→整理格式→编辑确认→写回当日日志（覆盖而非新增）

  ———

  3) 技术架构建议（低风险、可扩展）

  - 前端：CRM 内嵌浮层 + 语音录入按钮 + 结果编辑页
  - 语音：ASR（声网）→文本
  - NLU：意图识别 + 实体抽取（客户名、拜访内容、时间）
  - 检索：关键词粗筛 + 向量召回（RAG）
  - 生成：拜访记录、日志结构化输出（JSON + 文本）
  - 写回：标准化 CRM API（保证幂等）

  ———
─  4) 数据与权限设计

  - 权限：严格继承 CRM 权限，禁止跨权限搜索
  - 数据来源：仅 CRM（禁止外部来源）
  - 索引：公海客户 + 授权客户；按客户维度切分
  - 更新：客户库增量更新（建议 15min~1h 更新）

  ———

  5) 重点风险

  - 同名客户 / 模糊检索 → 多轮澄清必须做
  - 写回失败或重复写回 → 必须幂等、要有重试与回滚
  - 语音识别错误 → 强制“用户确认”再写回
  - 性能目标过高 →先保 5 秒以内，逐步优化

  ———

  6) 测试建议（直接可落地）

  - 功能：查客户、录拜访、录日志全流程
  - 权限：未授权客户必须不可见
  - 边界：模糊客户名、多客户匹配、网络断开
  - 语音：安静 / 嘈杂 60db，准确率验证
  - 性能：10 轮对话连测，P95 < 5s

  ———

  7) 你可以补充的关键问题

  - CRM 现有接口是否支持：拜访记录写入、工作日志更新（覆盖）？
  - 客户搜索是否有唯一标识（ID）可用于消歧后写回？
  - 是否允许“语音转文字后再编辑”，默认是否必须人工确认？
  - 语音转写是否需要离线/弱网方案？
