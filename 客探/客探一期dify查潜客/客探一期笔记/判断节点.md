• 可以。Dify 的 HTTP 请求 节点返回的数据（响应体里的字段、状态码等）都会作为后续节点可引用的变量；If-Else 节点可以直接引用这些变量做条件判断（也支持取深层字
  段）。(docs.dify.ai (https://docs.dify.ai/en/use-dify/nodes/ifelse))

  下面给你一个“先判断知识库是否已有文档 sales_customers_{employee_code}”的推荐做法（最稳、最容易维护）。

  一、用 Knowledge API 按文档名查是否存在（推荐）

  1. 先拼出目标文档名

  - 你可以用 模板 节点生成：doc_name = sales_customers_{{employee_code}}
  - 或者直接在 HTTP 节点的参数里写 sales_customers_{{employee_code}}（Dify 支持在 HTTP 节点里插入变量）。(docs.dify.ai (https://docs.dify.ai/zh/use-dify/
    nodes/http-request))

  2. HTTP 请求 节点：调用“获取知识库的文档列表”接口

  - 方法：GET
  - 路径（Dify Cloud 示例；自托管就把域名换成你自己的）：

    https://api.dify.ai/v1/datasets/{dataset_id}/documents
  - Query 参数（关键点是 keyword）：
      - keyword = {{doc_name}}（在文档名称中搜索关键词）
      - page = 1
      - limit = 20（或更大一点，比如 100，避免同名/近似名时漏掉你要的那条）(docs.dify.ai (https://docs.dify.ai/api-reference/documents/get-the-document-list-
        of-a-knowledge-base))
  - Header：
      - Authorization: Bearer {API_KEY}(docs.dify.ai (https://docs.dify.ai/api-reference/documents/get-the-document-list-of-a-knowledge-base))

  接口返回结构里会有：total（匹配到的数量）和 data（文档数组）。(docs.dify.ai (https://docs.dify.ai/api-reference/documents/get-the-document-list-of-a-
  knowledge-base))

  二、If-Else 里怎么判断

  你有两种判断方式：

  A) 简单版（够用但可能“误判”）

  - 在 If-Else 节点里选中“HTTP 节点响应体”的 total 字段
  - 条件：total > 0 就当“存在”
    因为 keyword 是“按名称搜索关键词”，它通常是包含/模糊匹配，所以如果你知识库里有类似 sales_customers_001_backup 也可能被算进来。(docs.dify.ai (https://
    docs.dify.ai/api-reference/documents/get-the-document-list-of-a-knowledge-base))

  B) 严谨版（推荐，精确匹配文档名）
  做法：加一个 代码 节点，把 data 里真正 name == doc_name 的那条找出来，输出一个布尔值 exists 和 document_id。代码节点支持 Python/JS，适合这种“筛选数组”的 逻
  辑。(docs.dify.ai (https://docs.dify.ai/en/guides/workflow/node/code?utm_source=openai))

  Python 示例（你只需要在代码节点里把输入变量绑定到：HTTP 响应体、以及 doc_name）：

  def main(doc_list: dict, doc_name: str) -> dict:
      for item in doc_list.get("data", []):
          if item.get("name") == doc_name and item.get("archived") is False and item.get("enabled") is True:
              return {"exists": True, "document_id": item.get("id", "")}
      return {"exists": False, "document_id": ""}

  然后 If-Else 判断 exists == True 即可。(docs.dify.ai (https://docs.dify.ai/en/use-dify/nodes/ifelse))

  三、你后续可能会用到的分支动作

  - IF（存在）：直接走你的后续流程（例如检索/回答/更新等）。
  - ELSE（不存在）：你可以再接一个 HTTP 请求 节点调用“Create a Document from Text / File”之类的接口创建文档，然后（如果要马上用它检索）通常还需要等索引完成再
    用。(docs.dify.ai (https://docs.dify.ai/versions/3-0-x/en/user-guide/knowledge-base/knowledge-and-documents-maintenance/maintain-dataset-via-api))

  四、两个常见坑（提前避开）

  - 文档名是否带扩展名：很多情况下知识库文档的 name 会是带后缀的（例如 .txt / 原始文件名），如果你现在“固定名”是 sales_customers_001，而实际入库叫
    sales_customers_001.txt，那“精确相等”会判不存在；这时你要统一命名规则，或把精确匹配改成 starts with。(docs.dify.ai (https://docs.dify.ai/versions/3-0-x/
    en/user-guide/knowledge-base/knowledge-and-documents-maintenance/maintain-dataset-via-api))
  - API Key 权限：知识库 API token 往往对同账号下可见的知识库都有操作权限，别把它暴露到前端或不安全位置（工作流里用一般没问题，但仍要注意泄露风险）。
    (docs.dify.ai (https://docs.dify.ai/versions/3-0-x/en/user-guide/knowledge-base/knowledge-and-documents-maintenance/maintain-dataset-via-api))

## 第二轮对话
• 只要你用的是 Chatflow 应用，第二轮开始“跳过知识库文档是否存在的 HTTP 检查”有两种常用做法（本质都是：用“第几轮/会话状态”来做一次性 gate）。
  sys.dialogue_count 和 Conversation Variables 都是 Chatflow 才有的。(docs.dify.ai (https://docs.dify.ai/versions/3-0-x/en/user-guide/workflow/variables?
  utm_source=openai))

  方案 A：用 sys.dialogue_count（最快）

  - 在流程最前面加一个 If-Else 节点：
      - 条件：sys.dialogue_count > 1
      - True 分支：直接连到你“后面的流程”（完全不走 HTTP 检查链路）
      - False 分支（第一轮）：再去走“检查/创建文档 sales_customers_{employee_code}”的分支(docs.dify.ai (https://docs.dify.ai/versions/3-0-x/en/user-guide/
        workflow/variables?utm_source=openai))
  - 注意：如果你第一轮检查失败/没写入任何结果，第二轮仍会被放行，所以很多时候会配合方案 B 更稳。

  方案 B：用会话变量缓存“已确认/已拿到 doc_id”（更推荐）

  1. 在 Chatflow 里创建 Conversation Variables（例如）：
      - doc_ready（Boolean，默认 false）
      - doc_id（String）
      - doc_name_cached（String）(docs.dify.ai (https://docs.dify.ai/versions/3-0-x/en/user-guide/workflow/variables?utm_source=openai))
  2. 每轮开始先算出当前目标名：doc_name = sales_customers_{{employee_code}}
  3. 流程开头加 If-Else：
      - IF：doc_ready == true 且 doc_name_cached == doc_name → 直接走后续流程（不发 HTTP）
      - ELSE：走你的 HTTP 查文档/建文档逻辑 → 成功后用 Variable Assigner 写入：
          - doc_ready = true
          - doc_id = (查到/创建得到的 id)
          - doc_name_cached = doc_name(docs.dify.ai (https://docs.dify.ai/versions/3-0-x/en/user-guide/workflow/node/variable-assigner?utm_source=openai))

  ## 方式 2：用会话变量“只要成功一次就跳过”（更稳，推荐）

  适合你真实需求：第一轮确保文档存在；从第二轮开始直接走后续；如果第一轮没成功，第二轮还会继续补做。

  1. 在 Chatflow 里建 Conversation Variables（会话变量）：

  - doc_ready（Boolean，默认 false）
  - 可选：doc_id（String）
  - 可选：doc_name_cached（String，用于员工号变化时重新校验）

  2. Start 后加 If-Else：

  - 判断：doc_ready == true
      - True：直接走后续流程（不再调用 HTTP 检查）
      - False：走 HTTP 检查/创建文档链路

  3. 在你“确认文档存在/创建成功”之后，加一个 Variable Assigner：

  - doc_ready = true
  - （可选）doc_id = 查到/创建返回的 document_id
  - （可选）doc_name_cached = sales_customers_{{employee_code}}

  如果你担心用户第二轮会换 employee_code，就把条件写成更严谨的：

  - doc_ready == true 且 doc_name_cached == sales_customers_{{employee_code}}
    不满足就重新走一次检查。
