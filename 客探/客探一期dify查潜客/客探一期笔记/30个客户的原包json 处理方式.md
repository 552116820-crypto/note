• 30 个客户“整包原始 JSON”一次性丢给大模型做信息抽取，大概率不划算也不稳：

  - Token 方面：很多模型会直接超上下文或把输出挤爆。
  - 幻觉方面：即使 token 够，也更容易出现“漏抽/错字段/把空值补成合理值”的情况（这类在抽取任务里也算幻觉）。

  下面按你这份结构，给你一个可落地的判断和做法。

  Token 够不够（怎么估）
  Token 消耗 = 系统/指令 + 输入(JSON) + 输出(JSON/表格)。

  你这一个客户对象字段非常多（基本信息 + 开票信息 + 联系人数组 + 收货地址数组），而且 key 很长、字符串很多、还有不少空字段。经验上这种 “带大量 key 的 JSON”
  token 密度很高。

  粗略估算（按常见大模型 tokenizer 的量级）：

  - 1 个客户全量 JSON：通常 ~1,000 到 3,000+ tokens（视字段多少、中文/英文占比、空值多少而波动很大）
  - 30 个客户输入：~30,000 到 90,000+ tokens
  - 再加上你的抽取指令 + 结构化输出（30 条结果）：很容易再多 几千到上万 tokens

  所以结论是：

  - 如果你用的是 8k/16k/32k 上下文的模型：基本不够（或者勉强塞进去了也会截断，结果不可靠）
  - 即使用 128k 上下文的模型：可能“塞得下”，但会 贵、慢，且更容易因为信息太密而 错抽/串字段

  最稳的做法不是“猜够不够”，而是：

  - 在 Dify 的运行日志里看每次调用的 prompt_tokens / completion_tokens（很多模型供应商都会回传 usage）
  - 或者你把“实际要发给模型的文本”拿去做 token 计数（用你所选模型对应的 tokenizer）

  会不会产生幻觉（会，而且有规律）
  信息抽取里的“幻觉”通常长这样（跟 token 不够/信息太多都有关）：

  - 把 null/""/"-" 自动补成“合理值”（例如把缺失区县补上、把空邮箱编一个）
  - 字段串位：例如把 tel 当 phone，或联系人里的电话写到客户基本信息里
  - 归一化时瞎改：例如把地址/地区“纠正”成另一个省市，或者把“月结30天…”总结成它以为的付款条件
  - 数组合并错误：联系人/收货地址多条时，把 A 的名字和 B 的电话拼到一起

  这些不是“模型坏”，而是它默认会“帮你补全”，输入越长、结构越复杂、你要求的输出越多，越容易发生。

  推荐的工作流方案（在 Dify 里很好落地）
  核心思路：别一次喂 30 个完整对象，改成“裁剪 + 分批/逐条 + 结构化输出 + 校验”。

  1. 能不用模型的就不用模型
     如果你要抽取的就是现成字段（比如 creditNo、invoice_type、payment_method_name、contact/phone 等），这属于“结构化数据取值”，用代码/JSONPath 100% 准确且 0
     幻觉。
     大模型更适合做：归类、清洗、把多字段合并成一句话、从 description 这种非结构化文本里提取要点。
  2. 输入先“瘦身”（非常关键）
     在进入 LLM 节点前，用一个 Code 节点把每个客户对象变成你真正关心的子集，比如只留：

  - 客户主键（比如 "73408"）
  - 你要抽取的字段集合（10~30 个以内更稳）
  - 联系人/地址数组也只留你关心的那 1-2 个字段（如 name/phone/identity）

  3. 用“迭代/循环”按客户逐条抽取（或小批量 3~5 个）

  - 最稳：1 个客户 = 1 次 LLM 调用
  - 想省调用次数：3~5 个客户一组（仍然比 30 个一组稳定很多）

  4. 强制结构化输出，禁止推断
     把提示词写成“抽取器”，明确：

  - 只允许从输入复制/重组，不允许补全
  - 缺失就输出 null 或空字符串
  - 输出必须是合法 JSON（最好用 JSON schema / function calling）

  可以用类似约束（示例，思路为主）：

  - “如果某字段在输入 JSON 中不存在或为 null/""/"-"，则输出 null”
  - “不要猜测省市区、不要根据地址推断地区编码”

  5. 加一层“自动校验”，把幻觉拦在模型后面
     LLM 输出后再用 Code 节点做校验，例如：

  - 对于应当“原样拷贝”的字段：要求输出值必须能在原输入中找到（精确匹配或规范化后匹配）
  - 不通过就标记该客户为 needs_review=true 或触发重试（重试时把规则再强调一次）

  额外提醒：输出也会爆 token
  很多人只算输入，忘了输出。你如果让模型“把 30 个客户全部抽成一大坨详细结构”，输出 token 也会很夸张，可能触发：

  - 输出截断（最后几条客户丢失）
  - 格式崩坏（JSON 末尾缺括号）
  - 费用飙升

  所以输出字段要克制：只输出你需要进入后续流程的“关键字段”。

  隐私/合规
  你这份数据里有手机号、地址、税号/信用代码等敏感信息。无论 token 是否够用，都建议：

  - 先脱敏/加密（至少手机号、详细地址）
  - 或用私有化/自托管模型
  - 并确认你所用模型供应商的数据保留/训练策略符合你公司的合规要求

  1. 你在 Dify 里用的具体模型（以及上下文长度/最大输出，或者你告诉我供应商+型号也行）
  2. 你最终要抽取哪些字段（列 10~30 个目标字段名即可）
