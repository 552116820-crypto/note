/• 要把“代办事项”和“拜访总结”做成你这个 Dify Chatflow 助手的能力，最稳的原则是：数据与权限在 CRM，Dify 只负责理解/生成结构化结果。这样才能做到“按 sales_id 隔离、不能越权查客户、一个销售固定会话还能跨设备一致”。

  下面按“你现在的架构（PHP 后端 + Vue 前端 + Dify 自部署 + user=sales_id 固定会话）”给一套可落地的做法。

  前提（强烈建议）

  - 不要依赖 Dify WebApp 分享页（http://dify.amer.cn/...）来承载这些功能；因为它的会话通常跟浏览器存储绑定，很难做到“一个销售固定会话”，也不适合承载“写 CRM数据”的权限边界。
  - 建议你走：Vue → CRM(PHP) → Dify POST /v1/chat-messages；user 固定传 sales_id，conversation_id 由 CRM 数据库保存（你前面提的“一销售一个固定会话”就是靠这个实现）。

  ———

  总体设计（3 层）

  - CRM（PHP）提供“工具 API”：创建/查询/更新代办；保存/查询拜访总结；查询客户/拜访记录等（带强鉴权）。
  - Dify Chatflow 通过 HTTP Request 节点调用这些工具 API（header 带服务端 token）。
  - Vue UI 展示：聊天 + 代办列表 + 拜访总结列表/详情（列表类 UI 建议直接调 CRM，不一定非走 Dify）。

  ———

  功能 1：代办事项（Todo）怎么做

  - 数据模型（放 CRM DB）
      - id
      - sales_id（必须）
      - customer_id（可选，但强烈建议支持，方便“跟客户绑定”）
      - title、description
      - due_at、priority
      - status（open/done/canceled）
      - source（ai/manual）、conversation_id（可选，便于审计）
      - created_at/updated_at
  - CRM 工具 API（给 Dify 调用）
      - GET /internal/dify/todos?status=open&customer_id=&limit=
      - POST /internal/dify/todos
        Body 示例（关键：sales_id不要从用户输入拿，后面解释）：

        {"sales_id":"S123","customer_id":"C888","title":"下周二回访确认需求","due_at":"2026-02-10 10:00:00","priority":"P2"}
      - PATCH /internal/dify/todos/{id}（完成/改期/改标题）
      - 可选：POST /internal/dify/todos/batch（拜访总结后批量创建跟进事项很有用）
  - 鉴权（非常关键）
      - Dify → CRM 的所有 /internal/dify/* 请求必须带固定 header（例如 X-Dify-Tool-Token），CRM 中间件校验。
      - CRM 端必须再次做数据权限：即使请求里带了 sales_id/customer_id，也要校验该 customer_id 是否属于该 sales_id，否则拒绝。
      - 在 Dify Chatflow 里，sales_id 一律用系统变量：{{sys.user_id}}（不要让 LLM 自己“写 sales_id”）。
  - Dify Chatflow 编排建议
      - 节点 A：意图识别（LLM 输出严格 JSON）
          - 识别动作：todo_list / todo_create / todo_done / todo_update
          - 结构：{ "action": "...", "args": {...} }
      - 节点 B：Switch/IF 按 action 分流
      - 节点 C：调用对应 HTTP Request 节点
      - 节点 D：把返回结果用一个 LLM 节点“人话化”输出给销售
  - Vue UI（推荐做成体验加分项）
      - 右侧加一个“代办”tab：列表、筛选、勾选完成、跳转到客户。
      - AI 负责创建/更新；列表展示/手工操作直接走 CRM API（更快更稳）。

  ———

  功能 2：拜访总结怎么做（生成 + 存档 + 自动生成跟进代办）

  拜访总结本质上是“把一次拜访的原始记录 → 结构化总结 → 存到 CRM，并可生成 follow-up 代办”。

  - 你要先决定：原始拜访记录在哪
      - A. 拜访记录在 CRM 里（最常见）：销售在 CRM 填了拜访纪要/录音转写。
      - B. 拜访记录由 AI 对话里输入：销售直接在聊天框粘贴笔记。

  我更推荐 A（合规、可审计、流程清晰），AI 只负责“生成总结”。

  - 数据模型（放 CRM DB）
      - id
      - sales_id
      - customer_id
      - visit_id（如果你 CRM 已有拜访记录表，强烈建议关联）
      - visit_at、visit_type（上门/电话/企微等）
      - raw_notes（可选：如果 CRM 已经存了原文，这里可以只存引用）
      - summary_json（结构化 JSON，方便后续 BI/检索）
      - summary_text（给人看的精简版）
      - created_at
  - 建议的 summary_json 结构（让 Dify 生成，CRM 校验入库）
      - title
      - attendees
      - customer_needs（数组）
      - key_points（数组）
      - objections_and_answers（数组）
      - next_steps（数组）
      - follow_up_todos（数组：title/due_at/priority/customer_id）
      - risk / opportunity / sentiment
      - tags
  - CRM 工具 API（给 Dify 调用）
      - （可选）GET /internal/dify/visits/{visit_id}：取原始拜访记录（如果总结基于已存在的拜访记录）
      - POST /internal/dify/visit-summaries：保存总结（以及可选的自动创建代办）
          - Body 可以是：

            {
              "sales_id":"S123",
              "customer_id":"C888",
              "visit_id":"V777",
              "summary_json": { "...": "..." }
            }
      - GET /internal/dify/visit-summaries?customer_id=&limit=：给 AI 查询“最近一次拜访总结”
      - 你想做“一键生成并生成代办”：可以在 CRM 保存总结时，把 follow_up_todos 解析后落到 todo 表（同事务更好）。
  - Dify Chatflow 编排（一个很实用的流程）
      - 触发语句例子：“帮我生成今天对 XX 客户的拜访总结（基于拜访记录/基于以下笔记）”
      - 流程：
          - 1）确定 customer（优先用 CRM 页面上下文传入 inputs.customer_id；没有就让 AI 调 CRM 的“客户搜索”接口或让用户明确）
          - 2）拿原始材料（GET visits/{visit_id} 或直接用用户输入 notes）
          - 3）LLM 生成严格 summary_json（不要 Markdown，不要多余文字）
          - 4）POST visit-summaries 存档
          - 5）如果 summary_json.follow_up_todos 非空：调用 POST todos/batch 创建代办
          - 6）回复用户：总结要点 + 已创建哪些代办（含到期时间）
  - Vue UI（强烈建议）
      - 在“拜访记录详情页”放一个按钮：生成总结
          - 点击后你可以走两种方式：
              - 方式 1：把 visit_id 作为上下文发给 Dify（走你现有的 /api/ai/chat），让 Chatflow 自动拉取并生成总结。
              - 方式 2：不走聊天，单独做一个“总结生成”接口（仍然调用 Dify，但 UI 更像“生成文档”）。
      - 在客户详情页增加“拜访总结”tab：可查历史、可复制、可一键生成跟进代办。

  ———

  落地步骤（按最少返工排序）

  - 1）先在 CRM 做 DB 表 + API：todos、visit_summaries（以及可选的 visits 读取接口）。
  - 2）给 /internal/dify/* 加服务鉴权（固定 token + IP 白名单/反代限制），并把“customer 属于 sales”的校验写死。
  - 3）在 Dify Chatflow 里加 4~6 个 HTTP Request 工具节点：todos list/create/patch、visit get/save、todos batch。
  - 4）加一个“意图识别 → 分流”骨架，让聊天能自然触发创建代办/生成总结。
  - 5）Vue 做两个展示面板：代办列表、拜访总结列表/详情（展示直接调 CRM，AI 只负责生成/写入）。
