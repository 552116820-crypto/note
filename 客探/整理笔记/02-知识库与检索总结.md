# 知识库与检索总结

## 数据形态共识
- 业务数据是“每销售约 30 个潜在客户”的结构化档案，来源于 CRM/企知道接口。
- 文档组织建议：一客户一块（或一客户一文档），并保留稳定唯一标识。
- 参考：[[客探一期dify查潜客/客探一期笔记/30个客户的原包json 处理方式]]、[[客探一期dify查潜客/分块处理]]、[[客探一期dify查潜客/md最终模板]]。

## 入库策略
- 推荐：先规范化字段，再入库。
- 元数据最少要有：`sales_id`、`customer_id`、`company_name`、更新时间。
- 实操里已覆盖：创建文档、提取 document_id、补元数据、检查索引状态。
- 参考：[[客探一期dify查潜客/客探一期笔记/提取document_id]]、[[客探一期dify查潜客/客探一期笔记/元数据 赋予]]、[[客探一期dify查潜客/客探一期笔记/更新 文档 http请求]]。

## 关键流程节点（最容易出错的部分）
1. 文档存在判断：
- 不建议只依赖 HTTP 节点里字符串 JSON 直接判断。
- 推荐“HTTP -> Code 解析 -> IfElse”输出 `exists/ready/doc_id`。
- 参考：[[客探一期dify查潜客/客探一期笔记/文档存在判断方案]]。

2. 索引可用判定：
- 创建后要轮询 `indexing_status`，不要立即进入检索。
- 推荐 Loop + 退避等待 + 超时上限。
- 参考：[[客探一期dify查潜客/客探一期笔记/检查索引]]。

3. 检索召回策略：
- 先保召回（TopK、融合权重、阈值），再做 rerank 精排。
- “实体详情问题”和“全量列表问题”应拆成不同路径。
- 参考：[[客探一期dify查潜客/混合检索]]、[[客探一期dify查潜客/检索边界探索]]、[[客探一期dify查潜客/客探一期笔记/知识库的问题]]。

## 提示词策略
- 实践是“双节点提示词”：
  - 节点1：问题改写（增强字段召回）。
  - 节点2：结构化结果输出（严格基于检索内容，缺失写“未提供”）。
- 已覆盖去重、字段对齐、风险解读、安美业务交集分析。
- 参考：[[客探一期dify查潜客/客探一期笔记/提示词]]、[[客探一期dify查潜客/客探一期笔记/CRMapi响应后 提示词]]。

## 当前瓶颈
- 异常与边界问题主要集中在：
  - 缩写/别名归一化不足。
  - 低相关召回导致误判（尤其 degraded 判定前阶段）。
  - 全量目录查询能力不足（更适合工具/接口层处理）。
- 参考：[[客户公司名的缩写]]、[[客探一期dify查潜客/客探一期笔记/判断节点]]、[[客探一期dify查潜客/全量问题检索方案]]。

## 建议的稳定化方向
- 把“客户目录/列表查询”从纯 RAG 中剥离，改工具接口返回。
- 对每轮写入补充幂等键与版本号，避免重复建档和脏数据。
- 将“索引未完成”做显式中间状态提示，减少用户体感失败。
- 参考：[[SLA 看板字段，加幂等键]]、[[SLA存储]]、[[客探一期dify查潜客/客探一期笔记/目前待解决的问题]]。
