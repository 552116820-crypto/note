# 架构与流程总结

## 项目目标（从 PRD 抽取）
- 面向销售场景：通过语音/文字完成客户检索、拜访记录、拜访总结、代办提取。
- 目标是“提效 + 准确 + 合规”：
  - 检索更快（分钟级降到秒级）。
  - 结果可编辑可回写。
  - 严格继承 CRM 权限。
- 参考：[[客探一期dify查潜客/客探一期笔记/项目PRD结构化版本]]、[[客探一期dify查潜客/客探一期笔记/项目 业务需求]]。

## 当前主架构（笔记一致性最高）
- 推荐三层：
  - 前端（企业微信/CRM H5）：负责交互与展示。
  - CRM 网关（后端）：负责身份、权限、缓存、审计、限流。
  - Dify Chatflow：负责意图、检索、生成、流程编排。
- 核心原则：不要让 Dify 直接持有 CRM 强权限凭证，强身份与隔离应由网关控制。
- 参考：[[客探一期dify查潜客/客探一期笔记/项目 流程]]、[[客探一期dify查潜客/客探一期笔记/项目 框架]]、[[客探一期dify查潜客/客探一期笔记/登录 sales_id的问题]]。

## Chatflow 主链路（稳定版本）
1. Start 接收可信用户标识（`wecom_userid`/`sales_id`）。
2. EnsureCache（HTTP）做首次同步或增量刷新。
3. If 判断缓存是否 ready。
4. 轻量分类器分流：客户查询/列表汇总/刷新。
5. 检索与格式化输出。
6. 失败统一进入告警与兜底回复链路。
- 参考：[[客探一期dify查潜客/客探一期笔记/项目 流程]]、[[客探一期dify查潜客/客探一期笔记/dify 编排]]、[[状态机]]。

## 工作流建设方式演进
- 从“手工点 UI”演进到“API 级改图 + UI 验收”的混合模式：
  - 通过 workflow draft API 直接增改 graph（快、可批量、可审计）。
  - 用 Chrome 侧做观测、验收、回归。
- DSL 能力已经具备，可从零编写或修改大型工作流。
- 参考：[[API 级编辑工作流图]]、[[ai自动搭建工作流的原理]]、[[dsl文件的编写]]、[[从零编写dify的dsl文件]]。

## 技术路线分歧与共识
- 共识：Dify 适合快速交付与业务编排；复杂治理能力最终要代码化补齐。
- 迁移备选：
  - LangGraph：更强控制力，适合复杂状态与服务化。
  - n8n：更偏自动化编排，不替代对话/RAG中枢。
- 参考：[[dify迁移到代码开发]]、[[客探一期dify查潜客/客探一期笔记/langgraph落地]]、[[客探一期dify查潜客/客探一期笔记/n8n 和 dify]]。

## 当前阶段判断
- 已跨过“可用性验证”阶段，进入“可持续运营”阶段。
- 下一步关键不是继续扩功能，而是先固化工程边界：身份可信注入、幂等与审计、SLA落库、异常闭环。
- 参考：[[SLA验收]]、[[审计留痕]]、[[SLA存储]]。
