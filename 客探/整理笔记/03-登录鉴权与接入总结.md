# 登录鉴权与接入总结

## 已达成的核心共识
- 前端不可信，`sales_id/employee_code` 不能由前端直接决定权限。
- Dify API Key 只能保存在服务端；浏览器端直接调用是高风险方案。
- 会话隔离建议依赖可信 `user`（对应 `sys.user_id`）或后端注入变量，而不是 URL 直传。
- 参考：[[客探一期dify查潜客/客探一期笔记/登录 sales_id的问题]]、[[客探一期dify查潜客/客探一期笔记/sys.user_id]]。

## 三种接入方式对比
1. 直接跳 Dify `/chat/...`（最快，风险最高）
- 优点：开发快，几乎零前端改造。
- 风险：隔离弱、鉴权弱、参数注入不稳定、易被链接绕过。
- 参考：[[客探一期dify查潜客/客探一期笔记/登录 问题]]、[[客探一期dify查潜客/客探一期笔记/登录 妥协方案]]。

2. 中间接口跳转（MVP可用）
- 由后端解析参数、调用 Dify、再重定向。
- 适合阶段验证，但要补 HTTPS、签名、限流、防预取误触发。
- 参考：[[客探一期dify查潜客/客探一期笔记/中间接口]]、[[客探一期dify查潜客/客探一期笔记/中间接口代码]]、[[客探一期dify查潜客/客探一期笔记/中间接口 3 代码 解释]]。

3. 自建聊天页 + 后端代理 Dify（推荐最终形态）
- 身份、会话、隔离、审计都在自有后端控制。
- 可稳定支持“同用户会话复用 + 严格按用户检索过滤”。
- 参考：[[客探一期dify查潜客/客探一期笔记/自建WEB APP]]、[[客探一期dify查潜客/客探一期笔记/UI+API请求]]、[[客探一期dify查潜客/客探一期笔记/项目 登录接口]]。

## 无感登录结论
- Dify 支持在嵌入时注入 `inputs`，可避免手工填写 Start 必填项。
- 但要注意：
  - 参数名必须与变量名一致。
  - `/chat/` 与 `/chatbot/` 行为存在差异。
  - 编码链路（gzip/base64/urlencode）不能手工破坏。
- 参考：[[客探一期dify查潜客/客探一期笔记/无感登录]]、[[客探一期dify查潜客/客探一期笔记/无法实现自动登录]]、[[客探一期dify查潜客/客探一期笔记/现在无法实现无感登录的问题逻辑]]。

## conversation_id 管理
- 建议后端维护 `sales_id -> conversation_id` 映射，避免每次开新会话。
- 若要支持“会话重绑/切换”，需加管理接口和并发安全策略。
- 参考：[[客探一期dify查潜客/多轮对话的conversation_id处理]]、[[API文档id]]、[[客探一期dify查潜客/客探一期笔记/中间跳转页]]。

## 上线前最低安全清单
- 强制 HTTPS。
- API Key 全量轮换后改环境变量管理。
- 对中间接口加签名/鉴权/限流。
- 禁止以 GET 触发有副作用操作，或至少增加一次性 token 与过期控制。
- 参考：[[部署相关/nginx]]、[[部署相关/dify外网映射的问题]]、[[客探一期dify查潜客/客探一期笔记/注意事项]]。
