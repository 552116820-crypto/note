# SLA验收与生产治理总结

## 现状判断
- 四态验收（success / duplicate / degraded / failed）已形成可重复执行流程。
- 告警链路已实跑验证，失败场景可触发 webhook，并可回滚环境配置。
- 参考：[[SLA验收]]、[[一键四态验收脚本]]、[[dify工作流，生产告警，降级处理]]。

## 已落地能力
1. 四态定义明确
- `success`：正常命中并成功输出。
- `duplicate`：幂等命中后短路。
- `degraded`：业务降级回复（非崩溃）。
- `failed`：异常进入 fail-branch，给兜底答复并告警。
- 参考：[[4态验收]]、[[状态机]]。

2. 告警与回归
- 失败演练支持“临时改坏 -> 触发告警 -> 自动回滚 -> 回归验证”。
- 证据链包含 SSE 原始日志、报告 JSON/Markdown。
- 参考：[[一键四态验收脚本]]、[[tmp文件]]、[[快照文件]]。

3. 验收自动化
- 已有一键脚本和报告输出机制，可纳入定时回归。
- 参考：[[一键四态验收脚本]]。

## 仍需补齐（生产级差距）
- 当前 SLA 写回很多仍停留在会话变量层，不等于真实可审计落库。
- 需要把四态结果统一写入业务数据层（明细表 + 聚合表），并以幂等键 upsert。
- 参考：[[SLA存储]]、[[SLA 看板字段，加幂等键]]、[[审计留痕]]。

## 推荐落地数据模型（从笔记抽象）
- 运行明细表：`run_id`、`trace_id`、`status`、`latency_ms`、`idempotency_key`、`error_code`、`alert_sent`、`created_at`。
- SLA 聚合表：按天/小时聚合成功率、失败率、P95耗时、告警数。
- 关键策略：写库失败不阻断主链路，但必须触发二级告警。
- 参考：[[SLA存储]]、[[dify工作流，生产告警，降级处理]]。

## 下一步最有价值动作
1. 固化一份“发布前强制门禁”：改图后必须跑四态验收。
2. 补齐 SLA 真实落库节点与审计字段。
3. 把验收报告自动推送到团队渠道，减少人工复核成本。
- 参考：[[任务清单]]、[[SLA验收]]。
