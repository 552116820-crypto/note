- 低磷 `<50ppm` → **数值上限**(客户值 ≤ 产品上限)
- 泵型 `柱塞/叶片/齿轮` → **集合交集**
- 压力等级、国标 HL/HM/HV/HS → **枚举相等或就近**

这就是两个大类 matcher 天生不同的原因。将来真做 `category_dimension` 配置,每维就要标"类型(范围/上限/集合/枚举)＋硬过滤 or 软打分＋权重"。

## 对你当前工作的修正

上一步"Step 0 定 schema"补一句:**schema 分两层**——共性层(catalog)一次定死、所有大类共用;个性层(每大类工况维度＋词表)一个大类定一次。你手头 A01、A30 字段差异最大,正好各抽一版当模板,后面大类照着套。

# 安美产品推荐 · 可上线方案

## 方案一句话
**产品异构维度全部数据化,不进代码。** 加大类/改字段 = 插配置 + 灌数据,零改代码。选品靠结构化(确定、不漏、不编),deepseek 只写理由。

## 一、数据模型(MySQL,`prod_` 前缀,跟 churn_ 区分)

| 表 | 作用 | 关键字段 |
|---|---|---|
| `prod_catalog` | 产品主数据(**16 类共用**) | 物料编码(PK)、名称、大类、子域、国标分类、外观、图片、小程序path、h5、**状态(在售/开发中/停用)** |
| `prod_spec` | 产品工况值(**异构,JSON 承载**) | 物料编码、大类、`dims`(JSON)、`embedding`(找替代用) |
| `category_dimension` | **元配置(整套设计的核心)** | 大类、维度key、比较类型、匹配角色、权重、词表组 |
| `vocab` | 受控词表(**产品侧+客户侧共用,是地基**) | 词表组、code、label、**别名[]**、父级 |
| `customer_profile` | 客户工况(**按大类**) | 客户id、大类、`dims`(JSON)、来源、更新时间 |

**`dims` 用 JSON 装异构维度**,同一张 `prod_spec` 存所有大类:
```jsonc
// 液压油 A01
{"压力等级":"高压","泵型":["柱塞","叶片"],"温度下限":-40,"温度上限":120,"国标":"HS","抗燃":false}
// 水性切削液 A30
{"材质":["铸铁","碳钢"],"工艺":["车削","铣削"],"负荷":"重","防锈天数":7,"硬水上限":600,"环保":["无氯","无磷"]}
```

**`category_dimension` 是让 16 类跑一套代码的关键**——它声明每个维度**怎么匹配**:
```
(A01, 温度下限/上限, 范围包含, 硬过滤, -,  -)
(A01, 泵型,         集合交集, 硬过滤, -,  泵型词表)
(A01, 压力等级,     枚举就近, 软打分, 0.4, 压力词表)
(A30, 材质,         集合交集, 硬过滤, -,  材质词表)
(A30, 硬水上限,     数值上限, 硬过滤, -,  -)
(A30, 工艺,         集合交集, 软打分, 0.5, 工艺词表)
```
比较类型就四种算子:**范围包含 / 数值上限下限 / 集合交集 / 枚举相等就近**。

## 二、运行时管线(一套代码,配置驱动)

```
recommend(mode, ...):
  cfg = category_dimension[大类]                 # 读该类维度定义
  cands = prod_spec[大类] ⋈ catalog(status=在售) # 开发中/停用的不进候选
  # 硬过滤(红线,代码做,不交给会飘的模型)
  for p in cands: for d in cfg(硬过滤):
      if not 算子[d.类型](输入[d.key], p.dims[d.key]): 淘汰 p
  # 软打分
  score(p) = Σ cfg(软打分): 契合度[d.类型](...) × 权重
  return top-N
```
- **选型模式**:输入 `customer_id(+大类)` → 用 `customer_profile` 的工况过滤打分。
- **替代模式**:输入**在用物料编码** → 以它的 `spec` 为查询,对同大类做相似度(结构化字段加权 or 行级 `embedding`)。这就是 C511P 那种问题,稳定捞全(含跨系列近邻),不靠字面锚。
- **两模式同一出口**:top-N → deepseek 逐个写理由(喂命中维度+参数,**护栏:禁销量断言、禁编型号**)→ 组装**统一 cards**(物料编码/名/图/理由/小程序path/h5)→ `/recommend` 返回。跟大类无关,推送侧不用管。

> `embedding` 别用 deepseek(它没 embedding 接口),用 BGE/M3E 或云厂商 embedding;16 类×几十行≈几百到千余产品,应用层 cosine 就够,**不上向量库**。

## 三、16 张表怎么入库(最大的一次性工作量)

每张表字段不同、单元格自由文本,纯脚本啃不动、纯手工太慢。**用 LLM 做离线 ETL + 人工校验**:
1. 每张表 → 先定它的 `category_dimension`(维度+算子+硬/软)+ 抽 `vocab`(材质/泵型…枚举及别名);
2. 把整张表喂 LLM,按目标 schema 输出 `catalog`+`spec` 的 JSON;
3. **人工过一遍 vocab 归一**(别名合并,如"碳钢件"→"碳钢")——这步不能省,词表错了全盘匹配失效;
4. 按大类优先级分批灌(先上量大的几类)。

> 注意:架构表会改版(A01/A30 都标了 2025-04 变更日期)。ETL 要**可重跑**,`catalog` 记表版本,改版重抽即可。这跟我们否掉的运行时 RAG 不冲突——**LLM 用在一次性抽取(有人校)和运行时写理由,不用在选品**。
