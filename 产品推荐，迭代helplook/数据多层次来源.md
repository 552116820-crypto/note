## 二、运行时管线(一套代码,配置驱动)

```
recommend(mode, ...):
  cfg = category_dimension[大类]                 # 读该类维度定义
  cands = prod_spec[大类] ⋈ catalog(status=在售) # 开发中/停用的不进候选
  # 硬过滤(红线,代码做,不交给会飘的模型)
  for p in cands: for d in cfg(硬过滤):
      if not 算子[d.类型](输入[d.key], p.dims[d.key]): 淘汰 p
  # 软打分
  score(p) = Σ cfg(软打分): 契合度[d.类型](...) × 权重
  return top-N
```
- **选型模式**:输入 `customer_id(+大类)` → 用 `customer_profile` 的工况过滤打分。
- **替代模式**:输入**在用物料编码** → 以它的 `spec` 为查询,对同大类做相似度(结构化字段加权 or 行级 `embedding`)。这就是 C511P 那种问题,稳定捞全(含跨系列近邻),不靠字面锚。
- **两模式同一出口**:top-N → deepseek 逐个写理由(喂命中维度+参数,**护栏:禁销量断言、禁编型号**)→ 组装**统一 cards**(物料编码/名/图/理由/小程序path/h5)→ `/recommend` 返回。跟大类无关,推送侧不用管。

> `embedding` 别用 deepseek(它没 embedding 接口),用 BGE/M3E 或云厂商 embedding;16 类×几十行≈几百到千余产品,应用层 cosine 就够,**不上向量库**。

**Assistant:**

两句都对，因为它们根本不在回答同一个问题——一句讲**"怎么算"**，一句讲**"拿什么算"**。是同一个系统的两个面，正交的。

## 第 1 句 = 机制面(怎么算)
"硬过滤 / 软打分"是**匹配引擎的方法**——回答"用什么方式把产品挑出来、排序"。这是**动词、是引擎、是处理逻辑**。它不关心数据从哪来，只管"给我一堆候选 + 一组约束，我怎么筛、怎么排"。对应架构就是 P2 那个 matcher。

## 第 2 句 = 依据面(拿什么算)
那一串来源是**喂给引擎的信号**——回答"凭什么推"。这是**名词、是燃料、是输入**。

而且第 2 句自己还是**混着讲的**，里面其实有 4 种不同性质的依据：

| 依据 | 性质 | 回答 |
|---|---|---|
| 行业选型表、产品架构表 | **规则 / 专家知识** | 该推什么 |
| 业务员提供的客户需求 | **查询 / 客户画像** | 客户要什么 |
| 客户历史订单、同行采购历史 | **行为 / 证据信号** | 有什么倾向和佐证 |
| 安美主推产品 | **商业意志** | 公司想卖什么 |

## 两个面怎么合到一起
不是"两种实现方式二选一"，是**引擎(1)消化信号(2)**。而且第 2 句每类信号，进引擎的**位置不一样**：

- **选型表 / 架构表** → 候选**规则库**（硬过滤+软打分就跑在这些行上）
- **客户需求** → **查询**（驱动过滤打分的那组工况）
- **历史订单** → 反推在用产品**补全画像** + 复购/找替代 + **打分加分**
- **同行采购** → **协同信号** → 打分加分（"同行业 23 家在用"——就是原型卡片理由那句）
- **安美主推** → 软打分里的**商业加权 / 重排**

## 一句话合起来
**产品推荐 = 用【硬过滤+软打分】这套机制(第 1 句)，去消化【规则+需求+行为+商业】这些依据(第 2 句)。** 前者是引擎、后者是燃料——第 1 句说"怎么烧"，第 2 句说"烧什么"，缺一个都跑不起来。

所以你那两句不是矛盾、也不是重复，是**一个系统被从"算法面"和"数据面"各描述了一次**。真要写文档，这正好是两个并列小节:「匹配机制」+「数据依据」。
