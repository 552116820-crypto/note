**Assistant:**

vocab 就是**每个"类别型"维度的标准值清单**,作用是让**产品侧和客户侧说同一种话**。叫"枚举"是因为这些维度的取值是一个固定有限集合(材质就那十几种、工艺就那几种),不是随便填的自由文本。

**它解决什么问题?** 看你 A30 表:材质写的是"铸铁""碳钢""铝合金";而业务员填客户工况时可能打"生铁""碳钢件""45#钢"。**对人是一回事,对机器是不同字符串,直接匹配不上。** vocab 就是中间那本"标准字典",两边都归一到它,才对得上。

**一张 vocab 表长这样(材质为例):**

| vocab_group | code | label(显示) | aliases(别名,归一用) | parent |
|---|---|---|---|---|
| 材质 | carbon_steel | 碳钢 | ["碳素钢","低碳钢","碳钢件","45#钢"] | 黑色金属 |
| 材质 | cast_iron | 铸铁 | ["生铁","灰铸铁"] | 黑色金属 |
| 材质 | al_6061 | 6061铝 | ["6061","6系铝"] | 铝 |
| 材质 | al_diecast | 压铸铝 | ["压铸铝合金","ADC12"] | 铝 |

- **code**:机器用的标准 ID,存进 `sel_row` 和 `customer_profile` 的都是它
- **label**:人看的,下拉框里显示的就是这个
- **aliases**:别名,把自由文本/口语归到 code(打"碳钢件"→ 认成 carbon_steel)
- **parent**:层级,6061铝的父是铝——支持"客户填 6061铝,也能匹配到只写'铝'的产品行"

**它用在三处,"vocab 枚举接口"就是第二处:**
1. **产品灌数据**:A30 那格"铸铁"→ 存成 `cast_iron`(ETL 按别名归一)
2. **客户填画像**:业务员在工作台选材质,**下拉框的选项就是前端调 `/vocab?group=材质` 拿回去渲染的**——这就是那个"枚举接口";选完存的是 code
3. **匹配**:matcher 比的是 `cast_iron == cast_iron`,不是中文串

**哪些维度要 vocab、哪些不要:**
- **要**:值是"类别"的——材质、工艺、泵型、**行业**、气味等级、定性类型、环保标签。值域固定,要归一。
- **不要**:值是"数值/范围"的——防锈天数、硬水 ppm、工作温度、翻边高度 mm。直接比大小,没有别名问题(-40 就是 -40)。

**为什么我一直叫它"地基":** 产品侧存"铸铁"、客户侧存"生铁",不归一 → 永远匹配不上,而且**不报错,就是静默推不出来**,最难查的那种 bug。所以有条硬规矩:**业务员填画像必须从下拉选、不许自由输入**——这样客户侧天然就是干净的 code,不会漂。

## 一、五个算子(硬/软两种用法)

| 算子 | 用在哪些维度 | 硬过滤怎么判(淘汰) | 软打分 fit(0~1) |
|---|---|---|---|
| **集合交集** | 材质、工艺、泵型 | 客户值 ∉ 产品集合 → 淘汰 | 交集 / 客户需求 = **覆盖率** |
| **数值上限** | 硬水、低磷 | 产品上限 < 客户实测 → 淘汰 | 通常纯硬,不打分 |
| **数值下限** | 防锈期 | 产品提供 < 客户要求 → 淘汰 | 刚好=1,富余略减(过配费钱) |
| **范围包含** | 工作温度、翻边高度 | 客户工况值 ∉ 产品区间 → 淘汰 | 越居中越高,贴边缘略减 |
| **枚举相等/就近** | 压力等级、定性、国标 | 不相等 → 淘汰(就近可放一档) | 相等=1,相邻=0.5,否则0 |
