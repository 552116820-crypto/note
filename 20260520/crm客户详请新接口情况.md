## 三个必须钉死的边界

### ① 两个都传时的语义
最容易出歧义。我建议的语义是"**name 对两边都生效，主要看公海，潜在做命中校验**"。原因：

- 销售自己也分不清"安美智"是不是自己名下的
- 如果传 `employee_code + name`，后端先在销售名下查这个 name → 命中就返回（标 `potential`），未命中再去公海查（标 `sea`）
- 这样销售只输入名称，工作流一次调用就能搞清"是我的客户还是公海客户"

如果按你原话"`employee_code` 用于查名下、`name` 用于查公海"严格执行，同时传时就变成"全量名下 + 公海匹配"——会一次性返回几百条数据，token 会爆。**不推荐**这种语义。

### ② name 模糊匹配多条怎么表达
继续用之前讨论过的 `meta` 字段，不要让接口偷偷只返回 top-1：

```json
{
  "code": 0,
  "data": {
    "C0000123": { ..., "customer_source": "sea" },
    "C0000457": { ..., "customer_source": "sea" }
  },
  "meta": {
    "match_type": "exact | ambiguous | none",
    "sea_match_count": 4
  }
}
```

**关键约定**：`ambiguous` 时 `data` 里的每条**只返回轻量字段**（customer_id、customer_name、address_short、credit_no），不返回完整的 customer_qizhidao_info / customer_contact——避免 4 条公司的详情把 LLM context 撑爆。等销售消歧选定后，再传更精确的 name 复查。

### ③ name 字段的 keyword 范围
让后端明确告诉你 name 字段匹配哪些列，不要含糊：
- 企业名称（去空格、繁简体归一化）
- 曾用名 / 别名 / 简称
- 统一社会信用代码精确匹配
- 拼音首字母（可选）
