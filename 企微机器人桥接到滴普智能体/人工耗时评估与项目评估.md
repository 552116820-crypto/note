# 一句话定位

这是一个**协议转换 + 流式中继**服务：一端用 WebSocket 长连接挂在「企业微信智能机器人」网关上，另一端用 HTTP 调滴普的 `/chat/chat-messages` 流式接口，把滴普吐出的 `answer` 增量**实时刷进同一个企微气泡**。它本身不含任何 AI 能力，纯粹做"翻译 + 搬运 + 兜底"。

```
企业微信用户                                          滴普 Agent
    │ @机器人提问                                          ▲
    ▼                                                     │ HTTP streaming
┌─────────────┐   aibot_msg_callback   ┌──────────────────┴──────────────┐
│  企微网关    │ ─────────────────────▶ │  本服务（一个常驻进程）            │
│ openws...    │ ◀───────────────────── │                                  │
└─────────────┘   aibot_respond_msg     │  wecom_ws  ─ message_handler ─ agent_chat
   (stream/        (流式原地刷新)        └──────────────────────────────────┘
    image/file)
```

# 核心架构：缝合两个"世界"

整个项目最关键的设计点，是它要把**两种完全不同的运行模型**粘在一起：

| | 企微侧 | 滴普侧 |
|---|---|---|
| I/O 模型 | **异步** `asyncio` + `websockets` | **同步阻塞** `requests` |
| 形态 | 一条长连接，帧驱动，全双工 | 一次性 POST + SSE 流 |
| 并发 | 单事件循环 | 每请求一个线程 |

直接在 asyncio 里跑 `requests` 的阻塞流式会卡死整个事件循环。所以它用了一个经典的**线程池 + 线程安全队列**桥接（`message_handler.py:825-844`）：

```
事件循环(asyncio)                          工作线程(ThreadPoolExecutor)
   │  reply_stream "正在思考..."
   │  run_in_executor(worker) ───────────▶ agent_client.stream_chat()  ← 阻塞 SSE
   │                                          │ for event in 流:
   │  ◀── queue ◀─ call_soon_threadsafe ◀────┘   把每个增量丢回事件循环
   │  从 asyncio.Queue 取增量
   │  节流后 reply_stream 刷新气泡
   ▼  ...直到 done，发 finish=true
```

- 同步的滴普流式跑在 `agent-stream` 线程池里（大小 `AGENT_STREAM_MAX_WORKERS`，默认 8，`__main__.py:36`），避免并发消息时无限建线程；
- 线程里每收到一个增量，用 `loop.call_soon_threadsafe(queue.put_nowait, event)` 安全地塞回事件循环的 `asyncio.Queue`；
- 事件循环这边消费队列、做节流、刷企微气泡。

这是理解整个项目的"主梁"。

# 端到端主链路：一条消息的完整旅程

以"群里 @机器人 问问题"为例（`message_handler.py:445` `handle_message`）：

1. **收帧**：企微网关推 `aibot_msg_callback` 帧，`wecom_ws._handle_frame` 识别后 `create_task` 异步丢给 `handle_message`（`wecom_ws.py:193-203`），不阻塞收帧循环。
2. **去重**：用 `msgid` 查 `Deduplicator`，重复消息直接丢弃（默认 TTL 7200s，`message_handler.py:460`）——企微会重推，必须防重复回答。
3. **抽 prompt**：`extract_prompt` 支持 `text / voice(转文字) / mixed(图文里的文本)`，正则剥掉开头的 `@机器人`，把引用消息拼到末尾（`:152`）。
4. **抽文件**：若带文档，`extract_file_parts` 递归找文件块 → `download_encrypted_media` 下载并 **AES-CBC 解密**（企微下行媒体是加密的）→ 交给滴普（`:725`）。
5. **重置命令**：内容精确命中 `/reset`、`重置会话` 等就清掉会话缓存、回确认语，不调滴普（`:499`）。
6. **路由选 agent**：`resolve_agent_route_for_prompt` 按 `chatid` 选群专属 agent，或按群内命令前缀（`/法务 …`）切 agent 并把前缀从 prompt 里剥掉（`:513`）。不同 route 各自缓存一个 `AgentChatClient`（`:531`）。
7. **取会话**：用 `chatid+userid+route` 组 key 查 `conversation_id`，实现多轮上下文续话（`:524-530`）。
8. **流式中继**：`_stream_agent_reply` 启动——先发一条 `正在思考...` 占位气泡，再走上面那套"线程池跑滴普流 → 队列回灌 → 节流刷新"（`:808`）。
9. **收尾**：流结束发 `finish=true`；缓存返回的 `conversation_id`；若正文超长转成 `.md` 附件（`:556-572`）。
10. **图片/文件投递**：正文里的内嵌图片在 finish 后逐张下载并以企微原生 image 消息单独发出；滴普带回的文件以 file 消息发出（`:584-617`）。

# 分层拆解

## A. 入口 `__main__.py`（64 行）

只干装配：`load_dotenv` → `Settings.from_env()` → 建 `WeComWsClient` + 线程池 → 建 `MessageHandler` 并把 `handle_message`/`handle_event` 注册成回调 → `asyncio.run(ws_client.run())`。注意它把"配置读取"和"import"分离，环境变量在运行时才读。

## B. 企微长连接层 `wecom_ws.py`（642 行）

把企微那套私有 WS 协议封装成"发请求拿响应"和"注册回调"两种范式。

- **连接/鉴权**：连 `wss://openws.work.weixin.qq.com`，关掉库自带 ping（`ping_interval=None`）自己管心跳；连上立刻发 `aibot_subscribe`（带 bot_id + secret）鉴权（`:173`）。
- **帧分发器** `_handle_frame`（`:188`）：按 `cmd` 分流——消息帧、事件帧、**请求-响应帧**、鉴权响应、心跳 ack、回复 ack。
- **请求-响应模型**（`:554` `_request`）：这是 WS 层的精髓。每个主动请求带唯一 `req_id`，注册一个 `asyncio.Future` 进 `_pending_requests`；对端回帧时按 `req_id` 找到 future 并 `set_result`，把"全双工乱序帧流"伪装成"同步 await 调用"。`errcode≠0` 抛 `WeComApiError`。
- **心跳与重连**：每 30s 发 `ping`，连续 2 次失败就主动断；外层 `run()` 用指数退避无限重连（`:120`）。收到 `disconnected_event`（别处建了新连接）主动关连接让外层重连。
- **回复族**：
  - `reply_stream`（`:299`）= `msgtype=stream` + `stream.id` + `finish` 标志——**靠同一个 stream.id 实现"原地刷新"那条逐渐变长的气泡**；
  - `reply_text` = 一发即 finish 的 stream；
  - `reply_file` / `reply_image`（`:325/:338`）= 先 `upload_file` 拿 media_id 再发 file/image 消息；
  - `reply_welcome` = 进群欢迎语。
- **分片上传** `upload_file`（`:354`）：init → chunk×N → finish 三段式，512KB/片、最多 100 片（约 50MB），带 md5 校验，返回 `media_id`。`reply_image` 就是复用它并把 `type` 传成 `image`。
- **下行解密** `download_encrypted_media`（`:405`）：下载用户上传的文件并做 AES-CBC 解密（密钥来自回调里的 `aeskey`，做了 hex/base64 多种解码归一到 32 字节），下载前还有 SSRF 防护（拒私网/环回 IP）。
- **错误重试** `_request_with_retry`（`:585`）：只对 `errcode=6000`（数据版本冲突，流式刷新高发）做退避重试，最多 4 次。

## C. 编排层 `message_handler.py`（940 行）—— 项目大脑

除了上面主链路那 10 步，最值得看的是**流式回放引擎** `_stream_agent_reply`（`:808`）里的几个细节：

- **节流**：不是每个增量都刷企微，而是攒够 **0.8s 或 80 字** 才刷一次（`:23-24`, `:867`），否则企微会被高频 `aibot_respond_msg` 打爆、也容易触发 6000 冲突。还有一个 `wait_for(timeout)` 兜底，保证即使流暂停也能把已攒内容刷出去（`:886-891`）。
- **去抖 dedup**：`last_sent_content` 记"上次发出去的清洗后文本"，内容没变就不重发（`:862`）。
- **气泡净化** `safe_stream_display`（`:109`）：流式期间把已完整出现的 `<img>`/`![]()` 换成 `[图片N]` 占位，并把**末尾还没收全的半截标签**（dangling `<i`、`![alt](http` 之类）从可见文本里截掉——避免用户在气泡里看到裸 `<img src="http...`（这正是上一个会话修的那个问题）。
- **图片独立投递** `_dispatch_reply_images`（`:642`）：finish 后逐张 `download_image_by_url` → `reply_image`，单张失败只追加一条"图片N加载失败"，不阻断主回复。
- **超长兜底**：正文超 `REPLY_FILE_THRESHOLD_BYTES`（默认 5000）或单帧字节上限，就把完整内容转 `.md` 当附件发（`:559`）。

## D. 滴普客户端 `agent_chat.py`（1282 行）

负责"把一次提问翻译成滴普能懂的 HTTP 请求，再把 SSE 流翻译回干净的增量事件"。

- **签名**（`:1262`）：`signature = HMAC_SHA256(secret_key, app_id + timestamp)`，连同 `app_id/timestamp/可选 pid/app_user` 放进请求头。
- **请求体**（`:385`）：`inputs / agent_id / query / response_mode=streaming / conversation_id`，外加一堆可选透传字段（command、chat_model、files…）。
- **SSE 解析** `_iter_stream_data`（`:758`）：兼容 `data:` 前缀行、`[DONE]` 结束符、以及裸 JSON 行；流提前断开且一个字都没吐就抛 `StreamEmptyError`。
- **🔑 累计→增量归一化** `_normalize_stream_answer_delta`（`:807`）：**这是最容易踩坑、也最巧妙的一处**。不同滴普工作流的 `answer` 字段，有的是"本次增量"，有的是"到目前为止的全文快照"。它通过判断"新值是否以已累计文本开头"来区分——是快照就只取新增的尾巴，避免气泡里文字成倍重复拼接。
- **重试** `stream_chat`（`:239`）：对 `408/429/5xx` 和连接/超时错误，在"尚未吐出任何数据"的前提下最多重试 3 次（吐过数据就不能重试，否则重复）。
- **输入文件三策略**（`:439`）：`auto`（先传滴普文件接口，可抽取文本时同步把文本追加进 query，失败退回纯文本）/ `upload`（必须传成功）/ `text` / `inline`(base64)。本地能抽 PDF/TXT/MD/CSV/JSON 文本。
- **下行文件/图片下载**：`_validate_download_url` + `_host_is_allowed` + `_read_limited_response` 三件套做 **SSRF 防护、域名白名单、大小上限**；图片走单独的扩展名识别链（URL → Content-Type → mimetypes → 默认 jpg，`:999`）。
- **线程安全**：每个工作线程持有独立 `requests.Session`（`threading.local`，`:171`），避免跨线程共享 Session 出问题——这正好配合上面那个线程池模型。

# 几个"不显眼但关键"的设计

1. **stream.id 是流式 UX 的核心**——所有增量复用同一个 id，企微才会"原地把气泡越改越长"，而不是刷一串新气泡。
2. **req_id ⇄ Future** 把乱序全双工帧流伪装成同步调用，是 WS 层能写得这么干净的原因。
3. **节流 + 6000 退避**是流式稳定性的两道保险，缺一个都会在真实群聊里翻车。
4. **三处 SSRF 防护**（下行解密、下行文件、下行图片）说明作者把"URL 来自外部、不可信"这件事贯彻到底。
5. **会话 key = chatid+userid+route**：同一个群不同人、同一个人不同 agent 路由，上下文互相隔离，不串台。

# 真正难的就这几处（有代码为证）

**1. 企微私有协议对接 —— 全项目最贵的部分**

这是"无 SDK + 半文档化"的对接，很多东西**文档里查不到，只能连真网关试出来**：

- 分片上传三段式（init→chunk→finish，512KB/片、带 md5，`wecom_ws.py:354`）
- 下行媒体的 AES-CBC 解密（`wecom_ws.py:479-518`）——`aeskey` 可能是 hex / base64 / urlsafe-base64，还得归一到 32 字节，PKCS7 要试 128 和 256 两种块大小。这段代码长这样，本身就是"被真实网关教育过"的证据：纯看文档绝写不出这种多候选兜底。
- `errcode=6000`（数据版本冲突）只对这一个码退避重试（`wecom_ws.py:29`）——这种码**只有在生产里高频刷新气泡时才会撞到**，撞到之前你根本不知道要处理它。

这一块如果你**没做过企微长连接**，光"读文档 + 连网关 + 撞各种没写的行为"就能吃掉一两周。

**2. 两种 I/O 模型的缝合 —— 唯一的"高级并发"**

`asyncio` 事件循环里不能跑阻塞的 `requests` 流式，于是用线程池 + `call_soon_threadsafe` + `asyncio.Queue` 把增量喂回事件循环（`message_handler.py:825-844`）。这类代码的 bug 全是隐蔽型的：丢事件、卡死事件循环、共享状态竞态。要写对，得真懂 asyncio 的 future / executor / 线程安全边界——**这是中级偏不过去、得靠资深经验的坎**。

**3. 流式 UX 的"暗坑"**

- `_normalize_stream_answer_delta`（`agent_chat.py:807`）：滴普不同工作流，`answer` 有的是增量、有的是全文快照。这个函数判断"新值是不是以已累计文本开头"来区分。**它看起来平平无奇，但你只有在气泡里看到文字成倍重复之后，才会意识到需要它。**
- 节流（0.8s / 80 字才刷一次）+ dedup——不做，真实群聊里要么把企微打爆，要么频繁触发 6000 冲突。

这三处的共同点：**都是"生产里流过血才知道要写"的需求**，不是设计阶段能预见的。

# 但它又没那么难——别被 6000 行吓到

- **单进程、单职责**：没有分布式、没有数据库、没有水平扩展、没有一致性问题。状态全在内存（会话缓存、去重）。
- **业务逻辑极薄**：本质是"管道"，AI 智能完全外包给滴普，这个项目只是个**客户端**，不是大脑。
- **库都成熟**：`requests` / `websockets` / `cryptography` 现成，不用自己造轮子。
- **边界清晰**：整个系统就是"两个 API + 中间的胶水"，表面积小。

换句话说：**难在"集成一个你不掌控、又没好好写文档的外部系统"和"处理海量真实失败模式"，不难在任何一道算法或架构题。**

# 真实短板（带严重度 + 改法）

**🟡 中｜测试不 hermetic（已证实，最该修）**
`_set_required_env`（`test_config.py:4`）清了一批可选变量，但**漏了** `DEEPEXI_APP_USER`、`DEEPEXI_PID`、`DEEPEXI_COMMAND`、`DEEPEXI_CHAT_MODEL`、`DEEPEXI_ENABLE_DEEPRESEARCH`、`DEEPEXI_FILES_JSON`、`DEEPEXI_COMMAND_PARAM_JSON`、`DEEPEXI_APP_USER_PREFIX`。你机器上 shell export 了真实 `.env`，于是 `DEEPEXI_APP_USER` 漏进来，`assert agent_app_user == ""` 就炸了。CI 干净环境不会暴露，但**本地开发体验很差、且掩盖了"测试该自己隔离环境"的原则**。

`conversation_id` 缓存和去重都在内存（`conversation_store.py`、`Deduplicator`）。后果：①重启丢全部多轮上下文 + 去重记录（重启瞬间可能重复回答）；②**无法多副本横向扩展**（状态不共享，去重和续话都会乱）。对单实例机器人没问题、文档也诚实标注了边界，但这是真实的天花板。
→ 若要扩展：会话/去重挪到 Redis。

**🟢 低｜配置面偏大**
`Settings` ~35 个字段、`config.py` 713 行多为 parsing 样板。`command`/`command_param`/`enable_deepresearch` 你自己在 PROJECT_CONTEXT 第 10 节也标了"不确定是否真需要、当前只透传"——有轻微 YAGNI。能用，但维护面大。

**🟢 低｜healthcheck 偏弱**
`healthcheck.py` 只在 `/proc` 里找进程 cmdline 标记——是**存活检查不是就绪检查**。进程活着但 WS 卡在重连/未鉴权时仍报 healthy。

**🟢 低｜其他**
自实现的 `load_dotenv`（`config.py:13`）不处理带空格的引号值/多行；依赖版本锁定不统一（部分 `==` 部分 `>=`），影响可复现性。都属吹毛求疵级别。
