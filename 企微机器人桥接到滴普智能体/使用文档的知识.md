# 使用文档的知识

## 为什么开放平台用 APP_ID + SECRET_KEY 签名，而不是像 Dify 那样用 API Key？

两种方案本质上是不同的安全模型：

### Dify 方案：API Key

```
headers: { "Authorization": "Bearer sk-xxx" }
```

- 一个密钥直接当"通行证"
- 简单直接，开发体验好
- 风险：密钥在每次请求中明文传输，被截获即可重放

### 开放平台方案：APP_ID + SECRET_KEY + 签名

```
headers: { "app_id": "xxx", "signature": hmac(app_id + timestamp), "timestamp": "..." }
```

- SECRET_KEY 永远不上网，只在本地参与签名计算
- 每次请求的 signature 都不同（因为 timestamp 变化）
- 有 5 分钟时效窗口，截获了也很快过期

### 对比

| 对比点 | API Key | 签名模式 |
|--------|---------|----------|
| 接入难度 | 极简 | 需写签名逻辑 |
| 密钥暴露风险 | 每次请求都传输密钥 | 密钥不传输 |
| 防重放攻击 | 无（截获即可复用） | 有（时间戳过期） |
| 适用场景 | 面向开发者的 SaaS | 企业级 / B2B 集成 |

### 核心原因

这套系统面向的是企业级/B2B场景。像微信支付、阿里云、AWS 等企业级 API 都采用类似的签名机制（AccessKey + SecretKey），因为：

1. **安全要求更高** — 企业集成往往涉及敏感数据，密钥不能在网络上明文传输
2. **防重放** — 即使请求被中间人截获，过了时效窗口也无法复用
3. **审计溯源** — APP_ID 明确标识调用方身份，便于限流和审计

而 Dify 选择 API Key 是因为它面向开发者个人使用，优先考虑接入体验，降低上手门槛。

**总结：安全性优先选签名，易用性优先选 API Key。**
