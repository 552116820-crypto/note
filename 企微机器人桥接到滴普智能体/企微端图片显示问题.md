# 修复企微端 `<img>` 标签裸文本问题

## Context

截图显示企微气泡里出现裸 `<img src="http://know` 文本，没有渲染图片：

- 滴普 Agent 的 thinking 明确说"最终答案：渲染该图片"，所以 agent 是有意输出 HTML `<img>` 标签来表达图片的
    
- 当前主链路只走 `aibot_respond_msg` 的 `msgtype=stream`，这是纯文本气泡，不会把 HTML/Markdown 图片语法渲染成图
    
- 截图里看起来"被截断"是因为后端真正输出的 URL 较长，气泡里的纯文本超出可见范围，但根本问题不是截断而是没有走原生图片消息
    

需要让 agent 返回里的图片实际能在企微看到，并且不要让 `<img>` 标签污染文字气泡。

## Approach

四步走，全部在服务端处理，不依赖 agent 改 prompt：

1. **流式期间用占位符替换图片标签** 每次 `flush_stream_reply` 把累计的 `answer` 推回 WeCom 之前，先用正则把已经完整出现的图片引用替换成 `[图片N]`，同时把末尾"还没收完整"的图片标签（dangling `<img src="...` 或 `![alt](url`）从可见文本里截掉。气泡里全程只看到 `[图片1] [图片2]` 这种可读占位，不会泄露 `<img` 半截标签。
    
2. **finish=true 之后串行下载并投递图片** 流式结束后，对完整 `answer` 再扫描一次，得到去重后的图片 URL 列表。逐个走「下载 → 校验大小/扩展名/host → `aibot_upload_media_init`(type=image) 分片上传 → `aibot_respond_msg`(msgtype=image)」流程，让每张图片作为独立气泡发回会话。
    
3. **复用现有 URL 安全校验** 下载图片复用 `AgentChatClient._validate_download_url` / `_host_is_allowed` / `_is_disallowed_ip_literal` / `_read_limited_response`，沿用路由级 `file_download_allowed_hosts` 和 `file_download_max_bytes`，不引入新配置。
    
4. **失败兜底，不阻塞主回复** 单张图片下载/上传失败时，文本气泡照常发送；失败的图片用一条短文本 `图片 N 加载失败：{url}` 跟在后面追加，方便用户兜底自查。整体响应不因图片失败而中断。
    

正则覆盖两类图片语法：

- HTML：`<img\b[^>]*\bsrc\s*=\s*['"]([^'"]+)['"][^>]*/?>`，大小写不敏感
    
- Markdown：`!\[([^\]]*)\]\(\s*(<[^>]+>|[^\s)]+)(?:\s+"[^"]*")?\s*\)`
    

扩展名集合 `{jpg, jpeg, png, gif, webp, bmp}`；同 URL 多次出现共用同一编号、只下载/上传一次。

## Changes

### `wecom_deepexi_aibot/wecom_ws.py`

- `upload_file(..., *, filename, content)` 增加可选参数 `media_type: str = "file"`，把 `aibot_upload_media_init` 体里的 `"type": "file"` 改成传入参数。`media_type` 限制在 `{"file", "image"}`。
    
- 新增 `reply_image(req_id, *, filename, content)`：调 `upload_file(... media_type="image")` 拿 `media_id`，再发 `aibot_respond_msg` 体 `{"msgtype": "image", "image": {"media_id": media_id}}`，结构照搬 `reply_file`（`wecom_ws.py:325-339`）。
    
- `WeComApiError` / 重试逻辑不变，已有的 `_request_with_retry` 会自动覆盖。
    

### `wecom_deepexi_aibot/agent_chat.py`

- 新增模块常量 `SUPPORTED_REPLY_IMAGE_EXTENSIONS = {"jpg","jpeg","png","gif","webp","bmp"}` 和对应的 `IMAGE_CONTENT_TYPE_EXTENSION_MAP`。
    
- 新增公开方法 `AgentChatClient.download_image_by_url(url) -> AgentChatFile`：
    
    - 复用 `_validate_download_url`（http/https + 拒本地 IP + allowed_hosts）
        
    - 复用 `_read_limited_response`（按 `file_download_max_bytes` 截断）
        
    - 扩展名顺序：URL path → `Content-Type` map → `mimetypes.guess_extension` → 默认 `jpg`
        
    - 文件名用 URL path 抽取的尾部，没有就用 `image-{hash前缀}.{ext}`
        
- 不动现有 `_build_file_from_url`，避免影响 PDF/DOCX 文件流。
    

### `wecom_deepexi_aibot/message_handler.py`

- 新增模块常量：图片正则 `IMAGE_HTML_RE`、`IMAGE_MARKDOWN_RE`，以及"未闭合图片标签"探测器 `DANGLING_IMAGE_TAIL_RE`（用来把流式末尾的半截 `<img` 或 `![...](...` 截掉）。
    
- 新增数据结构 `@dataclass(frozen=True) class ImageRef { index: int; url: str; alt: str }`。
    
- 新增工具函数：
    
    - `extract_image_refs(text) -> tuple[str, list[ImageRef]]`：返回 `(占位符替换后的可见文本, 去重后的图片列表)`。同 URL 共享同一 `index`。
        
    - `safe_stream_display(text) -> str`：先调 `extract_image_refs`，再把末尾未闭合的图片片段切掉。流式 flush 专用。
        
- `_stream_agent_reply`:
    
    - `flush_stream_reply` 中，把 `current_answer` 先丢给 `safe_stream_display`，再交给 `fit_stream_content`；`last_sent_content` 改成存"已发送的清洗后文本"，比较时也用清洗后的版本，避免 dedup 漏发或重发。
        
    - finish 那次发送也走 `safe_stream_display`（finish 时不会有 dangling，所以等价于 `extract_image_refs[0]`），并把 `extract_image_refs` 的图片列表返回给 `handle_message`。
        
- `handle_message`:
    
    - 在文字气泡 finish=true 之后、`reply_files` 投递之前，新增 `await self._dispatch_reply_images(req_id=req_id, agent_client=agent_client, image_refs=image_refs)`。
        
    - `_dispatch_reply_images` 内部逐张：调 `agent_client.download_image_by_url(url)` → `ws_client.reply_image(req_id, filename=..., content=...)`；任意一步抛异常就 `logger.exception` + 收集失败文案 `图片 N 加载失败：{url}`，最后用一条 `reply_text` 汇总；不重抛，不阻断 `reply_files` 流程。
        

### Tests

- `tests/test_message_handler.py`：
    
    - 单测 `extract_image_refs`：单个 HTML、单个 Markdown、混排、同 URL 去重共享编号。
        
    - 单测 `safe_stream_display`：完整标签 → `[图片N]`、末尾 `<img src="ht` → 整段被切掉、末尾 `![alt](http://` → 整段被切掉。
        
    - 集成测：用 `FakeWsClient` 跑 `_stream_agent_reply` + `handle_message`，断言：
        
        - 所有发出去的 `reply_stream` 内容里都没有 `<img` 或 `![`
            
        - finish=true 后调用 `reply_image` 次数 = 唯一图片 URL 数
            
        - 下载失败时调用 `reply_text` 发出兜底文案
            
- `tests/test_wecom_ws.py`：mock `_request`，断言 `reply_image` 链路发出的 frame 为 `aibot_upload_media_init` 带 `type=image`、`aibot_respond_msg` 带 `msgtype=image` 和 `image.media_id`。
    
- `tests/test_agent_chat.py`：单测 `download_image_by_url` 在 host 不在 allowed_hosts、本地 IP、超出大小上限时分别抛 `AgentChatError`；成功路径返回正确扩展名（content-type=image/png → .png）。
