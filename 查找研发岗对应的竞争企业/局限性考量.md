**Assistant:**

回顾这次项目踩到的坑，可以提炼出 7 条"开局就该想到"的事情：

## 1. 提前画好"结构化字段 vs 人类 prose"的边界

这次最大的坑是 `product_text` 同时给人看（叙述性，含"无 X 业务""非 X 厂""应用于 Y"）和给 pipeline 看（字面子串匹配）。两个目的对 prose 的要求是冲突的。

下次：**verifier 输出的 JSON 里 prose 字段（给 HR 看的）不参与评分**。打分只看 `main_categories[]`、`related_categories[]`、`negative_categories[]`、`company_type` 这些枚举字段。prose 是 audit trail，不是 truth source。

## 2. 中文做子串匹配前先列"歧义清单"

`锡膏` ⊂ `锡膏检查机`、`助焊剂` ⊂ `助焊剂清洗剂`、`硅脂` ⊂ `导热硅脂`、`焊膏` ⊂ `烧结银焊膏`、`无` ⊂ `无铅`/`无尘`/`无关`、`非` ⊂ `非晶`。这些不是"以后再说"的边角，是核心评分准确性的来源。

下次：**categories.yaml 定稿前过一遍"这些词作为子串出现在 reject 类目里多常见"**。每一条 main_category 列 3-5 个"看上去像但其实不是"的对照词，写进 negative_keywords。在 assign 跑通之前先用 50 条人工标注样本验证一遍。

## 3. company_type 这种"硬标签"放在评分循环之前

我们的 `company_type ∈ {设备厂, 贸易商, 服务商, 矿山客户}` 是个 strong prior，但 assign.py 只在 score=0 时拿去归类。结果"锡膏检查机"这种字面命中分数 1.0 的设备厂跑去了 matched。

下次：**写 assign 第一行就先看硬标签**。决策顺序应该是 `force_unmatched > company_type 黑名单 > URL 硬规则 > 评分`，不是反过来。

## 4. 否定语义不要交给 regex

`无 X 业务` / `代理 X` / `非 X 厂` / `主营并非 X` / `X 应用于 Y` / `不生产 X` / `兼营 X 销售` / `非 X 制造商` — 我们花了两轮把这些 idiom 一个个塞进 neg_pattern，但 `X 不生产 Y` 当 X、Y 都在窗口内的语义级 case，regex 怎么调都解决不了（WEVO 例）。

下次：**与其在 regex 里堆 alias，不如让 verifier 写 `excluded_for_position: [P07]` 这种显式字段**。"我看了官网，结论是不命中 P07"比"我写一段 prose 让 pipeline 猜结论"靠谱得多。verifier prompt 应该明确产出布尔决策，而不是描述文本。

## 5. Day-1 就建 audit-diff 工具

我这次是为了对比 bug-fix 前后才临时写的 `/tmp/audit_diff.py`。如果一开始就有这工具，每次跑完 assign 就 print "+N -M 岗位变化 K"，bug 和 regression 当天就能发现，不会等到一个月后用户复核才暴露。

下次：**`pipeline/assign.py` 自带 `--diff <previous>` 模式**，输出新增/失配/岗位变化清单。每次重跑前自动 backup 旧 assignments.jsonl。

## 6. 守住"deterministic pipeline → xlsx"的链路

最坑的是发现：原 P07 result.xlsx 的 22 家公司**不能从当前 verified.jsonl 跑 assign 复现**。中间有手动 patch 但没记入 decisions/。结果我每次重跑都跟"原版"对不上，要逆推哪里 diverge 了。

下次：**xlsx 必须是 `assign → build_xlsx` 的纯函数输出**。所有人审决策都进 `decisions/*.tsv`，不允许直接编辑 xlsx 或 assignments.jsonl。每次 build 之前自动 `git status` 检查 verified/assignments 干净；不干净就拒绝 build。

## 7. 命中阈值 0.6 是个 magic number，要有 sanity check

`SCORE_THRESHOLD = 0.6` + 各项权重（1.0/0.6/0.3/0.2/-0.5）是拍脑袋的。这次出现的"分数刚好 0.3" 失配（卡夫特、翰华锡业一度）就是阈值附近的灰带。

下次：**预先准备 ~30 条人工 ground truth**（10 强匹配 + 10 弱匹配 + 10 边缘 unmatched），跑出来看 precision/recall。阈值 + 权重根据这组样本调，不是凭感觉。

---

**核心一句话总结**：这种调研项目的关键风险是**"verifier 写的人类语义"和"pipeline 做的字面匹配"之间的断层**。从 day 1 把它们隔离开（verifier 出结构化决策、pipeline 不读 prose），后面 80% 的局限都不会出现。
