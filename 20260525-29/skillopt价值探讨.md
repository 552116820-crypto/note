## 四、最终结论

1. **方法论：** SkillOpt 不是范式革命，是 prompt 优化路线（TextGrad/GEPA/DSPy）里**工程上最稳的一种**，核心贡献是三个约束——小步编辑、拒绝缓存、验证准入。"训练神经网络"是教学比喻，不要当真。

2. **结果：** 52/52 + 平均 +23.5 在论文配置下成立，但**基线弱 + 任务可自动评分**这两个前提把适用范围限得很死。SpreadsheetBench 这种"有标准答案、有重复模式"的任务上确实能涨，对应到 CLAUDE.md 类开放指令上，提升不会复制。

3. **个人开发者怎么用：**
   - **划算的：** 对**高重复、有验收标准**的任务（例如固定格式的 Excel 报表、固定接口的代码生成），手动跑一遍 *失败案例 → 单条规则补充 → 回归集验证* 的循环，比凭直觉写 Skill 强。slow/meta update 的精神可以借——每隔几周回看一次 Skill，做一次"纵向编辑"，比每天微调更稳。
   - **不划算的：** 给 CLAUDE.md / Codex Agents.md 这种开放性指令上跑全自动 SkillOpt。准备 ground truth 数据集的成本会吞掉所有 ROI。
   - **直接用工具：** 除非你已经有现成 benchmark（用得上他们 repo 里的 6 个 config），否则 fork 下来跑没意义；先抄走方法论。

4. **一句话：** **"别再手写 Skill 了"这个标题是钓鱼。真正的信号是"别只手写一次就放着"——Skill 应该按失败样例迭代，但是否值得上自动训练，看你的任务有没有可自动评分的 ground truth。**

## 真实能用的场景

**1. 结构化抽取流水线**
- 发票 / 合同 / 简历 → 固定字段 JSON
- 财报 PDF → 表格结构化
- 验收：字段级 exact match。你只要有 200 份"已人工标注的发票"，就能跑出一份比手写 prompt 强很多的抽取 Skill。

**2. SQL / 代码方言转换**
- Postgres → BigQuery / Snowflake SQL
- Python → Go 端口移植（有现成测试套）
- pandas → Polars 改写
- 验收：跑测试用例。这是 SkillOpt 最甜的场景之一，因为失败模式高度收敛（类型转换、保留字、语法糖）。

**3. 固定输出格式的代码生成**
- 给一个 OpenAPI schema 自动生成 client SDK
- 给一个 protobuf 生成对应的 TypeScript 类型 + zod validator
- 给一个 React 组件生成 Storybook story
- 验收：编译过 + lint 过 + 渲染快照对得上。

**4. 单元测试自动生成**
- 给函数生成单测，验收用 mutation testing（看新测试能不能杀掉变异体）
- 这个场景非常适合，因为有客观分数（mutation score）。

**5. Text-to-SQL / Text-to-工具调用**
- 自然语言 → SQL 查询（对照 oracle SQL 或对照执行结果）
- 自然语言 → 工具调用序列（有 oracle trace）
- 验收：execution match。

**6. 表格 / Excel 自动化**
- 月度报表自动生成（输入是原始 csv，输出是固定模板的 Excel）
- 验收：单元格值对比。论文的 SpreadsheetBench 就是这类。

**7. RAG 答案精度优化**
- 内部知识库 QA，有"正确答案"集合
- 验收：F1 / exact match。SearchQA 类。

## 看起来能用但实际不行的反例

- **"写 PR review 的 Skill"** — 输出是自然语言评论，没有自动评分器。

- **"重构遗留代码的 Skill"** — 测试是约束不是目标，重构好坏没有 ground truth。
- **"写文档的 Skill"** — 风格类任务，自动评分无效。
- **"调试线上 bug 的 Skill"** — 每个 bug 不一样，没有重复模式。

这些场景就该按上一轮说的——**借思想（失败样例 → 小步改 → 拒绝缓存 → 回归集）但不要套工具**。

## 为什么 100% 不现实

图片型 PDF 抽取的天花板由**输入信息质量**决定，模型再强也突破不了。常见的物理限制：

- **印章/水印盖住关键字段**：金额上盖了红章，人都得猜，模型只能瞎蒙
- **手写批注 / 修改**：原打印的"1000"被划掉手写改成"1200"，到底用哪个是业务规则不是 OCR 能定的
- **低分辨率 / 折痕 / 倾斜**：传真件、复印多次的合同
- **相似字符歧义**：`0` vs `O`、`1` vs `l`、`5` vs `S`，特别是发票号/合同编号这种没有语义可校验的字段
- **表格跨页 / 跨栏断裂**：金额在第一页表头、明细在第二页，匹配关系靠版面理解
- **多语言 / 异体字 / 简繁混排**：财报里中英文交替很常见

## 当下（2026）现实的准确率分布

把字段分类看更清楚：

| 字段类型 | 干净扫描件 | 一般质量 | 差质量 |
|----------|-----------|---------|--------|
| 印刷规范字段（日期/编号/标题） | 99%+ | 95-98% | 85-92% |
| 金额 / 数字 | 98-99% | 93-97% | 80-90% |
| 地址 / 自由文本 | 95-98% | 88-94% | 70-85% |
| 表格明细行 | 95-99% | 85-95% | 60-85% |
| 手写内容 | 80-90% | 60-80% | 40-60% |
| 被遮挡字段 | — | — | 接近随机 |

顶级 VLM（GPT-5.5 / Claude Opus 4.7 多模态 / Gemini 2.x）+ 专用 OCR（PaddleOCR / Azure Document Intelligence / Mathpix）的组合，在**整页字段级准确率**上一般落在 92-97% 区间，**单据级 100% 正确**（所有字段都对）通常只有 70-85%。

## SkillOpt 在这场景里能做什么 / 不能做什么

**能做：**
- 把模型反复犯错的模式提炼成规则（"金额字段如果同时出现￥和元，以￥后数字为准"、"印章遮挡时优先用大写金额校验小写"）
- 让模型先做版面理解再抽字段，而不是平铺式扫
- 教模型用业务规则交叉验证（小计=单价×数量、税额=不含税×税率）

**不能做：**
- 把信息从被盖住的像素里变出来
- 把手写和印刷的歧义自动消解
- 替代人工审核

## 真要逼近"100%"的工程方案

行业里实际做法不是追求模型 100%，而是**让 100% 的字段都被审核**，分三层：

**1. 置信度路由（最重要）**
- 模型为每个字段输出 confidence
- 高置信度（>0.95）直通
- 中置信度（0.7-0.95）走二次校验
- 低置信度直接转人审

实际效果：高置信度部分能做到 99.5%+ 准确，但通过率（不需要人审的占比）通常只有 60-80%。

**2. 业务规则交叉校验**
- 金额加总等式
- 日期合理性（开票日不能晚于今日）
- 编号格式正则
- 同一字段在不同位置出现要一致（发票上"合计"和"价税合计"）

校验失败的全部转人审。这一步能再吃掉一些假阳性。

**3. 双模型 / 多模态投票**
- VLM 直接看图 + OCR 文本喂给 LLM，两条路独立抽取
- 不一致的字段转人审

加上这三层，**字段级"模型+规则放行后"的准确率可以做到 99.5-99.9%**，剩下 0.1-0.5% 要么走人审、要么用户事后改。

## 给你的现实建议

- **如果产品对外承诺"100% 准确"** → 别承诺。承诺"100% 人工兜底"或"95% 直通 + 5% 人审"。
- **如果是企业内部工具** → 99% 字段级 + 人审通道，已经能省 80% 人力，这是甜点区。
- **如果是合同/财务等高合规场景** → 抽取永远只是"建议值"，最终签字必须是人。模型负责把人工从"录入"降级为"复核"。
- **SkillOpt 在这里的角色** → 持续吃错例、迭代规则。把人审标注的"模型抽错的样本"喂回去训 Skill，准确率会以可量化的方式爬升。这才是它真正的甜点 —— **持续优化已经够好但还不够完美的抽取流水线。**

---

**一句话：** 图片型 PDF 抽取的工程目标不是"模型 100% 准确"，是"100% 的不确定性都被显式处理"。SkillOpt 帮你压缩前者，但替代不了后者。
