### Turn 4  *(2026-05-06 00:43)*

**User:**

**Assistant:**

我查到的更可能是 **Agent Reach**，不是 “Agency Reach”。

它是一个给 AI Agent 用的 **联网/检索 Skill + CLI 脚手架**。核心作用是让 Claude Code、OpenClaw、Cursor、Windsurf 这类 coding agent 能通过命令行去读和搜互联网内容，比如网页、Twitter/X、Reddit、YouTube 字幕、GitHub、B站、小红书、微信公众号、微博、RSS 等。

它不是一个大模型，也不是浏览器，而是把一堆上游工具装好并写进 `SKILL.md`，让 Agent 知道该用什么命令。例如：

- 读网页：通过 Jina Reader
- 搜 GitHub：`gh search repos`
- 读 GitHub 仓库：`gh repo view`
- 提取 YouTube 字幕：`yt-dlp`
- Twitter/X、Reddit、小红书等：按平台配置 cookie、token 或代理后使用
- 诊断状态：`agent-reach doctor`

比较实用的场景：

- “帮我总结这个 YouTube 视频”
- “去 Reddit/Twitter 搜这个产品的评价”
- “看看这个 GitHub repo 是干什么的”
- “搜一下小红书/微博上的口碑”
- “监控 RSS 或网页更新”
- “做跨平台资料收集和市场研究”

需要注意的是，它会让 Agent 执行 shell 命令、安装依赖，并可能保存平台 cookie/token 到本地配置。官方 README 也提醒：Twitter、小红书这类 cookie 登录平台有封号和凭据风险，建议用专用小号，不要用主账号。安装前最好先用 safe/dry-run 模式看它会做什么。

### Turn 5  *(2026-05-06 00:45)*

**User:**

safe/dry-run 模式 这是什么

**Assistant:**

`safe` 和 `dry-run` 是安装/运维工具里常见的两个保护模式：

`dry-run`：只预演，不真正执行  
它会告诉你“如果真的运行，我会安装哪些包、改哪些文件、写哪些配置”，但不会实际改系统。适合先审查风险。

```bash
agent-reach install --dry-run
```

`safe`：安全安装模式  
它通常不会自动安装系统级依赖或大幅修改环境，只列出需要你手动确认/安装的东西。适合生产机、公司电脑、多人共用机器，或者你不想让 Agent 自己乱装东西。

```bash
agent-reach install --safe
```

也可以组合：

```bash
agent-reach install --safe --dry-run
```

对 Agent Reach 来说，这两个模式主要是为了避免 Agent 自动执行 `pip install`、装 Node/gh/mcporter/twitter-cli 等依赖、修改 skill 目录或 MCP 配置时超出你的预期。先 `dry-run` 看清楚，再决定是否真正安装，会更稳妥。

**推荐架构**

1. **输入表**
   你提供一个 Excel/CSV：

   ```text
   company_name, website, notes
   ABC Corp, https://abc.com,
   XYZ Inc, https://xyz.com,
   ```

2. **站点发现**
   对每家公司网站做 crawl：

   - 优先读 `sitemap.xml`
   - 解析首页导航、产品页、解决方案页
   - 控制范围：只爬同域名、最大深度、最大页面数
   - 对 JS 很重的网站，用 Playwright/Chrome 自动化
   - 对普通页面，用 requests/Jina Reader/Agent Reach 这类工具即可

3. **产品信息抽取**
   从网页里抽这些结构化字段：

   ```json
   {
     "company": "ABC Corp",
     "source_url": "https://abc.com/products/...",
     "raw_category_path": ["Products", "Sensors", "Temperature Sensors"],
     "product_name": "Wireless Temperature Sensor",
     "keywords": ["temperature sensor", "wireless", "industrial IoT"],
     "description": "...",
     "confidence": 0.86
   }
   ```

4. **单家公司分类整理**
   对同一家公司内部做去重和归并：

   - `Temperature Sensors`
   - `Temp Sensor`
   - `Industrial Temperature Sensor`

   这些可能要合并成一个标准类目。

5. **全局产品分类架构**
   把所有公司的分类放在一起，让 LLM/embedding 做归一：

   ```text
   Level 1: Sensors
   Level 2: Temperature Sensors
   Level 3: Wireless Temperature Sensors
   ```

   每个类目保留：

   - 标准名称
   - 别名
   - 上级类目
   - 证据 URL
   - 来自哪些公司

6. **生成 Excel**
   最终 Excel 不建议只有一个 sheet，最好有 4 个：

   - `Category Matrix`：最终矩阵，行是产品分类，列是公司，有则打勾
   - `Taxonomy`：完整产品分类树
   - `Evidence`：每个勾选项对应的来源 URL 和原始文本
   - `Raw Extracts`：原始抽取结果，方便排查

   矩阵大概这样：

   | Level 1 | Level 2 | Level 3 | ABC Corp | XYZ Inc | DEF Ltd |
   |---|---|---|---|---|---|
   | Sensors | Temperature Sensors | Wireless Temperature Sensors | ✓ |  | ✓ |
   | Sensors | Pressure Sensors | Digital Pressure Sensors |  | ✓ | ✓ |

**技术路线建议**

Agent Reach 可以用，但我不会把它当成唯一核心。它更适合“读网页、搜内容、拉取公开信息”。如果网站有复杂菜单、懒加载、产品筛选器，还是需要 **Playwright / Chrome DevTools 自动化**。

```text
      global_taxonomy.yaml
    output/
      product_category_matrix.xlsx
```

每跑完一家公司就 checkpoint。失败后从 `status.json` 继续。
