❯ agentic-harness-patterns-zh算得上好的skill吗

● 按 skill 该有的标准给它过一遍：

  形式上确实是合格的：
  - 元数据规范，when_to_use 把触发词放在前面（懒加载发现成本低）
  - 主 SKILL.md 当索引，9 个 pattern 放 references/ 按需展开——这是 skill 系统设计意图里"渐进披露"的标准做法
  - 篇幅克制，每章都是固定五段式（痛点 / 黄金法则 / 工作原理 / 从哪开始 / 权衡）
  - 双向交叉引用，章节末尾指向 references/，references/ 也能反过来读

  内容上有真东西：
  - "fail-closed 默认 + 按调用粒度的并发分类"、"两步保存不变式（先主题文件再索引）"、"协调者必须自己综合，不能委派理解"—
  —这些是非显然的具体主张，不是"要做好上下文管理"这种废话
  - 权衡是真权衡（"激进缓存降低延迟但有过时风险"、"fail-closed
  让新工具开箱即安全，但开发者忘记标记并发安全会静默降低吞吐量"），不是单边推销

  但要说算不算"好 skill"，得分清类型：

  - 操作型 skill（改变 agent 行为）：simplify、loop、schedule 这类，触发时 agent 会按它的指令干活
  - 参考型 skill（改变 agent 知道什么）：这个属于这类，触发时给 agent 灌一份术语表 + 设计 checklist

  两种都合法，但要小心：它不会让 Claude 替你做出更好的 harness 设计，因为它没说"当用户问 X 时你应当
  Y"。它实际上是把一份高质量的中文 agent 架构手册包装成 skill 形式，让相关话题触发时它能进上下文，由
  Claude（或你自己）拿来参考。

  两个隐性弱点：

  1. 时效性脆弱。基于 Claude Code 当下实现归纳的模式，runtime 迭代后某些细节会过时（比如"上下文窗口
  1%"这种数字、/remember 这种命令）。抽象层模式（fail-closed、两步保存）会比较抗老化，具体数字和命令名不一定
  2. 作者不可验证。frontmatter 没写 author，只有 license: MIT。内容质量高不代表是官方出品，可能是某个深入研究过 Claude
  Code 的人写的逆向观察。引用时心里有数

  结论：在我见过的 skill 里属于上半区——比 grill-me / skill-vetter
  这种通用脚手架高一个层次，但它的形态本质是"中文参考资料 + skill
  触发外壳"，而不是"自动化操作流"。判定它好不好，取决于你是否承认"把好的领域参考文档做成按需触发的 skill"是 skill
  系统的合法用法——我个人倾向于承认。
