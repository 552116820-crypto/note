## 核心矛盾：模糊匹配的"歧义"问题

销售输入"安美智"，公海里很可能存在多家：

**接口要怎么决定返回哪一家？**

无论后端用什么策略，都只有三种选择：

| 后端策略 | 后果 |
|---|---|
| 按 ES 相似度返回 top-1 | 销售看到"安美智控电子"的详情，以为就是他想找的"安美智科技"，**拿着错误信息打电话/写方案**——CRM 业务里这是最严重的事故 |
| 返回多条详情 | 一次返回 4 家公司的完整信息（每家几十个字段），LLM context 直接爆炸；销售也要肉眼挑 |
| 匹配多条时报错 | 销售看到"匹配不唯一"，但**根本不知道有哪些候选**，下一步无从输入 |

这就是为什么所有"在大池子里找特定对象"的产品（百度、京东、企查查、CRM）都用"列表 → 详情"两步式——**不是技术做不到一步，是产品上必须留一个消歧的窗口给用户**。

## 为什么"潜在客户"能一步到位但公海不行

| 维度 | 潜在客户 | 公海客户 |
|---|---|---|
| 池子大小 | 一个销售名下几十~几百 | 全公司几千~几十万 |
| 同前缀重名概率 | 极低（销售自己挑的客户，会避开重名） | 极高（公海什么都有） |
| 用错的代价 | 较低（自己客户，对话中能纠错） | 高（可能联系错公司，影响商务） |
| 召回保障 | 知识库 rerank + Result Validate 二次校验 + LLM context 看到全文还能再判一次 | 直接调接口拿到一条详情，**没有复核机会** |

注意现有的潜在客户路径其实**不是真的"一步到位"**——它有三道兜底：知识检索 top-3、Result Validate 算 confidence、LLM 看到全部 3 条 context 后还能选/拒答。公海如果只返回一条详情，这三道兜底全没了。

## 折衷方案：一个接口、三种返回

如果你嫌"两个接口"麻烦，可以合并成**一个接口、按匹配数返回不同结构**，保留消歧能力：

```http
POST https://crm.amer.cn/v2/api/OutPut/getSeaCustomerByName
{
  "passport": "...",
  "name": "安美智"
}
```

**返回**根据匹配情况分三种：

```json
// 情况 1：高置信单条命中（名称完全相等 或 信用代码命中 或 唯一前缀匹配）
{
  "code": 0,
  "match_type": "exact",
  "data": {
    "C0000123": { "customer_qizhidao_info": {...}, "customer_base": {...}, ... }
  }
}

// 情况 2：模糊匹配多条 → 返回候选列表（不含详情，轻量）
{
  "code": 0,
  "match_type": "ambiguous",
  "candidates": [
    {"customer_id": "C0000123", "customer_name": "安美智科技有限公司", "address_short": "深圳南山"},
    {"customer_id": "C0000457", "customer_name": "安美智能装备股份有限公司", "address_short": "广州天河"}
  ]
}

// 情况 3：未找到
{
  "code": 0,
  "match_type": "none"
}
```

工作流分支：
```
HTTP 调 getSeaCustomerByName
  ↓
代码节点解析 match_type
  ├─ "exact"     → 复用渲染代码 → LLM 输出（标注：公海客户）
  ├─ "ambiguous" → LLM 列出候选名单让销售选 → 下轮销售给出更精确名称再查
  └─ "none"      → "未找到"
```

## 我的建议

**接口可以合并成一个**，省去 search 节点 + detail 节点两次 HTTP 调用，工作流也清爽。**但务必保留 `match_type=ambiguous` 这个分支**——这不是技术洁癖，是 CRM 业务必须留的保险。

剩下唯一要和后端死磕的是"什么算 exact"的判定标准，建议后端硬规则只认这三种：
1. 名称**完全相等**（去空格/繁简体归一化后）
2. **统一社会信用代码**完全相等
3. 模糊匹配命中且**唯一**（没有第二名）

只要不满足上面任何一个，一律走 `ambiguous` 让用户消歧。这样既给了销售"输个简称就能查"的便利，又把"自动猜错"的风险压到零。
