## 147 是怎么算出来的

celery `run_detection` → `service.run_detection`（`service.py:213-249`）：

1. `sales_ids = crm.list_sales_ids()` 先拿销售名单
2. 按每批 50 个销售分批（`default_batch_size=50`）
3. 每批 `customers = crm.list_customers_for_sales_ids(batch)`（`service.py:292`）
4. `total_customers = Σ len(customers) = 147` ← 这就是你看到的「扫描客户数」

生产是 `REPOSITORY_TYPE=api`（`.env.example:18`）。**关键：api 模式下客户不是查客户表，而是两层圈选出来的**（默认没配 `CRM_CUSTOMER_API_URL`，走订单衍生路径）：

- **销售层**（`crm_api.py:499`）：配了 `CRM_API_SALES_IDS` → 只扫这几个销售；没配 → discovery 模式从订单窗口里「发现」有下单的销售
- **客户层**（`crm_api.py:505-512`）：客户**从订单衍生**（只有分析窗口内有订单的才在），再过**拜访过滤**——`CRM_VISIT_API_URL` 默认开（`factory.py:44`），只有出现在该销售拜访记录里的才算「当前客户」，其余剔除；某销售拜访 API 返回空，**该销售名下客户全被砍光**（`crm_api.py:470-477`）

## 最可能的原因（按嫌疑排序）

1. **`CRM_API_SALES_IDS` 配了灰度名单** —— 嫌疑最大。CLAUDE.md 铁律#2 写明 churn 在「~50 业务员灰度」。147 客户 ÷ ~50 销售 ≈ 每人 3 个拜访过的客户，量级完全吻合灰度。
2. **拜访过滤砍人**（默认开）—— 订单客户里只留拜访过的，拜访数据稀疏就会砍掉一大批。
3. **分析窗口/起止日期窄** —— `ANALYSIS_WINDOW_DAYS` 或 `CRM_API_START_DATE/END_DATE` 限制了 discovery 能发现的销售和客户。
4. **休眠客户天然不在** —— 没下过单的客户根本不会被衍生进来。
