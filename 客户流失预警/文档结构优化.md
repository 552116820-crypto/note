# 文档结构优化

日期：2026-04-13
工具：Claude Code (Opus 4.6)

---

## 一、优化前的问题诊断

对 `churn-warning-engine` 项目的文档结构进行了评估，发现以下问题：

### 优化前的文档清单

| 文件 | 定位 | 行数 |
|------|------|------|
| `AGENTS.md` | AI 编码助手指南 + 开发者指南（双重角色） | 52 行 |
| `explain.md` | 对话式教学记录 | ~1000+ 行 |
| `pyproject.toml` | 构建配置 | 37 行 |
| `.gitignore` | 忽略规则 | 10 行 |

### 做得好的地方

1. **AGENTS.md 质量不错** — 覆盖了项目结构、构建命令、编码风格、测试指南、提交规范、环境变量，信息密度高，结构清晰。
2. **环境变量文档化** — 必选/可选变量、生产环境校验规则都有说明。
3. **新模块模板** — 给出了明确的"新增功能模块"脚手架路径，降低扩展时的摸索成本。

### 需要改进的 6 个问题

**1. 没有 README.md — 最大的缺失**

项目根目录没有 README。对任何新来的维护者来说，第一反应就是找 README。现在他只能看到一个 `AGENTS.md`（名字暗示是给 AI 用的）和一个 `explain.md`（打开一看是教学对话）。

建议：新建 `README.md`，包含：项目简介、快速启动、架构概览图、API 端点列表、环境变量速查表。

**2. `explain.md` 内容已过时且定位混乱**

这个文件有三个严重问题：

- **过时**：它描述的是 v1.0.0 的扁平结构（`app/engine.py`、`app/service.py`、`app/repository.py`），但项目已经重构为 v2.0.0 模块化结构（`app/churn/`、`app/scoring/`、`app/shared/`）。新维护者看到的目录和文档完全对不上。
- **混杂**：项目说明、代码审查方法论、Python 基础教学（什么是 `def`、什么是 `@dataclass`、什么是 `BaseModel`）全部堆在一个文件里。
- **对话体**：每段结尾都是"如果你愿意，我下一步可以继续讲..."，这不是维护文档，是一份聊天记录。

建议：要么删除，要么提取其中仍有价值的内容（如业务流程、去重规则、状态流转）整合进正式文档，丢弃 Python 基础教学部分。

**3. 没有 API 文档**

虽然 FastAPI 自动生成 OpenAPI，但没有任何文档说明：
- 完整的业务流程（start_job → list_batches → detect_batch → enrich 的调用顺序）
- 请求/响应示例
- 错误码含义

维护者需要读 `routes.py` 源码才能理解 API 怎么用。

**4. 没有数据库迁移文档**

`migrations/mysql/churn/` 下有 SQL 文件，但没有文档说明：
- 如何执行迁移（手动还是自动？什么顺序？）
- 各迁移文件的依赖关系
- 回滚策略

**5. 没有 CHANGELOG**

版本从 1.0.0 跳到 2.0.0，意味着发生了破坏性重构（从扁平结构到模块化）。但没有任何变更记录，后续维护者无法理解版本演进脉络。

**6. `AGENTS.md` 承担了过多职责**

它既是 AI 编码助手的上下文，又是唯一的开发者文档。如果团队中有人不用 AI 辅助工具，这个文件名本身就会让他困惑。开发者指南的内容（构建命令、编码风格、测试、提交规范）应该有一个更通用的载体。

## 二、执行的改动

| 操作 | 文件 | 说明 |
|------|------|------|
| 新建 | `README.md` | 项目入口：简介、架构图、快速启动、API 端点一览、完整环境变量表、项目结构树 |
| 新建 | `docs/api-guide.md` | API 使用指南：完整业务流程、每个端点的请求/响应示例、去重规则、状态流转图、错误码、Celery 定时任务 |
| 新建 | `docs/migration-guide.md` | 数据库迁移指南：表结构说明、手动执行步骤、安全检查要点、编号约定、新模块迁移模板 |
| 新建 | `CHANGELOG.md` | 版本变更记录：1.0.0 → 1.0.1 → 2.0.0 的 breaking changes / added / changed / fixed |
| 重写 | `AGENTS.md` | 精简为 AI 编码助手专用上下文，去掉与 README 重复的内容，开头指引人类开发者去看 README |
| 删除 | `explain.md` | 过时的对话式教学记录，描述的是 v1.0.0 扁平结构，已不反映当前代码 |

### 建议的目标文档结构

```
README.md              ← 项目入口：简介、快速启动、架构图
AGENTS.md              ← 保留，专注于 AI 编码助手上下文
CHANGELOG.md           ← 版本变更记录
docs/
  api-guide.md         ← API 业务流程 + 调用示例
  migration-guide.md   ← 数据库迁移操作手册
```

## 三、优化后的评估

### 优化后的文档全景

```
README.md                       (181 行)  项目入口
AGENTS.md                       (52 行)   AI 助手上下文
CHANGELOG.md                    (49 行)   版本记录
docs/
  api-guide.md                  (267 行)  API 使用指南
  migration-guide.md            (97 行)   数据库迁移指南
```

### 做得好的部分

1. **职责分离清晰** — 每个文件只解决一个问题，不再有一个文件身兼数职的情况
2. **README 作为入口够用** — 架构图、快速启动、API 一览表、环境变量速查、项目结构树、到子文档的链接，新人 5 分钟内能跑起来
3. **api-guide 是可操作的** — 按业务流程顺序组织，每个端点有请求/响应 JSON 示例，去重规则、状态流转图、错误码都有交代
4. **migration-guide 把隐性知识显性化了** — 手动执行、安全检查步骤、锁表提醒、编号约定这些原本只在 SQL 注释里的信息现在有了独立文档
5. **AGENTS.md 精简了** — 从密集上下文变得可扫读，且不再和 README 大面积重复
6. **CHANGELOG 补了版本断档** — 1.0.0 → 2.0.0 的 breaking changes 有记录，后来者能理解为什么目录结构跟早期提交对不上

### 仍可改进的方向（非阻塞，留待后续）

1. **README 环境变量表偏长** — 目前 ~40 行环境变量表占了 README 将近 1/3 篇幅。如果 scoring 模块完成后再增加变量，README 会变得头重脚轻。到时候可以考虑把环境变量表拆到独立文档（如 `docs/configuration.md`），README 只保留必填项速查。

2. **AGENTS.md 和 README 之间仍有内容重叠** — 项目结构描述、开发命令在两处都有。目前量不大（AGENTS.md 才 52 行），但如果后续两边各自演化，可能出现不一致。可以考虑让 AGENTS.md 用一句话引用 README 对应章节，而非自己复述。

3. **api-guide 的 JSON 示例是手写的** — `feature_summary` 和 `parameter_recommendation` 的示例字段是根据代码逻辑推断的，没有从真实运行输出中截取。如果实际 engine 输出的字段名不同，文档会误导。值得跑一次内存模式的完整流程，用真实响应替换示例。

4. **migration-guide 缺初始建表 DDL** — 文档提到 `warning_jobs` 和 `warning_records` 由 SQLAlchemy `create_all` 自动创建，但没给出完整的建表 SQL。如果有人需要在不经过应用的情况下手动建库（比如 DBA 审核），现在拿不到完整 DDL。

5. **没有 CONTRIBUTING.md** — 提交规范、PR 规范目前在 AGENTS.md 里。如果项目开源或多人协作，这些内容更适合放在标准的 `CONTRIBUTING.md` 中。

### 总结

对于当前项目规模（一个已完成模块 + 一个脚手架），这套文档结构已经够用且合理。上面 5 点是在项目继续扩大时的改进方向，不需要现在处理。
