# 客户流失预警 - AI工程师职责 QA 记录

---

## Q1: 我作为AI工程师，我需要做什么，交付什么

根据需求文档，你作为 AI 工程师的职责集中在**算法模型 + 预测服务**这一层。以下是你需要做和交付的全部内容：

---

### 你的核心交付物

#### 1. 下单周期预测模型

**输入：** 客户历史订单数据（下单时间列表）

**输出：** 预测的下单周期天数 + 置信度

**用于：** 频率异常预警（场景一）

| 项目    | 内容                               |
| ----- | -------------------------------- |
| 做什么   | 分析每个客户的历史下单时间间隔，输出"该客户平均多少天下一次单" |
| 算法思路  | 加权移动平均（近期权重更高）+ IQR 去异常值         |
| 系统参考值 | 即模型输出的天数，展示在配置页供业务参考             |
| 更新频率  | 每周跑一次批量预测，结果写入数据库                |
|       |                                  |

#### 2. 关键产品识别 + 产品级周期预测

**输入：** 客户过去一年所有订单明细（产品、金额、时间）

**输出：** 关键产品列表（占比 >= 阈值）+ 每个关键产品的预测下单周期

**用于：** 产品异常预警（场景三）

| 项目 | 内容 |
|------|------|
| 做什么 | 识别占客户年度采购额 ≥ X% 的产品，再对每个关键产品单独预测下单周期 |
| 占比阈值 | 默认 30%，支持从配置表读取（业务可改） |
| 周期预测 | 同上，对单个产品的下单时间列表做周期预测 |

#### 3. 量级异常检测（相对简单）

**输入：** 客户本月下单量、上月下单量

**输出：** 环比变化率 + 是否触发异常

**用于：** 量级异常预警（场景二）

> 这个严格来说是规则引擎，不需要 ML 模型，但通常也归 AI 工程师出。

### 重点要想清楚的问题

1. **冷启动**：新客户或订单少于 N 单的客户怎么处理？建议设最低样本量（如 ≥ 3 单才预测），不足的不预警或用行业均值兜底
2. **季节性**：如果客户有明显淡旺季（如建材客户冬季少下单），简单均值会误报，是否需要引入季节性因子
3. **预测服务部署形式**：是独立微服务、还是以 Python 脚本被后端 Java 通过 HTTP 调用、还是直接用 SQL/存储过程实现？需要和后端对齐
4. **"AI 给出系统参考值"**：需求里这句话的意思是你的预测结果要能展示在前端配置页上，供业务人员参考后决定是否调整倍数

## Q2: 我们CRM后端用php7.3写的，我用python3.10版本可以吗，兼容性如何呢

完全可以，**没有兼容性问题**。两者是独立服务，通过 HTTP 接口通信。

### 架构关系

```
┌──────────────────┐    HTTP/JSON     ┌──────────────────┐
│  CRM 后端 (PHP)  │ ◄──────────────► │  AI 服务 (Python) │
│  PHP 7.3         │                  │  Python 3.10      │
│  端口 80/443     │                  │  端口 8100        │
└────────┬─────────┘                  └────────┬─────────┘
         │                                     │
         ▼                                     ▼
   ┌───────────┐                         ┌───────────┐
   │  MySQL    │ ◄───── 共享读取 ────────│  同一个DB  │
   │  (CRM库)  │                         │  或独立库  │
   └───────────┘                         └───────────┘
```

PHP 和 Python **不在同一个进程里运行**，互不依赖，唯一的交互方式就是：

1. **HTTP 接口调用** — PHP 用 `curl`/`Guzzle` 请求 Python 的 FastAPI 接口
2. **共享数据库** — 两边都读写 MySQL，各管各的表

### PHP 侧调用示例

```php
<?php
// PHP 调用 AI 预测接口
function getCustomerPrediction($customerId) {
    $ch = curl_init();
    curl_setopt($ch, CURLOPT_URL, "http://ai-service:8100/api/ai/predict/{$customerId}");
    curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);
    curl_setopt($ch, CURLOPT_TIMEOUT, 10);

    $response = curl_exec($ch);
    curl_close($ch);

    return json_decode($response, true);
}

// PHP 触发批量预测
function triggerBatchPredict($customerIds = null) {
    $ch = curl_init();
    curl_setopt($ch, CURLOPT_URL, "http://ai-service:8100/api/ai/predict/batch");
    curl_setopt($ch, CURLOPT_POST, true);
    curl_setopt($ch, CURLOPT_HTTPHEADER, ['Content-Type: application/json']);
    curl_setopt($ch, CURLOPT_POSTFIELDS, json_encode([
        'customer_ids' => $customerIds
    ]));
    curl_setopt($ch, CURLOPT_RETURNTRANSFER, true);

    $response = curl_exec($ch);
    curl_close($ch);

    return json_decode($response, true);
}
```

### 唯一需要对齐的点

| 项目 | 说明 |
|------|------|
| **数据库字符集** | 两边都用 `utf8mb4`，已在代码中配置 |
| **日期格式** | JSON 传输统一用 ISO 格式 `YYYY-MM-DD`，PHP 和 Python 都原生支持 |
| **网络互通** | 部署在同一内网，PHP 能访问 Python 服务的 `host:8100` |
| **定时任务谁触发** | 两种方案都行（下面展开） |

### 定时任务调度方案选择

| 方案 | 做法 | 适合场景 |
|------|------|---------|
| **A: PHP 侧调度** | PHP 的 cron 任务在周一 8:00 调用 AI 的 HTTP 接口 | 后端团队想统一管理所有定时任务 |
| **B: Python 自调度** | Python 用 Celery Beat 自己跑定时任务 | AI 服务独立运维，不依赖 PHP |

**建议用方案 A**，因为你们现有的 cron 管理大概率在 PHP 侧，减少运维复杂度。Python 只做"被调用"，不主动触发。

### 总结

PHP 7.3 + Python 3.10 完全没问题，业界标准做法。你专注写好 Python 算法服务和接口，PHP 后端当成普通 HTTP 客户端来调你的接口就行。

## Q3: 所以这个需求的对接流程是什么样的，我产出接口给php后端就行了吗

对，核心就是你产出接口给 PHP 后端。但完整流程不只是写接口，按时间线梳理一下：

### 完整对接流程

```
第1周                第2-3周              第4-5周              第6周
需求评审/对齐    →    并行开发       →     联调测试      →     灰度上线
```

### 第1周：对齐阶段（最关键）

你需要和 PHP 后端坐下来确认以下内容：

**1. 你从他那读什么（数据输入）**

```
你: "我需要从你的订单表读数据，表名是什么？字段叫什么？"
他: "主表是 crm_orders，明细表是 crm_order_items，字段是..."
```

就是 `data_requirements.md` 里那 7 个待确认问题，不确认清楚后面全白写。

**2. 你给他返什么（接口契约）**

```
你产出的接口                      他消费的方式
─────────────────────────────────────────────────
GET  /api/ai/predict/{id}    →   前端配置页加载时调用，展示"AI预测周期"
POST /api/ai/predict/batch   →   他的 cron 每周日凌晨调一次
POST /api/ai/detect/frequency →  他的 cron 每周一8点调一次
POST /api/ai/detect/volume   →   他的 cron 每月1号8点调一次
POST /api/ai/detect/product  →   他的 cron 每周一8点调一次
```

**3. 他给你提供什么（回调接口）**

```
你检测出预警 → 调用他的接口 → 他负责写预警记录 + 发企微消息

他需要提供:
POST /api/backend/warning/receive   ← 接收预警列表
```

**4. 数据库分工**

| 表 | 谁建 | 谁写 | 谁读 |
|---|---|---|---|
| crm_orders (已有) | 后端 | 后端 | **你读** |
| crm_order_items (已有) | 后端 | 后端 | **你读** |
| churn_prediction | **你建** | **你写** | 他读（给前端用） |
| volume_anomaly_snapshot | **你建** | **你写** | 他读 |
| churn_warning_global_config | 他建 | 他写 | **你读** |
| churn_warning_customer_config | 他建 | 他写 | **你读** |
| churn_warning_record | 他建 | 他写 | 他读 |

### 第2-3周：并行开发

```
你（Python）                          他（PHP）
──────────────                        ──────────
写算法代码                             写配置 CRUD 接口
写 FastAPI 接口                        写预警记录表 + 企微推送
跑单元测试                             写前端配置页/大屏
用模拟数据验证算法                      写 cron 调度脚本
                                      写预警回调接口
```

两边互不阻塞，各干各的。

### 第4-5周：联调

```
步骤1: 他把真实数据库地址给你，你连上去跑一次批量预测，看数据对不对
步骤2: 他调你的接口，看返回格式对不对
步骤3: 你调他的回调接口，看预警消息能不能发出去
步骤4: 端到端跑通: cron触发 → 你检测 → 出预警 → 他推企微 → 手机收到消息
```

### 第6周：灰度上线

选 50 个客户先跑，只记录不推送，人工验证准确率。

| 序号 | 交付物 | 交给谁 | 什么时候 |
|------|--------|--------|---------|
| 1 | `data_requirements.md` | PHP 后端 | 第1周评审时 |
| 2 | `algorithm_design.md` | 全组评审 | 第1周评审时 |
| 3 | 接口文档 (FastAPI 自动生成 `/docs`) | PHP 后端 | 第2周开发开始时 |
| 4 | `sql/schema.sql` | DBA/后端 | 第2周建表 |
