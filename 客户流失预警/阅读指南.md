# 客户流失预警项目阅读指南

项目路径：`E:\churn-warning-engine`

本指南整理自本次会话中对该项目代码的阅读说明。推荐按业务流程阅读，而不是按文件名从上到下一口气读。

## 项目主线

这个项目的主流程是：

1. 创建检测任务
2. 按销售人员拆分批次
3. 读取客户和订单数据
4. 运行异常检测算法
5. 对预警做去重、刷新或保存
6. 人工 enrich 或 resolve 预警

对应的核心分层是：

- `app/main.py`：FastAPI 入口，只负责 HTTP 请求、响应和错误码映射。
- `app/service.py`：业务编排核心，把 API、仓储、算法串起来。
- `app/engine.py`：异常检测算法，包括频率、量级、产品三个场景。
- `app/repository.py`：仓储抽象和内存仓储实现。
- `app/db.py`：可选 MySQL 仓储实现。
- `app/schemas.py`：API 入参和出参模型。
- `app/models.py`：内部领域模型。
- `app/config.py`：环境变量和默认配置。
- `app/tasks.py`：Celery 定时任务。

## 推荐阅读顺序

### 1. 先看数据结构

文件：`app/models.py`

重点对象：

- `Customer`：客户信息。
- `OrderItem`：订单明细中的产品。
- `Order`：订单，包含订单日期、金额和明细。
- `WarningJob`：一次检测任务。
- `WarningRecord`：一条预警记录。

先理解这些对象，后面所有代码都是围绕它们流转。

### 2. 再看 API 契约

文件：`app/schemas.py`

重点模型：

- `StartJobRequest`
- `StartJobResponse`
- `DetectBatchRequest`
- `DetectBatchResponse`
- `EnrichWarningRequest`
- `WarningResponse`

这些 Pydantic 模型决定了接口的请求体和响应体长什么样。

### 3. 看 FastAPI 入口

文件：`app/main.py`

重点接口：

- `POST /api/warning-jobs/start`
- `GET /api/warning-jobs/{job_id}/batches`
- `POST /api/warning-jobs/{job_id}/detect-batch`
- `GET /api/warnings/pending-enrich`
- `GET /api/warnings/{warning_id}/feature-summary`
- `POST /api/warnings/{warning_id}/enrich`
- `POST /api/warnings/{warning_id}/resolve`
- `GET /api/warnings/{warning_id}`

`main.py` 的特点是：它基本不写业务逻辑，只调用 `WarningService`，再把 `KeyError`、`ValueError`、`InvalidWarningStateError` 转换为 HTTP 状态码。

### 4. 重点读业务编排

文件：`app/service.py`

这是最重要的文件。重点方法：

- `start_job`：解析业务日期，读取销售 ID，按批次创建 `WarningJob`。
- `list_batches`：返回任务下的批次列表。
- `detect_batch`：校验批次，读取客户和订单，调用检测算法，保存或刷新预警。
- `list_pending_warnings`：列出待 enrich 的预警。
- `get_feature_summary`：返回预警特征摘要。
- `enrich_warning`：人工补充预警内容。
- `resolve_warning`：关闭预警。
- `get_warning`：查询单条预警。

阅读 `detect_batch` 时要特别关注三件事：

- 它如何校验 `batch_id`。
- 它如何根据 `job_type` 选择不同算法。
- 它如何做预警去重。

## 一次完整请求如何流动

以创建并执行一次频率检测为例：

1. 调用 `POST /api/warning-jobs/start`。
2. `app/main.py` 的 `start_job` 接收请求。
3. FastAPI 通过 `Depends(get_cached_service)` 获取 `WarningService`。
4. `app/service_factory.py` 决定使用内存仓储还是 MySQL 仓储。
5. `WarningService.start_job` 创建 `WarningJob` 并保存。
6. 调用 `GET /api/warning-jobs/{job_id}/batches` 获取批次。
7. 调用 `POST /api/warning-jobs/{job_id}/detect-batch` 执行某个批次。
8. `WarningService.detect_batch` 读取客户和订单。
9. 根据任务类型调用 `app/engine.py` 中的检测函数。
10. 检测函数返回 `WarningRecord` 列表。
11. service 层对预警做去重：已有同类活跃预警则刷新或复用，否则新建。

## 检测算法阅读方式

文件：`app/engine.py`

建议先读工具函数，再读三个业务场景。

### 周期预测

核心函数：`predict_cycle`

逻辑：

1. 订单日期排序去重。
2. 计算相邻订单的间隔天数。
3. 使用 IQR 去除异常间隔。
4. 用 EWMA 计算预测周期。
5. 根据样本量和稳定性计算置信度。

相关函数：

- `_remove_iqr_outliers`
- `_ewma`
- `_prediction_confidence`

### 频率异常

核心函数：`detect_frequency_anomaly`

判断逻辑：

```text
实际未下单天数 > 预测周期 * frequency_multiplier
```

满足条件时生成 `frequency` 类型预警。

### 量级异常

核心函数：`detect_volume_anomaly`

判断逻辑：

- 业务日期必须是月末。
- 比较当前月和上个月采购金额。
- 下降超过 `volume_drop_threshold` 触发下降预警。
- 上升超过 `volume_rise_threshold` 触发上升预警。

量级预警会把方向写入：

- `direction = "down"`
- `direction = "up"`

这会影响去重 key。

### 产品异常

核心函数：`detect_product_anomaly`

判断逻辑：

1. 取最近 `analysis_window_days` 天订单。
2. 统计各产品金额占比。
3. 金额占比超过 `key_product_threshold` 的产品被视为关键产品。
4. 对每个关键产品单独预测采购周期。
5. 若某产品停采时间超过 `预测周期 * product_multiplier`，生成产品预警。

## 仓储层怎么读

先看 `app/repository.py`。

`Repository` 是抽象基类，定义 service 层需要的数据操作：

- 查询销售 ID。
- 查询销售名下客户。
- 查询客户订单。
- 保存和读取任务。
- 保存、更新和查询预警。
- 查找活跃同类预警。
- 提供去重锁。

`InMemoryRepository` 是开发和测试用的内存实现，里面有演示数据。第一遍读项目时，重点读这个实现即可。

再看 `app/db.py`。

`MySQLRepository` 是生产实现：

- 从 CRM 表读取客户、订单、订单明细。
- 将任务写入 `warning_jobs`。
- 将预警写入 `warning_records`。
- 使用 MySQL `GET_LOCK` 做同类预警去重锁。

第一遍不需要细读所有 SQL，先理解它实现了和 `Repository` 一样的接口即可。

## 配置和工厂

文件：`app/config.py`

`Settings` 管理检测参数：

- `min_order_count`
- `frequency_multiplier`
- `volume_drop_threshold`
- `volume_rise_threshold`
- `product_multiplier`
- `key_product_threshold`
- `analysis_window_days`
- `default_batch_size`
- `max_pending_enrich`
- `completed_status_codes`
- `demo_reference_date`

文件：`app/service_factory.py`

核心逻辑：

- 默认使用 `InMemoryRepository`。
- 当 `REPOSITORY_TYPE=mysql` 时使用 `MySQLRepository`。
- `get_cached_service` 用 `lru_cache` 缓存 service 实例。

## 定时任务

文件：`app/tasks.py`

Celery 定时任务会周期性调用 `run_detection`：

- 每周一 08:00：频率异常检测。
- 每周一 08:10：产品异常检测。
- 每月 1 日 08:00：量级异常检测。

`run_detection` 会自动完成：

1. 创建 Job。
2. 获取批次。
3. 逐批执行检测。
4. 汇总预警 ID 和异常数量。

## 一句话总结

这个项目的阅读关键是分层：

```text
HTTP API -> WarningService -> Repository -> Engine -> WarningRecord
```

也就是：

- `main.py` 管 HTTP。
- `service.py` 管流程。
- `engine.py` 管算法。
- `repository.py` 和 `db.py` 管数据。
- `schemas.py` 和 `models.py` 管数据结构。
