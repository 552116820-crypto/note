[Assistant]
最少要从 CRM 拿 3 类数据，才能把这套预警算法跑起来。

1. 客户归属数据
- `customer_id`
- `customer_name`
- `sales_id` / `salesperson_id`
- `sales_name`
- 用途：分批跑任务、确定预警发给谁。你现在这个改进版默认就按 `crm_customers(customer_id, customer_name, sales_id, sales_name)` 取，[db.py#L7](/mnt/e/churn-warning-engine/app/db.py#L7)

2. 订单主表数据
- `order_id`
- `customer_id`
- `customer_name`（如果客户表里有，也可不冗余）
- `sales_id`
- `order_date`
- `total_amount`
- `status`
- 用途：
  - 频率异常：主要用 `order_date`
  - 量级异常：当前改进版主要用 `total_amount`
  - 活跃客户筛选：用 `status + order_date`
- 参考字段定义在 [data_requirements.md#L5](/mnt/e/churn-warning-ai/docs/data_requirements.md#L5)；代码里频率和活跃客户也就是这么查的，[data_fetcher.py#L15](/mnt/e/churn-warning-ai/src/utils/data_fetcher.py#L15) [data_fetcher.py#L46](/mnt/e/churn-warning-ai/src/utils/data_fetcher.py#L46)

3. 订单明细表数据
- `order_id`
- `product_id`
- `product_name`
- `quantity`
- `amount`
- 用途：
  - 产品异常：必须用 `product_id/product_name/amount`
  - 如果你要做“数量维度”的量级异常，`quantity` 也必须有
- 参考字段定义在 [data_requirements.md#L24](/mnt/e/churn-warning-ai/docs/data_requirements.md#L24)；产品异常和数量聚合分别见 [data_fetcher.py#L78](/mnt/e/churn-warning-ai/src/utils/data_fetcher.py#L78) 和 [data_fetcher.py#L118](/mnt/e/churn-warning-ai/src/utils/data_fetcher.py#L118)

如果按你现在的 `E:\churn-warning-engine` 代码说，最低可跑集是：

- `crm_customers(customer_id, customer_name, sales_id, sales_name)`
- `crm_orders(order_id, customer_id, sales_id, order_date, total_amount)`
- `crm_order_items(order_id, product_id, product_name, amount)`

这就是它现在默认假设的 3 张表，[db.py#L7](/mnt/e/churn-warning-engine/app/db.py#L7)。  
但如果你想把“量级异常”真正做完整，建议 `crm_order_items` 再补上 `quantity`，否则只能看金额，不能看数量。

还需要你确认 6 个业务口径，不然后面结果会偏：

- `status` 里“已完成”到底是哪一个值，[data_requirements.md#L19](/mnt/e/churn-warning-ai/docs/data_requirements.md#L19)
- `order_date` 是下单时间、发货时间，还是完成时间
- 已取消、已退货、红冲订单要不要排除
- `amount` 是否已经扣折扣、退款
- 产品预警按哪个层级做：品类、系列还是 SKU
- 一个客户是否可能挂多个销售；如果是，预警归谁

时间范围建议：

- 至少给近 2 年已完成订单，用于频率预测
- 至少给近 1 年订单明细，用于关键产品分析
- 至少保证上月和本月数据完整，用于量级异常

最实用的交付方式不是直接丢原表，而是让 CRM/数据团队给你 3 个只读视图。  
如果你愿意，我下一步可以直接按你现有 CRM 表结构，帮你出一版“字段映射表 + SQL 视图”。

**接口模板**
接口名：`客户预警数据查询接口`  
方法：`POST`  
路径：`/api/crm/warning-data/query`  
用途：返回指定时间范围内的客户、订单、订单明细数据，供频率异常、量级异常、产品异常检测使用。

**请求头**
```http
Content-Type: application/json
Authorization: Bearer {token}
```

**请求参数**
```json
{
  "biz_date": "2026-04-09",
  "start_date": "2024-04-09",
  "end_date": "2026-04-09",
  "sales_ids": [8001, 8002],
  "customer_ids": [1001, 1002],
  "order_status_codes": [1],
  "page": 1,
  "page_size": 100
}
```

**请求字段说明**
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `biz_date` | string(date) | 是 | 本次检测业务日期 |
| `start_date` | string(date) | 是 | 查询开始日期 |
| `end_date` | string(date) | 是 | 查询结束日期 |
| `sales_ids` | array<number> | 否 | 指定销售ID集合，不传则查全部 |
| `customer_ids` | array<number> | 否 | 指定客户ID集合，不传则查全部 |
| `order_status_codes` | array<number> | 是 | 参与检测的订单状态码，通常只传“已完成” |
| `page` | number | 否 | 页码，默认1 |
| `page_size` | number | 否 | 每页条数，默认100，建议最大500 |

**返回字段说明**
| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `customer_id` | number | 是 | 客户ID |
| `customer_name` | string | 是 | 客户名称 |
| `sales_id` | number | 是 | 归属销售ID |
| `sales_name` | string | 否 | 归属销售名称 |
| `order_id` | number | 是 | 订单ID |
| `order_date` | string(date) | 是 | 订单日期 |
| `status` | string | 否 | 订单状态文本 |
| `status_code` | number | 是 | 订单状态码 |
| `total_amount` | number | 是 | 订单总金额 |
| `product_id` | string | 是 | 产品ID |
| `product_name` | string | 是 | 产品名称 |
| `quantity` | number | 否 | 产品数量，若做数量异常则必须返回 |
| `amount` | number | 是 | 该产品金额 |

**字段口径要求**
- `order_status_codes` 必须能准确过滤“已完成”订单。
- `order_date` 要明确是“下单日期”还是“完成日期”，整套算法要统一口径。
- `total_amount`、`items.amount` 建议返回最终成交金额，避免把退款前金额算进去。
- `product_id` 必须稳定，不要今天 SKU、明天品类。
- 一个客户如果有多个销售，必须提前约定主销售归属规则。

**错误返回**
```json
{
  "code": 4001,
  "message": "invalid date range",
  "data": null
}
```

**错误码建议**
| code | 含义 |
|---|---|
| `0` | 成功 |
| `4001` | 日期范围非法 |
| `4002` | 请求参数缺失或格式错误 |
| `4003` | 状态码无效 |
| `5001` | CRM内部错误 |

**实现要求**
- 支持按 `sales_ids` 和 `customer_ids` 过滤。
- 支持分页。
- 返回结果建议按 `customer_id`、`order_date` 升序。
- 单次返回最好控制在 `500` 个客户以内。
