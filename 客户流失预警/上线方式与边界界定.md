**1. 触发方式**
有两种：

1. API 手动触发：
   先调用 `POST /api/warning-jobs/start` 创建任务，再调用 `GET /api/warning-jobs/{job_id}/batches` 看批次，最后对每个批次调用 `POST /api/warning-jobs/{job_id}/detect-batch` 跑检测。对应路由在 [app/main.py](/mnt/e/churn-warning-engine/app/main.py):54、[app/main.py](/mnt/e/churn-warning-engine/app/main.py):72、[app/main.py](/mnt/e/churn-warning-engine/app/main.py):86。

2. Celery 自动触发：
   [app/tasks.py](/mnt/e/churn-warning-engine/app/tasks.py):34 配了定时任务：每周一 08:00 跑频率异常，08:10 跑产品异常，每月 1 日 08:00 跑量级异常。`run_detection` 会自动完成“创建任务 -> 列批次 -> 逐批检测 -> 汇总结果”，在 [app/tasks.py](/mnt/e/churn-warning-engine/app/tasks.py):59。

**2. 创建任务和业务日期**
创建任务时，系统会先确定 `biz_date`，也就是本次检测基准日期。逻辑在 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):34：

- `frequency` 和 `product`：如果请求传了 `biz_date` 就用它；否则用 `DEMO_REFERENCE_DATE`；再否则用当天日期。
- `volume`：必须按完整自然月比较。如果没传日期，默认用上个月最后一天；如果传了日期但不是月末，会直接报错。

然后系统从仓储里拿所有 `sales_id`，按 `batch_size` 切成批次，生成一个 `WarningJob` 并保存。批次 ID 格式是：

```text
{job_id}:{批次序号}
```

例如：`abc-uuid:1`。这段逻辑在 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):74。

**3. 批量检测**
调用 `detect_batch` 后，服务会先校验 `batch_id` 格式、所属 `job_id`、批次序号是否越界。然后根据这个批次里的销售 ID 找客户，再批量加载这些客户的订单。主逻辑在 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):103。

订单加载时会带上时间过滤：

- `frequency`：用客户历史订单，截止到 `biz_date`。
- `product`：只取最近 `analysis_window_days` 天的订单。
- `volume`：只取上月月初到本月月末这段订单，因为要比较完整自然月。

然后按客户循环，根据任务类型调用不同算法：

- `frequency` -> `detect_frequency_anomaly`
- `volume` -> `detect_volume_anomaly`
- `product` -> `detect_product_anomaly`

对应分发在 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):133。

**4. 三类异常算法**
频率异常在 [app/engine.py](/mnt/e/churn-warning-engine/app/engine.py):200。它先预测客户正常下单周期：订单日期排序去重、算相邻订单间隔、用 IQR 去掉异常间隔、用 EWMA 做指数加权平均，再计算置信度。预测周期逻辑在 [app/engine.py](/mnt/e/churn-warning-engine/app/engine.py):101。若“距离最后一次下单天数 > 预测周期 * frequency_multiplier”，就生成预警。

量级异常在 [app/engine.py](/mnt/e/churn-warning-engine/app/engine.py):262。它要求 `biz_date` 是月末，然后算本月采购额和上月采购额。如果上月金额为 0，不告警。否则计算环比变化：

- 下降超过 `volume_drop_threshold`：生成下降预警，偏流失风险。
- 上升超过 `volume_rise_threshold`：也生成预警，但更像集中备货、录入异常或异常需求，不完全是流失风险。

产品异常在 [app/engine.py](/mnt/e/churn-warning-engine/app/engine.py):364。它先在分析窗口内按商品汇总金额，找出采购占比达到 `key_product_threshold` 的关键产品。然后对每个关键产品单独预测采购周期；若该产品距离上次采购时间超过 `预测周期 * product_multiplier`，就生成产品级预警。一个客户可能产生多个产品预警。

**5. 预警内容**
每条算法生成的预警都是 `WarningRecord`，构造入口在 [app/engine.py](/mnt/e/churn-warning-engine/app/engine.py):166。它包含：

- `warning_id`
- `anomaly_type`：`frequency`、`volume`、`product`
- 客户和销售字段
- `biz_date`
- `severity`：`low`、`medium`、`high`
- `status`：初始是 `pending_enrich`
- `feature_summary`：算法特征摘要
- `parameter_recommendation`：推荐参数
- `reason`：基础原因
- `sales_message`：给销售看的建议话术
- `dedup_key` 和 `metadata`：用于去重

严重程度的分级规则在 [app/engine.py](/mnt/e/churn-warning-engine/app/engine.py):135，比如频率/产品偏离比达到 2.0 是 `high`，达到 1.5 是 `medium`。

**6. 去重和刷新**
检测出预警后不会无脑插入。系统会按“客户 + 异常类型 + 细分 key”查找是否已有活跃预警。逻辑在 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):156。

细分 key 大致是：

- 频率：`frequency`
- 量级：`volume:up` 或 `volume:down`
- 产品：`product:{product_id}`

处理规则：

- 没有活跃预警：保存新预警。
- 已有 `pending_enrich` 预警：刷新算法字段，比如 `biz_date`、`severity`、`feature_summary`、`reason`、`sales_message`，但沿用原来的 `warning_id`。
- 已有 `enriched` 预警：不覆盖人工补充内容，只复用原预警 ID。
- 已经 `resolved` 的预警不算活跃预警，后续检测可以生成新的预警。

这块是项目里比较关键的业务保护：重复跑任务不会制造一堆同类重复预警，也不会覆盖人工已经处理过的内容。

**7. 人工补充和关闭**
检测出来的预警默认是 `pending_enrich`。后续有几类接口：

- `GET /api/warnings/pending-enrich`：拿待补充预警列表，见 [app/main.py](/mnt/e/churn-warning-engine/app/main.py):103。
- `GET /api/warnings/{warning_id}/feature-summary`：查看算法摘要，给人工判断用，见 [app/main.py](/mnt/e/churn-warning-engine/app/main.py):110。
- `POST /api/warnings/{warning_id}/enrich`：人工补充严重程度、原因、参数建议、销售话术，状态改成 `enriched`，见 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):220。
- `POST /api/warnings/{warning_id}/resolve`：关闭预警，状态改成 `resolved`，见 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):248。
- `GET /api/warnings/{warning_id}`：查看单条预警。

注意：已 `resolved` 的预警不能再 enrich，会返回 409。这个保护在 [app/service.py](/mnt/e/churn-warning-engine/app/service.py):232。

**8. 数据来源和存储**
项目通过 `Repository` 抽象隔离数据来源，接口定义在 [app/repository.py](/mnt/e/churn-warning-engine/app/repository.py):19。

默认使用 `InMemoryRepository`，内置 6 个客户和一些演示订单，适合本地演示。它会把任务和预警存在内存字典里，逻辑在 [app/repository.py](/mnt/e/churn-warning-engine/app/repository.py):101。

如果设置 `REPOSITORY_TYPE=mysql`，会用 `MySQLRepository`。生产路径里，客户和订单从 CRM 表读取，预警任务和预警记录写入自建表。它会一次 JOIN 订单和订单明细，避免按客户 N+1 查询，见 [app/db.py](/mnt/e/churn-warning-engine/app/db.py):198。预警写入、更新和活跃预警查询在 [app/db.py](/mnt/e/churn-warning-engine/app/db.py):316、[app/db.py](/mnt/e/churn-warning-engine/app/db.py):349、[app/db.py](/mnt/e/churn-warning-engine/app/db.py):372。

总结一下：这个项目不是“用户点一次接口直接跑全量”的简单流程，而是一个分层的预警引擎。`main.py` 负责 API，`tasks.py` 负责定时触发，`service.py` 负责任务、批次、去重和状态流转，`engine.py` 负责三类异常判断，`repository.py` / `db.py` 负责数据读写。整个业务闭环是先自动发现候选流失/异常预警，再让人工补充判断和销售话术，最后关闭或等待下一次刷新。
