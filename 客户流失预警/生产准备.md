## 一、阻塞上线的问题（Must Fix）

### 1. 没有认证/鉴权

所有 API 端点（`/api/warning-jobs/start`、`/api/warnings/{id}/enrich` 等）完全裸奔，任何人都可以创建 Job、修改预警状态。生产环境必须加认证（至少 API Key 或 JWT）。

### 2. factory 中 `create_engine` 被调用两次，创建了两个独立的连接池

`app/churn/factory.py:19` 和 `app/churn/factory.py:36` 分别对 CRM 和 Churn 各创建了一次 `create_engine`，但使用相同的 `DATABASE_URL`。这意味着同一个数据库建了两个连接池，浪费资源且连接管理不一致。应该共享同一个 engine。

### 3. Session 未正确使用——没有 rollback 处理

`mysql_repo.py` 中所有数据库操作使用 `with self._session() as s:` 后直接 `s.commit()`，但 SQLAlchemy 的 `Session()` 不是 context manager（`sessionmaker` 返回的不自带 rollback-on-exception）。如果 `commit()` 前抛异常，session 不会 rollback，连接可能泄漏。应该用 `Session.begin()` 或手动 try/except/rollback。

### 4. `lru_cache` 在多 worker 场景下的问题

`get_cached_service()` 用 `@lru_cache(maxsize=1)` 缓存，但如果用 `uvicorn --workers N` 多进程部署，每个 worker 会各自建一份缓存，这本身没问题。但 `lru_cache` 不是线程安全的初始化——在 `uvicorn` 默认的异步模式或多线程场景下，可能同时触发多次 `build_churn_service()`。

### 5. `Base.metadata.create_all(engine)` 在生产环境不安全

`app/churn/mysql_repo.py:87` 在 `MySQLChurnRepository.__init__` 中调用 `create_all`，每次构造仓储就尝试建表。生产环境应该通过 migration 管理表结构，而不是 ORM 自动创建。这可能意外覆盖表结构或掩盖 migration 问题。

### 6. 没有日志和可观测性

整个 FastAPI 应用没有配置结构化日志、没有请求追踪（trace_id）、没有 metrics 端点。出了问题很难排查。

---

## 二、重要但不阻塞的问题（Should Fix）

### 7. 没有分页

`/api/warnings/pending-enrich` 虽然有 `max_pending_enrich` 限制（默认 100），但没有 offset/cursor 分页。数据量大时无法翻页。

### 8. `detect_batch` 不是幂等的

对同一个 batch 调用两次 `detect_batch`，如果预警已被 resolve，会重新创建新预警。Celery 任务在 retry 场景下可能触发重复预警。

### 9. 缺少 CORS 配置

如果前端直接调用 API，需要配置 CORS middleware。

### 10. Celery 任务缺少错误处理和重试策略

`run_detection` 任务没有配置 `max_retries`、`retry_backoff`，单个 batch 失败会导致整个 job 中断，已处理的 batch 结果无法回滚。

### 11. `InMemoryCrmRepository` 的 demo 数据硬编码

如果 `REPOSITORY_TYPE` 没设为 `mysql`，生产环境会默认使用内存仓储的 demo 数据。应该在生产环境强制要求配置 MySQL，否则 fail fast。

### 12. 环境变量没有 Startup 校验

`DATABASE_URL` 等关键配置仅在首次请求时才会被读取（lazy 构建 service）。如果配置缺失或错误，要等到第一个请求进来才会发现。应该在应用启动时校验。
