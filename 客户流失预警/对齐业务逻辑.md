### Turn 1  *(2026-04-13 09:45)*

**User:**

python -m uvicorn app.main:app --reload这行命令是什么意思

**Assistant:**

这行命令是启动 FastAPI 开发服务器：

```bash
python -m uvicorn app.main:app --reload
```

逐段解释：

- `python -m uvicorn`：用当前 Python 环境运行 `uvicorn` 这个 ASGI 服务器模块。
- `app.main:app`：告诉 uvicorn 去加载 `app/main.py` 里的 `app` 变量。
  - `app.main` 对应文件路径 `app/main.py`
  - `:app` 对应该文件里通常写的 `app = FastAPI(...)`
- `--reload`：开发模式热重载。代码文件变化后，服务会自动重启。

在这个项目里，它的意思就是：启动 `app/main.py` 中定义的 FastAPI 应用。默认访问地址通常是：

```text
http://127.0.0.1:8000
```

接口文档一般在：

```text
http://127.0.0.1:8000/docs
```

`--reload` 适合本地开发，不建议生产环境使用。

### Turn 6  *(2026-04-14 00:50)*

**User:**

POST /api/warning-jobs/start的请求体"batch_size"的含义和POST https://crm.amer.cn/v2/api/OutPut/getSalesmanSaleOrder的page_size有什么区别和联系

**Assistant:**

它们不是同一个概念，只是名字都带 `size`。

`POST /api/warning-jobs/start` 里的 `batch_size` 是**本系统的检测任务分批大小**：

```json
{
  "type": "frequency",
  "biz_date": "2026-04-14",
  "batch_size": 20
}
```

含义是：创建 warning job 时，每个检测批次里放多少个 `sales_id`。

例如全量发现到 275 个销售 ID：

- `batch_size=20`：生成约 14 个检测批次
- `batch_size=100`：生成约 3 个检测批次
- `batch_size=200`：生成约 2 个检测批次

对应后续接口：

```text
GET  /api/warning-jobs/{job_id}/batches
POST /api/warning-jobs/{job_id}/detect-batch
```

也就是说，`batch_size` 控制的是**后续检测任务怎么按销售 ID 分批执行**。

`POST https://crm.amer.cn/v2/api/OutPut/getSalesmanSaleOrder?page=1&page_size=100` 里的 `page_size` 是**CRM 订单接口每页返回多少条订单记录**。

例如：

```text
page=1&page_size=100
```

含义是：从 CRM 拉第 1 页订单，每页最多 100 条订单。它控制的是**从 CRM API 拉订单数据时的分页大小**。

两者的联系是：

```text
start job
  -> 需要拿到销售 ID 列表
  -> 如果没配置 CRM_API_SALES_IDS，就会调用 CRM 订单接口分页拉订单
  -> 从订单里提取 sales_id
  -> 再按 batch_size 把 sales_id 分成检测批次
```

举例：

CRM 接口返回：

```text
total=134740 条订单
page_size=100
需要拉 1348 页
最终发现 275 个 sales_id
```

然后你请求：

```json
{
  "type": "frequency",
  "biz_date": "2026-04-14",
  "batch_size": 20
}
```

系统会把这 275 个 `sales_id` 按每批 20 个分成大约 14 个检测批次。

总结：

```text
CRM page_size = 每次从 CRM 拉多少条订单
start batch_size = 每个检测批次包含多少个销售 ID
```

`batch_size` 不会减少 CRM 全量发现时要拉的订单总页数。只要你不传 `CRM_API_SALES_IDS`，系统要发现全量销售 ID，就仍然需要把窗口内订单分页拉完。

amount > 0  -> 作为采购记录
amount <= 0 -> 按退款/冲销处理，不参与产品周期预测，也不参与关键产品占比

所以退款不会刷新某个产品的最后采购日期。

IQR 过滤异常间隔
