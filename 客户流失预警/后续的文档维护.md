# 后续的文档维护

日期: 2026-04-10

项目路径: `/mnt/e/churn-warning-engine`

## 本次讨论主题

后续如果要继续开发功能，应该是在当前已有的 Python 文件中继续添加代码，还是创建新的文件夹。

讨论结论:

- 如果只是客户流失预警内部新增一种检测规则，可以先在现有分层里扩展。
- 如果是和客户流失预警并行的新业务功能，建议创建独立功能文件夹，不要继续塞进当前的 `engine.py`、`service.py`、`repository.py`、`main.py`。

---

## 当前项目结构判断

当前项目的主要代码是围绕“客户流失/异常预警”设计的:

```text
app/
  engine.py          # 客户流失/异常预警检测逻辑
  service.py         # 客户流失预警任务编排
  repository.py      # 内存仓储
  db.py              # MySQL 仓储
  schemas.py         # API 请求/响应模型
  main.py            # FastAPI 入口
```

这些文件已经有明确业务语义。如果把一个“并行功能”继续写进这些文件，会导致:

- `service.py` 变成万能 service。
- `engine.py` 变成多个业务混杂的规则集合。
- `schemas.py` 混入不相关 API 契约。
- `main.py` 里的路由越来越难维护。
- 以后拆服务、排查问题、写测试都会更困难。

所以并行功能不建议继续堆在现有 py 文件里。

---

## 什么时候继续使用现有文件

如果新需求仍然属于“客户流失预警”的一个场景，可以先使用现有文件。

例如:

- 新增客户回款异常预警。
- 新增订单取消异常预警。
- 新增客户采购频率的另一个规则。
- 新增一个 warning 状态，例如忽略、重新打开。
- 给现有 warning 增加字段。

推荐做法:

```text
app/
  engine.py        # 新增检测函数
  service.py       # 接入 job_type 分发
  schemas.py       # 更新 JobType / 请求响应模型
  repository.py    # 如需新增查询方法
  db.py            # 如需新增 MySQL 查询或字段
tests/
  test_engine.py
  test_service.py
  test_api_and_factory.py
```

原则:

- 小改动先沿用现有分层。
- 不要为了“看起来规范”提前建很多空文件夹。
- 等 `engine.py` 或 `service.py` 明显变长，再拆模块。

---

## 什么时候创建新文件夹

如果新功能是和客户流失预警并行的业务功能，建议新建独立文件夹。

例如:

- 销售机会管理。
- 客户画像分析。
- 合同风险预警。
- 回款预测。
- 报表中心。
- 通知推送中心。
- LLM 自动丰富或自动生成跟进建议，如果它逐渐变成独立流程，也可以单独拆。

推荐结构:

```text
app/
  new_feature/
    __init__.py
    engine.py
    service.py
    repository.py
    schemas.py
    routes.py
```

如果新功能没有算法层，可以省略 `engine.py`。

如果新功能不需要单独仓储，可以省略 `repository.py`。

不要机械套模板，按实际职责建文件。

---

## 推荐的并行功能结构

假设新功能叫“销售机会预警”，可以这样建:

```text
app/
  opportunity/
    __init__.py
    engine.py
    service.py
    schemas.py
    routes.py
    repository.py

tests/
  opportunity/
    test_engine.py
    test_service.py
    test_api.py
```

然后在 `app/main.py` 中只做路由挂载:

```python
from app.opportunity.routes import router as opportunity_router

app.include_router(opportunity_router, prefix="/api/opportunity")
```

`main.py` 的职责应保持简单:

- 创建 FastAPI app。
- 挂载不同业务模块的 router。
- 不写具体业务逻辑。

---

## 是否需要重构旧的客户流失预警模块

短期不建议马上大搬家。

当前代码已经稳定，测试也覆盖了主要路径。为了开发并行功能，直接把旧代码整体移动到 `app/churn/` 下面，风险偏高，收益不大。

建议策略:

1. 旧的客户流失预警代码暂时保持不动。
2. 新的并行功能使用独立文件夹开发。
3. 等新功能稳定后，如果发现项目内有多个业务模块，再考虑统一调整结构。

未来可能的目标结构:

```text
app/
  churn/
    engine.py
    service.py
    repository.py
    schemas.py
    routes.py

  opportunity/
    engine.py
    service.py
    repository.py
    schemas.py
    routes.py

  common/
    date_utils.py
    warning_keys.py

  main.py
  config.py
```

但这属于后续重构，不是开发并行功能的前置条件。

## 公共能力放哪里

并行功能之间可能会共享一些能力。共享代码不要放进某个业务模块里。

适合保持公共的文件:

- `app/config.py`
- `app/date_utils.py`
- `app/service_factory.py`，如果后续仍然只有一个主服务入口

未来如果共享代码变多，可以新增:

```text
app/common/
  date_utils.py
  errors.py
  pagination.py
  database.py
```

判断标准:

- 只服务于客户流失预警: 放在客户流失预警模块。
- 只服务于新功能: 放在新功能模块。
- 多个业务都会用: 放在 common 或顶层公共模块。

## 开发新功能时的文档维护建议

每开发一个并行功能，建议同步维护以下文档:

1. 功能说明
   - 这个功能解决什么问题。
   - 和客户流失预警的边界是什么。

2. API 文档
   - 路径。
   - 请求参数。
   - 响应字段。
   - 常见错误码。

3. 数据模型说明
   - 新增表。
   - 新增字段。
   - 关键索引。
   - 是否需要迁移 SQL。

4. 测试说明
   - 核心规则测试。
   - service 编排测试。
   - API 测试。
   - 如涉及数据库，补集成测试说明。

5. 配置说明
   - 新增环境变量。
   - 默认值。
   - 合法范围。
   - 是否敏感。

## 决策规则总结

可以按下面规则判断是否新建文件夹:

| 场景 | 建议 |
|---|---|
| 客户流失预警新增一个检测规则 | 先放现有 `engine.py` / `service.py` |
| 客户流失预警新增一个 API | 先放现有 `main.py` / `schemas.py` |
| 客户流失预警新增字段或状态 | 改现有模型、service、db |
| 完全新的并行业务功能 | 新建独立功能文件夹 |
| 新功能只有一两个文件 | 可先新建单个模块文件 |
| 新功能有 engine/service/schema/API 多层 | 新建文件夹 |
| 多个功能共享的工具 | 放公共模块或未来的 `app/common/` |

一句话结论:

> 如果是客户流失预警的延伸，先沿用现有分层；如果是和客户流失预警并行的新业务功能，应创建独立功能文件夹。
