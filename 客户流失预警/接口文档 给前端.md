**Assistant:**

分页的作用是**防止一次查询/传输的数据太大，把 CRM 或我们这边服务打爆**。

在这个项目里，风险主要在订单和订单明细：

- 一批 50 个销售员下面可能有很多客户。
- `frequency` 预警可能要拉客户历史订单，数据量不可控。
- `product` 预警要拉订单明细，一张订单多条明细，行数会放大。
- 如果一次返回几十万行，容易出现接口超时、数据库慢查询、内存占用过高、网络传输失败。
- 分页后可以分多次稳定拉取，比如每次 1000 或 5000 行，失败了也只重试当前页。

你可以这样跟 CRM 后端说：

> 分页主要是为了控制单次查询和返回数据量，避免订单/明细数据过大导致超时或内存压力。销售员我们这边会按批次传，比如 50 个销售员一批；但这一批下面的订单明细可能很多，所以订单和明细接口建议按 `page_size + cursor/page_token` 分页返回。客户列表数据量如果不大可以不强制分页，但订单/明细最好支持。

### 二、查询方式和参数

引擎对 CRM 数据**只读**，有以下几种查询模式：

#### 查询 1：获取全部销售ID
```sql
SELECT DISTINCT sales_id FROM crm_customers ORDER BY sales_id
```
- **无参数**，用于创建检测任务时按销售分批

#### 查询 2：按销售ID批量获取客户
```sql
SELECT customer_id, customer_name, sales_id, sales_name
FROM crm_customers
WHERE sales_id IN (:sales_ids)
```
- **参数**：`sales_ids` — 一组销售ID，每批最多 50 个（可配置）

#### 查询 3：按客户ID批量获取订单 + 明细（核心查询）
```sql
SELECT o.order_id, o.customer_id, o.sales_id,
       o.order_date, o.total_amount,
       i.product_id, i.product_name, i.amount
FROM crm_orders o
LEFT JOIN crm_order_items i ON o.order_id = i.order_id
WHERE o.customer_id IN (:customer_ids)
  AND o.status IN (:status_codes)
  AND o.order_date <= :until_date
  [AND o.order_date >= :since_date]   -- 可选
ORDER BY o.customer_id, o.order_date, o.order_id
```

| 参数 | 类型 | 说明 |

这取决于你们每个销售下面平均挂多少客户。我帮你算一下链路上的数据放大：

**一个批次的数据量放大链路：**

```
batch_size 个销售
  → 每个销售 × N 个客户
    → 每个客户 × M 笔订单
      → 每笔订单 × K 个明细行（产品检测时）
```

代码里的硬约束（`app/schemas.py:19`）：

```python
batch_size: int | None = Field(default=None, ge=1, le=200)
```

默认值 50，上限 200。

**实际建议要看你们的数据情况：**

| 每个销售平均客户数 | 建议 batch_size | 单批预估客户数 | 说明 |
|---|---|---|---|
| ≤ 10 | 50（默认值就行） | ~500 | 数据量不大，默认即可 |
| 10 ~ 30 | 20 ~ 30 | ~500 | 比较典型的 B2B 场景 |
| 30 ~ 100 | 5 ~ 10 | ~500 | 大销售团队，客户多 |
| > 100 | 3 ~ 5 | ~500 | 极端情况 |

核心原则是**控制单批的客户总数在 500 左右**，因为瓶颈在订单查询那条 SQL：

```sql

```

500 个客户 × 平均 20 笔订单 × 3 个明细 = 约 3 万行 JOIN 结果，MySQL 处理起来没压力。

**所以你问 CRM 后端的关键问题是：你们每个销售平均挂多少个客户？** 拿到这个数字后用 `500 ÷ 平均客户数` 就能算出合适的 batch_size，通过 API 参数或环境变量 `DEFAULT_BATCH_SIZE` 配置。
