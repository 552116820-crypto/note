它不是一个“单一模型”，而是一套可解释的规则链：

`历史订单 -> 周期预测(EWMA + IQR) -> 偏离判断 -> severity分级 -> 参数推荐`

其中：

- 频率异常、产品异常都用“周期预测”这条链
- 量级异常单独走“月度环比 + 双向阈值”这条链·

核心实现都在 [engine.py](/mnt/e/churn-warning-engine/app/engine.py#L98)，默认阈值在 [config.py](/mnt/e/churn-warning-engine/app/config.py#L8)。

**1. EWMA + IQR：先预测“正常周期”**
这部分用于频率异常和产品异常。

第一步是把订单日期排序去重，再算相邻两次订单的间隔天数：
```text
interval[i] = order_date[i+1] - order_date[i]
```
代码入口在 [predict_cycle](/mnt/e/churn-warning-engine/app/engine.py#L98)。

第二步是 IQR 去异常值，在 [_remove_iqr_outliers](/mnt/e/churn-warning-engine/app/engine.py#L38)：
```text
Q1 = 25%分位
Q3 = 75%分位
IQR = Q3 - Q1
保留区间 = [Q1 - 1.5*IQR, Q3 + 1.5*IQR]
```
作用是过滤掉极端间隔，比如：
- 春节停工导致特别长的间隔
- 补录/重复下单导致特别短的间隔

第三步是 EWMA 指数加权平均，在 [_ewma](/mnt/e/churn-warning-engine/app/engine.py#L57)：
```text
weights[i] = 2^i
predicted_days = Σ(interval[i] * weights[i]) / Σ(weights[i])
```
含义是“越近的间隔，权重越大”。  
例如间隔是 `[30, 30, 15]`，权重是 `[1, 2, 4]`，预测值会更靠近最近的 `15`，说明系统认为“客户最近下单变快了”。

第四步是置信度，在 [_prediction_confidence](/mnt/e/churn-warning-engine/app/engine.py#L72)：
```text
count_score = min(len(intervals) / 6, 1.0)
stability_score = max(0, 1 - CV)
confidence = 0.45 * count_score + 0.55 * stability_score
最终截断到 [0.1, 0.95]
```
这里的 `CV = 标准差 / 均值`。  
意思是：
- 样本越多，可信度越高
- 间隔越稳定，可信度越高

所以这个周期预测本质上是在回答一句话：

“基于历史订单，这个客户正常应该多久下一单？”

**2. frequency：整体下单频率异常**
逻辑在 [detect_frequency_anomaly](/mnt/e/churn-warning-engine/app/engine.py#L195)。

先得到：
- `predicted_days`：预测周期
- `days_since`：距离最近一单已经过去多少天
- `deviation_ratio = days_since / predicted_days`

触发条件是：
```text
days_since > predicted_days * frequency_multiplier
```
默认 `frequency_multiplier = 1.5`，[config.py](/mnt/e/churn-warning-engine/app/config.py#L12)。

例子：
- 预测周期 `20` 天
- 已经 `38` 天没下单
- 阈值 = `20 * 1.5 = 30`
- `38 > 30`，触发预警

这个异常的本质不是“绝对天数长”，而是“比这个客户自己的正常节奏慢很多”。

**3. product：关键产品停采异常**
逻辑在 [detect_product_anomaly](/mnt/e/churn-warning-engine/app/engine.py#L351)。

它比 frequency 多了一步“关键产品识别”：

```text
product_share = 产品金额 / 近 analysis_window_days 内总采购金额
```

只有 `share >= key_product_threshold` 的产品才进入检测。  
默认阈值是 `0.30`，也就是 30%，见 [config.py](/mnt/e/churn-warning-engine/app/config.py#L20)。

然后对每个关键产品单独做一遍上面的周期预测，再判断：
```text
days_since_last_product_order > predicted_days * product_multiplier
```

例子：
- 某产品占客户一年采购额 42%
- 该产品历史采购周期预测是 28 天
- 现在已经 45 天没买
- `45 > 28 * 1.4`，触发产品异常

所以 product 异常的含义是：

“客户不一定整体停采，但关键产品的采购节奏已经明显变了。”

**4. severity：把异常程度分成 low / medium / high**
这部分是为了运营优先级，而不是为了检测本身。  
实现分别在 [_severity_from_ratio](/mnt/e/churn-warning-engine/app/engine.py#L132)、[_severity_from_drop](/mnt/e/churn-warning-engine/app/engine.py#L141)、[_severity_from_rise](/mnt/e/churn-warning-engine/app/engine.py#L150)。

频率/产品看“偏离倍数”：
```text
ratio >= 2.0   -> high
ratio >= 1.5   -> medium
否则            -> low
```

量级下降看跌幅：
```text
drop >= 60% -> high
drop >= 40% -> medium
否则         -> low
```

量级上升看涨幅：
```text
rise >= 100% -> high
rise >= 60%  -> medium
否则          -> low
```

所以 severity 的本质是：

“已经异常了，再判断到底有多严重。”

**5. 参数推荐：系统顺手告诉你阈值该不该调整**
这部分不是预测未来，而是“根据这次检测结果，给配置调参建议”。

在 frequency 里，[engine.py#L219](/mnt/e/churn-warning-engine/app/engine.py#L219)：
```text
recommended_frequency_multiplier
= clamp(1.1 + (1 - confidence) * 0.8, 1.2, 2.0)
```
意思是：
- 置信度低，建议 multiplier 调高一点，减少误报
- 置信度高，建议 multiplier 可以更敏感一点

在 product 里，[engine.py#L400](/mnt/e/churn-warning-engine/app/engine.py#L400)：
```text
recommended_product_multiplier
= clamp(1.0 + (1 - confidence) * 0.7, 1.1, 1.8)
```

同时还会给关键产品占比阈值建议，[engine.py#L425](/mnt/e/churn-warning-engine/app/engine.py#L425)：
```text
recommended_key_product_threshold
= clamp(product_share * 0.9, 0.2, 0.5)
```

在 volume 里：
- 下降异常给 `recommended_volume_drop_threshold`，[engine.py#L298](/mnt/e/churn-warning-engine/app/engine.py#L298)
- 上升异常给 `recommended_volume_rise_threshold`，[engine.py#L316](/mnt/e/churn-warning-engine/app/engine.py#L316)

它们本质上都是：
“观察到本次异常幅度后，给一个更合理的阈值建议，方便人工调参。”

**6. 双向量级：不仅看跌，也看异常上涨**
逻辑在 [detect_volume_anomaly](/mnt/e/churn-warning-engine/app/engine.py#L256)。

先算：
```text
change_rate = (current_month_amount - previous_month_amount) / previous_month_amount
```

然后做双向判断：

```text
if change_rate < -volume_drop_threshold:
    下降异常
elif change_rate > volume_rise_threshold:
    上升异常
else:
    正常
```

默认：
- `volume_drop_threshold = 0.35`
- `volume_rise_threshold = 0.50`
见 [config.py](/mnt/e/churn-warning-engine/app/config.py#L14)

这比只看下降更完整，因为：
- 大跌通常代表流失风险、项目停工、竞品替换
- 暴涨不一定是好事，也可能是囤货、赶工、录入异常

所以双向量级的业务含义是：

“订单规模偏离正常水平，不管是向下还是向上，都值得看。”

**7. 这套算法的优点**
- 可解释，不是黑盒模型
- 对中小样本友好，不依赖大量训练数据
- 能适配“每个客户节奏不一样”的真实 CRM 场景
- 产出不只是告警，还会给出 `feature_summary`、`severity`、`parameter_recommendation`

**8. 你这版实现里有两个值得注意的技术点**
- 当前 IQR 实现会先 `sorted(values)` 再返回过滤结果，[engine.py#L47](/mnt/e/churn-warning-engine/app/engine.py#L47)。这会打乱“时间顺序”，从而削弱 EWMA 的“最近更重要”语义。严格来说，过滤后最好保留原始时间顺序。
- 当前量级比较的是“本月至 `biz_date` 的累计金额”对比“上月整月金额”，[engine.py#L269](/mnt/e/churn-warning-engine/app/engine.py#L269)。如果在月中跑，会天然更容易报下降，这需要业务上明确是否接受。

 EWMA + IQR通常不是一个固定单一算法名，而是两种方法的组合：

  - EWMA = Exponentially Weighted Moving Average
    中文：指数加权移动平均
  - IQR = Interquartile Range
    中文：四分位距

  它们常一起用于时间序列的平滑和异常检测：

  - EWMA：先把序列平滑，减弱短期噪声
  - IQR：再用四分位距设异常阈值，常见规则是：
    下界 = Q1 - 1.5*IQR
    上界 = Q3 + 1.5*IQR
