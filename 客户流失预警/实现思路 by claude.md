# 客户流失预警 — 实现思路

整理时间：2026-04-07

---

## 一、整体架构

### 数据流

```
CRM MySQL 只读库
  → Python 规则引擎（粗筛）
    → ML 模型打分（精排）
      → LLM 富化（仅命中候选）
        → Python 校验 / 落库
          → FastAPI 接口
            → CRM / PHP 展示
```

### 各层职责

| 层 | 职责 | 说明 |
|---|---|---|
| 规则引擎 | 粗筛异常候选 | 确定性规则，快速排除正常客户 |
| ML 模型 | 流失概率打分 | 捕捉规则难以覆盖的复杂模式 |
| LLM | 生成解释文案和挽回建议 | 不承担判定，只做表达层 |
| Python 校验 | 兜底、限制、落库 | 确保输出可控可追溯 |
| FastAPI | 对外提供标准接口 | CRM 只需调接口，边界清晰 |

### 方案优势

- 判定权始终在 Python（规则 + ML），稳定、可复盘、可评估
- LLM 只做解释和文案，不承担底层判断
- CRM 只需要调标准接口，边界清楚
- 后面要加规则、补跑、灰度、回测，都不会推翻架构

## 二、预警特征体系

### 核心特征清单

| 类别 | 特征 | 计算口径 |
|---|---|---|
| 频率类 | 下单频率月环比 | 近 30 天订单数 / 前 30 天订单数 |
| 频率类 | 下单频率季环比 | 近 90 天订单数 / 前 90 天订单数 |
| 频率类 | 最近一次下单距今天数 | 当前日期 - 最近一笔有效订单日期 |
| 金额类 | 客单价趋势 | 近 N 笔订单均价 vs 历史均价 |
| 金额类 | 月度消费金额环比 | 近 30 天消费 / 前 30 天消费 |
| 结构类 | 关键产品/高毛利产品占比变化 | 近期关键品 SKU 占比 vs 历史占比 |
| 结构类 | 产品宽度变化 | 近期采购 SKU 数 vs 历史 SKU 数 |
| 负面类 | 投诉/退款频率 | 近 N 天投诉或退款次数 |
| 负面类 | 退货率变化 | 近期退货金额占比 vs 历史 |

> 以上为初始特征集，上线后根据效果回测持续调整。

### 冷启动处理

- 设最低数据门槛：至少 **3 笔有效订单** 且 **历史跨度 >= 60 天** 才纳入预警
- 不满足门槛的新客户标记为"数据不足，暂不评估"，不输出预警结论
- 后续可对新客户单独建立"活跃度监测"规则（如首单后 N 天未复购）

## 三、三层识别逻辑

### 第一层：规则引擎（粗筛）

用确定性规则快速筛出异常候选，产出：

- 是否命中异常
- 异常类型（频率下降 / 金额萎缩 / 结构偏移 / 负面信号）
- 结构化特征快照
- 兜底参数推荐（当后续 ML 或 LLM 不可用时直接使用）

### 第二层：ML 模型（精排）

对规则命中的候选做流失概率打分：

- 模型选型：初期推荐 **XGBoost / LightGBM**，训练快、可解释性好
- 输入：规则引擎产出的结构化特征
- 输出：流失概率分值（0~1）+ 特征重要性 Top-N
- 按分值分级：高危 / 中危 / 关注

> 初期如果标注数据不足，可先跳过 ML 层，用规则引擎直接输出候选；积累足够正负样本后再训练模型。

### 第三层：LLM 富化（仅高危/中危）

只对命中异常的候选调用 LLM，输入结构化特征摘要，产出：

- 流失风险自然语言解释
- 针对性挽回建议
- 销售话术参考

---

## 四、LLM 调用规范

### 关键约束

- **不喂原始订单明细**，只喂结构化特征摘要
- **不是所有客户都调 LLM**，只对命中异常候选调用
- **LLM 不能改异常结论**，只能补充说明、建议、文案
- 落库时必须存：特征快照、模型输出、模型版本、Prompt 版本，方便追溯

### 容错与降级

| 场景 | 处理策略 |
|---|---|
| LLM 调用超时（>10s） | 重试 1 次，仍失败则使用兜底模板文案 |
| LLM 返回格式异常 | 校验失败，使用兜底模板文案，记录异常日志 |
| LLM 服务整体不可用 | 降级为纯规则 + ML 输出，跳过富化层，不阻塞预警产出 |
| 单日 Token 消耗超限 | 熔断 LLM 调用，剩余候选使用兜底模板 |

### 成本控制

- 预估单次调用 Token 量，设单日 Token 预算上限
- Prompt 精简，只传必要特征字段
- 批量调用时做并发限流，避免触发 API 限速

### Prompt 版本管理

- Prompt 文本入库，带版本号
- 上线新 Prompt 前先灰度（如 10% 候选用新版），对比输出质量后再全量切换

## 五、数据读取策略

### 结论

- **读取**：优先直连 MySQL 只读库
- **写回**：写自己的预警结果表，不侵入 CRM 核心业务表

### 为什么直连 MySQL

- 预警引擎是批量分析场景，不是单条业务操作，直接查库更快更稳
- 规则计算需要联表和批处理（下单周期、占比、环比），SQL/数据处理更擅长
- 走 CRM API 会遇到分页、限流、超时、字段不全等问题

### 必须满足的前提

- 连 **只读库/从库**，不扫生产主库
- 拿到明确的表结构和字段口径
- 确认业务真实口径（订单状态、是否作废、是否退款、明细金额、客户归属销售）
- 和后端确认表结构稳定性，避免字段变更导致引擎故障

### 数据流向

```
MySQL 只读库（读） → Python 引擎（算） → 预警结果表（写） → CRM/PHP（查/展示）
```

### 什么时候走 CRM API

- 数据库权限拿不到
- 表结构很乱，真实业务口径封装在 API 里
- 订单逻辑要经过 CRM 复杂业务层处理
- 团队明确禁止旁路读库

## 六、后端技术栈

### 框架选型：FastAPI

| 框架 | 判断 |
|---|---|
| **FastAPI** | 适合 API-first、强类型校验、自动 OpenAPI 文档、异步调用外部服务，最贴合当前链路 |
| Django | 适合整站后台，但已有 PHP CRM，能力重复 |
| Flask | 短期简单，长期缺规范易散 |
| Litestar | 可行但生态和团队认知不如 FastAPI 普及 |

### 选 FastAPI 的理由

- 系统本质是 API 服务，不是完整 CMS/后台站点
- 自动接口文档方便与 PHP CRM 联调
- 异步 IO 适合调大模型 API
- 后续扩展功能都可在同一项目中承载：预警日报接口、人工确认/误报反馈、参数配置、批量补跑、模型和 Prompt 版本审计

### 统一技术栈

| 职责 | 选型 |
|---|---|
| Web/API | FastAPI |
| ORM | SQLAlchemy 2.0 |
| 数据迁移 | Alembic |
| 数据校验 | Pydantic |
| 定时/异步任务 | 初期 cron + HTTP，复杂后升级 Celery / ARQ |
| ML | scikit-learn / XGBoost / LightGBM |
| 测试 | pytest |

## 七、效果度量与反馈闭环

### 核心指标

| 指标 | 定义 | 初期目标 |
|---|---|---|
| 召回率 | 实际流失客户中被预警命中的比例 | >= 70% |
| 准确率 | 预警客户中实际流失的比例 | >= 50% |
| 误报率 | 预警客户中未流失的比例 | <= 50% |
| 销售采纳率 | 销售收到预警后实际跟进的比例 | 持续观测 |

> 初期容忍较高误报率，优先保证不漏掉真实流失客户（召回优先）。

### 反馈闭环

1. 销售端提供"确认流失 / 误报 / 已挽回"标记入口
2. 反馈数据回流至预警引擎，用于：
   - 规则阈值调优
   - ML 模型重训练的标注数据
   - Prompt 优化依据
3. 定期（月度）做回测：用历史数据验证当前规则和模型的表现

### "流失"的定义

需要与业务方对齐，建议初始定义：

- **流失**：连续 N 天（如 90 天）无有效订单
- **流失前兆**：近期下单频率或金额显著下降但尚未完全停止

## 八、批处理与调度

### 初期方案：每日批跑

- 每日凌晨定时执行（cron 触发 HTTP 接口）
- 全量扫描符合数据门槛的活跃客户
- 产出当日预警列表，写入预警结果表
- CRM 端次日展示

### 后期演进

- 如需更高时效性，可引入事件驱动（如订单表变更触发增量计算）
- 但初期不建议上实时架构，批跑足够且复杂度低

---

## 九、运维与安全

### 接口认证

- Python 服务与 CRM 之间通过 **API Key + 内网白名单** 双重验证
- API Key 定期轮换

### 监控告警

| 监控项 | 告警条件 |
|---|---|
| 预警引擎执行状态 | 批跑失败或超时未完成 |
| LLM 调用成功率 | 低于 90% |
| 单次批跑耗时 | 超过预期时长的 2 倍 |
| 预警结果数量 | 突然为 0 或异常暴增（可能是数据源异常） |

### 数据保留

- 预警结果表保留 **12 个月**，过期归档
- LLM 原始输出保留 **6 个月**，用于 Prompt 调优和审计
- 特征快照随预警记录一起保留

## 附录：概念解释

### ML 模型是什么

ML 是 **Machine Learning（机器学习）** 的缩写。简单说：你给它大量历史数据（比如过去哪些客户流失了、流失前有什么特征），它自己从中**学出规律**，然后对新客户预测"这个客户有多大概率会流失"。

#### 和规则引擎的区别

| | 规则引擎 | ML 模型 |
|---|---|---|
| 逻辑来源 | 你手写的规则（如"60天没下单就报警"） | 从历史数据中自动学出来的 |
| 优点 | 直观、可控、不需要训练数据 | 能发现人想不到的复杂组合模式 |
| 缺点 | 只能捕捉你想到的情况 | 需要足够多的标注数据才能训练 |

**举个例子：**

- 规则引擎：你定义"月下单量下降50%就报警" — 这是你凭经验写的。
- ML 模型：你把过去一年所有客户的特征（下单频率、金额、产品类型...）和结果（流失/未流失）喂给它，它自己学出"下单频率下降30% + 高毛利产品占比减少 + 客单价微降 = 高危"这种**组合规律**，这些组合可能是你凭直觉想不到的。

#### 文档中提到的 XGBoost / LightGBM

这两个是 ML 模型的具体算法名称，都属于"梯度提升树"这一类，特点是：

- **训练快**，对数据量要求不算高
- **可解释**，能告诉你"哪个特征对判断影响最大"
- 在表格类数据（像订单/客户数据）上效果通常很好
- 是业界做客户流失预测最常用的算法之一

#### 在本项目中的定位

ML 层**初期可以先不上**。先用规则引擎跑起来，等销售标记了足够多的"确实流失 / 误报"数据后，再用这些数据训练 ML 模型，提升预警精度。所以它不是前期的阻塞项（对应实施节奏中的 P4 阶段）。
