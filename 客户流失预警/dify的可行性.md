# 客户流失预警系统 - 用 Dify 工作流实现的可行性分析

## 结论：技术上能做，但不推荐。这个系统不适合用 Dify 工作流实现。

---

## 原因分析

### 1. 系统本质是"数据处理管道"，不是"LLM 编排流"

整个系统的核心算法全是统计计算，没有一处用到 LLM：

| 模块   | 实际做的事                     | 需要 LLM？ |
| ---- | ------------------------- | ------- |
| 周期预测 | IQR去异常 → EWMA加权平均 → 置信度   | 不需要     |
| 量级检测 | 环比变化率 > 阈值？               | 不需要     |
| 产品分析 | Pandas 聚合 → 占比筛选 → 复用周期预测 | 不需要     |
| 预警引擎 | 天数 >= 阈值 × 倍数？            | 不需要     |
|      |                           |         |

Dify 的核心价值是 LLM 工作流编排，这里完全用不上。

### 2. Dify Code 节点的硬伤

算法依赖 NumPy（`np.percentile`, `np.average`, `np.std`）和 Pandas（`groupby`, `agg`），而 Dify 的沙箱化 Code 节点对第三方库支持有限。要么：
- 用纯 Python 重写所有数值计算（可行但退化严重，丧失可维护性）
- 用外部 API 包一层（那这层 API 就是现在的 FastAPI 服务本身，等于绕了个圈）

### 3. 批量处理不是 Dify 的长项

系统要遍历上万客户逐个计算。Dify 的迭代节点（Iteration）是为小批量设计的，大规模批处理会遇到：
- 超时限制
- 节点间变量传递的性能开销
- 无法控制并发和错误重试

### 4. 数据库交互笨重

有 6 个 SQL 查询函数（`get_active_customers`, `get_customer_order_dates` 等）+ 多处写入操作。Dify 没有原生 MySQL 节点，每次 DB 交互都要通过 HTTP Request 调外部接口，等于还是得维护一个后端服务。

### 5. 调度能力缺失

现在的 Celery 定时任务（周日凌晨预测、周一早上检测、月初量级检测）在 Dify 中没有直接对应物。

---

## 如果真想用 Dify，合理的方式是"结合"而非"替代"

保留现有 Python 服务做核心计算，用 Dify 做**增值层**：

| Dify 做什么 | 怎么做 |
|------------|-------|
| **智能预警消息生成** | 调用 LLM 把结构化预警数据转为更自然、个性化的话术，而非固定模板 |
| **销售问答助手** | 搭一个 Chatbot，销售问"客户A最近什么情况"，Dify 调 API 拿数据，LLM 整合回答 |
| **预警摘要日报** | 每天用 LLM 汇总当日预警，生成一段总结发到企微群 |
| **配置建议** | 销售想调倍数/阈值时，LLM 基于历史数据给建议 |

推荐架构：

```
CRM 数据库 ← Python 服务(算法+批处理) → 预警结果
                                            ↓
                                    Dify 工作流(LLM 增值)
                                            ↓
                                   企业微信/销售前端
```

---

## 总结

系统定位清晰、架构合理，用 Python 服务是正确选择。强行迁移到 Dify 会增加复杂度、降低性能，还用不到 Dify 最强的 LLM 能力。如果想引入 AI 增值，建议在现有系统之上叠加 Dify 做消息润色或问答入口。

## 追问：按销售ID拆分 + 调CRM接口，Dify能承接并发吗？

### 思路

把"1次跑1万客户"拆成"100个销售各跑100客户"：

```
触发(按销售ID) → 调CRM接口拿该销售的客户订单 → 迭代每个客户 → 算法计算 → 判断预警 → 回写/推送
```

### 迭代节点的实际性能

假设一个销售有 100 个客户，Dify Iteration 节点是**串行**执行的：

```
每个客户 ≈ 1次 HTTP 调 CRM + 1次 Code 计算 + 1次条件判断
        ≈ 1~3 秒/客户

100 个客户 ≈ 100~300 秒 ≈ 2~5 分钟/销售
```

单个销售勉强能跑完（不超时），但体验很差。

### 并发是最大的坑

假设有 100 个销售，同时触发 100 个工作流：

**自部署 Dify** 的瓶颈：
- Worker 进程数有限（默认 Celery worker 通常 4~8 个）
- 每个 workflow 的 Code 节点要启动沙箱容器，并发时 CPU/内存吃紧
- 100 个 workflow × 每个 100 次 HTTP 请求 = **10,000 次 CRM API 调用同时打过去**，CRM 大概率扛不住

**Dify Cloud** 的瓶颈：
- 有并发和调用频率限制（rate limit）
- 不支持自己控制并发度（比如"最多同时跑 10 个"）

**没有重试和错误隔离**：一个客户计算失败 → 整个迭代中断 → 该销售的所有后续客户跳过。Python 服务里有 `try/except` 逐个兜底，Dify 做不到这么细粒度。

### 对比

| 维度       | Python 服务          | Dify 按销售拆分                  |
| -------- | ------------------ | --------------------------- |
| 算法准确性    | NumPy 原生支持         | 要么重写要么绕回 API                |
| 1万客户耗时   | 2~5 分钟（文档写的）       | 100个销售 × 3分钟 = 不可控          |
| 并发控制     | Celery worker 可配   | 无法精细控制                      |
| 错误隔离     | 逐客户 try/except     | 迭代节点中断即停                    |
| 定时调度     | Celery Beat 原生支持   | 需外部 cron 触发                 |
| 运维复杂度    | 1 个服务              | Dify + CRM API + 可能还有外部计算 API |

## 追问：Dify 不支持 NumPy 吗？

### Dify Code 节点 + NumPy 的真实情况

| 问题                   | 结论                              |
| -------------------- | ------------------------------- |
| 默认支持 NumPy/Pandas？   | **不支持**，沙箱只带 Python 标准库         |
| 能加上吗？                | **能**，自部署的话自己构建 sandbox 镜像      |
| 运行时 pip install？     | **不行**，沙箱安全策略禁止                 |
| 有坑吗？                 | NumPy 有 C 扩展，可能需要调 seccomp 策略  |

### 具体做法（自部署 Dify）

构建自定义 `dify-sandbox` Docker 镜像，预装 NumPy：

```dockerfile
FROM langgenius/dify-sandbox:latest
RUN pip install numpy pandas
```

然后在 `docker-compose.yml` 里替换默认镜像，并确保沙箱配置中放开对应的包路径。

### 修正后的结论

如果愿意折腾自定义沙箱镜像，**算法本身能在 Dify Code 节点里跑**。之前说的"硬伤"不成立。

但剩下的问题仍然在：

- **批量迭代性能**：按销售拆分后串行遍历客户，慢但能跑
- **并发控制**：同时触发多个销售的 workflow，Dify 没有精细的并发管理
- **错误隔离**：迭代中某客户失败的处理不如 Python try/except 灵活
- **数据库交互**：仍需通过 HTTP 调 CRM 接口

核心算法这块，**如果肯花时间配沙箱环境，是可以搬进去的**。
