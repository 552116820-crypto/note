# 客户流失预警思路整理

整理时间：2026-04-07

## 会话 1：整体方案确认

### 我的问题

我整理了一下思路，你看可行吗？从 CRM 里拿数据，Python 做预警引擎，给出结构化特征，把这些特征喂给大模型，再然后 Python 校验落库，写接口返回给 CRM 后端。

### 结论

可行，而且这是推荐的主方案。

### 推荐链路

1. CRM 提供订单和客户数据
2. Python 预警引擎做三类异常识别，产出：
   - 是否命中异常
   - 异常类型
   - 结构化特征
   - 兜底参数推荐
3. 只有命中异常的记录，再把特征摘要喂给大模型
4. Python 对大模型输出做校验、兜底、落库
5. Python 提供接口给 CRM 后端查询和展示

### 方案优势

- 判定权在 Python，稳定、可复盘、可评估
- 大模型只做解释和文案，不承担底层判断
- CRM 只需要调标准接口，边界清楚
- 后面要加规则、补跑、灰度、回测，都不会推翻架构

### 关键约束

- 不要把原始订单明细直接喂给大模型，只喂结构化特征摘要
- 不是所有客户都调大模型，只对命中异常候选调
- Python 校验时要限制大模型不能改异常结论，只能补充说明、建议、文案
- 落库时要把特征、模型输出、模型版本、Prompt 版本一起存下来，方便追溯

### 最推荐的数据流

`CRM 数据 -> Python 规则引擎 -> 异常候选 -> LLM 富化 -> Python 校验/落库 -> CRM 查询接口`

## 会话 2：后端框架选型

### 我的问题

有什么后端框架可以选用，你推荐哪个，要保证后续功能开发也用同一套框架。

### 结论

推荐选 `FastAPI`，并把后续这类功能都统一在同一套 Python API 服务栈上开发。

### 可选框架判断

- `FastAPI`
  - 适合 API-first、强类型校验、自动 OpenAPI 文档、异步调用外部服务
  - 对“CRM 取数 -> 规则引擎 -> 调大模型 -> 落库 -> 给 PHP 返回接口”这条链路最贴合
- `Django`
  - 适合整站式后台，内置能力很多
  - 但当前已存在 PHP CRM，很多 Django 的优势会重复建设
- `Flask`
  - 适合极简项目
  - 短期简单，长期容易因为缺少规范而变散
- `Litestar`
  - 也能做
  - 但生态和团队认知通常不如 FastAPI 普及

### 为什么推荐 FastAPI

- 当前系统本质上是 API 服务，不是完整 CMS/后台站点
- 和 PHP CRM 联调时，自动接口文档非常省事
- 接 DeepSeek 或其他大模型 API 时，异步 IO 更自然
- 后续要加的能力都适合继续放在同一个 FastAPI 项目里：
  - 预警日报接口
  - 人工确认/误报反馈
  - 参数配置接口
  - 批量补跑接口
  - 模型和 Prompt 版本审计
- 比 Flask 更有规范感，比 Django 更轻，适合作为 AI/规则引擎侧车服务

### 建议统一的后端技术栈

- Web/API：`FastAPI`
- ORM：`SQLAlchemy 2.0`
- 数据迁移：`Alembic`
- 数据校验：`Pydantic`
- 定时/异步任务：
  - 简单定时先用 `cron + HTTP`
  - 后面复杂了用 `Celery` 或 `ARQ`
- 测试：`pytest`

### 最终建议

直接定：`FastAPI + SQLAlchemy + Alembic + Pydantic`

这套栈适合当前客户流失预警项目，也适合后续继续扩展相关功能。



• 通常是，读 MySQL 比调 CRM 接口更适合你这个预警引擎，但前提是读法要对。

  我建议的结论是：

  读取：优先直连 MySQL 只读库
  写回：优先写你自己的预警表，或通过后端接口回 CRM

  原因很直接：

  - 你这是批量分析，不是单条业务操作
    预警引擎要拉大量订单、明细、客户、产品数据，直接查库通常比反复调 API 更快、更稳、更便宜。
  - 规则计算需要联表和批处理
    下单周期、关键产品占比、月环比这些，本来就是 SQL/数据处理更擅长的场景。
  - API 往往更适合业务操作，不适合数仓式读取
    如果你用 API 拉全量历史订单，通常会遇到分页、限流、超时、字段不全、调用成本高这些问题。

  但要注意，不是“直接连生产 MySQL 就完事了”。你要满足这几个条件：

  - 最好连 只读库/从库，不要直接扫生产主库
  - 你要拿到明确的表结构和字段口径
  - 你要搞清楚哪些字段才是业务真实口径
    比如：
      - 订单状态
      - 是否作废
      - 是否退款
      - 明细金额口径
      - 客户归属销售
  - 你要和后端确认表结构是否稳定，避免他一改字段你就挂

  我更推荐你这样分：

  - 读：MySQL 只读库
  - 算：Python 预警引擎
  - 写：写你自己的预警结果表
  - 查：CRM/PHP 通过接口查你的预警表或结果接口

  最稳的模式是：
  MySQL 读 + Python 算 + MySQL 结果表 + PHP 展示

  只有在这些情况下，我才建议继续走 CRM API：

  - 数据库权限拿不到
  - 表结构很乱，真实业务口径封装在 API 里
  - 订单逻辑要经过 CRM 复杂业务层处理
  - 你们团队明确禁止旁路读库

  一句话：
  对你的场景，直接从 MySQL 读通常更好；但要读只读库，写自己的结果表，不要直接侵入 CRM 核心业务表。
