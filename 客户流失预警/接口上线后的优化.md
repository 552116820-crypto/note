### Turn 3  *(2026-04-13 08:49)*

**User:**

说说这几个问题的优先级

**Assistant:**

按上线风险排序：

**P0**
1. **API 模式生产环境启动不了**  
   `APP_ENV=production + REPOSITORY_TYPE=api` 会被 `app/main.py` 拒绝。这是阻断级问题：如果你准备生产环境直接切到 CRM API，这个必须先修。

2. **API 模式预警/任务落到内存仓储**  
   即使放开生产校验，`app/churn/factory.py` 现在只有 `REPOSITORY_TYPE=mysql` 才使用 `MySQLChurnRepository`，`api` 会走 `InMemoryChurnRepository`。这会导致服务重启后 job 和 warning 丢失，也不适合多进程/Celery。

**P1**
3. **API 超过 100 条只取第一页**  
   `CrmApiRepository` 看到 `total > len(data)` 只是打日志，不继续拉取。销售订单多一点就会丢历史订单，直接影响频率、量级、产品异常判断，属于结果正确性问题。

4. **只从订单行构造客户，会漏掉无订单客户**  
   如果 CRM 订单接口不返回“无订单客户”，那最近没下单的客户就不会被检测。对流失预警来说这很关键，因为最应该关注的客户可能正是没有近期订单的人。是否 P0 取决于 CRM 是否还有独立客户列表接口；如果没有，就是 P0 级业务缺陷。

**P2**
5. **API 数据长期缓存导致陈旧**  
   当前 service 被 `lru_cache` 缓存，API 仓储 `_loaded=True` 后不会重新拉数据。长时间运行的 API 服务或 Celery worker 会用旧订单。短期手动重启能绕过，但定时任务场景会出错。

6. **缺少 `httpx` 依赖声明**  
   新环境 `pip install -e .[dev]` 可能装不出 API 模式依赖。当前机器能跑只是因为环境里已有 `httpx`。这是部署可靠性问题，修复成本很低。

**P3**
7. **`test_crm_api.py` 不会被主测试收集，且依赖真实接口**  
   这不会直接影响线上运行，但会让 API 仓储没有可靠回归测试。建议改成 `tests/shared/test_crm_api.py` 里的 mock HTTP 测试；真实接口测试可以保留成手动脚本或打 marker。
