### Turn 2  *(2026-04-17 06:07)*

**User:**

我这边需不需要将拜访记录的数据存入mysql中

**Assistant:**

短答案：**现在不必须存 MySQL**，但生产上我更建议存一份“当前销售-客户归属快照”，而不是存完整拜访记录明细。

当前实现的逻辑是：

- `REPOSITORY_TYPE=api` 时，系统从订单接口拿订单客户；
- 再按 `sales_id` 调拜访记录接口；
- 只有在拜访记录里能匹配到的客户，才进入流失预警检测；
- 拜访记录结果只在当前进程内缓存，不落库。

所以如果你们现在直接用 CRM API 跑检测，**不需要额外建 MySQL 表也能工作**。

但下面几种情况建议落 MySQL：

1. **销售很多、跑批频繁**  
   每次跑批都实时调拜访接口，会增加 CRM 接口压力，也可能受限流、超时影响。

2. **希望检测结果可追溯**  
   比如以后问“为什么 2026-04-17 这天这个客户没有预警”，如果没有当时的归属快照，就只能依赖当前接口数据，无法复盘历史状态。

3. **生产环境主要用 `REPOSITORY_TYPE=mysql`**  
   当前 MySQL 仓储读的是 `crm_customers` / `crm_orders`。如果 `crm_customers` 里也混有历史归属客户，那就需要在 MySQL 里存当前归属并用于过滤，否则 MySQL 模式下仍可能把历史客户纳入检测。

推荐落库方式不是存完整拜访记录，而是同步一张轻量表，例如：

```sql
crm_sales_current_customers
- sales_id
- customer_id
- customer_name
- source
- synced_at
```

预警检测只需要判断：`customer_id/customer_name` 是否属于 `sales_id` 当前名下。完整拜访时间、拜访内容、图片等对流失检测没必要，存进去会增加维护成本和隐私风险。

我的建议是：

- **短期上线**：先用现在的实时接口校验，不存 MySQL。
- **生产稳定版**：加一个定时同步任务，把拜访接口转成“当前销售-客户归属快照”写入 MySQL，然后检测时用这张表过滤客户。
- **如果你们已经用 MySQL 作为 CRM 数据源**：建议尽快加这张归属快照表，或者确保 `crm_customers` 本身只包含当前归属客户。
