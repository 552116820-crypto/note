### Turn 1  *(2026-04-20 03:42)*

**User:**

这不就是异步处理吗

**Assistant:**

对，这就是异步处理。

准确说是两层：

```text
检测任务本身
-> 只负责生成预警、派发 AI 任务
-> 不等待 1265 个 DeepSeek 分析全部完成

AI 分析任务
-> 被丢进 Redis broker
-> Celery worker 后台慢慢消费
-> 每个 worker 调 CRM + DeepSeek + 落库
```

所以主流程是：

```text
product run completed
AI tasks dispatched
worker async processing in background
```

这就是为什么你能先看到：

```text
{'status': 'dispatched', 'customer_count': 1265}
```

但 `churn_ai_analyses_count` 不是立刻变成 `1265`。因为 `dispatched` 只表示任务已经投进队列，不表示 DeepSeek 已经全部分析完成。

当前实际状态是：

```text
Redis 队列里还有任务
Celery 同时跑 4 个客户分析
每个客户分析完成后才写 1 条 churn_ai_analyses
失败或超时的任务会 retry
```

所以是异步的，只是异步任务本身还要做几件慢事：

```text
拉 CRM 数据
拉拜访记录
组装上下文
调用 DeepSeek
等待模型返回
写库
```

总结一句：

```text
派发是快速同步完成的；AI 分析是在后台异步慢慢完成的。
```

### Turn 3  *(2026-04-20 03:52)*

**User:**

API 容器是什么

**Assistant:**

API 容器就是跑 FastAPI 服务的那个 Docker 容器。

在这个项目里，`docker-compose.yml` 里有几个服务：

```text
redis   -> Redis broker，存 Celery 任务队列
celery  -> 后台任务 worker，消费 AI 分析任务、调用 DeepSeek、写库
api     -> FastAPI HTTP 服务，对外提供接口
nginx   -> HTTP 入口，转发请求到 api
```

对应关系是：

```text
用户/前端
  -> nginx
  -> api 容器
  -> FastAPI app.main:app
  -> 调用 service/repository
```

`api` 容器主要负责这些接口：

```text
GET /healthz
GET /docs
POST /api/warnings
GET /api/warnings
POST /api/warnings/ai-analysis
GET /api/warnings/ai-analysis/latest
```

你现在这批 product run 的后台 AI 派发和分析，主要是在 `celery` 容器里跑，所以我们一直在重启/重建 `celery`。

什么时候需要重建 API 容器？

如果你改了这些内容：

```text
FastAPI 路由
接口入参/出参
service 逻辑
AI 分析接口同步调用逻辑
提示词代码，并且前端会直接调 POST /api/warnings/ai-analysis
```

就要重建 `api`。

如果只是让后台队列里的 AI 任务继续跑，通常只需要重建：

```bash
docker compose build celery
docker compose up -d --force-recreate celery
```

如果你希望 HTTP API 和后台 worker 都用同一套最新代码，就一起重建：

```bash
docker compose build api celery
docker compose up -d --force-recreate api celery
```

查看当前有哪些容器：

```bash
docker compose ps
```

你会看到类似：

```text
churn-warning-engine-redis-1
churn-warning-engine-celery-1
churn-warning-engine-api-1
churn-warning-engine-nginx-1
```

如果当前只启动了 `redis` 和 `celery`，那说明 HTTP API 服务没跑，后台任务仍然可以继续跑。
