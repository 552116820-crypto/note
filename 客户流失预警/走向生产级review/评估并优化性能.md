**User:**

MySQL 预警仓储有明显的 N+1 读放大。这是什么意思

**Assistant:**

意思是“本来可以几条 SQL 读完的数据，现在变成先查一批主记录，再为每条记录额外发很多次查询”，数据库往返次数被成倍放大了。

在你这个项目里，典型位置是 [app/churn/mysql_repo.py](/mnt/e/churn-warning-engine/app/churn/mysql_repo.py:793) 的 `list_active_warnings_for_customers()`。它先查出一批 `warning_id`，然后：

- 对每个 `warning_id` 再调用一次 `get_warning()`
- 而 `get_warning()` 本身又不是一条简单查询，它会先拿长度，再对 `feature_summary`、`parameter_recommendation`、`reason`、`sales_message`、`metadata` 这些长文本列做分段读取，[app/churn/mysql_repo.py](/mnt/e/churn-warning-engine/app/churn/mysql_repo.py:616)、[app/churn/mysql_repo.py](/mnt/e/churn-warning-engine/app/churn/mysql_repo.py:211)

所以如果一次要读 100 条 warning，不是“1 次批量查询”，而更像：

- 1 次查 ID 列表
- 100 次 `get_warning()`
- 每条 warning 再拆成多次 `SUBSTRING` 查询

这就叫 N+1，甚至在这里接近 “1 + N x 多次子查询”。

为什么它不好：
- 延迟高，因为数据库往返次数很多
- 并发差，请求一多连接池容易被打满
- 数据量越大越慢，不是线性地“多一点”，而是被 round-trip 成本放大

一句话概括：不是 SQL 算得慢，而是“查太碎、查太多次”。如果你愿意，我下一步可以直接把这个点改成批量读取，先把 `list_active_warnings_for_customers()` 从 N+1 收敛成 1 到 2 条 SQL。

你这里在 [app/churn/tasks.py](/mnt/e/churn-warning-engine/app/churn/tasks.py:33) 的 `_customer_keys_for_warning_ids()` 里，为了把 `warning_ids` 去重成 `(sales_id, customer_id)`，做的是：

- 结果变成了 200 次详情查询
