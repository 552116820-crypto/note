### Turn 1  *(2026-04-20 03:27)*

**User:**

接口的请求体的customer_id字段要不要换成字符串

**Assistant:**

我先看一下当前接口 schema、模型和仓储里 `customer_id` 的类型流向，再判断改成字符串会不会影响兼容性和数据库查询。

目前搜索结果显示几乎整条链路都把 `customer_id` 当 `int`，但 CRM API 内部有一些“拜访客户标识”做了字符串归一化。我再看具体请求体和入口，确认这里说的是哪个接口。

从代码结构看，`customer_id` 不是只在请求体里出现：它参与客户匹配、订单查询、预警去重、AI 分析缓存索引，并且 MySQL 迁移列类型也是 `INT`。我快速验证一下 Pydantic 当前是否已经能接受数字字符串，避免误判前端兼容性。

这里环境里没有 `python` 命令，我换成 `python3` 验证同一件事。

结论：**现在不建议只把请求体里的 `customer_id` 改成字符串**。

当前项目里 `customer_id` 是一条完整的 `int` 链路：请求 schema 是 `int`，领域模型是 `int`，仓储接口是 `int`，MySQL 字段也是 `INT`。例如 [schemas.py](/mnt/e/churn-warning-engine/app/churn/schemas.py):94、[models.py](/mnt/e/churn-warning-engine/app/churn/models.py):36、[shared/models.py](/mnt/e/churn-warning-engine/app/shared/models.py):9、[000_initial_schema.sql](/mnt/e/churn-warning-engine/migrations/mysql/churn/000_initial_schema.sql):16、[007_churn_ai_analyses.sql](/mnt/e/churn-warning-engine/migrations/mysql/churn/007_churn_ai_analyses.sql):6 都是按数字客户 ID 设计的。

而且当前 Pydantic 已经能接受请求体里传字符串数字，例如：

```json
{
  "sales_id": "000455",
  "customer_id": "115849"
}
```

会被解析成内部的 `int 115849`。我刚验证了 `ChurnAiAnalysisRequest` 和 `WarningQueryRequest` 都是这个行为。所以如果只是前端表单或 JSON 里把它当字符串传，**后端现在已经兼容，不需要改接口契约**。

需要改成字符串的情况主要有三种：

1. CRM 的 `customer_id` 可能有前导零，比如 `"000115849"`。
2. CRM 的 `customer_id` 可能包含非数字字符。
3. 客户 ID 可能超过 MySQL `INT` 或 JavaScript 安全整数范围。

如果满足这些情况，就不应该只改请求体，而应该成套改：schema、响应模型、共享模型、仓储接口、MySQL 字段类型、索引、迁移脚本、测试数据都要从 `int` 迁到 `str` / `VARCHAR`。否则会出现请求进来了，但客户匹配、订单查询、缓存命中、预警去重不一致的问题。

我的建议是：**短期保持 `customer_id: int`；前端可以传数字或数字字符串。只有确认 CRM 客户 ID 本质上不是纯数字 ID 时，再做一次完整迁移到字符串。**
