### Turn 3  *(2026-04-24 07:55)*

**User:**

active_slot这一列是什么，有必要吗

**Assistant:**

我看一下 `active_slot` 的使用点和建表/迁移逻辑，再直接判断它现在是否还有保留价值。

`active_slot` 是内部去重字段，不是业务字段。

它的作用是：保证同一个客户、同一种预警、同一个去重 key 同时只能有一条“活跃预警”。

当前规则是：
- `pending_enrich`、`enriched`：`active_slot = '1'`
- `resolved`：`active_slot = NULL`

配合唯一约束：

```sql
UNIQUE (customer_id, anomaly_type, dedup_key, active_slot)
```

这样 MySQL 会做到：
- 活跃状态只能有一条，因为 `active_slot = '1'` 会冲突
- 已解决历史可以保留多条，因为 MySQL 的唯一索引允许多个 `NULL`

所以它解决的是 MySQL 没有 partial unique index 的问题。等价需求在 PostgreSQL 里可以写成：

```sql
CREATE UNIQUE INDEX ...
WHERE status IN ('pending_enrich', 'enriched')
```

但 MySQL 不能这么写，所以用 `active_slot` 做 workaround。

我建议保留。否则会有两个风险：
- 定时任务重复跑或并发跑时，可能插入重复活跃预警
- 如果直接对 `(customer_id, anomaly_type, dedup_key)` 做唯一约束，又会不允许保留历史 resolved 记录

如果要删，也不是简单删列，需要重做去重方案，比如改成 MySQL generated column 或放弃历史多条记录。当前项目结构下，`active_slot` 是有必要的。
