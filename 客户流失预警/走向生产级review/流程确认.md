当前链路是：

1. `celery_app` 定时触发三个任务类型：

   - `product`：每周一 08:10
   - `volume`：每月 1 日 08:00，业务日期解析为上一个完整自然月月末

2. `run_detection(job_type)` 启动后调用 `service.start_job()`：
   - 先从两个 CRM 配置接口同步客户级配置和全局配置
   - 再准备 CRM 订单窗口
   - 获取销售 ID
   - 按 `DEFAULT_BATCH_SIZE` 分批创建 job

3. Celery 逐个 batch 调用 `service.detect_batch()`：
   - 读取该批销售名下客户
   - 批量读取订单接口数据
   - 加载该批客户的生效配置
   - 按当前 `job_type` 跑对应算法：`frequency` / `volume` / `product`

4. 算法产出的预警会先持久化到 MySQL：
   - 新预警写入 `warning_records`
   - 已有 `pending_enrich` 刷新
   - 已有 `enriched` 保留并返回 ID
   - 未再次命中的活跃预警置为 `resolved`

5. 持久化成功后，调用 `saveCustomerWarn` 推送 CRM：
   - `batch_size=50`
   - `concurrency=20`
   - `timeout=30`
   - `max_retries=3`

6. 所有 batch 跑完后，Celery 保存 `detection_runs` 汇总：
   - `total_customers`
   - `total_anomalies`
   - `warning_ids`
   - `failed_batches`
   - `status=completed/partial`

7. 最后如果 AI 分析开启，会按 `warning_ids` 去重到客户维度派发 AI 分析任务；这一步不影响预警持久化和 CRM 推送主流程。

需要注意一个行为：单个 batch 失败时，Celery 当前策略是记录到 `failed_batches` 并继续跑后续 batch，不会在同一次 `run_detection` 内自动重试该 batch。整体任务启动失败才会走 Celery `max_retries=3`。如果 CRM 推送失败，因为发生在 MySQL 持久化之后，该 batch 会被标记为失败，但数据已经落库。
