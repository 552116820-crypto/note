1. 给所有预警类型加一个业务日历/停工窗口机制，比如配置 `SPRING_FESTIVAL_PAUSE_START`、`SPRING_FESTIVAL_PAUSE_END`、`HOLIDAY_GRACE_DAYS`，计算 `days_since` 时扣除或顺延这些非采购期。

3. volume 不应只做上月环比。更合理的是按客户/产品的预测使用周期选择比较窗口：产品周期约 60 天就比较最近 60 天与上一段 60 天，或和历史同等周期基线比较；周期不足或没有产品明细时再回退到月环比。

4. volume 最好拆到产品组维度或至少按关键产品加权。客户总 quantity 下降可能是某个产品周期没到，也可能是产品结构变化，单看总量会丢信息。

### Turn 2  *(2026-04-18 01:10)*

**User:**

给所有预警类型加一个业务日历/停工窗口机制，这是什么

**Assistant:**

“业务日历/停工窗口机制”就是：系统不要把所有自然日都当成客户应该正常下单的日子，而是显式告诉算法哪些日期属于行业性停工、长假、春节影响期，然后在判断“逾期多少天”时做修正。

举个简单例子：

客户正常 30 天采购一次，最后一次下单是 `2026-01-20`。

如果按自然日算，到了 `2026-03-10`：

```text
距离上次下单 = 49 天
预测周期 = 30 天
```

如果预警阈值是 `1.5 倍周期`，阈值就是 `45 天`，系统会认为已经逾期，触发预警。

但制造业春节可能从 `2026-02-10` 到 `2026-02-24` 基本停工 15 天。那这 15 天其实不应该全算作“客户异常未采购”。修正后可以这样算：

```text
自然天数 49 天
扣除春节停工 15 天
有效经营天数 = 34 天
```

这样就不会触发预警。

在代码里，这个机制通常包括三部分：

1. **配置停工窗口**

比如：

```env
BUSINESS_PAUSE_WINDOWS=2026-02-10:2026-02-24:春节停工
```

或者更结构化一点：

```json
[
  {
    "name": "春节停工",
    "start_date": "2026-02-10",
    "end_date": "2026-02-24",
    "industry": "manufacturing"
  }
]
```

2. **计算有效逾期天数**

新增一个工具函数，不直接用：

```python
days_since = (biz_date - last_order_date).days
```

而是用类似：

```python
effective_days_since = business_days_between(
    last_order_date,
    biz_date,
    pause_windows=settings.business_pause_windows,
)
```

这里的 `business_days_between` 不一定是传统工作日概念，而是“扣掉停工窗口后的经营天数”。

3. **所有预警类型统一使用这个口径**

目前项目里：

- `frequency` 用 `days_since_last_customer_order`
- `product` 用 `days_since_last_product_order`
- `volume` 用本月和上月自然月 quantity 环比

加业务日历后：

- `frequency`：判断客户多久没下单时，扣掉春节停工期。
- `product`：判断关键产品多久没补货时，扣掉春节停工期。
- `volume`：如果比较窗口包含春节停工期，要么顺延比较窗口，要么把春节月份标记为特殊月份，降低或暂停环比预警。

它解决的问题是：**春节这种所有客户、所有产品、整个行业都会受影响的低交易期，不应该被算法误判成客户流失。**

我说“给所有预警类型加这个机制”，意思不是只在 AI 分析里提示“可能受春节影响”，而是在预警触发前就把春节停工影响纳入计算。这样预警数量会更干净，销售看到的也更像真实风险。

法定春节假期可能是 8 天、9 天，但工厂、物流、上下游复工往往更长。这个更像行业规则，不是国家统一法定日历。所以可以自动生成一个制造业停工窗口：

```text
春节停工窗口 = 官方春节假期 + 节前缓冲天数 + 节后缓冲天数
```

例如：

```env
HOLIDAY_CALENDAR_COUNTRY=CN
INCLUDE_STATUTORY_HOLIDAYS=true
SPRING_FESTIVAL_INDUSTRY_PAUSE_DAYS=15
SPRING_FESTIVAL_PRE_BUFFER_DAYS=3
SPRING_FESTIVAL_POST_BUFFER_DAYS=3
```

或者更直接：

```text
如果官方春节假期不足 15 天，则按官方春节假期中心向前/向后扩展到至少 15 天。
```

我建议项目里分三层：

1. **国家法定节假日层**  
自动按年份加载，不让你手工写每个窗口。包含假期日、调休上班日。

2. **行业停工层**  
制造业默认春节按至少 15 天处理。这个需要一个可调参数，因为不同公司、地区、产品线复工节奏不一样。

3. **客户/产品例外层**  
可选。比如某些客户是连续生产企业，春节不停工；某些客户节后复工慢，可以单独覆盖。

所以答案是：**法定节假日应该自动生成或自动导入；春节制造业长停工建议自动生成默认窗口，但保留配置项。**  
完全不配置也能跑，用“中国法定节假日 + 制造业春节默认 15 天”；配置只是为了把默认规则调得更贴近你们行业。

**frequency 改法**

当前 frequency 是：

```text
days_since = biz_date - last_order_date
threshold = predicted_cycle_days * FREQUENCY_MULTIPLIER
days_since > threshold 就预警
```

建议改成：

```text
effective_days_since = 自然天数 - 中间重叠的停工/节假日天数
adjusted_estimated_order_date = 预测下单日如果落入停工期则顺延
adjusted_threshold_date = 预警阈值日如果落入或跨过停工期则顺延
biz_date > adjusted_threshold_date 才预警
```

这样客户上次下单到当前日期如果跨春节，不会把春节停工期全部算成“客户没下单”。

同时，历史周期训练也建议用业务日历修正一次：历史两次下单间隔如果跨春节，可以用“有效经营天数”代替纯自然天数。IQR 仍保留，但不再把春节只当统计异常值处理。

**product 改法**

product 当前已经比较接近正确方向：它按关键产品单独预测周期，并且支持 quantity 折算使用周期。这里要补的是“当前逾期窗口春节顺延”。

建议和 frequency 使用同一套业务日历：

```text
last_product_order_date -> biz_date
计算 effective_days_since
预测补货日、预警阈值日都经过停工窗口顺延
```

产品周期预测里也建议把历史采购间隔改成有效经营天数，尤其是春节前后补货产品，否则春节会把产品使用周期拉长或造成误报。

feature_summary 里建议新增字段：

```json

  "effective_days_since_last_product_order": 57,
  "holiday_pause_days": 15,
  "applied_pause_windows": ["春节制造业停工"],
  "adjusted_warning_threshold_date": "2026-03-18"
}
```

这样销售能看懂为什么没有触发，或者为什么延后触发。

**volume 改法**

volume 是最需要调整的。当前只比较本月和上月正向 quantity，确实偏粗。

建议改成两层判断：

第一层：客户总量级趋势，保留但降级为兜底。

```text
如果客户/产品缺少足够历史周期，继续使用完整自然月环比。
但如果当前月或上月包含长假，自动降低置信度，或跳过本月环比预警。
```

第二层：产品周期窗口比较，作为主要逻辑。

```text
先识别客户的关键产品组
对每个关键产品预测使用周期，比如 60 天
比较最近 60 天采购量 vs 上一个 60 天采购量
如果明显下降，再触发产品组 volume 预警
```

也就是：

```text
product_cycle_days = 60
current_window = biz_date 往前 60 个有效经营天
previous_window = current_window 前一个 60 个有效经营天
change_rate = (current_qty - previous_qty) / previous_qty
```

这比“4 月 vs 3 月”更符合使用周期。比如客户两个月采购一次，用月环比会经常误报，用 60 天窗口就合理很多。

```text

```

去重键也要跟着变：

```text

volume:product:<product_group_key>:down

```

**节假日对 volume 的特殊处理**

volume 不建议简单扣天数，因为采购不是每天均匀发生的。更合理的是：

```text
如果比较窗口内停工天数占比过高，比如超过 25%，则：
1. 延长窗口，凑足同等有效经营天数；
2. 或降低预警等级；
3. 或标记为 holiday_adjusted，不直接生成强预警。
```

我更建议第一种：窗口用“有效经营天数”对齐，而不是固定自然月。
