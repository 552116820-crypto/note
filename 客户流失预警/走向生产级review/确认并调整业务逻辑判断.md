我的方案是把 `product` 的“产品识别”从单一 `product_id` 改成“产品组”识别，而不是简单把 key 从 `product_id` 换成 `product_name`。

**核心规则**

同一个客户的订单明细里，只要满足以下任一条件，就归到同一个产品组：

1. `product_id` 相同
2. 标准化后的 `product_name` 相同
3. 通过传递关系能连起来  
   例如：
   - A：`product_id=P1`, `product_name=水泥A`
   - B：`product_id=P2`, `product_name=水泥A`
   - C：`product_id=P2`, `product_name=水泥新版`

   那么 A、B、C 都应该归为同一个产品组。

**实现方式**

我会在 [engine.py](/mnt/e/churn-warning-engine/app/churn/engine.py) 的 `_positive_product_activity()` 里做产品组聚合：

- 保留正向明细过滤：负金额、`quantity < 0` 的退款/冲销单不参与。
- 建两个索引：
  - `product_id -> group`
  - `normalized_product_name -> group`
- 遍历每条正向订单明细时：
  - 如果 `product_id` 命中已有组，使用该组。
  - 如果 `product_name` 命中已有组，使用该组。
  - 如果两个都命中但指向不同组，就把两个组合并。
  - 如果都没命中，就创建新产品组。
- 产品组内累计：
  - 正向采购金额，用于关键产品占比。
  - 下单日期，用于周期预测。
  - `quantity`，用于现有的 quantity 周期预测。
  - 所有关联过的 `product_ids` 和 `product_names`，用于排查和前端展示。

**字段和兼容性**

我建议保留现有字段：

```json
"product_id": "...",
"product_name": "..."
```

但语义调整为“产品组的代表 ID / 代表名称”。同时新增：

```json
"product_ids": ["P1", "P2"],
"product_names": ["水泥A", "水泥新版"],
"product_group_key": "..."
```

这样前端和旧逻辑还能继续读 `product_id`、`product_name`，但你也能看到这个预警到底合并了哪些产品编码和名称。

**去重规则**

当前 `product` 预警去重按 `product_id`。改成产品组后，我会把去重 key 改成基于 `product_group_key`，否则同名不同 ID 仍可能重复预警。

`product_group_key` 我建议用确定性规则生成，例如：

- 优先使用产品组内排序后的 `product_id` 列表
- 再加排序后的标准化 `product_name` 列表
- 拼成稳定字符串后生成短 key

这样同一组产品在不同检测批次里能尽量稳定去重。

**名称标准化**

`product_name` 不能直接裸比较，我会做轻量标准化：

- 去掉首尾空格
- 合并连续空白字符
- 英文字母做大小写归一
- 不做激进替换，比如不自动删除型号、规格、括号内容

原因是产品名里型号和规格可能有业务含义，过度清洗容易把不同产品误合并。

**测试补充**

我会加这些回归测试：

1. 相同 `product_id`、不同 `product_name`，应合并为同一产品组。
2. 不同 `product_id`、相同 `product_name`，应合并为同一产品组。
3. 通过传递关系连接的多个产品，应合并为同一产品组。
4. 合并后的金额占比达到阈值时，应识别为关键产品。
5. 合并后的订单日期应一起参与周期预测。
6. `quantity=-1` 或负金额退款明细，不参与产品组活动，也不重置周期。
7. 预警 `feature_summary` 包含 `product_ids`、`product_names`、`product_group_key`。

这个方案能解决你说的“同名不同 ID 被拆开”的问题，同时也覆盖“同 ID 改名”的场景，避免简单按名称或简单按 ID 带来的漏判。
