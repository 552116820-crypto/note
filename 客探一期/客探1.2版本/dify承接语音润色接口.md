		  # 用 Dify 承接语音文本润色接口

  ## Summary

  保留你们现有的小程序录音和腾讯云 ASR 流程，只把 ASR 产出的 rawText 发给你们自己的后端，再由后端转调 Dify。Dify 负责“规则后润色”或“直接润色”，后端只做鉴权、场景参数传递、结果兜底和日志留存。

  最短路径是：

  - 小程序不直接调 Dify
  - 由你们后端封装一个 POST /speech/polish
  - 后端调用 Dify Workflow 或 Chatflow
  - 返回优化稿给前端

  ## Key Changes

  ### 1. 推荐架构

  调用链建议固定为：

  1. 小程序录音并拿到腾讯云 ASR 文本
  2. 小程序调用你们后端 POST /speech/polish
  3. 后端把 rawText + scene 发给 Dify
  4. Dify 返回润色结果
  5. 后端做一次结果校验后返回前端

  这样做的原因：

  - Dify 的 Key 不暴露给小程序
  - 便于以后替换模型或提示词
  - 便于加敏感词、审计、失败回退

  ### 2. Dify 里怎么搭

  最短实现优先用一个 Workflow。

  输入参数：

  - rawText
  - scene，值为 work_log 或 visit_record

  Workflow 最小节点：

  1. 开始节点：接收文本和场景
  2. LLM 节点：执行润色
  3. 结束节点：输出 polished_text

  提示词核心约束：

  - 删除语气词、口头禅、重复词
  - 补全标点和断句
  - 改成适合业务日志/拜访记录的客观表达
  - 不允许新增原文中没有的事实
  - 不允许修改客户名、时间、金额、产品名、地址、电话等关键实体

  ### 3. 后端最少要做的事

  即使用 Dify，也不要把逻辑全丢给它。后端至少保留：

  - 登录态和接口鉴权
  - 调 Dify
  - 超时/失败回退
  - 保存 rawText 和 polishedText
  - 对关键字段做基础校验

  回退策略建议：

  - Dify 失败时，直接返回原文或规则清洗稿
  - 如果发现数字、日期、客户名被改动，丢弃 Dify 结果并标记人工确认

  ### 4. 什么时候适合用 Dify

  适合：

  - 你们想快速上线
  - 提示词会频繁调整
  - 希望运营或产品也能参与调试话术
  - 当前量不算特别大

  不太适合直接一步到位重度依赖 Dify 的情况：

  - 对延迟极其敏感
  - 并发很高
  - 合规要求必须完全内网闭环
  - 需要非常细粒度的代码级控制

  ## Public Interfaces

  建议你们自己对前端暴露一个稳定接口：

  POST /speech/polish

  请求：

  - rawText: string
  - scene: "work_log" | "visit_record"

  响应：

  - text: string
  - needReview: boolean

  Dify 对你们是内部实现细节，不直接暴露给小程序。

  先验证这几类输入：
  1. 大量语气词和停顿词
  2. 没有标点的一长句
  3. 含客户名、时间、金额
  4. 拜访记录场景
  5. Dify 超时或返回异常内容

  验收标准：

  - 文本明显更通顺
  - 不新增事实
  - 关键实体不被改坏
  - Dify 故障时前端仍可正常提交

  ## Assumptions

  - 腾讯云 ASR 已经能稳定产出文本
  - 当前目标是最快上线，不是最终极架构
  - 默认由你们后端转调 Dify，而不是小程序直连

  结论很直接：能做，而且是你现在最省事的方案。




  # Dify 语音文本润色落地方案

  ## Summary

  保留现有腾讯云 ASR 链路，在后端新增 POST /speech/polish，由后端转调 Dify Workflow 完成润色。Dify 只负责“文本优化”，
  不直接暴露给小程序。

  默认决策：

  - Dify 用 Workflow
  - 一个工作流先支持两个场景：work_log、visit_record
  - 前端只拿最终优化稿
  - 后端负责鉴权、超时兜底、结果校验、日志留存
  - 润色边界固定为“只整理表达，不补事实”

  ## Dify 设计

  ### Workflow 输入

  定义两个入参：

  - rawText: string
  - scene: string

  scene 只允许：

  - work_log
  - visit_record

  ### Workflow 节点

  最短可用版本只要 3 个节点：

  1. 开始节点
     接收 rawText、scene
  2. LLM 节点
     根据 scene 和提示词输出 polished_text
  3. 结束节点
     返回 polished_text

  如果想多一层稳妥，升级成 5 个节点：

  4. 开始节点
  5. 代码/模板节点
     拼出场景专属 system prompt
  6. LLM 节点
     执行润色
  7. 代码节点
     做基础校验，判断是否疑似改坏关键实体
  8. 结束节点
     返回 polished_text、needReview

  ### 模型建议

  先用一个稳定中文能力较好的通用模型，不做复杂多模型编排。默认要求：

  - 中文口语转书面效果稳定
  - 成本可控
  - 响应延迟适中

  Dify 内部参数默认：

  - temperature: 0.2
  - top_p: 0.8
  - max_tokens 按你们日志长度预留
  - 禁止高随机性配置

  ## Prompt 模板

  ### System Prompt

  你是一个业务语音转文字整理助手。

  任务：
  将语音识别出来的中文口语文本，整理成适合“业务员工作日志/拜访记录”填写的正式文本。

  必须遵守：
  1. 只能整理表达，不能新增、猜测、补充原文中没有的事实。
  2. 不得修改客户名、联系人、时间、日期、金额、产品名、型号、地址、电话等关键实体。
  3. 删除语气词、口头禅、无意义重复，例如：嗯、啊、额、那个、这个、然后、就是说、你知道吧。
  4. 修正断句、标点、空格，使表达通顺自然。
  5. 保持原文核心意思不变。
  6. 不输出解释，不输出分析，只输出整理后的最终文本。
  7. 如果原文信息过少或混乱，也只能尽量整理，不能自行补全结论。

  ### User Prompt

  场景：{{scene}}

  原始文本：
  {{rawText}}

  请输出适合该场景填写的优化文本。

  ### 场景约束补充

  如果 scene = work_log，附加约束：

  输出风格要求：
  简洁、客观，适合填写工作日志。

  如果 scene = visit_record，附加约束：

  输出风格要求：
  适合填写拜访记录，强调沟通内容、客户反馈、后续动作，但只能基于原文整理，不能补充未提到的信息。

  ## 后端接口

  ### 对前端暴露的接口

  POST /speech/polish

  请求体示例：

  {
    "rawText": "嗯今天上午去拜访了王总 然后聊了下下个月采购计划 他这边说先看预算",
    "scene": "visit_record"
  }

  响应体示例：

  {
    "text": "今天上午拜访了王总，沟通了下个月的采购计划。对方表示需要先看预算。",
    "needReview": false
  }

  ### 后端调用 Dify 的职责

  后端固定做这些事：

  - 校验 rawText 非空、长度上限、scene 合法
  - 调用 Dify Workflow API
  - 设置超时
  - 保存原文和优化稿
  - 做兜底回退
  - 返回统一格式给前端

  ### 回退策略

  按这个顺序执行：

  1. Dify 超时/失败
     返回原文或规则清洗稿
  2. Dify 返回空文本
     返回原文
  3. 检测关键数字/日期/金额/手机号变化
     丢弃 Dify 结果，needReview = true
  4. 改写幅度异常大
     回退到规则清洗稿

  ## 结果校验

  后端最小校验规则：

  - 原文中的数字在结果中是否仍存在
  - 原文中的日期格式是否仍存在
  - 明显手机号、金额是否仍存在
  - 文本长度是否异常缩短到原文的极低比例

  默认策略：

  - 校验通过：needReview = false
  - 校验可疑但还能用：返回优化稿，needReview = true
  - 校验失败：返回原文/清洗稿，needReview = true

  ## Test Plan

  必须覆盖：

  1. 语气词很多
     例如“嗯啊那个今天去客户那边然后聊了一下”
  2. 一整句没标点
     确认能合理断句
  3. 含客户名、日期、金额
     确认关键实体不变
  4. 拜访记录口语化严重
     确认不会被 AI 擅自“脑补修正”
     确认后端能正常回退
  5. 超短文本
     例如“今天见了王总聊采购”，确认不会被过度扩写

  验收标准：

  - 可读性明显提升
  - 不新增事实
  - 关键实体不误改
  - 异常情况下仍可提交

  ## Assumptions

  - 腾讯云 ASR 已经完成接入并稳定返回文本
  - 当前只做提交前润色，不做实时边录边改
  - 默认前端不展示原文和改写差异
  - 默认先不做结构化字段抽取
  - Dify 只作为内部编排层，由你们后端统一封装



  # 小程序语音润色接入 Dify 的实现规格

  ## Summary

  现状假设已经固定：

  - 小程序前端已完成录音和腾讯云 ASR
  - 你们有统一后端服务
  - 后端技术栈暂按“语言无关 HTTP 服务”设计
  - Dify 采用自建部署
  - 目标是最短路径上线“语音转文字后自动优化”

  最终链路定为：

  1. 小程序调用现有语音识别流程，拿到 ASR 原文
  2. 小程序把原文提交给你们后端 POST /speech/polish
  3. 后端调用自建 Dify Workflow
  4. 后端校验和兜底后，把优化稿返回前端
  5. 前端用优化稿回填日志/拜访记录输入框

  ## Implementation Changes

  ### 1. 前端改动

  前端只做最小改动，不改录音逻辑。

  触发时机：

  - 腾讯云 ASR 成功返回文本后
  - 用户停止录音并确认转写完成后，再调 /speech/polish

  请求参数：

  - rawText
  - scene

  scene 默认固定两种：

  - work_log
  - visit_record

  前端行为：

  - 调用成功：用返回的 text 覆盖当前输入框
  - needReview = true：显示轻提示“内容已优化，请提交前确认”
  - 调用失败：保留 ASR 原文，不阻断用户提交

  前端不要做的事：

  - 不直连 Dify
  - 不保存 Dify Key
  - 不承担润色规则逻辑
  - 不做复杂 diff 对比

  ### 2. 后端接口设计

  新增接口：

  - POST /speech/polish

  请求体：

  {
    "rawText": "今天上午去拜访了王总 然后聊了下采购计划",
    "scene": "visit_record"
  }

  响应体：

  {
    "text": "今天上午拜访了王总，沟通了采购计划。",
    "needReview": false,
    "source": "dify"
  }

  字段定义：

  - text: 最终返回给前端的文本
  - needReview: 是否提示用户二次确认
  - source: dify | fallback_rule | fallback_raw

  后端处理步骤固定为：

  1. 参数校验
  2. 本地规则清洗
  3. 调用 Dify Workflow
  4. 校验 Dify 输出
  5. 选择最终稿并返回
  6. 记录调用日志

  ### 3. 后端本地规则清洗

  在调 Dify 前先做轻量标准化，避免把明显脏数据直接送模型。

  规则范围只做确定性处理：

  - 去首尾空白
  - 压缩连续空格和换行
  - 合并重复标点
  - 移除高频语气词
  - 移除明显重复短词
  - 基础断句

  语气词初始词表建议：

  - 嗯
  - 啊
  - 额
  - 呃
  - 那个
  - 这个
  - 然后
  - 就是说
  - 你知道吧
  - 就是
  - 然后呢

  注意：

  - 不要无脑删除“这个月”“那个客户”里的“这个/那个”
  - 规则需要按词边界或上下文处理
  - 数字、日期、金额、手机号、产品型号绝不改动

  建议返回一个中间文本：

  - normalizedText

  它只用于：

  - 调 Dify
  - Dify 失败时回退

  ### 4. Dify Workflow 设计

  自建 Dify 中创建一个 Workflow，名称建议：

  - speech_text_polish

  输入变量：

  - rawText
  - scene

  输出变量：

  - polished_text

  最短节点方案：

  1. Start
  2. LLM
  3. End

  推荐稳妥节点方案：

  4. Start
  5. Template / Code
  6. LLM
  7. Code 校验
  8. End

  节点职责：

  - Start：接收输入
  - Template / Code：拼系统提示词，按场景附加风格要求
  - LLM：输出润色稿
  - Code 校验：做简单字段检查并输出 needReview
  - End：返回 polished_text、needReview

  ### 5. Dify Prompt 最终稿

  System Prompt：

  你是一个业务语音转文字整理助手。

  你的任务是把中文语音识别文本整理成适合填写业务工作日志或拜访记录的文本。

  严格遵守以下规则：
  1. 只能整理表达，不能新增、猜测、补充原文中没有的事实。
  2. 不得修改客户名、联系人、时间、日期、金额、产品名、型号、地址、电话等关键信息。
  3. 删除语气词、口头禅、无意义重复，例如：嗯、啊、额、那个、然后、就是说。
  4. 修正标点、断句、空格和语序，使表达通顺自然。
  5. 保持原文核心意思不变。
  6. 如果原文内容不完整或有歧义，只能在原意范围内整理，不能补写结论。
  7. 不输出解释，不输出分析，不输出标题，只输出最终整理后的文本。

  User Prompt：

  场景：{{scene}}

  原始文本：
  {{rawText}}

  请输出适合该场景填写的优化文本。

  场景补充：

  - work_log：
      - 简洁、客观、适合工作日志
  - visit_record：
      - 适合拜访记录，突出沟通内容和客户反馈，但只能基于原文

  模型参数默认：

  - temperature = 0.2
  - top_p = 0.8
  - 尽量关闭高随机性
  - 输出长度上限按单次日志常见长度配置

  ### 6. 后端调用 Dify 方式

  后端只通过服务端调用 Dify API。

  调用数据：

  - 使用服务端保存的 API Key
  - 使用固定 Workflow ID
  - 把 normalizedText 或 rawText 作为工作流输入
  - 建议优先传 normalizedText
  - 同时保留 rawText 到本地日志

  后端超时建议：

  - 连接超时：2 到 3 秒
  - 读取超时：8 到 12 秒

  失败处理：

  - Dify 超时
  - Dify 返回空内容
  - Dify 返回非预期字段
  - Dify 网络异常

  统一进入回退逻辑。

  ### 7. 回退与校验策略

  最终稿选择规则固定为：

  优先级 1：

  - Dify 结果校验通过，返回 Dify 文本
  - source = dify

  优先级 2：

  - Dify 失败或结果可疑，返回规则清洗稿
  - source = fallback_rule
  - needReview = true

  优先级 3：

  - 连规则稿都不可用时，返回 ASR 原文
  - source = fallback_raw
  - needReview = true

  后端校验项：

  - 原文数字是否仍在结果中
  - 原文日期是否仍在结果中
  - 金额格式是否仍在结果中
  - 手机号/联系电话是否仍在结果中
  - 文本长度是否异常压缩
  - 是否出现原文没有的新结论性词语

  长度阈值默认：

  - 若结果长度小于原文的 40%，标记可疑
  - 若结果长度大于原文的 180%，标记可疑

  可疑时处理：

  - 优先回退规则稿
  - 返回 needReview = true

  ### 8. 数据留存

  后端至少记录这些字段：

  - operatorId
  - scene
  - rawText
  - normalizedText
  - polishedText
  - finalText
  - source
  - needReview
  - difyLatency
  - status
  - createdAt

  用途：

  - 问题排查
  - 提示词调优
  - 后续积累样本

  如果你们已有日志表，直接扩展即可；没有也可以先落应用日志，不要求第一版必须单独建表。

  ## Public Interfaces / Types

  对前端的稳定契约固定如下：

  请求：

  type SpeechPolishRequest = {
    rawText: string;
    scene: "work_log" | "visit_record";
  };

  响应：

  type SpeechPolishResponse = {
    text: string;
    needReview: boolean;
    source: "dify" | "fallback_rule" | "fallback_raw";
  };

  Dify 在后端内部的返回映射规则：

  - 从 Workflow 输出中取 polished_text
  - 若没有该字段，视为失败
  - 其余 Dify 字段不透传给前端

  ## Test Plan

  必须覆盖以下场景。

  功能测试：

  1. 语气词密集文本
     输出应明显更干净
  2. 无标点长句
     输出应合理断句
  3. 含客户名、日期、金额
     关键实体保持不变
  4. work_log 场景
     风格简洁客观
  5. visit_record 场景
     风格更像拜访记录，但不补事实

  异常测试：

  6. Dify 超时
     返回规则稿或原文
  7. Dify 空输出
     不影响前端提交
  8. Dify 改坏数字
     自动回退并标记 needReview
  9. 超短文本
     不允许模型擅自猜测修正
  验收标准：

  - 业务员看到的文本明显比 ASR 原文更顺
  - 关键事实不被修改
  - Dify 故障不影响提交流程
  - 第一版能在日志和拜访记录两个场景稳定运行

  ## Assumptions

  - 现阶段只做“提交前一次性润色”，不做实时流式优化
  - 前端输入框最终只展示优化稿
