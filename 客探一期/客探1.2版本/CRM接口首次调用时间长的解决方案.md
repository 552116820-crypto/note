  先做这 4 件事

  1. 在网关/Nginx 上看这个接口的 $request_time 和 $upstream_response_time。如果两者都接近 21s，慢点在应用内或应用下游，不在网关前面。
  2. 在接口里加分段耗时日志，不要只打总耗时。至少拆成：鉴权、查客户列表、查企业信息、查 Redis、查 DB、调外部 HTTP、序列化响应。
  3. 打开数据库慢查询和锁等待排查。重点看慢请求那一刻有没有 lock wait、大 SQL、连接池耗尽。
  4. 检查这个接口是否会调用第三方或内部别的服务。21s 很像一次下游读超时。把每个 HTTP 调用的 connect/read timeout 和重试次数打出来。

  最常见的真实原因

  - 下游接口偶发卡住，应用一直等到超时才继续。
  - 某个 SQL 偶发慢，或者被锁住。
  - 连接池/线程池满了，请求先排队再执行。
  - 缓存失效后走冷路径，首次重建数据很慢。
  - 如果这个接口会现查企业信息之类的数据，缓存 miss 很可疑。你返回体里就有 customer_qizhidao_info，这类字段很值得优先查。

  对应的解决方式

  - 如果是下游 HTTP：把超时收紧，比如 connect 1-2s，read 3-5s，减少重试，必要时加熔断和降级。
  - 如果是数据库：给慢 SQL 加索引，避免大范围扫描，排查事务过长和锁竞争。
  - 如果是缓存冷启动：做预热，或用 single-flight，避免多个请求同时重建同一份缓存。
  - 如果是连接池：看池大小、等待时间、活跃连接数，不要让请求在池前排队 20 秒。
  - 不要先靠“把超时调大”解决，这只会把偶发慢请求藏起来。

• 你这类“偶发 20s”慢，解决点在服务端，不在客户端网络。

  我们测到慢请求时是这种形态：dns/connect/tls 都是毫秒级，但 ttfb≈19–21s。这表示请求已经进到服务端了，只是服务端在返回响应头之前卡住了；而且 19–21s 很像“等某个下游到 20s 超时”或“锁/连接池排队接近 20s”。

  怎么解决（按这个顺序做，才能一锤定音）

  1. 先把慢请求能在服务端对齐出来：客户端每次带 X-Request-Id；网关/Nginx 和应用日志都打印这个 id（注意别把 passport 明文落日志）。
  2. 网关/Nginx access log 增加耗时字段：request_time、upstream_header_time、upstream_response_time（用来确认慢是在上游还是应用内部）。
  3. 在 getSalesmanCustomerDetail 里加“分段耗时”或上 APM trace，把时间拆到：鉴权、DB 查询、Redis、外部 HTTP（尤其是生成/补全 customer_qizhidao_info 这类信息的步骤）、序列化输出。
  4. 找到哪一段接近 20s 后再改：
      - 如果是外部/下游 HTTP：把 connect/read timeout 收紧（别用 20–30s 默认），减少重试，加熔断；超时就降级返回主数据，附加信息走缓存或异步刷新。
      - 如果是 DB：开慢查询和锁等待排查，补索引，避免 N+1，控制一次返回的数据量。
      - 如果是缓存重建：做 singleflight/互斥锁防击穿，用“stale-while-revalidate”或定时预热，别让用户撞到重建。
