  ## 1）推荐总体架构（强烈建议加一层“网关服务”）

  不要让 Dify 直接连 CRM（会暴露 CRM 凭证、难做权限、难做审计）。推荐三层：

  1. 企业微信工作台前端（H5）
      - 负责拿到企业微信登录态（userid / openid / corpId 等）
      - 负责 UI（聊天窗口、客户卡片、跳转 CRM）
      - 调用你自己的后端接口
  2. CRM 网关服务（你的后端，核心）
      - 负责企业微信 SSO 校验（把“是谁”搞清楚）
      - 负责从 CRM 拉取“该销售名下30个客户”
      - 负责“首次同步到 Dify Dataset”的编排、增量更新、TTL
      - 负责权限控制（永远用 userid 做过滤）、审计日志、限流
  3. Dify Chatflow 应用
      - 专注做“对话编排 + 调用网关工具 + 检索总结”
      - 不直接持有 CRM 密钥
      - 可以只知道一个 sales_id（或 wecom_userid）作为输入变量

  你会发现：权限隔离和首次同步这两件事，只要网关服务做对了，Dify 里面的流程会非常干净。

  ———

  ## 2）Chatflow 节点怎么编排（给你一条“主链路 + 3条分支”）

  下面是一个实用、可上线的 Chatflow 编排。节点名称按常见 Dify Workflow/Chatflow 的习惯写（“开始/HTTP/条件/LLM/知识检索/代码/结束”这类）。

  ### 主链路（所有请求都走）

  4. 开始节点（Start）
      - 输入变量建议至少有：
          - wecom_userid：企业微信用户唯一标识（或你后端给的 sales_id）
          - query：用户输入
          - 可选：crm_tenant_id / corp_id（如果多租户）
      - 你在企业微信工作台里每次调用 Dify 时，把 wecom_userid 填进去（非常关键）
  5. HTTP 请求节点：EnsureCache（确保已缓存/已同步）
      - 调用你的网关：POST /ensure_customer_cache
      - 入参：wecom_userid
      - 出参建议：
          - cache_ready: true/false
          - dataset_id: 该用户对应的 dataset（或 collection）ID
          - last_sync_time
          - sync_mode: “first_sync / refresh / noop”
          - customer_count（应为 30）

     这一步的目的：把“首次使用要缓存到知识库”这件事变成一个幂等接口。
      - 首次调用：拉 CRM → 写入 Dify Dataset → 标记 ready
      - 非首次：检查 TTL/版本号 → 需要就增量刷新 → 返回 ready
  6. 条件判断节点（IF cache_ready）
      - cache_ready == false：走“同步提示分支”
      - cache_ready == true：进入下一步
  7. 问题分类节点（Classifier / Router）
     用一个轻量 LLM 分类即可，把用户问题分成 3 类（够用且稳定）：
      - A：客户查询/客户画像（问某一个或少数几个客户）
      - B：列表汇总/优先级/漏斗（30个客户的整体问题）
      - C：刷新/同步/更新（用户明确要刷新数据）

  ———

  ### 分支 C：刷新/同步

  5C. HTTP 请求节点：RefreshCache

  - POST /refresh_customer_cache（网关做增量）
  - 返回 last_sync_time
    6C. LLM 节点：生成简短回执
  - 告诉用户同步完成时间、客户数量、提示如何查询

  ———

  ### 分支 A：客户查询/画像（最常用）

  5A. 参数抽取节点（Extractor）

  - 从 query 里抽：
      - customer_name（可能是公司名简称）
      - contact_name（联系人）
      - intent（进展/风险/下一步/关键人/跟进记录/报价等）

  6A. 检索节点（两种实现方式，选一种）

  方式 A1（更安全、更灵活，推荐）：HTTP 检索由网关完成

  - Chatflow 用 HTTP 节点调：POST /search_customer_knowledge
  - 入参：wecom_userid, query, customer_name(optional), top_k
  - 网关内部做：
      - 只在“该销售对应 dataset”里检索（或在向量库里做 metadata filter）
      - 返回 chunks[]（内容 + customer_id + source + updated_at）
  - 好处：你不依赖 Dify 内置“知识检索节点”能否动态选 dataset，也不会串数据。

  方式 A2（Dify 内置知识检索节点）：只在你能保证数据隔离时用

  - 你要么：
      - 每个销售一个独立 Dify 应用（不推荐，运维爆炸），或
      - Dify 的检索节点支持“按 metadata 过滤 wecom_userid”（很多版本做不到/不好用），否则有串数据风险
  - 所以通常我建议 A1。

  7A. LLM 节点：客户回答（带结构化输出）

  - 输入：用户问题 + 检索到的 chunks
  - 输出建议固定为 4 段（销售读起来最快）：
      1. 结论（1-2句）
      2. 关键事实（3-6条 bullets，带日期）
      3. 建议下一步（3条，尽量可执行）
      4. 风险/缺口（例如：缺少关键人、超期未跟进、信息过旧）
  - 强制规则：
      - 不允许编造；没有信息就说“未找到/建议刷新/请确认客户名”
      - 标注数据更新时间：last_sync_time 或 chunk 的 updated_at

  ———

  ### 分支 B：30客户汇总/优先级/漏斗

  5B. （可选）参数抽取

  - 抽取时间范围：本周/本月/近7天
  - 抽取维度：阶段、预计金额、风险、下一步到期等

  6B. 检索策略

  - 你可以让网关直接返回“结构化汇总”更快（不用全靠向量检索）：
      - GET /customer_overview?wecom_userid=...
      - 返回：按阶段分组、待办到期、超过N天未触达、TOP5高价值等
  - 或者仍然用向量检索拿一些代表性 chunk，再让 LLM 汇总（但对“统计类问题”稳定性略差）

  7B. LLM 节点：输出表格化汇总

  - 输出一个“优先跟进清单”（例如 8 条以内）
  - 每条包含：客户名/阶段/上次触达/下一步/风险标签/CRM链接（如果你能提供）

  ———

  ### 同步提示分支（首次使用）

  5Sync. LLM 节点：提示 + 兜底体验

  - 首次同步可能需要 2~10 秒（取决于 CRM 和 embedding），你可以做两种体验：
      - 体验1（简单）：提示“首次同步中，请稍后重试/自动继续”
      - 体验2（更好）：网关同步时先返回“基础结构化数据”（不走向量），先让 LLM 给一个临时回答；同步完成后再支持更深问答

  ## 3）“首次使用把30客户缓存到 Dify 知识库”怎么做（推荐幂等流程）

  核心是你网关里的 ensure_customer_cache(wecom_userid)：

  ### 3.1 缓存判定（必须幂等）

  判定依据建议用组合：

  - 是否存在 user_dataset_map[wecom_userid] => dataset_id
  - last_sync_time 是否超过 TTL（例如 6 小时）
  - CRM 数据版本号/更新时间（如果 CRM 能给）

  ### 3.2 创建/更新 Dataset（两种策略）

  策略 S1：每个销售一个 Dataset（隔离最简单）

  - dataset 命名：sales-{corpId}-{wecom_userid}
  - 优点：天然隔离，不需要 metadata filter
  - 缺点：销售人数多时 dataset 数量膨胀（但每个只有30客户，量很小）

  策略 S2：全公司一个 Dataset + 严格 metadata filter（依赖检索能力）

  - 每个文档带 wecom_userid metadata
  - 检索时必须 filter
  - 风险：如果某处忘了 filter，就会串数据（销售系统一般不建议赌这个）

  在你这个“企业微信 CRM 工作台、销售敏感数据”的场景里，我倾向 S1。

  ### 3.3 文档粒度（强烈建议“一客户一文档”）

  每个客户做一个 Markdown 文档，内容尽量“短 + 结构化”，例如：

  - 标题：客户：{公司名}（{customer_id}）
  - 基本信息：行业、规模、地区、官网
  - 商机：阶段、预计金额、预计成交日期、产品线
  - 关键人：姓名/角色/立场/关系强度
  - 跟进记录：最近3条（时间+类型+一句话）
  - 下一步动作：动作+截止时间+负责人
  - 风险标签：预算不明/竞品强/决策链不完整/超期未触达
  - CRM 深链（如果能生成）

  这样检索命中率会显著高于“把30个客户塞进一个大文档”。

  ### 3.4 同步节奏（避免每次都重算 embedding）

  - 首次：全量30个客户写入
  - 后续：只更新“有变化”的客户（按 CRM 更新时间对比）
  - 建议 TTL：2~6 小时（看销售使用频率）
  - 如果 CRM 支持 webhook：客户变更时触发增量同步（最佳）

  ## 4）你提到的“实现一些功能”，我给你 6 个最实用的（对应节点怎么拼）

  ### 功能 1：自然语言查某个客户“进展/风险/下一步”

  节点要点：

  - Extractor 抽客户名
  - 检索拿该客户文档 + 最近跟进 chunks
  - LLM 输出四段式（结论/事实/建议/风险）

  ### 功能 2：问“我这30个客户里，谁最该优先跟进？”

  不要纯 RAG，建议“结构化规则 + LLM解释”：

  - 网关返回每个客户的：days_since_last_touch、next_action_due、stage、amount、risk_tags
  - 代码节点算一个 priority score（可解释）
  - LLM 把 Top N 用人话讲清楚，并给具体下一步话术

  ### 功能 3：会议/拜访前一键“客户速览 + 话术”

  节点：

  - Extractor 抽客户
  - 检索：客户文档 + 最近沟通记录
  - LLM 生成：
      - 30秒速览
      - 3个高价值问题
      - 2套话术（探需求/推进决策）

  ### 功能 4：对比两个客户（谁更接近成交？）

  节点：

  - Extractor 抽 customer_a、customer_b
  - 检索两份文档
  - LLM 按同一维度输出对比表：阶段、关键人覆盖、风险、下一步、预计金额、预计周期

  ### 功能 5：允许“写回 CRM”（创建跟进任务/记录纪要）

  强烈建议加“二次确认”：

  - LLM 先生成一个结构化 action proposal（例如：客户、动作、时间、内容）
  - 条件节点：要求用户回复“确认/取消”
  - HTTP 工具节点：POST /crm/create_task 或 /crm/add_note
  - LLM 回执：返回任务ID/链接

  ### 功能 6：客户名模糊、同名冲突的处理

  节点：

  - Extractor 抽到的名字不唯一时：
      - 网关做候选匹配（返回 3~5 个候选客户）
      - LLM 让用户点选/回复序号
  - 选定后把 current_customer_id 放到会话变量（或由前端保存），支持追问“那他下周怎么跟进？”

  ## 5）最容易踩坑的点（提前帮你规避）

  1. 数据串用（权限事故）
      - 解决：每个销售独立 dataset（或检索层强制 filter），并在网关做审计日志
  2. 统计类问题靠向量检索不稳定
      - 解决：统计/列表类问题尽量走结构化接口（网关直接算），LLM 负责解释与建议
  3. 首次同步耗时影响体验
      - 解决：ensure_customer_cache 做快速返回 + 异步同步，或同步时提供临时“基础回答模式”
  4. 客户信息过期
      - 解决：TTL + “刷新我的客户”指令 + webhook 增量同步
  5. 模型幻觉
      - 解决：LLM 提示词里明确：只用检索内容；缺失就说缺失；输出带更新时间；重要字段要引用原文片段

  ## 6）落地 MVP（1~2 周可交付）的最小清单

  - 企业微信工作台 H5：能拿到 wecom_userid，能发消息到后端
  - 网关服务：
      - ensure_customer_cache
      - search_customer_knowledge
      - （可选）customer_overview
      - （可选）refresh_customer_cache
  - Dify Chatflow：
      - Start → EnsureCache(HTTP) → Classifier → A/B/C 分支 → LLM 输出
  - 知识库结构：
      - 一客户一文档（Markdown），30份
      - 同步策略：首次全量 + TTL 增量
  - 安全：
      - 网关做用户鉴权、日志、限流
      - 数据隔离方案确认（建议 per-user dataset）
