
• 可以把它拆成一句话：企业微信工作台只负责“打开你们 CRM 的一个 H5 页面”，n8n 只负责“在后台被触发跑流程”；两者通过你们 CRM 后端做一层安
  全、可控的“自动化网关”连接起来（移动端体验也最好）。

  下面按“你们是自研 CRM + 只跑后台 + 手机端”为前提，给一套最稳、最常用的落地方式。

  ———

  ## 1) 工作台侧：把 CRM 的 H5 入口挂进去

  1. 企业微信管理后台 → 应用管理 → 自建应用创建一个应用（比如叫“CRM”或“CRM自动化”）
  2. 在该应用里设置：
      - 应用主页：指向你们 CRM 手机端页面，例如 https://crm.xxx.com/wecom/home 或 .../wecom/crm
      - 网页授权可信域名：crm.xxx.com
      - （如你们页面要用 JS-SDK 的能力）JS 接口安全域名：crm.xxx.com
  3. 结果：员工在手机企业微信 → 工作台点这个应用，就会在企业微信内置浏览器打开你们 CRM H5。

  > 关键点：入口一定是你们 CRM 的页面，不是 n8n。n8n 后台跑就不要暴露给员工端。

  ———

  ## 2) CRM 侧：手机端 SSO（企业微信内打开自动识别员工）

  在 H5 页面里做“静默登录”（推荐）：

  ### 推荐流程（静默，体验最好）

  - 进入 https://crm.xxx.com/wecom/home
  - 若本地无登录态 → 重定向到企业微信网页授权（scope=snsapi_base）
  - 回跳后拿到 code → 你们服务端用企业微信接口换取当前员工 UserId
  - CRM 服务端把 UserId 映射到 CRM 用户，签发你们自己的 session/JWT，H5 正常进入
  注意事项（很重要）

  - corpsecret（应用 Secret）只能放服务端，前端不能拿到
  - 你们要防止 iOS/弱网出现重复回跳：用 state + 一次性 nonce 做去重
  - 移动端尽量全程同域（crm.xxx.com），避免跨域 Cookie/跳转问题

  ———

  ## 3) n8n 触发方式：统一走 CRM 后端，不让前端直连 n8n

  你们的 CRM 页面上，用户点击“同步客户/创建线索/发跟进提醒”等动作时：

  H5 → CRM API → n8n Webhook → 回写 CRM / 发企业微信通知

  ### 最推荐的连接点：n8n 的 Webhook Trigger

  - n8n 工作流起点用 Webhook Trigger
  - 你们 CRM 后端 POST 到 webhook，传入上下文，例如：
      - actorUserId（操作者企业微信 UserId）
      - crmUserId
      - customerId / leadId
      - action（比如 sync_customer_to_erp）
      - requestId（幂等用，必带）
      - callbackUrl（可选，用于异步回调 CRM）

  ### 为什么不让前端直连 n8n？

  - webhook URL 一旦在前端暴露，很容易被复制/刷爆/伪造请求
  - 鉴权、限流、审计都不好做
  - 以后你们要换 n8n、改编排，前端不用动

  ## 4) 强烈建议：把“同步/集成”做成异步任务（手机端体验更稳）

  移动端在企业微信里，网络波动大；n8n 流程也可能跑几秒到几十秒。建议两段式：
 
  ### A. CRM 先快速返回（1 秒内）

  - H5 调用 POST /api/automation/run
  - CRM 立刻返回 jobId（或 requestId）+ “已提交处理中”
  - H5 显示进度页/列表（轮询 GET /api/automation/jobs/:jobId 或用你们的 WebSocket/长轮询）

  ### B. n8n 跑完后回调 CRM

  - n8n 最后一步请求你们 CRM：POST /api/automation/callback
  - CRM 落库结果、更新客户状态、写操作日志
  - 需要提醒员工时：CRM 再调用企业微信 API 给该员工发应用通知（或发到群）

  这套模式的好处：
 
  - 不怕超时
  - 好做失败重试
  - 好做“谁触发的、对哪个客户、跑了多久”的审计
 
  ———
 
  ## 5) 安全与隔离：别让 n8n 管理后台暴露
 
  建议你们这样部署/配置（非常实用）：
 
  - n8n 编辑器后台只允许内网/VPN 访问，或至少 IP 白名单
  - Webhook 入口单独域名或路径隔离（通过反向代理）
    例：https://automation-gw.xxx.com/n8n-webhook/... 只转发 webhook，不转发 editor
  - webhook 做鉴权（至少二选一）：
      - 固定 Token（Header：X-Automation-Token）
───      - HMAC 签名（timestamp + nonce + body，服务端验签，防重放）
  - 加 限流 与 幂等：
      - 幂等键：requestId（CRM 生成，全链路带着）
      - 限流：按 actorUserId / IP / action 限制
 
  ———
 
  ## 6) 你们自研 CRM 的“封装方式”建议（代码组织层面）
 
  在 CRM 后端加一层很薄的模块就够了，建议命名类似：
 
  - automation-service（或 integration/n8n）
      - runAutomation(action, payload, actor)：统一触发入口
      - verifyAndDispatchCallback()：统一接收回调
      - jobsRepo：记录 job 状态（pending/running/success/failed）
      - auditLog：记录全链路 requestId、耗时、结果、错误
 
  这样前端/业务模块只需要调用你们自己的 API，不需要理解 n8n。
 
  ———
 
  ## 7) 最小可行落地（你可以按这个 Checklist 做）
 
  - 企业微信自建应用创建完成：应用主页指到 crm.xxx.com/wecom/home
  - CRM H5：能在企业微信内静默识别到员工（拿到 UserId 并登录）
  - CRM 后端：POST /api/automation/run 能创建 job 并触发 n8n webhook
  - n8n：Webhook Trigger 收到 payload 能跑通一条最简单流程（例如写回 CRM “已完成”）
  - 回调 + 通知：n8n 完成后回调 CRM，CRM 再发应用通知给触发人
 
  ———
