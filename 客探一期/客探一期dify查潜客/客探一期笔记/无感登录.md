  下面把原因拆开讲清楚，并给你能直接落地的做法。

  ## 1) 先澄清：Dify 的 /chat/{token} 不是“登录态”

  http://dify.amer.cn/chat/qQrRtKvsKyEmBoho 里的 qQrRtKvsKyEmBoho 是 这个 Dify WebApp 的公开 token（用来标识“加载哪个应用”），它 不是用户 token，也不会自动继
  承你企业微信/CRM 的登录态。

  所以：

  - Dify WebApp 本身不会“知道”当前企业微信用户是谁
  - 你必须把“当前登录用户的 employee_code”通过嵌入参数/中间接口明确传给 Dify
  - 否则 Start 节点必填项缺失，就一定会弹出输入表单

  ## 2) 你这条链接为什么不会自动填（你贴的 URL 有 3 个明显问题）

  你贴的是：

  http://dify.amer.cn/chat/qQrRtKvsKyEmBoho?name=H4SIA...%3D%3D

  ### 问题 A：参数名用错了

  你的必填字段叫 employee_code，但你传的是 name=...。

  Dify 不会“猜”哪个参数对应哪个输入变量；它只会按变量名匹配。

  - 应该传：employee_code=...
  - 而不是：name=...

  ### 问题 B：路由可能用错了（/chat/ vs /chatbot/）

  Dify 的“自动注入 inputs（并且需要 gzip+base64 编码）”这套，主要是给 **嵌入模式（iframe / chatbot）**用的，常见路径是：

  - .../chatbot/{token}?employee_code=<encoded>

  而你现在打开的是 全屏 WebApp 的：

  - .../chat/{token}

  很多部署里 /chat/ 并不会按嵌入参数规则去解析并自动填表；即使它解析，也可能行为和 /chatbot/ 不一致。

  所以建议你把目标地址换成：

  - http://dify.amer.cn/chatbot/qQrRtKvsKyEmBoho?...

  ### 问题 C：你这个 “H4SIA…” 串看起来已经被破坏了（不是合法 base64）

  你发的参数值里出现了类似 О、р 这种字符（看起来像西里尔字母），而 base64 只能包含：

  - A-Z a-z 0-9 + / 以及结尾 padding 的 =

  只要中间混进了这种“长得像 O/p 但其实不是 ASCII 的字符”，Dify 解码就会失败，最终等同于“没传这个字段”，于是仍然提示必填。

  结论：这个编码串必须 程序生成 + 原样传递，不能手工改大小写、不能复制过程中被输入法/字体替换。

      return urllib.parse.quote(b64, safe="")
