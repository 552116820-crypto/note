  下面给你一套在“企业微信工作台里的 CRM 侧边栏 iframe”里最容易落地的方案（不强行让用户点按钮）：

  推荐架构（最稳）

  - UI：一个很薄的聊天页（H5），iframe 打开它就能运行。
  - 后端编排/缓存：n8n（非常适合做“拉 CRM → 清洗 → 缓存 → 触发更新”）。
  - 大模型生成：可以用 n8n 直接调用 LLM；或者你继续用 Dify 作为 LLM/RAG 层，但 n8n 仍负责“取 CRM + 缓存”。

  为什么这么拆：你要“进页面就自动拉 CRM 并缓存”，这件事必须由 页面 onload 触发一次请求（或父页面 postMessage 触发）。Dify 的“现成 WebApp”通常更偏“用户输
  入后才跑一次流
  程”，要做到“打开即跑”，往往仍需要一个很薄的壳页面来自动发起第一轮请求；n8n 则非常适合作为这个壳页面的后端编排与缓存层。

  ## 1) 进入页面就开始：不需要“开始对话”按钮

  做法是：你的 iframe 指向一个页面 URL（不管这个页面是 n8n Webhook 返回的 HTML，还是你们自建的 H5），页面加载后自动执行：
 
  - 立即调用 POST /assistant/init（n8n Webhook）
  - 后端返回“当前客户摘要/关键字段/最近跟进”等首屏内容
  - 页面把它渲染成第一条“助手消息”
  - 用户接着输入问题时再走 POST /assistant/chat
 
  所以是否需要按钮，完全取决于你写的页面 JS：你可以 0 点击自动跑，也可以 1 点击再跑。
 
  ———
 
  ## 2) employee_code/上下文怎么传进来（同时保证安全）
  你既要用 employee_code 去调 CRM，又要避免被人改 URL 冒充，所以建议这样传：

  - CRM 父页面生成一个 短期 token（里面包含 employee_code、过期时间、可选 nonce）
  - iframe URL 带 token + 当前业务上下文（例如当前客户 account_id / 商机 deal_id）：
      - .../chat?token=...&account_id=...
  - n8n 在 init/chat 两个接口里都先验证 token，再从 token 里取 employee_code 去调 CRM

  如果你们 CRM 页面无法生成 token，也可以退一步：CRM 后端提供一个“换 token”接口，页面先换再用。

  ———

  ## 3) 你要的“第一轮就从 CRM 拉数据 + 缓存 + 更新”怎么做（核心）

  在 n8n 里做两个 Webhook 工作流就够了：

  ### A. POST /assistant/init（首屏摘要初始化）

  输入：token, account_id（或 deal_id）

  步骤（n8n 节点逻辑）：
 
  - 校验 token → 得到 employee_code
  - 计算缓存 key，例如：crm_ctx:{tenant}:{employee_code}:{account_id}
  - 查缓存（推荐 Redis / Postgres；不建议用 n8n static data 存多人共享缓存）
  - 缓存命中且未过期：直接用缓存内容生成“首屏摘要”（或直接返回缓存的摘要）
  - 缓存未命中/已过期：调用 CRM 接口拉 JSON
      - 优先用 CRM 的 updatedAt/etag 做增量（减少全量）
      - 把 CRM JSON 规整成“可给 LLM 的文本”（只挑需要的字段）
      - 生成 content_hash（字段没变就不重复做昂贵步骤）
      - 写入缓存：raw_json + normalized_text + updated_at + content_hash + cached_at
  - 调大模型生成“首屏总结”（或返回 normalized_text 让前端/另一个接口生成）
  - 返回：summary + data_timestamp + cache_status +（可选）一些可点击来源（如客户链接）
  ### B. POST /assistant/chat（后续问答）

  输入：token, account_id, message

  步骤：

  - 校验 token → employee_code
  - 读缓存 key（同 init）
  - 如果缓存过期或 CRM 有更新标记：刷新缓存（同 init 的更新逻辑）
  - 将 cached_normalized_text + message 送给 LLM，总结/回答
  - 返回答案（可附上“基于哪个时间点数据”）

  ———
 
  ## 4) 页面放哪儿更合适：n8n 直接吐 HTML vs 你们自建 H5

  你说“嵌在 CRM 页内侧边栏 iframe”，我会这样建议：

  - PoC/快速上线：n8n Webhook 直接返回 HTML 页面（你现在说的方案 A）
      - 优点：不用前端再部署一个站点，最快拿到 URL
      - 风险：你用的 n8n 版本如果较新，HTML Webhook 会被包 sandbox iframe，功能会受限（但做“自动 init + 文本聊天”通常还能实现）
  - 生产更稳：你们 CRM 同域自建一个很薄的 H5 页面（几十行 JS 就够），n8n 只提供 init/chat API
      - 优点：企业微信 WebView + 多层 iframe 情况下更可控；也更容易做埋点、样式、权限联动
      - 缺点：需要前端发一个很小的静态页面
 
  如果你明确“我不想自建页面”，那就走 n8n 吐 HTML 先跑起来；只是要接受它在 iframe/sandbox 环境里的约束，复杂交互（语音、文件、持久会话历史）后面可能会
  卡。

  ## 5) 你现在要做“知识库检索”吗？还是“实时 CRM 摘要”？
  你这句“第一轮会话从 CRM 接口拿数据并缓存更新”，更像 实时 CRM 摘要/上下文，不一定非要向量库。

  两种模式：

  - 模式 1（最简单）：不建向量库
      - init 拉 CRM → 归一化文本 → LLM 总结 → 后续问答都带上这段上下文
      - 适合：单客户/单商机上下文很明确，问答范围就是“当前记录”
  - 模式 2（更强）：把 CRM 文本入向量库/知识库
      - n8n 增量同步 CRM → 向量库
      - chat 时按 account_id/employee_code 做过滤检索 → LLM 总结
      - 适合：用户会问跨客户/跨历史的“模糊检索”类问题

  你如果侧边栏绑定“当前客户”，模式 1 往往已经够用，而且实现快。
