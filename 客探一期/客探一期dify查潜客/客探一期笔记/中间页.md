  - CRM 里 iframe 不再直接指向 Dify
  - CRM 的 iframe 指向 你自己的中转 H5 页面
  - 在这个中转 H5 页面里，用 embed.min.js 脚本方式加载 Dify，并把 systemVariables.user_id 设成员工号

  这样 CRM 依旧是 iframe（满足 CRM 的嵌入规则），但 Dify 的加载方式变成“脚本嵌入”（你就能传 systemVariables 了）。

  2) /api/me/employee-code 这个接口怎么实现（关键是从哪里拿员工号）

  你有两种常见来源，选一种就行：

  - A. CRM 本身已经登录且能识别员工（最推荐）
      - 后端从 CRM 登录态/SSO/session 里拿到当前用户
      - 直接返回 { employee_code: "000424" }
      - 优点：不需要在 iframe 里再做企业微信 OAuth（iframe 里重定向经常会被拦/体验差）
  - B. 必须走企业微信静默 OAuth 才能拿到员工
      - 中转页 /dify-bridge 如果发现没登录态，就重定向到企业微信 OAuth 拿 code
      - 后端用 code 换 UserId，再映射成 employee_code
      - 注意：OAuth 重定向在“被 CRM iframe 包住”的情况下，有时不稳定；通常要让它在顶层窗口打开（这就是为什么更推荐 A）

  4) Dify 工作流里怎么继续用 employee_code（不想改后面节点的话）

  你把身份放进了 sys.user_id，后面需要 employee_code 有两种写法：

  - 直接用：把所有 {{employee_code}} 改成 {{sys.user_id}}
  - 不想改后面：在最开始加一个“变量赋值”节点，做一行别名：
      - employee_code = {{sys.user_id}}
      - 后面节点继续用 {{employee_code}}（HTTP body、文档命名都不用动）

  同时把“对话前置必填 employee_code 输入”从应用里移除/取消必填，否则仍然会出现“开始对话”。
