• 下面给你一套可直接落地的 Dify Chatflow 方案，严格按你描述的业务流程（小程序 →CRM 获取员工ID →Dify 意图识别/改写 →知识库检索 →整理输出）。

  一、总体架构（建议）

  - 小程序只负责语音/文字输入与展示；业务请求走你们后端。
  - 后端先从 CRM 拿到 employee_id，再调用 Dify 的 chat API，把 employee_id 作为用户标识传入。Dify 在 API 请求里需要 user 字段用于区分用户会话。(docs.dify.ai
    (https://docs.dify.ai/api-reference?utm_source=openai))
  - Dify Chatflow 里基于 userinput.query 处理用户问题。(docs.dify.ai (https://docs.dify.ai/en/guides/workflow/node/user-input?utm_source=openai))

  二、知识库设计（每销售 30 个客户缓存）
  方案 A：单知识库 + 元数据过滤（推荐）

  1. 在知识库中为每个客户建一条文档（可按“客户维度”组织）。
  2. 给每条文档打元数据字段，例如：
      - sales_id（销售员工ID，String）
      - company_name（公司名，String）
      - customer_id（客户ID）
        Dify 元数据支持 String/Number/Time 类型。(docs.dify.ai (https://docs.dify.ai/en/use-dify/knowledge/metadata?utm_source=openai))
  3. Chatflow 里使用知识检索节点，开启元数据过滤，把 sales_id = sys.user_id（或你传入的 employee_id 变量）。知识检索节点支持按元数据过滤检索范围。
     (docs.dify.ai (https://docs.dify.ai/versions/3-7-x/zh/user-guide/workflow/node/knowledge-retrieval?utm_source=openai))

  > 这样每个销售只会检索到自己那 30 个客户的文档。

  方案 B：每人一个知识库（替代）

  - 如果元数据过滤无法绑定变量，就为每个销售建一个知识库，用后端决定调用哪一个（适合人数少）。

  三、Chatflow 画布设计（节点顺序）
  1) User Input 节点

  - 直接使用 userinput.query 作为用户问题输入。(docs.dify.ai (https://docs.dify.ai/en/guides/workflow/node/user-input?utm_source=openai))
  2) LLM 节点（意图识别 + 问题改写）

  - 用结构化输出要求模型输出 JSON：

    {
      "intent": "company_info|other",
      "company_name": "...",
      "rewrite": "...",
      "need_clarify": true/false,
      "clarify_question": "..."
    }
  - Dify LLM 节点支持结构化输出（JSON Schema / 可视化配置）。(docs.dify.ai (https://docs.dify.ai/en/use-dify/nodes/llm?utm_source=openai))

  3)（可选）Code 节点

  - 如果你需要清洗 company_name（去“某某公司/有限公司”后缀），或处理模型输出异常，可以用 Code 节点做 JSON 解析与规范化。(docs.dify.ai (https://docs.dify.ai/
    en/guides/workflow/node/code?utm_source=openai))

  4) Knowledge Retrieval 节点

  - Query：用 rewrite 或 company_name
  - 数据源：选择你的知识库
  - 开启元数据过滤（sales_id = sys.user_id 或 sales_id = employee_id）
  - 检索结果输出 result 变量，可直接给下游 LLM 使用。(docs.dify.ai (https://docs.dify.ai/versions/3-7-x/zh/user-guide/workflow/node/knowledge-retrieval?
    utm_source=openai))

  5) LLM 节点（信息整理输出）

  - Context：使用 Knowledge Retrieval 的 result
  - 提示词：要求只基于检索结果回答（避免编造）

  2) 意图识别 Prompt 模板（可直接放到 LLM 节点）

  你是销售助手的意图识别器。
  用户问题：{{userinput.query}}

  请输出 JSON，且只输出 JSON：
  {
    "intent": "company_info|other",
    "company_name": "",
    "rewrite": "",
    "need_clarify": false,
    "clarify_question": ""
  }

  规则：
  3) 识别“工商信息/企业信息/公司信息/注册信息/基本信息”=>intent=company_info
  4) 抽取公司全称到 company_name
  5) 若公司名缺失、过短或模糊 => need_clarify=true，clarify_question 要求用户补全公司全称
  6) rewrite 用于检索，格式为“查询{company_name}的工商信息”

  7) 若不是查公司信息 => intent=other，rewrite 置空

  3) 小程序/后端调用 Dify API 的输入模板（含 employee_id）
  Dify 的 Chat API 关键字段是 query、user、inputs，user 是应用内唯一用户标识，inputs 用于传入你在 Chatflow 里定义的变量（如 employee_id）。(docs.dify.ai
  (https://docs.dify.ai/api-reference))
  API Key 必须放在后端，避免暴露给小程序。(docs.dify.ai (https://docs.dify.ai/api-reference))

  POST {DIFY_API_BASE}/chat-messages
  Authorization: Bearer {API_KEY}
  Content-Type: application/json

  {
    "query": "查询某某公司的工商信息",
    "user": "employee_id_000424",
    "inputs": {
      "employee_id": "000424"
    },
    "response_mode": "blocking"
  }

  你可以把 user 直接设为 employee_id，这样在 Chatflow 里也能用 employee_id 变量做知识库元数据过滤（比如 sales_id == employee_id）。(docs.dify.ai (https://
