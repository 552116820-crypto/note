  ## 3）“每个销售一个 Dataset”的缓存同步：你后端要做的幂等流程

  你需要一个后端函数/接口，专门干这件事（建议叫 ensure_customer_cache）：

  ### 数据表/缓存建议

  - sales_dataset_map：wecom_userid -> dataset_id
  - sales_sync_state：wecom_userid -> last_sync_time, last_crm_version, syncing_lock
  - customer_doc_map：(wecom_userid, customer_id) -> document_id, updated_at

  ### ensure_customer_cache(wecom_userid) 逻辑（幂等 + 防并发）

  1. 校验 wecom_userid（来自企业微信登录态，不信任前端直传）
  2. 查 dataset_id：
      - 没有：创建一个 Dataset（命名如 sales-{corpId}-{wecom_userid}），写入映射表
  3. 判断是否需要同步：
      - dataset 不存在 / last_sync_time 超过 TTL（如 2~6 小时）/ CRM 有更新
  4. 加一个分布式锁（避免用户同时开两次工作台导致重复写入）
  5. 从 CRM 拉 “该销售名下 30 个客户（含最近跟进/关键人/阶段等）”
  6. 写入 Dify Dataset（建议“一客户一文档”，Markdown 结构化）：
      - 新客户：创建文档，记录 document_id
      - 老客户有变化：更新文档（若你们 Dify 版本更新接口不方便，就“删除旧文档 + 新建”，30 个量很小也能接受）
  7. 返回：
      - cache_ready: true
      - dataset_id
      - last_sync_time
      - customer_count: 30



• 要在“首次访问就把该销售名下 30 个客户缓存到 Dify 知识库（每人一个 Dataset）”，最稳的实现方式是：

  - Dify Chatflow 负责触发（在流程最前面调用一个“确保已同步”的接口）
  - 你的后端负责真正写入 Dify Dataset（创建 dataset / 创建文档 / 增量更新 / 等待索引完成）
  - Chatflow 后续检索也建议走你后端（因为“每人一个 Dataset”很难在 Dify UI 里静态绑定到应用的知识库配置）

  下面给你一套可直接照着落地的方案。

  ———

  ## 1）在 Dify Chatflow 里怎么编排（实现“首次访问触发缓存”）

  ### 节点 1：Start（开始）

  - 输入变量至少两个：
      - wecom_userid：销售身份（从你的小程序/后端传进来）
      - query：用户问题（默认输入）

  > 注意：wecom_userid 必须可信，建议由你后端根据企业微信登录态生成，再调用 Dify API 时写进 inputs（或 user 字段）。

  ### 节点 2：HTTP Request（EnsureCache）

  - 调用你后端：POST /ensure_customer_cache
  - Body：{ "wecom_userid": "{{wecom_userid}}" }
  - 返回建议包含：
      - cache_ready：true/false
      - dataset_id
      - last_sync_time
      - progress（可选：0~100）
      - sync_mode（可选：first_sync/refresh/noop）
      - customer_count（应为 30）

  ### 节点 3：IF 条件（cache_ready ?）

  - 如果 cache_ready == false：
      - 直接用一个 LLM 节点返回提示（例如“首次初始化客户中，预计需要 5~15 秒，请稍后再问一次/点击刷新”）
      - 这一步不要继续走检索（因为索引可能没完成）
  - 如果 cache_ready == true：
      - 继续往下

  ### 节点 4：Variable Assigner（写入会话变量）

  把 dataset_id、last_sync_time 写入会话变量（比如 kb_dataset_id、kb_last_sync_time）。

  原因：续聊时很多集成会只稳定传 conversation_id 和 query，把关键变量落在会话里更抗变。

  ### 节点 5：HTTP Request（Search / Retrieve）

  这里有两种做法：

  推荐做法：检索也走你的后端

  - POST /search_customer_kb
  - 入参：wecom_userid（或 dataset_id）+ query + top_k
  - 后端用 Dify Dataset 的“检索 chunks”接口查该 dataset
  - 返回：chunks[]（内容、客户名、更新时间、来源等）

  然后把 chunks 喂给 LLM 节点回答。

  > 为什么推荐这条路：因为你“每个销售一个 Dataset”，Dify 应用里的“知识库绑定”一般是静态配置，运行时很难动态切到某个用户的 dataset；你自己做检索最可控、最不容
  > 易串数据。

  ———

  ## 2）后端怎么实现“首次访问同步到 Dify 知识库”（核心幂等逻辑）

  你需要一个幂等接口：POST /ensure_customer_cache

  ### 2.1 建议你后端存 3 张映射/状态（DB 或 Redis 都行）

  - sales_dataset_map：wecom_userid -> dataset_id
  - sales_sync_state：wecom_userid -> last_sync_time, syncing_lock, last_crm_update_marker
  - customer_doc_map：(wecom_userid, customer_id) -> dify_document_id, crm_updated_at

  ### 2.2 ensure_customer_cache 伪代码（照着实现即可）

  ensure_customer_cache(wecom_userid):
    dataset_id = lookup sales_dataset_map[wecom_userid]
    if not dataset_id:
      dataset_id = create_dify_dataset(name="sales-{corp}-{wecom_userid}")
      save map

    if recently_synced(last_sync_time) and no_force_refresh:
      return {cache_ready:true, dataset_id, last_sync_time, sync_mode:"noop"}

    acquire lock for wecom_userid (防并发双写)
    customers = CRM.list_my_customers(wecom_userid, limit=30)

    for each customer in customers:
      detail = CRM.get_customer_detail(customer.id)
      doc_text = render_markdown(detail)

      if customer_doc_map exists AND detail.updated_at <= mapped.crm_updated_at:
        continue

      if mapped.dify_document_id exists:
        update_document_in_dify(dataset_id, dify_document_id, doc_text)
        (如果你版本没提供 update，就 delete + create 也行，30 条量很小)
      else:
        dify_document_id = create_document_by_text(dataset_id, doc_text, title=customer.name)
        save mapping

    wait until indexing ready (可选但建议：轮询文档状态)
    update last_sync_time
    release lock

    return {cache_ready:true, dataset_id, last_sync_time, sync_mode:"first_sync/refresh"}

  ### 2.3 Dify 侧你会用到的“知识库 API”

  在自建 Dify 里通常有两类 key/权限（不同版本名字略有差异）：

  - 应用对话 API Key：用来 POST /v1/chat-messages
  - 知识库（Dataset）API Key：用来创建 dataset、创建文档、检索 chunks

  你后端在“首次同步”阶段需要的能力是：

  - 创建 Dataset（每销售一个）
  - 创建/更新文档（每客户一文档，推荐 Markdown）
    -（可选）轮询文档索引状态，直到可检索
  - 后续检索：按 dataset_id 检索 chunks

  具体 endpoint 名称请以你自建 Dify 控制台里的 API Reference 为准（不同版本路径可能有 document/documents 的差异），但流程就是上面这些动作。

  ## 3）客户文档怎么写最适合检索（强烈建议“一客户一文档”）

  每个客户文档建议 Markdown 固定结构，利于命中“风险/下一步/关键人/阶段”等问法：

  - 客户：{公司名}（customer_id）
  - 阶段/预计金额/预计成交日期
  - 关键人（姓名、角色、立场、关系强度）
  - 最近跟进记录（最近3~10条）（日期 + 动作 + 结论）
  - 下一步动作（带截止日期）
  - 风险标签
  - 更新时间：{crm_updated_at}

  这样用户问“XX 客户风险是什么”“下一步怎么推进”，检索几乎必命中。

  ———

  ## 4）首次同步的体验怎么做得更好（避免用户第一句就空）

  你有两种成熟做法：

  - 做法 A（最简单）：Chatflow 里 cache_ready=false 就提示“正在初始化，请稍后重试”
  - 做法 B（更像产品）：用户进入工作台时（还没开始聊天）你先调用一次 ensure_customer_cache 预热；用户真正开聊时基本已 ready

  你是小程序 + 流式输出，我建议优先用 做法 B，体验会明显好。

  ———

  ## 5）两个关键提醒（不提前说很容易返工）

  - “每人一个 Dataset”通常无法在 Dify 应用里静态绑定知识库：因为 dataset 是运行时创建的。最稳就是“后端用 dataset 检索接口取 chunks → 作为上下文交给 LLM  节
    点”。
  - 不要指望用 Dify 自己判断“是否首次访问”：Chatflow/会话变量只对某个 conversation_id 生效，不等于“这个销售第一次使用”。“首次”需要你后端做持久化判断
    （sales_dataset_map）。
