• 这个 workflows/draft 不是我“猜”的，是从你这个 Dify 控制台前端实际请求里抓出来的。

  来源有两条：

  1. 打开编排页时，浏览器会自动请求草稿

  - 你进 .../app/{appId}/workflow 后，前端会发：
  - GET /console/api/apps/{appId}/workflows/draft（读当前草稿）
  - 你改节点后会发：
  - POST /console/api/apps/{appId}/workflows/draft（保存草稿） 
  2. 前端打包代码里也有对应路由

  - /_next/static/chunks/... 的 JS 里能看到拼接路径 apps/${id}/workflows/draft 的调用。

  所以它本质是：Dify Console 的内部管理 API（/console/api/...），不是公开的 WebApp 运行 API。

  你可以自己复现看到：

  1. 打开 Dify 编排页。
  2. 按 F12 打开 Network。
  3. 过滤关键字：workflows/draft。
  4. 拖个节点或改个字段，就能看到 POST .../workflows/draft。

  再补一个关键点：

  - 这个接口依赖登录态（Cookie）+ x-csrf-token，没这两个会 401。
  - 它是控制台内部接口，版本升级后可能变更，不建议当长期稳定对外接口用。
  - 对外稳定调用建议走 /v1 的公开 API（运行工作流/会话等）。


ai是使用API来编辑工作流图，实现从零搭建工作流，而不是编辑DSL文件，或者模拟人类在WEB上操作，为什么用API来编辑工作流图，而不用其他方式？

• 核心原因是：API 编辑工作流图最适合你现在这种“生产级、低出错、可验收”的目标。

  - API方式：可编程、可重复、可回滚。每次变更是确定性的，可加幂等键、自动验收、失败告警联调，适合复杂业务流长期维护。
  - DSL方式：更像“整份快照导入导出”，适合备份/迁移/版本归档。做增量改动时不如 API 精细，且容易遇到版本字段差异、整包覆盖风险。
  - Web模拟点击：最接近人工，但稳定性最低。UI 一变、元素一改、网络波动都可能导致脚本失效，速度也慢，难做批量和自动化验收。

  所以我不是单纯追求“能改就行”，而是优先选择“出错率低 + 可追溯 + 可自动验收”的实现路径。
  对你这种业务逻辑型复杂流，推荐组合是：

  1. 用 Web 先调一次原型和提示词。
  2. 用 API 做正式变更、发布、回滚和 4 态验收。
  3. 用 DSL 做版本快照和跨环境迁移备份。

  一句话：API 不是替代所有方式，而是生产阶段的主干；DSL 和 Web 分别承担备份迁移、可视化调试。
