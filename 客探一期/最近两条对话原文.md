```text
MCP和skill之间的区别与联系
```

### Assistant

```text
**核心区别**
1. `MCP` 是协议与运行时集成层。它解决“模型如何安全、标准化地调用外部能力（工具/资源）”。
2. `Skill` 是任务方法层。它解决“这个任务该怎么做、按什么步骤做、遵循什么规则做”。

**一句话理解**
- `MCP`：给模型“手和眼”（能调用外部系统）。
- `Skill`：给模型“作业方法”（知道该怎么组织这些能力）。

**对比表**
| 维度 | MCP | Skill |
|---|---|---|
| 本质 | 标准协议/接口层 | 指令模板/工作流知识 |
| 目标 | 连接外部工具和数据 | 约束任务步骤与输出风格 |
| 主要内容 | `tools/resources/prompts` + 调用 | 任务规则、步骤、最佳实践 |
| 执行位置 | MCP Server（独立服务） | Agent 上下文（提示/本地文件） |
| 是否直接连后端 | 是（通常通过适配器） | 否（通常通过调用 MCP/工具） |
| 动态数据能力 | 强 | 弱（本身不取数） |
| 标准化与互操作 | 强（跨客户端） | 弱到中（平台内约定为主） |

**它们的联系**
1. Skill 可以“编排” MCP。  
Skill 里写明“先调哪个 tool，再调哪个 tool，失败怎么兜底”，真正执行外部动作的是 MCP。
2. MCP 可以承载 Skill 需要的能力。  
Skill 需要查 CRM、写工单、查知识库时，调用的通常是 MCP 工具。
3. 组合后才完整。  
只有 Skill 没有 MCP，方法有了但干不了活。只有 MCP 没有 Skill，能力有了但调用不稳定、行为不一致。

**你当前场景的映射（含 LangGraph/自建框架）**
1. 在 LangGraph 中：  
- Skill 对应“节点提示词策略/流程策略”。  
- MCP 对应“ToolNode 可调用能力来源”。
2. 在自建框架中：  
- Skill 可落为 `playbook.yaml` / prompt policy。  
- MCP 作为统一工具总线，后面接 Dify、LangGraph、内部 REST API 都行。

**推荐架构**
1. `Backend APIs`：业务真源（CRM、订单、知识库）。  
2. `MCP Adapter`：把 API 封装成稳定工具（schema、鉴权、错误码）。  
3. `Skill/Agent Policy`：定义调用顺序、回退策略、输出格式。  
4. `LLM Client`：执行推理并按 Skill 调 MCP。

**实操建议**
1. 先做 MCP 工具“最小闭环”：`search_customer`、`get_customer_detail`、`update_note`。  
2. 再写 Skill 固化策略：何时查、查不到怎么问、输出字段白名单。  
3. 所有敏感鉴权放 MCP 服务端，不放 Skill 文本里。  
4. Tool 输出保持稳定 JSON，Skill 只负责解释和编排，不做重逻辑计算。

如果你愿意，我可以下一步直接给你一份“**Skill + MCP + Backend API** 三层目录模板”，按这个模板你可以同时接 Dify、LangGraph 和自建服务。
```
