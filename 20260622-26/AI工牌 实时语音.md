### Turn 1  *(2026-06-26 01:42)*

**User:**

你了解AI工牌吗？能实现实时的语音对话，给出产品推荐和销售话术。AI工牌这种软硬结合的产品，我作为AI工程师，只会做软件相关的啊，硬件初步给外包。我该怎么把我软件给到硬件？

**Assistant:**

## 先纠正一个认知:AI 不在工牌里

像 PLAUD、Limitless 这类 AI 工牌/吊坠,以及国内一堆"AI 录音工牌",**真正的 AI(ASR、LLM、推荐、话术、TTS)几乎都在云端**。工牌本体说穿了就是:

> 一个**麦克风阵列 + 音频前处理 + 一根无线电台(BLE/WiFi/4G)+ 电池**,把干净音频送出去而已。

所以你的问题"怎么把软件给到硬件",答案不是"把我的代码烧进芯片"。对这个品类,正确的心智模型是:

**你不是把软件塞进硬件,你是定义一条边界。硬件商的活在"干净音频离开设备"那一刻就结束了;你的软件活在云端 + 一个手机 App 里。交接物是一份接口契约,不是一次代码投递。**

## 三种拓扑,先把这个定了,其余全跟着走

| | 路径 | 固件难度 | 给谁建议 |
|---|---|---|---|
| **A** | 工牌 →BLE→ 手机App → 云 | 低(只做采集+蓝牙串流) | ✅ **v1 就选这个** |
| B | 工牌 →WiFi/4G→ 云(无手机) | 高(网络栈/TLS/OTA都在设备上) | 要"戴了就走、脱离手机"再上 |
| C | 设备端跑模型 | 极高、BOM贵 | 你这场景**不需要**,LLM 必须在云 |

你的推荐 + 话术天然得靠云端 LLM,所以唯一的真问题是"中间有没有手机"。**v1 选 A**:工牌当一个"更好的麦克风",手机 App 是桥 + 给业务员看推荐/话术的屏,云端是你全套 AI。固件那头最简单,你这头(App+云)全在你掌控。

## 分工:谁拥有什么

**硬件外包方拥有:** 工业设计/外壳/夹扣、PCB 选型、**麦克风阵列**、蓝牙模组、电池充电、固件(音频采集、设备端 VAD/AEC/降噪、BLE 栈、断网缓存、OTA、功耗管理)、认证(3C/SRRC/CE/FCC)、量产。

**你(软件)拥有:** 云端(流式 ASR → **说话人分离** → LLM → 产品 RAG → 推荐/话术 → 可选 TTS)、**手机 App**(这是你的,不是硬件商的)、设备管理后台、数据与 CRM 打通(接你现成那套)。

**接口(你来主导定义):** 音频格式 + 蓝牙/网络协议 + 控制通道。

## 你真正要交出去的东西:两份文档

1. **硬件需求规格书(PRD)** —— 形态/重量/续航目标、麦克风规格、指示灯交互、认证要求。
2. **接口契约(ICD)**,这份是你俩并行开发的"宪法",必须**冻结**:
   - **音频**:16 kHz 单声道、Opus 编码(裸 PCM 在 BLE 带宽上吃紧)、帧长、几路麦克风
   - **传输**:BLE GATT 的 service/characteristic 定义、分片、延迟预算
   - **控制通道**:按键事件、灯/震动反馈、录音起停、电量
   - **离线策略**:断网时缓存到 flash、恢复后补传
   - **配对/OTA**:设备怎么绑用户、固件怎么更新

## 软件出身最容易低估、且会让整个产品报废的几个坑

- **音频质量是命门**。垃圾音频 = 垃圾 ASR = 产品死。规格里**用可测量的话写**:"1.5 m 距离、60 dB 背景噪声下,我方 ASR 的 WER ≤ X%",并定验收测试,别写"音质好"。
- **说话人分离(客户 vs 业务员)**。你的推荐是基于**客户说了什么**,所以必须知道谁说的。这是软硬合办的事:硬件端用**指向性/阵列麦 + 波束成形**,软件端用 diarization 模型 —— **早决定**,因为它直接改麦克风规格。
- **续航 vs 常开麦**。全天佩戴 + 持续串流很费电,靠设备端 VAD 门控(没人说话不传)来救。
- **隐私合规**。录客户的声音涉 PIPL 与知情同意,硬件**必须**支持一颗可见的录音指示灯。B2B 商务会谈场景相对好谈,但别省这颗灯。

## 一条捷径:别从零做 ODM,先找"方案商/公模"

深圳有成熟的 AI 录音工牌**方案商**,基于乐鑫 ESP32、恒玄 BES、全志这类芯片,很多**已经把固件 + BLE + App SDK 做好了**,你只要接自己的云 + 换层 App 皮。搜索词:**"AI 录音工牌 方案商 / 公模 / ODM"**。这跟你"硬件初步外包"的打法最契合,可能几周就拿到一个能用的带 App SDK 的工牌,而不是几个月。

## 而且你现在就能开始,不用等硬件

**先用手机自带麦当工牌的替身**,把云端 + App 全链路跑通,做成内测版丢给真业务员用,**先证明推荐和话术真的有用**。证明了,工牌不过是个更好的麦 + 形态;没证明,你在掏硬件钱之前就便宜地知道了。

### Turn 2  *(2026-06-26 01:58)*

**User:**

怎么把语音内容给到软件，实现实时的语音交流

**Assistant:**

先把上一轮的拓扑接上:**工牌 →BLE→ 手机 App →WebSocket→ 你的云**。"把语音给到软件、实现实时交流"的核心,就是这条链上**每一跳传的是什么、用流式而不是整句**。

## 一、一条数据链,逐跳定格式

```
麦克风阵列
  → 板上 DSP(VAD/AEC/降噪)→ PCM 16kHz/16bit/单声道
  → Opus 编码(裸 PCM 256kbps,BLE 扛不住;Opus 压到 ~16-32kbps)
  → BLE GATT notify,每 20-40ms 一帧:[序号][时间戳][opus 负载]
手机 App(你的)
  → 重组、解 Opus
  → 一条常驻 WebSocket 把音频帧往上推(边到边推,不攒整句)
你的云
  → 流式 ASR → 说话人分离/断句 → LLM+RAG → 推荐/话术(→ 可选 TTS)
  → 同一条 WebSocket 把结果推回去:JSON(部分转写/最终转写/推荐卡/话术),可选二进制 TTS
```

一句话:**上行是二进制音频,下行是 JSON(+可选 TTS 音频),共用一条双向 WebSocket。** 手机↔云为什么是 WebSocket 而不是 HTTP —— 它全双工、长连接、穿透防火墙好,是流式音频 + 结构化结果回传的事实标准(要真·全双工低延迟语音才上 WebRTC,你这场景用不上)。

## 二、命门是"流式 ASR",而且别自己造

整个实时性成败在这一步。要的是**增量识别**:音频边到、转写边出,不等人说完。

- **直接买**(中文场景强,有 WebSocket 流式接口):火山(豆包)、阿里云、讯飞、腾讯云的实时语音识别。
- **自托管**(成本/隐私敏感再上):阿里达摩院 **FunASR / Paraformer-streaming**,中文很能打,开源。

不要用 Whisper 这类**整句**模型做实时 —— 它天生非流式,硬包一层延迟也难看。

## 三、两个人的对话,必须分轨

这是销售工牌和普通录音笔的根本区别:工牌同时收**业务员**和**客户**,而你的推荐是基于**客户说了什么**。所以 ASR 之上要做:

- **说话人分离(diarization)**:谁说的。硬件侧(指向性/阵列麦 + 波束)+ 软件侧(分离模型),早决定,直接改麦克风规格。
- **断句(endpointing)**:判断"客户这句说完了",作为触发 LLM 的时机 —— 客户说完一句 → 才去跑推荐,而不是每个字都触发。

## 四、要不要用"实时语音大模型"(语音到语音 API)?——你这场景不适用

市面上有一类 speech-to-speech 实时 API(把 ASR+LLM+TTS 缝成一条 WebSocket 会话)。**对你这个产品是个陷阱**,原因很本质:

> 那类 API 假设 **AI 是对话的一个参与者**(人 ↔ AI,轮流说话)。而你的工牌是**旁听两个人**(业务员 ↔ 客户)的对话,做分离 + 给一方喂提示 —— AI 是观察者,不是参与者,没有"轮到 AI 说话"这个回合。

所以正确架构恰恰是**解耦的流式管线**:持续流式 ASR + 分离 → LLM 在"客户说完一句/识别到某意图"时被触发 → 输出推荐/话术。这不是退而求其次,是和"旁听辅助"模式天然匹配。

补一句 LLM 这个盒子的现实:无论你用 Claude(质量)还是国产豆包/DeepSeek/通义(成本+延迟+中文+合规),它都是**文本进、文本出** —— 没有哪个厂商给你一个"语音进、推荐出"的一体化 API,你必须自己把 ASR→LLM→(TTS)串起来。好处是每段都能独立选型、独立换。两个要点:LLM **流式输出 token**(话术逐字蹦出来,业务员不用等整段);用 **tool use / 结构化输出**对接你的产品库做 RAG —— 这正好接上你现成那套 sales-assist 数据。

## 五、延迟预算(知道钱花在哪)

| 段 | 量级 | 备注 |
|---|---|---|
| 麦克风→BLE | 几十 ms | 帧长决定 |
| 手机→云 RTT | 几十 ms | 网络 |
| 流式 ASR 部分结果 | 100–300ms | 增量出字 |
| LLM 首 token | 数百 ms~1s | 流式缓解体感 |

**屏显辅助(给业务员看卡片)**:端到端 1–2s 完全够用。**要 TTS 进耳机实时播**:得压到 <1s,难度陡增 —— 这又回到上一轮那个问题:工牌到底要不要"说话"。

## 六、实时可靠性,别漏这几条

- **部分 vs 最终结果**:部分转写(partial)拿来即时上屏求响应;最终结果(final)才拿去触发 LLM 动作。
- **断线重连**:WebSocket 会断,要序号 + 时间戳、断连缓存、恢复续传。
- **背压/乱序**:BLE 丢帧用序号补;别假设帧严格有序到达。

## 七、今天就能搭,不用等工牌

**用手机自带麦当工牌替身**,跑通"手机麦 → WebSocket → 流式 ASR → 分离 → LLM+RAG → 屏显"整条链,内测给真业务员用。这等于把"实时语音 → 推荐"的全部软件风险**在硬件存在之前**清掉 —— 工牌后面只是换一个更好的麦 + 形态。
