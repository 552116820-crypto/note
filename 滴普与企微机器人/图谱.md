# 滴普知识库 — 图谱 vs 文档 RAG 实践指南

## 一、产品信息该不该用图谱？

### 结论：推荐用图谱，但要"图谱 + 文档"双知识库混合

润滑油产品数据天然是**多层级分类树 + 丰富属性 + 交叉关系**：

```
工业润滑油
├── 液压油
│   ├── 高压抗磨液压油（L-HM 46、L-HM 68...）
│   ├── 低温液压油
│   └── 抗燃液压油
├── 齿轮油
│   ├── 重负荷齿轮油
│   └── 蜗轮蜗杆油
├── 压缩机油
└── 导热油
```

### 图谱 vs 纯文档 RAG 能力对比

| 查询类型 | 图谱能力 | 纯文档 RAG |
|---------|---------|-----------|
| "高压抗磨液压油属于哪个大类？" | 直接沿边回溯 | 要靠文本命中 |
| "液压油下面有哪些子产品？" | 子节点遍历 | 不可靠 |
| "哪些产品适用于注塑机液压系统？" | 按设备关系查 | 依赖文档里恰好写了 |
| "L-HM 46 和 L-HM 68 有什么区别？" | 属性对比 | 需两段文本同时召回 |
| "这个客户用的齿轮油能换成什么？" | 替代品关系 | 几乎做不到 |

### 实体和关系设计

**实体类型：**
- `产品大类`（工业润滑油、车用润滑油...）
- `产品系列`（液压油、齿轮油...）
- `具体产品/SKU`（XX牌高压抗磨液压油 L-HM 46）
- `应用场景`（注塑机、数控机床、矿山设备...）
- `技术标准`（ISO VG 46、GB 11118.1...）

**关系：**
- `属于` — 子类 → 父类（高压抗磨液压油 → 液压油 → 工业润滑油）
- `适用于` — 产品 → 应用场景
- `符合标准` — 产品 → 技术标准
- `可替代` — 产品 ↔ 产品
- `推荐搭配` — 产品 → 产品（如液压油 + 液压油滤芯）

**关键属性：**
- 粘度等级、基础油类型、工作温度范围、闪点、倾点等

### 图谱 vs 文档的分工

| 存图谱 | 存文档知识库 |
|--------|-------------|
| 产品层级分类 | 产品详细说明书 / PDF |
| 产品核心参数（粘度、温度范围） | 使用指南、换油周期建议 |
| 适用设备 / 行业关系 | 技术问答 FAQ |
| 产品间替代/竞品关系 | 故障排查手册 |
| 标准认证关系 | 销售话术、客户案例 |

### 注意：实体名称 embedding 相似度的坑

润滑油产品名称很容易"语义近但实体不同"：
- "高压抗磨液压油 L-HM 46" vs "高压抗磨液压油 L-HM 68" → embedding 极其相似，但是不同产品
- "重负荷齿轮油" vs "中负荷齿轮油" → 也很近

录入时要确保**产品型号/SKU 作为唯一标识**，不能只靠名称 embedding 去重。自动合并阈值至少 0.93 以上，且必须叠加型号匹配规则。

## 二、怎么知道走的是图谱还是文档 RAG？

### 模式一：流程编排中手动定路由（推荐）

在滴普的可视化低代码流程编排中画条件分支：

```
用户提问
   │
   ▼
┌─────────────┐
│  意图分类节点  │  ← LLM 判断问题类型
│ （LLM / 规则）│
└─────┬───────┘
      │
  ┌───┴────┐
  ▼        ▼
图谱查询   文档RAG
  │        │
  └───┬────┘
      ▼
  合并回答生成
```

**意图分类节点的 prompt 示例：**

```
请判断用户问题的类型，只返回类别编号：

1 - 产品分类/层级查询（如"液压油有哪些子类""XX产品属于什么大类"）
2 - 产品对比/替代（如"HM46和HM68有什么区别""能用什么替代"）
3 - 产品参数查询（如"XX产品的粘度是多少""工作温度范围"）
4 - 使用指导/技术问答（如"怎么换油""多久换一次""出现乳化怎么办"）
5 - 其他

用户问题：{{query}}
```

路由规则：
- **1、2、3 → 走图谱**（结构化关系查询）
- **4、5 → 走文档 RAG**（非结构化内容检索）

### 模式二：全都查，合并结果（简单粗暴但有效）

不做路由，图谱和文档**并行查**，把两边结果一起塞给 LLM：

```
用户提问
   │
   ├──────────────┐
   ▼              ▼
图谱查询       文档RAG
   │              │
   ▼              ▼
图谱结果       文档片段
   │              │
   └──────┬───────┘
          ▼
   LLM 综合回答
```

给 LLM 的 prompt：

```
以下是两个来源的检索结果，请综合回答用户问题。
如果两个来源有冲突，以【图谱数据】为准（更结构化可靠）。

【图谱数据】
{{graph_results}}

【文档数据】
{{rag_results}}

用户问题：{{query}}
```

- 优点：不用做意图识别，不会走错路
- 缺点：每次查两遍，成本翻倍；噪声多

### 模式三：关键词规则 + LLM 兜底

先用简单规则快速分流，规则兜不住的再让 LLM 判：

```python
# 伪代码
graph_keywords = ["属于", "分类", "有哪些", "下级", "上级",
                  "替代", "区别", "对比", "参数", "粘度",
                  "适用于", "型号"]

if any(kw in query for kw in graph_keywords):
    route = "graph"
elif len(query) > 50:  # 长问题通常是技术咨询
    route = "rag"
else:
    route = llm_classify(query)  # LLM 兜底
```

### 实际建议

针对润滑油产品场景，**推荐模式一**：
1. 问题类型**边界清晰** — 问分类/参数/替代 vs 问使用方法/故障排查，用户意图很容易区分
2. 滴普的流程编排本身就支持条件分支节点，不需要额外开发
3. 成本可控，不像模式二每次双查

### 怎么验证走的哪条路

在流程编排中，给每个分支加一个**日志/标记节点**（调试阶段）：

```
回答末尾附加：
---
[来源: 图谱] 或 [来源: 文档RAG]
[命中实体: 高压抗磨液压油 L-HM 46]
[意图分类: 产品参数查询]
```

上线后去掉这个标记，或者只在后台日志里保留，用来持续监控路由准确率。如果发现某类问题经常走错路，就回去调意图分类的 prompt 或补关键词规则。
