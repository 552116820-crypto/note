### Turn 1  *(2026-06-05 01:11)*

**User:**

Feature of the week: /loop — run a prompt or slash command on a recurring interval，什么意思

**Assistant:**

`/loop` 是 Claude Code 的一个 skill，作用是**让某个 prompt 或 slash command 按固定节奏反复执行**，而不是只跑一次。

## 两种用法

**1. 固定间隔**
```
/loop 5m /foo
```
每 5 分钟自动跑一次 `/foo`。间隔可以写 `30s` / `5m` / `1h` 之类。命令位也可以直接放一句自然语言 prompt，不一定非得是 slash command。

**2. 自定节奏（省略间隔）**
```
/loop 盯着这个部署，好了就告诉我
```
不写时间，就由模型自己判断下一次什么时候醒过来——比如它知道 CI 大概 8 分钟，就不会每 30 秒空转一次。

## 典型场景

- `check the deploy every 5 minutes` —— 轮询部署状态
- `keep running /babysit-prs` —— 反复跑某个看护类命令
- 盯一个后台任务 / 远程队列，状态变了就回报

一句话：**适合需要"反复做、或盯着等"的活，不适合一次性任务**（skill 描述里明确写了 one-off 别用它）。

## 和 `/schedule` 的区别

| | 运行位置 | 适合 |
|---|---|---|
| `/loop` | 当前会话里循环 | 这一坐在电脑前的这段时间内盯着 / 轮询 |
| `/schedule` | 远程 cron 定时 agent | 关掉终端也要按时跑的长期定时任务 |

简单说，`/loop` 是"**这次会话内的循环**"，`/schedule` 是"**脱离会话的定时任务**"。

要现在试一个吗？告诉我想循环跑什么、多久一次就行。

### Turn 2  *(2026-06-05 01:13)*

**User:**

具体的应用场景

**Assistant:**

`/loop` 的甜区只有一个核心判断：**你现在坐在电脑前，有个东西需要"反复做"或"盯着等"，且就在这一两个小时内。** 跨天、关了终端还要跑的，那是 `/schedule` 的活。

按这个标准，具体落到你的工作上：

## 1. 盯长任务，完成即报（最高频）

任何要等几分钟到几十分钟的操作，提交后用 loop 轮询状态，好了再叫你：

- **Firecrawl 整站爬取** —— `firecrawl_crawl` 提交后返回 job id，本来就得 `firecrawl_check_crawl_status` 反复查。
  ```
  /loop 30s 查一下爬取任务的状态，爬完就把结果整理给我
  ```
- **churn_db 大查询 / 批量导出** —— 跑一个几分钟的流失客户分析，盯着出结果。
- **跑测试 / 构建 / 部署** —— `/loop 跑测试，过了通知我，挂了把报错贴出来`

这是 loop 最不可替代的地方：异步任务的轮询，你不用每隔一会儿手动敲一次 status。

## 2. 周期性巡检（会话内）

固定间隔重复同一个检查，**前提是你这段时间一直开着会话**：

- `/loop 5m /babysit-prs` —— 每 5 分钟看护一遍 PR
- 盯一个后台进程的日志，有异常就报

⚠️ 如果是"每天早上查一次新流失客户"这种跨天的，别用 loop，用 `/schedule`（远程 cron，关终端也跑）。

## 3. 自定节奏、增量推进（省略间隔）

一个大活分批做，模型自己决定下一轮何时继续：

- 批量处理一个清单（比如逐个给流失客户生成跟进话术），每轮做几个，盯着进度直到清空
- "loop-until"：反复跑直到满足某个条件才停

## 一句话边界

| 你的需求 | 用哪个 |
|---|---|
| 现在提交了个爬虫/查询，等它好 | `/loop` |
| 现在开着会话，每几分钟巡一次 | `/loop` |
| 每天 9 点自动跑、关终端也要跑 | `/schedule` |
| 只做一次 | 都不用，直接说 |

### Turn 3  *(2026-06-05 01:15)*

**User:**

套成一条具体的 loop，这长什么样

**Assistant:**

直接上两条，分别对应两种模式，连「敲进去的样子」和「跑起来的样子」一起给你看。

## 例 A — 固定间隔轮询（盯着等）

场景：你刚提交了一个 Firecrawl 整站爬取，想等它爬完。

**你敲的：**
```
/loop 1m 查这个爬取任务的状态。没爬完就只回一句"还在爬(已N页)"；
爬完了把每页标题+URL 汇总成表格给我，然后停止循环
```

**跑起来大概长这样：**
```
↻ loop · 每 1 分钟
─────────────────────────
[19:32] 第1轮 → 还在爬（已 47 页）
[19:33] 第2轮 → 还在爬（已 112 页）
[19:34] 第3轮 → 爬完了，共 138 页 ↓
        | 标题            | URL                  |
        | 润滑油代理政策   | https://...          |
        | ...             | ...                  |
        ✅ 已达成，停止循环
```

## 例 B — 动态自定节奏（省略间隔）

场景：churn 项目，批量给一批流失客户拟跟进话术，活比较大，分批做。

**你敲的：**
```
/loop 从流失客户清单里每轮取 5 个没处理的，生成跟进话术写回去，
报一下进度；全部处理完就停
```

**跑起来：**
```
↻ loop · 自定节奏
─────────────────────────
第1轮 → 处理 5/87，下一轮我稍后继续
第2轮 → 处理 10/87 …
  …
第18轮 → 87/87 全部完成 ✅ 停止循环
```
不带时间时，节奏由模型自己掂量——不会每 10 秒空转一次。

## 写好一条 loop，prompt 里务必交代 3 件事

1. **没达成时干什么** —— 给个「轻量回应」，否则每轮都长篇刷屏。例 A 里那句"只回一句"就是干这个的。
2. **达成的判断标准** —— "爬完"/"全部处理完"，让它知道何时算完。
3. **达成后停止** —— 必须明写"停止循环"，否则它会一直跑下去。

漏了第 3 条是最常见的坑：任务早完成了，loop 还在每分钟空转。
