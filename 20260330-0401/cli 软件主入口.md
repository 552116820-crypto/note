  • - CLI：Command-Line Interface，中文常译为“命令行界面”
  - GUI：Graphical User Interface，中文常译为“图形用户界面”
  
  AI 时代里，CLI 重新变热，不是因为大家怀旧，而是因为它最适合把“人、代码、系统、AI agent”接在一起。
  更准确地说，CLI 正在重新成为很多软件的关键主入口之一，尤其在开发、运维、自动化、知识工作流里最明显。

  一、为什么偏偏是 CLI，而不是 GUI

  1. CLI 天然适合“执行任务”
     AI agent 要做事，通常要完成这几步：

  - 读本地文件
  - 改代码或配置
  - 调 API
  - 跑命令
  - 看结果
  - 再决定下一步

  这套链路里，终端是最顺手的统一入口。
  GUI 更适合浏览和可视化，但不适合连续执行一长串带状态的操作。

  2. CLI 是最容易编排的接口
     对 AI 来说，最重要的不是“界面好看”，而是“动作可组合”。
     CLI 的优势在于：

  - 输入输出结构化
  - 参数明确
  - 容易写脚本
  - 容易接 CI/CD
  - 容易被 agent 调用

  所以它既适合人手动敲，也适合 AI 自动跑。

  3. CLI 离操作系统最近
     很多 AI 工作并不是单一产品内的动作，而是跨系统动作：

  - 在本地拉代码
  - 安装依赖
  - 运行测试
  - 调云服务
  - 部署
  - 查日志

  如果没有 CLI，AI 很难把这些能力串起来。
  终端本质上就是操作系统的“统一控制面”。

  二、为什么这件事在 AI 时代突然加速

  1. 以前 CLI 是“效率工具”，现在变成“执行容器”
     过去 CLI 主要服务熟练用户，提升效率。
     现在 AI agent 需要一个可以真正落地执行的环境，CLI 正好承担这个角色。

  所以这轮不是单纯的 CLI 回潮，而是 AI 把 CLI 的价值放大了。

  2. 大模型最擅长处理文本接口
     CLI 本质上是文本协议：

  - 命令是文本
  - 参数是文本
  - 输出是文本
  - 日志是文本
  - 错误信息也是文本

  而大模型天生擅长理解和生成文本。
  相比之下，纯 GUI 需要依赖额外的视觉解析、页面状态理解、控件定位，稳定性更差。

  3. Agent 需要可控性，CLI 比 GUI 更容易治理
     企业要让 AI 真做事，最怕两件事：

  - 权限失控
  - 行为不可审计

  CLI 在这方面反而更强，因为：

  - 每一步动作都可显式记录
  - 可以做 allowlist / denylist
  - 可以限制命令范围
  - 可以在执行前审批
  - 可以保留日志便于追溯

  这也是为什么很多 AI coding agent 都优先落在终端里，而不是先做纯网页助手。

  三、为什么大厂也开始拥抱 CLI

  1. CLI 能把产品接入开发者工作流
     今天的软件竞争，不只是“功能好不好”，还包括“能不能嵌进用户的工作流”。
     CLI 可以直接进入：

  - 本地开发环境
  - 自动化脚本
  - DevOps 流程
  - 团队工具链
  - AI agent 工作流

  这比单纯提供网页入口更有粘性。

  2. CLI 让产品能力更容易外溢
     以前一个产品只能在自己网页里用；
     有了 CLI，它的能力就能进入别人的脚本、IDE、CI、机器人、agent。

  这意味着产品不再只是一个“界面”，而是一个“能力平台”。

  3. 厂商交付成本更低，跨平台更容易
     维护复杂 GUI 的成本很高。
     CLI 的好处是：

  - macOS / Linux / Windows 更容易统一
  - 版本化和回滚更容易
  - 自动测试更容易
  - 文档化更容易
  - 分发更容易，用 npm、brew、安装包都行

  所以从厂商视角，CLI 是性价比很高的能力出口。

  四、为什么不只是开发工具，连飞书、Obsidian 这类产品也开始做 CLI

  因为越来越多软件都在经历同一个变化：
  从“给人点着用”变成“给人、脚本、agent 一起用”。

  飞书这类协作平台开始有 CLI，是因为它们不再只是“网页产品”，而是在进入：

  - 本地开发
  - 企业内部自动化
  - 低代码/集成场景
  - AI 工作流

  Obsidian 做 CLI 也一样。知识管理不再只是“打开笔记编辑”，而是变成：

  - 批量处理笔记
  - 自动整理知识库
  - 接外部工具
  - 接 AI 助手
  - 做可重复工作流

  一旦产品的数据和操作足够结构化，CLI 就会出现。

  五、为什么说 CLI 特别适合 AI，而 GUI 仍然不会消失

  这是个关键点：
  AI 时代不是 CLI 替代 GUI，而是分工更清楚了。

  - CLI 适合执行、编排、批处理、自动化、agent 调用
  - GUI 适合浏览、监控、分析、配置发现、低门槛上手
  - API 适合系统对系统
  - Agent 适合在 CLI/API 之上做自主执行

  所以未来更可能是四层并存，而不是某一种独占。

  六、一个更底层的解释

  这轮 CLI 兴起，背后其实是软件范式变了：

  - 过去：用户在 GUI 里“操作功能”
  - 现在：用户在 GUI + API + CLI + Agent 里“编排能力”

  CLI 之所以重要，是因为它站在中间：

  - 往下连操作系统和本地环境
  - 往上连产品能力和云服务
  - 往旁边连脚本、CI、IDE
  - 往前连人
  - 往后连 agent



  它是最容易成为“统一调度层”的那个入口。

  七、结论

  如果只用一句话总结：

  AI 让软件从“可使用”走向“可执行”，而 CLI 是最适合承载执行的通用接口，所以它重新成为主入口。

  但这个“主入口”不是唯一入口，而是最适合高频执行、自动化和 agent 化的一层。
