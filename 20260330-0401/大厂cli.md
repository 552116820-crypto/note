• 先给一个判断：CLI 不是“突然复古”，而是从过去的“工程师辅助工具”，升级成了“产品能力的可编排入口”。
  你提的两个例子里，严格说 飞书 CLI 更准确是飞书低代码平台的官方 CLI；Obsidian CLI 则是官方新推出的 CLI，说明这种趋势已经从传统 DevTools 外溢到了协作和知识管理软件。

  一、为什么现在会明显兴起

  1. 软件本身越来越“可编排”了
     很多产品现在不只要有网页，还要能接入本地开发、脚本、CI/CD、自动化流程。
     这不是理念，是产品文档里直接写出来的：Vercel CLI 支持在终端或自动化系统里做日志、证书、DNS、本地环境复制；
     Cloudflare 的 Wrangler 用来管理 Worker 项目并接入本地工作流；Supabase CLI 甚至能把整套本地开发栈跑起来；Stripe CLI
     直接面向“构建、测试、管理集成”。
     本质上，CLI 是连接“本地机器、远程环境、CI、API”的最小公分母。
  2. 终端用户基础并不小，反而一直很稳
     CLI 的前提不是人人都爱黑窗口，而是核心用户群长期就在终端里工作。
     Stack Overflow 2025 调查里，Bash/Shell 在专业开发者中的使用率是 48.8%。另一篇关于云开发者工具偏好的研究也发现：开发
     者整体更偏好 CLI，尤其是 CRUD 类任务；只有监控类任务更偏向 Web Console。
     所以高频、重复、可脚本化的动作，CLI 天然更顺手。
  3. AI Agent 把 CLI 从“命令入口”推成了“代理入口”
     这是这轮 CLI 热起来最关键的新变量。
     GitHub Copilot CLI、Google Gemini CLI、JetBrains Junie CLI 都把终端当成主战场，而且都在强调 MCP、插件、自动执行、会
     话恢复、CI/CD 接入。
     原因很简单：AI agent 要想真正干活，需要读文件、跑命令、调用工具、接系统权限，而终端正好天然拥有这些能力。
     Stack Overflow 2025 里，84% 的受访者已经在用或计划用 AI 工具；对 AI agent 用户来说，69% 认为它提升了生产力。
     所以不是 CLI 自己复兴，而是 AI 需要一个最适合执行任务的外壳，CLI 刚好最合适。
  4. 对厂商来说，CLI 的交付成本和扩展性很划算
     GUI 每加一个平台、一个复杂流程，成本都很高；CLI 则更容易跨平台、版本化、自动测试和文档化。
     你看这些产品的分发方式就知道了：飞书 CLI 通过 NPM 分发，Gemini CLI 支持 npx/npm/Homebrew/conda。
     这意味着厂商可以更低成本地把能力铺到 macOS、Windows、Linux、本地机、容器、CI 里。
     这点很多文档不会直接说，但从它们的安装和发布方式能很清楚地推断出来。
  5. CLI 更适合企业治理和权限控制
     以前 CLI 被认为“不安全”，现在反而因为它“显式、可控、可审计”而更适合企业。
     GitHub Copilot CLI 文档里专门有企业控制、工具允许/拒绝、MCP 接入；Junie CLI 也有命令审批、allowlist、plan mode、
     brave mode。
     当 AI 开始能改代码、跑命令、连内部系统时，企业最需要的不是更炫的 GUI，而是更明确的权限边界和自动化治理手段。
  6. CLI 已经从基础设施工具，扩散到“通用工作软件”
     这点是你观察得很准的地方。
     飞书低代码平台的 CLI 不只是安装工具，而是直接覆盖登录、代码生成、本地调试、代码同步、构建跟踪；Obsidian CLI 明确写
     的是面向脚本、自动化、外部工具集成。
     这说明一个趋势：只要产品的数据模型足够稳定、操作足够结构化，用户就会要求它暴露 CLI/API，而不再满足于“只能点界面”。

  二、更本质的原因

  真正兴起的不是 CLI，而是“软件的程序化使用方式”。
  过去用户是在 GUI 里“操作功能”；现在越来越多用户是在 API + CLI + Agent 里“编排能力”。

  所以未来主流形态不是“CLI 替代 GUI”，而是四层并存：

  - GUI 负责发现、可视化、低门槛使用
  - API 负责系统对系统
  - CLI 负责人对系统的高效执行
  - Agent 负责基于 CLI/API 进行半自动或自动执行

  CLI 这轮兴起，本质上是它刚好站在这四层的交汇点上。

  三、一个保留意见

  也别把它神化。
  研究和产品实践都说明，监控、分析、可视化探索 这类任务，Web UI 往往仍然更强。
  所以大厂“拥抱 CLI”不是要放弃界面，而是在把产品做成“既能点，也能编排，也能让 agent 调”。
