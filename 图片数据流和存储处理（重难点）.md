## 一、后端 → 前端传图片，常见 3 种方式

### 1. 直接返回二进制流（最朴素）
后端响应 `Content-Type: image/png`，body 就是图片字节。
```
GET /api/avatar/123
→ Content-Type: image/jpeg
→ <二进制>
```
前端用 `<img src="/api/avatar/123">` 就能直接显示。
**适合**：图片是接口主产物（头像、验证码）。

### 2. 返回 JSON，里面带 URL（**业界主流**）
```json
{ "image_url": "https://cdn.xxx.com/abc.png", "width": 1024 }
```
前端拿到 URL 再 `<img src={url}>`，浏览器自动发第二个请求去 CDN 拉图。
**适合**：图片是结果之一，还要带其他字段（这就是所有生图大模型 API 的做法）。

### 3. 返回 JSON，里面塞 Base64
```json
{ "image_base64": "iVBORw0KGgoAAAANSUhEUg..." }
```
前端：`<img src={"data:image/png;base64," + b64}>`。
**优点**：一次请求拿完，没有跨域/防盗链问题。
**缺点**：体积膨胀 ~33%，不能被浏览器/CDN 缓存，大图会拖慢接口。
**适合**：小图、临时图、不希望落盘的私密图。

---

## 二、生图大模型 API 到底怎么把图给你

并不是"JSON 里塞像素"，本质是**返回一个可下载的 URL 或 base64**，两种模式都有：

| 厂商 | 默认返回 |
|---|---|
| OpenAI DALL·E / gpt-image-1 | 可选 `url` 或 `b64_json` |
| Midjourney（官方 API/Discord） | URL（CDN） |
| 即梦 / 可灵 / 通义万相 / 文心一格 | URL（带签名，通常 24h-7d 过期） |
| CogView（智谱） | URL |
| Stable Diffusion 自部署 | 你自己决定，一般直接 base64 |

**异步任务模式**（视频/慢生图常见）：
1. `POST /generate` → 返回 `task_id`
2. 轮询 `GET /task/{id}` → `status: running`
3. 完成后返回 `result_url`

URL 通常长这样：
```
https://aigc-files.bj.bcebos.com/xxx.png?Expires=1715...&Signature=...
```
带签名 + 过期时间，**所以你必须及时下载存到自己存储里**，不然链接就失效了。

---

## 三、这些图在大模型公司是怎么存的

不会塞进 MySQL/Postgres，标准架构是：

```
用户请求
   ↓
推理集群（GPU）生成 PNG 字节
   ↓
对象存储（OSS / S3 / COS / BOS）  ← 实际像素文件躺这里
   ↓
CDN 边缘节点缓存
   ↓
数据库只存元数据：{id, user_id, prompt, oss_key, created_at, size}
   ↓
签发预签名 URL 返回给用户
```

要点：
- **对象存储**（阿里云 OSS、AWS S3、腾讯云 COS、百度 BOS）是为海量二进制设计的，按 key/value 存，单文件可到 TB 级。
- **数据库**只存"指针"（OSS 路径）和元信息，不存像素本身。
- **CDN** 负责就近分发，国内一张图首次下载可能 200ms，缓存后 20ms。
- **预签名 URL（Presigned URL）**：临时授权，给你一个带签名的链接，过期自动失效，防盗链。

---

## 四、给你的工程建议

如果你自己做生图业务：

1. **生成完立刻下载到自己 OSS**，别依赖第三方 URL（会过期、可能被审核下架）。
2. **数据库存 OSS key 而不是完整 URL**，迁移/换 CDN 方便。
3. **接口给前端返回 URL 而不是 base64**，除非图很小（< 50KB）。
4. **加防盗链**：OSS 配 Referer 白名单，或用预签名 URL（推荐）。
5. **缩略图单独存一份**，列表页用缩略图，详情页才加载原图。

## 一、图片存哪里：三种主流方案

### 方案 A：对象存储 + CDN（**推荐，能上规模**）
```
用户上传 → 后端 → 阿里云 OSS / 腾讯云 COS / 七牛
                       ↓
                    CDN 加速域名（cdn.yoursite.com）
                       ↓
                    前端 <img src=...>
```
- **成本**：OSS 存储 ~0.12元/GB/月，CDN 流量 ~0.2元/GB，小项目一个月几块到几十块。
- **优点**：不占服务器带宽和磁盘，天然支持高并发，自带图片处理（缩略图、水印、WebP 转换）。
- **缺点**：要注册云账号、配 Bucket、配密钥。

### 方案 B：本机磁盘 + Nginx 直接对外（**小项目够用**）
```
后端把图存到 /data/uploads/
Nginx 配 location /static/ → 直接读磁盘
```
- **优点**：零成本，零依赖，半小时上线。
- **缺点**：占服务器带宽，单机故障即丢图，迁移痛苦，不适合 > 10GB 或高访问量。

### 方案 C：自建 MinIO（**有多台机器、想要 S3 协议但不想上云**）
本质是自己搭一个 S3 兼容的对象存储。学习成本中等，运维要自己扛。

## 二、案例库具体怎么选

## 三、图片 URL 要不要走 Nginx 反代

**结论：看你用哪个方案，以及你在不在意域名统一。**

### 不反代（直接用 OSS/CDN 域名）
```html
<img src="https://your-bucket.oss-cn-hangzhou.aliyuncs.com/xxx.jpg">
<img src="https://cdn-provider.com/xxx.jpg">
```
**优点**：最简单，性能最好（直连 CDN 边缘节点）。
**缺点**：
- 域名暴露云厂商，看起来"不专业"。
- 换厂商时，老链接全部失效。
- 可能被某些客户的网络环境屏蔽。

### 反代（统一到自己域名）
```
img.yoursite.com  ──nginx反代──→  OSS/CDN
```
Nginx 配置大致：
```nginx
server {
    listen 443 ssl;
    server_name img.yoursite.com;

    location / {
        proxy_pass https://your-bucket.oss-cn-hangzhou.aliyuncs.com;
        proxy_set_header Host your-bucket.oss-cn-hangzhou.aliyuncs.com;
        proxy_cache img_cache;
        proxy_cache_valid 200 30d;
        expires 30d;
    }
}
```
**优点**：
- 域名统一，品牌一致。
- 换底层存储不影响前端 URL。
- 可以加自己的鉴权、防盗链、日志。

**缺点**：
- 流量过你的服务器，**带宽和成本你扛**，失去了 CDN 直连的核心优势。
- 多一跳，延迟增加。

---

## 四、真正推荐的姿势：**CDN 绑自定义域名**（不用 nginx 反代）

绝大多数云厂商的 CDN/OSS 都支持**直接绑你自己的域名**：

1. 在阿里云 CDN 控制台加速域名填 `img.yoursite.com`
2. DNS 给 `img.yoursite.com` 加一条 CNAME 指向 CDN 给的地址
3. 配 HTTPS 证书（可以白嫖云厂商免费证书）

之后前端就是：
```html
<img src="https://img.yoursite.com/cases/2024/abc.jpg">
```

**这样既有自己的域名，又走 CDN 直连，零运维。**只有当你需要在图片访问路径上做复杂逻辑（鉴权、动态拼接、跨多个存储后端聚合）时，才需要 Nginx 反代。

读完文档了。你卡住的点其实不是图片本身，而是一个更底层的误解 ——**「传输的本质是 JSON」这句话是错的**。把这个纠正过来，整件事就通了。

---

## 一、先拆掉那个误解：网络上传的从来不是 JSON,是字节

网络（TCP/HTTP)只会做一件事:**把一串字节从 A 搬到 B**。它根本不知道什么是 JSON、什么是图片。

- JSON 是字节 —— 一段**文本**,本质是 `{`、`"`、`a` 这些字符对应的字节。
- 图片也是字节 —— PNG/JPEG 编码后的一串**二进制**字节。

所以 JSON 和图片是**平级的两种东西**,都是「字节」。不存在「图片要塞进 JSON 才能传」。说「传输的本质是 JSON」,就像说「运输的本质是纸箱」—— 纸箱只是其中一种包装,运输的本质是搬货。

**「结构化 / 非结构化」是说你怎么*用*这份数据,不是说它怎么*走*。** 在网线上它们没区别,都是字节流。

---

## 二、一次 HTTP 响应长什么样

一个 HTTP 响应就两部分:**头(headers)** + **体(body)**。body 里装什么都行,`Content-Type` 这个头负责告诉接收方「body 里这串字节该怎么解读」。

**情况一:body 直接就是图片**
```
HTTP/1.1 200 OK
Content-Type: image/png        ← 标签:这串字节是 PNG
Content-Length: 84213

<84213 个二进制字节>            ← body 就是图片本身,没有 JSON
```

**情况二:body 是 JSON**
```
HTTP/1.1 200 OK
Content-Type: application/json  ← 标签:这串字节是 JSON 文本
Content-Length: 56

{"image_url":"https://cdn.x.com/a.png"}   ← body 是文本
```

看出来了吗?body 装图片还是装 JSON,是**你的选择**,二者地位完全一样。

---

## 三、用「寄信」重讲文档里那三种方式

把 **JSON 想成一张信纸**(上面只能写字),**图片想成一个包裹**(塞不进信封)。问题就变成:包裹怎么送到对方手里?

### 方式 1:裸二进制 —— 不写信,直接寄包裹
body 直接就是图片字节,`Content-Type: image/png`,连 JSON 都没有。包裹上贴个标签写「这是 PNG」。
前端 `<img src="/api/avatar/123">` —— 浏览器收到这串字节,看标签,直接画出来。

### 方式 2:返回 URL —— 信纸上写「取件码」(业界主流)
信纸(JSON)上只写一句话:**「包裹在 cdn.x.com 的仓库,地址是 /a.png」**。
```json
{ "image_url": "https://cdn.x.com/a.png" }
```
**关键:图片字节根本没跟这次 JSON 一起走。** 前端拿到 URL 后,浏览器**另外再发一个全新的 HTTP 请求**去那个地址,**那第二次请求的响应**,body 才是真正的图片字节(就是上面的「情况一」)。

所以生图大模型给你 URL 时,是这样的:
- 第 1 次请求:你 → 大模型 API,响应是 JSON,里面一个 URL。**没有图片。**
- 第 2 次请求:你/浏览器 → 那个 URL(通常是 CDN),响应 body 才是图片字节。

两次独立的传输。这就是为什么文档说「URL 会过期你得赶紧下载」—— 第二趟你不去取,仓库就把包裹清理了。

### 方式 3:base64 —— 把包裹拆开,用文字一字不差抄在信纸上
JSON 是**纯文本**格式,它的字符串里**装不下任意二进制字节**(图片里会有 `0x00`、引号、换行这种字节,直接塞进去 JSON 就解析崩了)。

于是发明了 base64:**把任意二进制,无损地翻译成只用 64 个安全字符(`A-Z a-z 0-9 + /`)的文本**,这样就能塞进 JSON 字符串了。
```json
{ "image_base64": "iVBORw0KGgoAAAANSUhEUg..." }
```
代价:每 3 个字节变成 4 个字符,体积涨 ~33%(信纸变超厚)。前端再用 `data:image/png;base64,...` 把它还原成图片。

base64 不是「传输方式」,它是一个**编码转换**:二进制 → 文本,为了能寄生在 JSON 这种文本容器里。

## 四、把整条链路串起来(你做案例库的场景)

跟着「字节」走一遍,你就彻底懂了:

**上传(前端 → 后端):**
1. 用户选了张图。浏览器把图片字节放进一个 `multipart/form-data` 请求的 body 发给后端。—— 注意,**上传图片也不用 JSON**,用的是专门装文件的 `multipart` 格式。
2. 后端收到字节,调 OSS SDK,把这串字节原封不动推到对象存储,得到一个 key,比如 `cases/2026/abc.jpg`。
3. 后端往 MySQL 写一行:`{id, key:"cases/2026/abc.jpg", prompt, ...}`。**数据库里没有图片字节,只有这个字符串地址(指针)。**

**展示(后端 → 前端):**
4. 前端要案例列表,请求后端接口,后端返回 **JSON**:`[{ "id":1, "image_url":"https://img.yoursite.com/cases/2026/abc.jpg" }]`。—— JSON 里只有地址。
5. 前端渲染 `<img src="https://img.yoursite.com/cases/2026/abc.jpg">`。
6. 浏览器对这个 URL **再发一个独立请求**,CDN 响应的 body 才是真正的图片字节(`Content-Type: image/jpeg`),浏览器画出来。

## OSS:对象存储 —— 云上一个「无限大的文件仓库」

**OSS = Object Storage Service**(对象存储服务)。阿里云叫 OSS,AWS 叫 **S3**(这是鼻祖),腾讯云叫 COS,本质同一个东西。

### 它的存储模型:key → value

你给它一个 **key**(一个字符串,像 `cases/2026/abc.jpg`),它帮你存一坨 **字节**(value,就是图片/视频/任意文件)。每一个文件就是一个「对象 object」,所以叫对象存储。

```
PUT  cases/2026/abc.jpg  +  <图片字节>     ← 上传
GET  cases/2026/abc.jpg                    ← 下载,拿回那坨字节
```

⚠️ 一个反直觉的点:key 里的 `/` **不是真的文件夹**,OSS 内部是**扁平的**键值表,`/` 只是你给人看的命名习惯。这点和你电脑里的文件夹层级不一样。

### 为什么图片不存数据库、不存服务器磁盘,非要 OSS?

| | 数据库(MySQL) | 服务器本地磁盘 | 对象存储 OSS |
|---|---|---|---|
| 擅长 | 结构化的行列、查询 | 临时小文件 | **海量二进制大文件** |
| 容量 | 存大二进制会撑爆 | 几百 GB,有上限 | 近乎无限,单文件可 TB 级 |
| 可靠性 | — | **机器挂了图就没了** | 自动存多副本,几乎不丢 |
| 并发 | 一拉图就占连接 | 占服务器带宽和 CPU | 独立扛,不碰你的服务器 |
| 付费 | — | 买固定容量的盘 | 按实际用量付费 |

所以上一轮说的「数据库只存 key,不存像素」就是这个意思:**数据库存指针(`cases/2026/abc.jpg` 这个字符串),OSS 存指针指向的真东西(字节)。**

OSS 还自带很多附加能力,比如你给 URL 加参数就能实时生成缩略图:`abc.jpg?x-oss-process=image/resize,w_400`。

## CDN:内容分发网络 —— 解决「距离」问题

**CDN = Content Delivery Network**(内容分发网络)。

### 它要解决什么

你的 OSS 桶实际只建在**一个地方**(比如阿里云杭州机房)。如果北京、广州、乌鲁木齐的用户都来拉图,离杭州远的人就慢 —— 物理距离 + 网络跳数实打实地拖延迟。

CDN 在全国(乃至全球)铺了**几百上千个「边缘节点」**(缓存服务器)。用户访问图片时,**就近**从离他最近的那个节点拿。

### 它怎么工作(缓存)

```
广州用户第一次访问 abc.jpg
  → 广州边缘节点:我这没有
  → 回源:去杭州 OSS 取一份
  → 在广州节点本地缓存下来
  → 给用户                       ← 这次稍慢,比如 200ms

广州第二个用户访问 abc.jpg
  → 广州边缘节点:我缓存里有!
  → 直接给                        ← 这次很快,比如 20ms,根本不碰 OSS
```

所以 CDN 给你三个好处:**快**(就近)、**抗并发**(几百个节点分摊流量)、**省钱**(大部分请求被缓存挡住,不回源,省了 OSS 的流量费)。

---

## 两者怎么配合

记住一句话:

> **OSS 是「东西真正存在哪」(唯一真相),CDN 是「东西的就近缓存副本」。**

标准组合是 **CDN 套在 OSS 前面**:

```
                            ┌─ 北京边缘节点 ─┐
用户  ──→  CDN 加速域名  ──┼─ 广州边缘节点 ─┼──回源──→  OSS(杭州,源站)
         img.yoursite.com   └─ 成都边缘节点 ─┘            ↑
                                                    图片真身只在这
```

用户访问的永远是 CDN 域名,感觉不到背后 OSS 的存在。CDN 节点没缓存时才「回源」找 OSS 要。

## 一个类比收尾

- **OSS = 中央大仓库**:货(图片)的本体都在这,容量巨大,但只有一个,远的人去拿费劲。
- **CDN = 遍布各地的便利店**:你买瓶水不会跑去中央仓库,就近便利店拿。便利店没货了,才派车去总仓调一批回来囤着。
- **数据库 = 仓库的台账本**:本子上只记「3 号货架第 2 格放了 abc.jpg」,本子上没有货本身。

回到你的案例库:用户上传 → 后端推到 **OSS**(货进仓)→ 数据库记 key(台账)→ 前端访问 **CDN 域名**(去便利店拿)→ CDN 没有就回源 OSS。这套就是文档里推荐的「方案 A」。

## 一、它要解决的矛盾

OSS 里每个对象(bucket)有两种权限:

- **公共读(public-read)**:任何人拿到 URL 都能下载。
- **私有(private)**:除了你自己,谁都下不了。

矛盾来了:很多东西你**既不想公开,又得让特定的人临时看到**。比如用户的私密照片、付费内容、内部文档。

- 设成公共读 → URL 一泄露,全网随便下,还能被爬。
- 设成私有 → 那前端的 `<img>` 也加载不出来,因为浏览器没有你的密钥。

**预签名 URL(Presigned URL)就是这个矛盾的解:让一个私有对象,在「限定时间内」「针对这一个文件」临时可访问,过期自动失效。**

---

## 二、它长什么样

就是普通 OSS URL,**屁股后面挂了几个 query 参数**:

```
https://your-bucket.oss-cn-hangzhou.aliyuncs.com/cases/abc.jpg
        ?OSSAccessKeyId=LTAI5t...           ← 谁签发的
        &Expires=1716200000                 ← 有效期到这个时间戳(过了就废)
        &Signature=vJ8kQ2%2Bx...            ← 防伪签名(核心)
```

上一轮文档里生图大模型返回的 `...?Expires=1715...&Signature=...`,**那就是预签名 URL**。这也解释了文档为什么反复说「URL 会过期,赶紧下载」—— 因为它本来就是个限时凭证。

---

## 三、签名是怎么来的,为什么伪造不了

这是最关键的一环。

你在云厂商有一对密钥:**AccessKey ID**(公开,相当于账号)+ **AccessKey Secret**(机密,相当于密码)。**Secret 只存在你后端,永远不下发给前端、不进 URL。**

生成签名时,后端做一次 **HMAC** 运算(一种带密钥的哈希):

```
Signature = HMAC( AccessKey Secret , "GET\n/cases/abc.jpg\nExpires=1716200000" )
```

注意:签名是**对「文件路径 + 过期时间」整体算出来的**。于是:

- 谁想**改文件名**(把 `abc.jpg` 改成别人的 `secret.jpg`)→ 算出的签名对不上 → OSS 拒绝。
- 谁想**延长过期时间** → 同理,签名对不上 → 拒绝。
- 想自己伪造一个签名 → 必须有 AccessKey Secret,而它只在你后端。

OSS 那边怎么验证?**OSS 也知道你的 Secret**(你的账号嘛),它收到请求后,用同样的 Secret + 同样的路径和时间,**自己重算一遍 HMAC**,和 URL 里带的 `Signature` 比对:一致就放行,不一致就 403。

> 一个反直觉但重要的点:**后端生成预签名 URL 时,根本不需要联网请求 OSS。** 它就是一次本地的纯数学运算(HMAC)。所以生成 100 万个预签名 URL,零网络开销、零成本。

## 四、完整流程(下载方向)

```
1. 前端:我要看 cases/abc.jpg
        │
        ▼
2. 后端:先鉴权 —— 这个用户有权看吗?(查数据库/会话)
        有权 → 用 SDK 本地算出一个预签名 URL,有效期设 5 分钟
        │
        ▼
3. 后端把这个带签名的 URL 返回给前端(放在 JSON 里)
        │
        ▼
4. 前端 <img src="那个预签名URL">,浏览器直接请求 OSS
        │
        ▼
5. OSS 验签名 + 验时间:都通过 → 吐图片字节;过期 → 403
```

「该不该让你看」的判断在**第 2 步**(你后端的业务逻辑),OSS 只负责「签名对不对、有没有过期」这种机械校验。职责分得很干净。

## 五、还有一个方向:预签名「上传」URL(很实用)

预签名不只能用于下载(GET),还能用于上传(PUT)。这个能力很多人不知道,但对你做案例库很有用:

**普通上传**:前端 → 把图片字节发给你后端 → 后端再转推 OSS。图片字节**过了你服务器一道**,吃你的带宽和内存。

**预签名上传**:
1. 前端告诉后端「我要传一张图」。
2. 后端算一个**预签名 PUT URL**(限定:只能往 `cases/2026/xxx.jpg` 这个 key 传,5 分钟内有效),返回给前端。
3. 前端拿着这个 URL,**把图片字节直接 PUT 到 OSS**,根本不经过你后端。

好处:大文件上传不占你服务器一点带宽,后端只干了「签个名」的活。

## 六、回到你的案例库:你其实大概率不需要它

说实话 —— **产品案例库的图通常是要公开展示给客户看的**,本来就该让所有人都能访问。这种场景:

- bucket 设**公共读** + 套 **CDN** 就行,简单、能被 CDN 缓存、最快。
- 不用预签名 URL。每个预签名 URL 都带不同签名参数,会让 CDN **缓存失效**(每次都像新 URL),反而打架。

**预签名 URL 该出场的场景**是:案例库后台的**管理端**(只有员工能看的原图、未发布的草稿)、用户私密内容、付费下载。

如果你想要的只是「公开图,但别让人盗链到他自己网站上」,那用的是另一个工具 —— **防盗链(Referer 白名单)**:OSS/CDN 配置「只有从 `yoursite.com` 跳来的请求才给图」。它和预签名是两套机制,解决两个不同问题:

| | 预签名 URL | 防盗链(Referer 白名单) |
|---|---|---|
| 解决 | 私有内容,临时授权给特定人 | 公开内容,别被别的站盗用 |
| 粒度 | 单个文件 + 限时 | 整个 bucket / 域名 |
| 案例库 | 管理端、私密图用 | 公开展示图用 |

## 类比收尾

接着「中央仓库 / 便利店 / 台账本」那套:

> 私有 bucket = **上了锁的仓库**。
> 你后端 = **拿着总钥匙的仓库管理员**(AccessKey Secret 就是总钥匙)。
> 预签名 URL = 管理员开给来访者的一张**限时取货凭证**:「持此条,今天 18:00 前,可取 3 号架的 abc.jpg,仅此一件」。凭证上盖了**防伪钢印**(签名),改单子上任何一个字,钢印就对不上;想自己刻一个,得有总钥匙才行。仓库门口的保安(OSS)只认钢印和时间,不管你是谁。
