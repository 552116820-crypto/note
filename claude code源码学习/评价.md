## 助手

**写得怎么样**

整体偏强，像成熟商业产品，不像玩具项目。不是“优雅到教科书”的代码，而是“能扛复杂产品需求”的代码。

我会这么评价：

- 架构层面是强的。`QueryEngine / query / Tool / Task / AgentTool / coordinatorMode` 这套拆分是成体系的，见 [src/QueryEngine.ts](/mnt/e/claude-code-snapshot-backup/src/QueryEngine.ts)、[src/query.ts](/mnt/e/claude-code-snapshot-backup/src/query.ts)、[src/Tool.ts](/mnt/e/claude-code-snapshot-backup/src/Tool.ts)、[src/tools/AgentTool/AgentTool.tsx](/mnt/e/claude-code-snapshot-backup/src/tools/AgentTool/AgentTool.tsx)。
- 工程细节很老练。权限、并发、队列、恢复、远程任务轮询这些麻烦问题都认真处理了，不是只把 happy path 写通。
- 注释质量高，很多注释在解释“为什么这样做”，而不是只解释“这行代码在做什么”。

但它的问题也很明显：

- 复杂度很高，核心文件明显过大。比如 [src/main.tsx](/mnt/e/claude-code-snapshot-backup/src/main.tsx) 4683 行，[src/services/tools/toolExecution.ts](/mnt/e/claude-code-snapshot-backup/src/services/tools/toolExecution.ts) 1745 行，[src/query.ts](/mnt/e/claude-code-snapshot-backup/src/query.ts) 1729 行。
- feature flag 和条件分支债务很重。`src/` 里 `feature(` 出现了 960 次，`eslint-disable / biome-ignore / require()` 这类“绕复杂度走”的痕迹有 985 处。
- 可维护性开始靠“团队上下文”而不是靠局部可读性维持了。新成员接手会比较痛苦。
- 这个快照仓库的测试面偏弱。`src/` 有 1902 个文件，但 `tests/` 目前只有 1 个测试文件，不过这更像快照/移植仓库的限制，不一定代表原项目真实测试覆盖。

一句话说：  
这是“很强，但很重”的代码。值得学它的 runtime 设计和工程处理方式，不值得照抄它的巨型文件和 flag 膨胀。

**能看出它是 AI 写的吗**

不能可靠地下这个结论。只看代码，没法严肃证明“就是 AI 写的”。

如果非要判断，我不会说它是“纯 AI 生成”的代码库，我更倾向于：

`人主导的生产代码，可能夹杂 AI 辅助和编译/变换产物`

原因是：

- 很多地方有明显的人类工程经验，尤其是 race condition、缓存稳定性、恢复逻辑、权限边界这些，不太像一次性 AI 直接生成出来的。
- 注释里有很多非常具体的性能、历史兼容、线上问题背景，这更像长期维护中的人写法。
- 有些奇怪痕迹看起来不像 AI，而像“编译/代码变换后的产物”，比如部分文件里的 `react.memo_cache_sentinel` 相关模式。
- 但局部也可能用了 AI 辅助，因为一些注释风格、样板化防御代码、重复结构，确实有 AI-assisted 的味道。

所以更准确的说法是：

- `不像纯 AI 项目`
- `很可能有人类主导`
- `不排除部分代码经过 AI 辅助生成或大规模自动化变换`
