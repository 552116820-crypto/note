# 企业微信 AI 销售工作台 — 完整框架规划

> 会话记录整理，涵盖：微信小程序架构 → 企微AI应用架构 → AI原生销售全流程平台设计

---

## 一、微信小程序主流技术架构

### 前端

| 层面            | 主流方案                                        |
| ------------- | ------------------------------------------- |
| **原生开发**      | WXML + WXSS + JavaScript/TypeScript         |
| **跨端框架（最流行）** | **uni-app**（Vue 语法）、**Taro**（React 语法）      |
| **状态管理**      | MobX-miniprogram、Pinia（uni-app）、Redux（Taro） |
| **UI 组件库**    | WeUI、Vant Weapp、uView（uni-app）、NutUI（Taro）  |

> 实际项目中，**uni-app** 占比最高，因为一套代码可同时发布到微信/支付宝/抖音小程序 + H5 + App。

### 后端

| 层面                    | 主流方案                                                                                              |
| --------------------- | ------------------------------------------------------------------------------------------------- |
| **语言/框架**             | **Java（Spring Boot）** 最多；其次是 **Node.js（Nest/Express/Koa）**、**Go（Gin）**、**Python（FastAPI/Django）** |
| **BaaS / Serverless** | 微信云开发（CloudBase）、阿里云函数计算、腾讯云 SCF                                                                  |
| **数据库**               | MySQL、PostgreSQL、MongoDB；云开发用自带的云数据库                                                              |
| **缓存**                | Redis                                                                                             |
| **对象存储**              | 腾讯云 COS / 阿里云 OSS（图片、文件）                                                                          |
| **API 风格**            | RESTful 为主，部分用 GraphQL                                                                            |

### 典型架构模式

**轻量项目（个人/小团队）：**
```
uni-app → 微信云开发（云函数 + 云数据库 + 云存储）
```
零运维，开箱即用，适合 MVP 和小规模应用。

**中大型项目（企业级）：**
```
uni-app/Taro → Nginx → Spring Boot / Go 微服务 → MySQL + Redis
                         ↓
                   消息队列（RabbitMQ/Kafka）
                         ↓
                   第三方服务（支付、推送、OSS）
```

### 选型建议

- **快速验证想法** → uni-app + 微信云开发，几乎零配置
- **团队有 Java 背景** → uni-app + Spring Boot
- **高并发/性能敏感** → Taro/原生 + Go（Gin）
- **全栈 JS 开发者** → Taro + Node.js（NestJS）

### 前端（工作台入口）

企业微信工作台的应用主要有两种形态：

| 形态         | 适用场景                        | 推荐方案                                             |
| ---------- | --------------------------- | ------------------------------------------------ |
| **H5 应用页** | 复杂交互界面（知识库管理、Dashboard、配置台） | **Vue 3 + Vant** 或 **React + Ant Design Mobile** |
| **机器人对话**  | 对话式 AI（问答、指令触发）             | 企微群机器人 / 单聊机器人（纯后端）                              |

> 企微工作台里的"应用主页"本质是 **H5 页面**（不是小程序），通过 JSSDK 调用企微能力。

**前端关键依赖：**
```
@wecom/jssdk          # 企微 JS-SDK，调用原生能力
wecom-sidebar-jssdk   # 侧边栏 SDK（如果做客户联系场景）
```

### 后端（核心推荐）

#### 方案 A：Python 全栈（最适合 AI 场景）

```
FastAPI / Flask
├── 企微接入模块（回调验证、OAuth、消息收发）
├── AI 模块
│   ├── LangChain / LlamaIndex（RAG 编排）
│   ├── LLM 接入（DeepSeek / 通义千问 / Claude API）
│   └── 向量检索（Milvus / Weaviate / pgvector）
├── 业务模块（审批、日报、任务等）
└── Celery（异步任务队列，处理耗时 AI 推理）
```

**为什么选 Python：** AI/ML 生态最成熟，LangChain、LlamaIndex、各 LLM SDK 都是 Python 优先。

#### 方案 B：Node.js + Python 混合（团队前端强）

```
NestJS（主服务）
├── 企微接入、鉴权、业务逻辑
├── BullMQ（任务队列）
└── 调用 Python AI 微服务（gRPC / HTTP）

Python FastAPI（AI 微服务）
├── RAG Pipeline
├── LLM 调用
└── 向量检索
```

#### 方案 C：Java + Python 混合（企业 Java 团队）

```
Spring Boot（主服务）
├── 企微 SDK 集成（wecom-sdk）
├── 业务逻辑、权限、审计
└── 调用 Python AI 服务

Python FastAPI（AI 微服务）
├── 同方案 A 的 AI 模块
```

#### LLM 选型建议（国内部署）

| 模型 | 优势 | 适用 |
|------|------|------|
| **DeepSeek-V3/R1** | 性价比极高，中文能力强 | 通用问答、代码 |
| **通义千问 Qwen2.5** | 阿里云生态集成好 | 已用阿里云的团队 |
| **Claude API** | 推理能力最强，长文本 | 复杂分析、文档理解 |
| **私有化部署（Ollama + Qwen）** | 数据不出内网 | 对数据安全要求极高 |

### 企业微信接入要点

```python
# 企微回调核心流程（Python 示例）
@app.post("/wecom/callback")
async def wecom_callback(request: Request):
    # 1. 验证签名 (msg_signature, timestamp, nonce)
    # 2. 解密消息 (AES)
    # 3. 根据 MsgType 分发处理
    #    - text → AI 对话
    #    - event → 菜单点击、审批回调
    # 4. 加密回复消息
```

**需要对接的企微 API：**
- 身份验证：OAuth 2.0 登录、JSAPI 鉴权
- 消息推送：应用消息、群机器人 Webhook
- 通讯录：获取部门/人员信息
- 审批/日程/会议：如果 AI 需要操作这些

### 推荐的快速起步方案

如果想**最快出成果**，推荐：

```
Dify (开源 AI 平台，自部署)
  +
企微机器人接入层 (Python/Node 薄层)
```

**Dify 的优势：**
- 可视化编排 RAG、Agent、Workflow
- 内置支持主流 LLM
- 提供现成的 API，只需写一层企微消息转发
- 开源可私有化部署

```
企微消息 → 你的中间层(验签/解密) → Dify API → AI 回复 → 企微回复
```

这样只需要写 **200-300 行胶水代码** 就能跑起来一个企微 AI 助手。

**MEDDPICC 落地到系统示例：**

| 要素 | AI 行为 |
|------|--------|
| **M**etrics（量化价值） | 自动从会话中提取客户提到的业务指标，未提到时提醒销售追问 |
| **E**conomic Buyer（决策人） | 分析通讯录+会话记录，判断是否已接触到经济决策人 |

| **P**aper Process（合同流程） | 跟踪合同阶段，预警卡点 |
| **I**dentify Pain（痛点） | 从对话中抽取显性/隐性痛点，映射到产品能力 |
| **C**hampion（内部支持者） | 分析互动频次和态度，识别谁是你的 Champion |
| **C**ompetition（竞争） | 自动匹配竞品情报，生成攻防话术 |

### AI Agent 设计

用 **LangGraph** 编排多个专业 Agent：

```python
# 核心 Agent 架构示意

class SalesAISystem:
    """销售 AI 系统 - 多 Agent 协作"""

    agents = {
        # ---- 拓客 ----
        "prospecting": ProspectingAgent(
            tools=[
                search_company_info,      # 企业工商查询
                search_news,              # 新闻舆情
                analyze_customer_profile, # 画像分析
                score_lead,               # 线索评分
                find_lookalike,           # 相似客户发现
            ]
        ),

        # ---- 拜访 ----
        "visit_prep": VisitPrepAgent(
            tools=[
                get_customer_history,     # 历史沟通记录
                get_deal_status,          # 商机状态
                generate_checklist,       # 方法论检查清单
                generate_talking_points,  # 话术要点
            ]
        ),

        # ---- 实时辅助 ----
        "realtime_coach": RealtimeCoachAgent(
            tools=[
                detect_objection,         # 异议识别
                search_competitive_intel, # 竞品情报
                suggest_response,         # 话术推荐
                check_methodology,        # 方法论提醒
            ]
        ),

        # ---- 谈单 ----
        "deal_advisor": DealAdvisorAgent(
            tools=[
                assess_meddpicc,          # MEDDPICC 评估
                predict_win_rate,         # 赢率预测
                identify_risks,           # 风险识别
                suggest_next_steps,       # 下一步建议
                generate_proposal,        # 方案生成
            ]
        ),

        # ---- 复盘 ----
        "review_coach": ReviewCoachAgent(
            tools=[
                analyze_conversation,     # 会话分析
                score_methodology,        # 方法论评分
                extract_insights,         # 洞察提取
                generate_coaching,        # 教练建议
                trend_analysis,           # 趋势分析
            ]
        ),
    }
```

### 技术选型汇总

| 层面 | 选型 | 理由 |
|------|------|------|
| **前端** | Vue 3 + Vant + 企微 JSSDK | 企微 H5 生态适配最好 |
| **后端框架** | FastAPI (Python) | AI 生态最强，异步性能好 |
| **Agent 框架** | LangGraph | 复杂多 Agent 编排，状态管理 |
| **RAG** | LlamaIndex | 文档解析 + 检索一体化 |
| **LLM** | DeepSeek (主) + Claude (复杂推理) | 成本与能力平衡 |
| **Embedding** | BGE-M3 (自部署) | 中文效果最好，无 API 成本 |
| **向量库** | Milvus 或 pgvector | Milvus 性能强，pgvector 运维简单 |
| **数据库** | PostgreSQL | 支持 JSON + pgvector 一体化 |
| **缓存/队列** | Redis + Celery | 实时辅助场景需要低延迟 |
| **部署** | Docker Compose → K8s | 先跑通再扩展 |
