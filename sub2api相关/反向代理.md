## 反向代理是什么意思

## 反向代理（Reverse Proxy）

### 一句话解释

客户端以为自己在跟目标服务器通信，实际上中间有个"代理人"在转发请求。

### 图示

```
普通访问（无代理）:
  用户浏览器  ──────→  Sub2API 服务 (localhost:8080)

反向代理:
  用户浏览器  ──→  Nginx (443端口)  ──→  Sub2API 服务 (localhost:8080)
                   ↑
              用户只看到这里
              (https://api.example.com)
```

### 为什么需要它？

| 作用        | 说明                             |
| --------- | ------------------------------ |
| **HTTPS** | Nginx 处理 SSL 证书，后端服务不用管加密      |
| **隐藏端口**  | 用户访问标准的 443 端口，而不是 8080 这种内部端口 |
| **域名绑定**  | 让 `api.example.com` 指向你的服务     |
| **安全防护**  | 后端服务不直接暴露在公网，Nginx 挡在前面        |
| **负载均衡**  | 如果有多个后端实例，Nginx 可以分发请求         |

### 和"正向代理"的区别

```
正向代理（VPN/梯子）:  用户 → 代理 → 目标网站
                       代理帮用户去访问，目标网站不知道真实用户是谁

反向代理（Nginx）:     用户 → 代理 → 后端服务
                       代理帮服务器接客，用户不知道真实服务器在哪
```

- **正向代理**：代理站在**用户**这边，帮用户隐藏身份
- **反向代理**：代理站在**服务器**这边，帮服务器接收请求

### Sub2API 场景下的典型配置

```nginx
server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://127.0.0.1:8080;  # 转发到 Sub2API
    }
}
```

用户访问 `https://api.example.com` → Nginx 接收并解密 HTTPS → 转发到本地 8080 端口的 Sub2API 服务。

简单理解：**反向代理就是给你的服务加了个"前台接待"**，所有请求先经过它，再转给后面真正干活的服务。

## 反向代理详解（扩展）

### 正向 vs 反向 完整对比

```
【正向代理】— 代理客户端
客户端 → [正向代理] → 互联网 → 服务器
例：VPN、科学上网，服务器不知道真正的客户端是谁

【反向代理】— 代理服务器
客户端 → 互联网 → [反向代理] → 后端服务器(们)
例：Nginx、Cloudflare，客户端不知道真正的后端是谁
```

**一句话区分**：正向代理藏的是**客户端**，反向代理藏的是**服务器**。

### 反向代理核心功能详解

#### 1. 负载均衡（Load Balancing）

把请求分发到多台后端服务器，避免单点过载：

```nginx
upstream backend {
    server 192.168.1.10:8080 weight=3;  # 权重高，分配更多请求
    server 192.168.1.11:8080 weight=1;
    server 192.168.1.12:8080 backup;    # 备用，前面都挂了才启用
}

server {
    location / {
        proxy_pass http://backend;
    }
}
```

常见策略：
| 策略 | 说明 |
|------|------|
| 轮询（Round Robin） | 依次分配，默认方式 |
| 加权轮询 | 性能好的机器多分点 |
| IP Hash | 同一 IP 始终到同一后端（保持会话） |
| 最少连接 | 哪台连接数少就发给谁 |

#### 2. SSL/TLS 终止（SSL Termination）

反向代理统一处理 HTTPS，后端只跑 HTTP，省去每台服务器都配证书的麻烦：

```
客户端 ──HTTPS──→ [反向代理 处理 SSL] ──HTTP──→ 后端服务
```

```nginx
server {
    listen 443 ssl;
    ssl_certificate     /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    location / {
        proxy_pass http://127.0.0.1:3000;  # 后端只需 HTTP
    }
}
```

#### 3. 缓存（Caching）

反向代理可以缓存后端响应，减少重复请求：

```nginx
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=my_cache:10m max_size=1g;

server {
    location /api/ {
        proxy_cache my_cache;
        proxy_cache_valid 200 10m;   # 200 响应缓存 10 分钟
        proxy_pass http://backend;
    }
}
```

#### 4. 安全防护

| 功能 | 说明 |
|------|------|
| 隐藏后端 | 攻击者看不到真实服务器 IP、端口、架构 |
| WAF | 可以在代理层过滤恶意请求（SQL注入、XSS） |
| 限流 | 限制每个 IP 的请求频率，防 DDoS |
| IP 黑白名单 | 在入口就拦截 |

```nginx
# 限流示例
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

location /api/ {
    limit_req zone=api_limit burst=20 nodelay;
    proxy_pass http://backend;
}
```

#### 5. 路径路由 / 微服务网关

一个域名，根据路径转发到不同服务：

```nginx
server {
    server_name example.com;

    location /api/users {
        proxy_pass http://user-service:8001;
    }
    location /api/orders {
        proxy_pass http://order-service:8002;
    }
    location / {
        proxy_pass http://frontend:3000;
    }
}
```

这就是**API 网关**的雏形。

#### 6. 压缩与优化

```nginx
gzip on;
gzip_types text/plain application/json text/css application/javascript;
gzip_min_length 1024;
```

后端不用管压缩，代理层统一处理。

### 常见的反向代理软件

| 软件 | 特点 |
|------|------|
| **Nginx** | 最主流，高性能，配置灵活 |
| **Caddy** | 自动 HTTPS，配置极简（Caddyfile） |
| **Traefik** | 容器原生，自动发现 Docker/K8s 服务 |
| **HAProxy** | 专注负载均衡，性能极致 |
| **Cloudflare** | CDN + 反向代理，SaaS 形式 |
| **Apache** | 老牌，mod_proxy 模块 |

### 跟 CLIProxyAPI 的关系

CLIProxyAPI 本质上也是一种**反向代理思想**：

```
API 客户端（如 ChatBox、Cursor）
        ↓ 标准 API 请求
  [CLIProxyAPI — 反向代理层]
        ↓ 翻译成 CLI 调用
  Claude Code / Gemini CLI（实际干活的）
        ↓ 返回结果
  [CLIProxyAPI — 封装成 API 响应]
        ↓
  API 客户端收到标准格式的回复
```

客户端以为自己在调官方 API，实际上后面是 CLI 工具在跑。

### 一个现实场景串起来

假设你有个小项目部署：

```
                    互联网用户
                        ↓
              [Cloudflare CDN/防护]
                        ↓
              [Nginx 反向代理 :443]
                ├─ /           → React 前端 :3000
                ├─ /api/       → Node.js 后端 :8080
                ├─ /ws/        → WebSocket 服务 :8081
                └─ /grafana/   → 监控面板 :3001
```

一台机器、一个域名、一个入口，Nginx 全权调度。
