没有这三个数，"做完了"永远说不清。

两个项目共享同一个分层骨架，从零搭也照这个分：

## 第 2 段：先定契约（写一行代码之前）

契约是项目能不能并行扩规模的命门。两个项目最后能跑通都靠这套。

### 2.1 目录契约
```
项目根/
├── CLAUDE.md                    # 主会话职责（subagent 不读）
├── .claude/
│   ├── skills/<skill-name>/SKILL.md   # subagent 入口，带 trace_token
│   ├── agents/*.md              # 单功能 subagent 定义
│   └── commands/*.md            # 一键入口（/run-xxx）
├── config/*.yaml                # 规则/词典/选择器（声明式）
├── pipeline/*.py                # L1 确定层
├── scripts/*.py                 # 一次性脚本（批次 build / 修表）
├── <unit>/                      # 主战场（A=positions/P0X, B=findings/批次/公司）
│   └── README.md / *.jsonl / decisions/ / evidence/ / result.xlsx
├── work/scratch/                # 临时区，gitignored
├── docs/                        # 规范文档（workflow / conventions / index）
├── tests/                       # 至少 3 个：归一化 / 评分 / xlsx 写入
└── output/.legacy/              # 旧版本归档（不删，备查）
```

### 2.2 数据契约（JSONL / TSV schema）
- **JSONL** 用于"中间态"（verified.jsonl / unmatched.jsonl），一行一对象，可流式追加、可 cat 看。
- **TSV** 用于"人审 override"（decisions/url_overrides.tsv），列固定、列名是英文 snake_case，可直接 `pandas.read_csv(sep='\t')`。
- 每条记录必带：`source_batch / first_seen_at / status / reason_tag / evidence_url`。**reason_tag 用枚举不要自由文本**，否则统计 unmatched 原因时就乱了。

### 2.3 状态机（决定一条记录能不能进表）
```
candidate ─┬─► verified ──► matched ──► 进表
           ├─► unmatched (reason: customer_logo / equipment_vendor / dead_url / category_miss / ...)
           ├─► pending  (待人审 / 待二次抓取)
           └─► blocked  (rate limit / cloudflare / 等条件恢复)
```
**关键**：blocked ≠ unmatched。blocked 是"暂时打不开"，下次重跑可能 verified；unmatched 是"判定不要"。两个项目早期都把这两个混了，后来都拆开。

### 2.4 subagent 返回契约（如果走 β 模式）
B 项目的做法直接抄：
```json
{
  "unit_id": "xxx",
  "status": "ok | blocked | unmatched",
  "findings_path": "findings/批次/公司.json",
  "evidence_count": 12,
  "blockers": ["cloudflare", "pdf_only"],
  "skill_trace": "SKILL-RESEARCH-ONE-COMPANY-v11-20260511"
}
```
- **`skill_trace` 必须主会话校验**——subagent 没读到最新 SKILL.md 时你能立刻发现。
- 主会话**不读** subagent 的 transcript，只接 JSON。这条是 token 预算的命。
- output_file 让 subagent 自己 Write，主会话 Read 那个文件，**不要走对话返回大字符串**。

## 第 3 段：流水线 8 步骨架（A 项目 playbook 抽象版）

每步必须满足三条：**可独立重跑、可幂等、有快照**。

| # | 步骤 | 输入 | 输出 | 谁来做 |
|---|---|---|---|---|
| 1 | **种子采集** | 展会页 / 名单 / 关键词 | `seeds.jsonl` | scrape_*.py + WebSearch |
| 2 | **归一化** | seeds | `candidates.jsonl`（公司名 canonical + 拼音 bucket） | normalize.py（rapidfuzz + pypinyin）|
| 3 | **URL 活检** | candidates | `url_check.jsonl`（live / dead / parking / unknown） | url_check.py（stdlib + parking blacklist）|
| 4 | **语义抽取** | url_check.live | `findings/*.json`（产品 / 品类 / 行业 / 证据 URL） | Claude（α 主会话 or β subagent）|
| 5 | **评分 / 匹配** | findings | `matched.jsonl` + `unmatched.jsonl` | assign.py（纯规则）|
| 6 | **人审 override** | unmatched + matched | `decisions/*.tsv` 合并 | 人 + apply_decisions.py |
| 7 | **xlsx 出表** | matched + decisions | `result.xlsx` | build_xlsx.py（openpyxl）|
| 8 | **回归 / diff** | 上一版 xlsx | `diff_report.md` | scripts/diff_xlsx.py |

**第 8 步两个项目都没做好**——增量调研最容易踩"这次少了 3 家但没人发现"的坑，从零搭项目时建议第 8 步一开始就写。

## 第 4 段：Claude 这一层怎么用（α / β / γ 三档）

| 模式 | 描述 | 适用 | 代表 |
|---|---|---|---|
| **α 单会话** | 主会话本体做 vision + 网搜 | 单元数 < 30、要看图、要灵活判断 | A 项目 |
| **β 主+subagent 并行** | 主会话调度，N 个 subagent 同时跑 | 单元数 30–500、单家工作流稳定 | B 项目 |
| **γ schedule / cron 远程** | 远程 agent 夜间跑 / 周期跑 | 持续监控（"每周扫一遍同行有没有新品"） | 两项目都没做 |

### 选哪个的判据
- 工作流**没稳定**前别上 β：subagent 跑 100 家一起翻车，你看不到中间过程。先在 α 跑通 3–5 家，再把流程提到 SKILL.md，再上 β。
- 上 β 必装三件套：
  1. SKILL.md frontmatter 的 **trace_token** 版本号
  2. 主会话校验 trace_token 不匹配的处置策略（B 项目：1 次警告 / 连续 3 次停 / count=0 重派）
  3. subagent **不读** CLAUDE.md（隔离），SKILL.md 必须自包含
- γ 是发散方向：用 `schedule` 技能把 β 工作流挂到 cron 上，每周拉一次同行新品，diff 出"本周新增"清单。

## 第 7 段：发散——两项目都没做但应该做的事

### 7.1 评估集（最重要的缺失）
两个项目都没有 ground truth golden set。从零搭时**第一天就标 30 家**，每次改 prompt / 改 SKILL.md 跑回归：
```
eval/golden.jsonl     # 30 家手标
eval/run.py           # 跑当前流水线 vs golden，输出 precision/recall/F1
eval/report_v11.md    # 这次改动 → 指标变化
```
没有 eval，SKILL.md 的 trace_token 升版就是猜的。

### 7.2 增量调研（公司库不是一次性）
- xlsx 每次出**带时间戳的版本号**：`result_v3_20260516.xlsx`
- 写 `scripts/diff_xlsx.py`：对比两版 xlsx，输出"新增公司 / 删除公司 / 字段变化"。
- 状态加一个 `last_verified_at`，**超过 90 天自动进重抓队列**。

### 7.3 证据三件套（不要只存 URL）
两项目目前只存 evidence_url。但官网会改、404，证据会消失。改成：
```json
{
  "evidence_url": "...",
  "evidence_text": "页面相关段落原文 200 字",
  "evidence_screenshot": "evidence/<unit>/<sha>.png",
  "captured_at": "2026-05-16T12:34:56Z"
}
```
存到 `evidence/` 目录（gitignored 或单独 LFS）。后面争议时甲方点 URL 是 404，你还能拿出当时的截图和文本。

### 7.4 LLM-as-judge 自审
跑完 N 家后，让 Claude 再过一遍：
- 行业列里 ✓ 是否真的对应该产品？
- 品类命中是否真的命中 main_categories？
- unmatched 的 reason_tag 是否选对？
随机抽 10% 自审，把不一致的进 pending。两项目都是用户手审，scale 上 1000 家时撑不住。

### 7.5 Token 预算 / Rate limit 管理
β 模式下：
- subagent 并发数动态调，根据上下文剩余 token 算 `max_parallel = min(20, remaining_tokens / 单次_token)`。
- subagent 撞 rate limit → 主会话 `curl` 兜底（B 项目原话）。**但 curl 不会看图**——所以图重的项目 rate limit 没法兜底，要换账号 / 等。
- 提前买配额，别让单次跑挂在第 87 家。

### 7.6 抓取退路链
B 项目已经有完整的（curl_cffi → chrome-devtools MCP），从零搭项目应该一开始就规划：
```
正常: WebFetch
  ↓ 失败
Cloudflare: curl_cffi
  ↓ 失败  
SPA 渲染: playwright / chrome-devtools MCP
  ↓ 失败
PDF only: pdfplumber / pymupdf
  ↓ 失败
图片 only: OCR（manual + Read / paddle）
  ↓ 失败
blocked，进 decisions/ 人工搞定
```
每层都要有一个"我什么时候放弃这层"的判据，不然会一直在底层卡着。

### 7.7 共享层（多项目复用）
搭第二个项目时再悟到就晚了。从一开始就把这些抽到 `~/code/research-pipelines-common/`：
- `fetch.py`（curl_cffi + 重试 + parking 检测）
- `normalize.py`（中文公司名归一化）
- `xlsx_helpers.py`（hyperlink 公式 + 合并单元格 + 修漂移）
- `ocr.py` / `pdf.py`
- `evidence.py`（证据三件套写入）

两个项目独立做的，至少有 6–8 个工具函数应该共享。

### 7.8 半结构化的可观测性
跑大批次时，加一个 `dashboard.md`，每跑完一批自动更新：
