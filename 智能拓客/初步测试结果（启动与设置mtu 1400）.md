## 一、现象：小查询秒回，大结果集必挂

排查时的关键实验（同一张 70 行的 `amer_industry` 表）：

```
SELECT 1                          → 0.4s  OK
SELECT COUNT(*) FROM amer_industry → 0.4s  OK
SELECT id   FROM amer_industry (70行,纯数字) → 1.0s  OK
SELECT id,name LIMIT 35           → 0.4s  OK
SELECT id,name LIMIT 69           → 12s   超时挂掉 ❌
SELECT id,name (全 70 行)         → 15s   超时挂掉 ❌
```

规律很干净：**结果集小（一个网络包能装下）就回；一旦超过约 1500 字节、需要发第二个"满包"，就卡死超时**。地理候选查询动辄返回几千行，必然踩雷——所以"所有查询卡死"。

## 二、链路：数据库流量走了一条"撒谎"的隧道

```
ip route get 47.114.77.122
→ 47.114.77.122 via 198.18.0.2 dev eth2
```

- `198.18.0.x` 是代理/VPN 工具（Clash 这类）TUN 虚拟网卡的惯用地址段 → 说明到数据库的流量是**经代理隧道**走的。
- 而 `eth2` 的 `MTU = 9000`（巨型帧）。但实测路径根本扛不住 9000：

```
ping -M do -s 8972 (9000字节包)  → FAIL 太大
ping -M do -s 1472 (1500字节包)  → OK
```

**真实路径 MTU 只有 ~1500，eth2 却对外宣称 9000。** 这就是病根。

## 三、机制：为什么"宣称 9000"会导致大结果黑洞

1. pymysql 连 MySQL 时，TCP 三次握手会**根据本地网卡 MTU 协商 MSS**（单个 TCP 段最大多大）。eth2 说自己 MTU 9000，于是客户端通告一个巨大的 MSS（~8960），服务器据此**可以一次发很大的数据段**。
2. **小结果**（几百字节）→ 一个小包就发完，<1500，顺利穿过隧道 → 正常。
3. **大结果**（服务器要发一个按 9000 裁的满包）→ 这个超大包进了只能走 ~1500 的真实路径就**太大了**。包带着 `DF`（不可分片）标志，本应触发 ICMP "需要分片" 回告服务器让它改小（这就是 PMTUD 路径 MTU 发现）——但**这条隧道里那个 ICMP 回不来**（典型的 *PMTUD 黑洞*）。
4. 结果：超大包被**静默丢弃**，服务器以为发出去了一直重传，客户端永远收不到完整结果 → 一直阻塞到 `read_timeout` 触发 → 报 `Lost connection ... timed out`。

> 补充证据：我试过开 `CLIENT.COMPRESS`（老的导出脚本用过它），结果**连握手都挂了**——因为压缩协议的握手包也变大、同样撞黑洞。这反过来印证了"就是大包过不去"。

## 四、为什么 `mtu 1400` 能修

```
sudo ip link set dev eth2 mtu 1400
```

把 eth2 的 MTU 降到 1400 后，**之后新建的 TCP 连接协商出的 MSS 只有 ~1360**，服务器最多只会发 ≤1400 字节的段——全部能塞进隧道，再也没有超大包、没有黑洞，大结果集正常流动。

实测改完立刻见效：之前必挂的 `SELECT id,name(70)` 0.4s 回，**3000 行 1.1s 回**，整条 pipeline 跑通。

选 1400 而不是 1500，是给隧道自身的封装开销留点余量（更稳）。

## 五、为什么每次 WSL 重启要重设

`ip link set` 改的是内核里这块网卡的**运行时**参数。WSL 重启会重建网络接口，eth2 又变回默认的 9000，所以失效。要**一劳永逸**，两条路：

- 在你的代理/VPN 工具配置里把 TUN 的 MTU 直接设成 1400（最干净）；
- 或写个开机/启动脚本自动执行那条 `ip link set`。
