# 实现路径建议

## 核心判断：要不要落库？

**部分落库，分层计算。** 关键观察：5 个维度里 **行业匹配 + 距离（共 70 分）是"按销售算"的**，**规模/标签/行为（30 分）是"客户自身属性"**。两类东西生命周期完全不同，存储策略也应该分开。

## 推荐分层方案

| 维度        | 性质            | 存储/计算策略                                |
| --------- | ------------- | -------------------------------------- |
| 企业规模（10）  | 客户静态属性        | **物化到 `customer_score` 表**，客户字段变更时触发更新 |
| 客户标签（10）  | 客户静态属性        | 同上                                     |
| 行为时间轴（10） | 客户半动态         | 同上，但需要事件触发刷新（有新行为时）                    |
| 行业匹配（40）  | **账号×客户**     | 离线算"账号行业画像"（每天 T+1），查询时与客户行业比对         |
| 距离（30）    | **账号位置×客户位置** | **实时算**（销售定位会变，不能预存）                   |

最终分 = 物化基础分（30） + 账号行业画像匹配（40） + 实时距离分（30）

## 实现路径（5 步）

1. **建表**：
   - `customer_score(customer_id, size_score, tag_score, behavior_score, updated_at)` — 物化客户基础分
   - `account_industry_profile(account_id, industry_code, weight, updated_at)` — 账号历史客户行业画像

2. **物化基础分（30 分部分）**：
   - 写一个事件监听 / 定时 job：客户的"实缴资金、标签、行为"字段变更时，重算该客户的基础分并 upsert 到 `customer_score`
   - 首次冷启动：跑一次全量

3. **账号行业画像（40 分部分）**：
   - 每天凌晨 T+1：扫该销售已成交/跟进客户，聚合行业分布，存到 `account_industry_profile`
   - 简化版：只取 TOP1 行业；进阶版：行业向量

4. **公海列表查询**：
   - 拿到当前销售的定位 + 行业画像
   - SQL/ES 里：`SELECT *, base_score + industry_match(account_profile, customer.industry)*40 + distance_score(account_loc, customer_loc) AS ai_score FROM customer JOIN customer_score ORDER BY ai_score DESC`
   - 量大就上 ES（geo_distance + function_score）

5. **AI 分要不要展示给销售看？**
   - 只用来排序：不用持久化最终分，每次算就行
   - 要展示分数：列表查询结果带回分数即可，**也不必落库**（因为不同销售看到的分不同）

## 三个关键决策点（在动手前要明确）

1. **公海客户量级？** < 1万：MVP 直接 SQL 实时算就行；> 10万：直接上 ES。
2. **销售定位是 GPS 实时还是公司地址固定？** 决定距离能不能预算。
3. **"历史客户行业"的范围？** 仅成交？包含跟进？时间窗口多长？这决定行业画像准不准。

# 落地方案（百万级 + 实时 GPS + 仅成交画像）

## 选型结论：**Elasticsearch + 物化基础分 + 实时打分函数**

为什么不是 SQL：100万 × 每查询都算 geo + 多字段排序 + 翻页 = SQL 必爆。  
为什么不是 PostGIS：geo 行，但 function_score 的多维度组合打分 ES 更顺手。

## 数据流

```
MySQL（客户主库）
  ├─ CDC（Canal/Debezium）──→ Elasticsearch（公海查询引擎）
  ├─ 事件触发 ──→ customer_score 表（物化30分）──→ 同步进 ES 文档
  └─ T+1 离线 job ──→ account_industry_profile 表（成交行业画像）──→ Redis 缓存
                                                                      │
公海列表 API：销售GPS + Redis画像 + ES function_score query ←─────────┘
```

## ES 索引核心字段

```json
{
  "customer_id":      "keyword",
  "industry_code":    "keyword",
  "size_score":       "byte",      // 0 or 10，物化
  "tag_score":        "byte",      // 0 or 10，物化
  "behavior_score":   "byte",      // 0 or 10，物化
  "location":         "geo_point", // 客户地理位置
  "pool_status":      "keyword"    // 公海/已认领，用于过滤
}
```

## 查询：function_score + script_score

```json
{
  "query": {
    "function_score": {
      "query": { "term": { "pool_status": "in_pool" } },
      "functions": [
        { "field_value_factor": { "field": "size_score",     "missing": 0 } },
        { "field_value_factor": { "field": "tag_score",      "missing": 0 } },
        { "field_value_factor": { "field": "behavior_score", "missing": 0 } },
        { "filter": { "terms": { "industry_code": ["<销售TOP行业列表>"] } },
          "weight": 40 },
        { "script_score": {
            "script": {
              "source": "double d = doc['location'].arcDistance(params.lat, params.lon)/1000.0; if (d<=20) return 30; else if (d<=50) return 30-(d-20); else return 0;",
              "params": { "lat": 31.23, "lon": 121.47 }
            }
        }}
      ],
      "score_mode": "sum",
      "boost_mode": "replace"
    }
  },
  "sort": [{ "_score": "desc" }, { "customer_id": "asc" }]
}
```

`score_mode: sum` 累加 5 项；`boost_mode: replace` 覆盖原始相关性（这里无文本搜索）。距离段是**分段线性**，标准 decay 函数表达不了，必须用 script。

## 实现路径（5 阶段，约 4-6 周）

| 阶段 | 内容 | 关键产出 |
|---|---|---|
| **1. 数据底座** | MySQL 建 `customer_score`、`account_industry_profile`；ES 建索引 mapping | 表、索引、字段对齐 |
| **2. 同步链路** | Canal/Debezium 监听 MySQL → 写 ES；全量初始化脚本 | 客户主数据进 ES，秒级延迟 |
| **3. 物化基础分** | 客户字段变更事件 → 重算 size/tag/behavior 三个分 → upsert MySQL + 同步 ES | 30 分维度查询 0 计算 |
| **4. 行业画像** | T+1 job 扫成交客户聚合行业分布，写 MySQL，热数据进 Redis | 每销售一份画像，毫秒命中 |
| **5. 查询 API** | 入参（销售ID + GPS）→ 拿画像 → 拼 function_score → 返 search_after 分页 | 列表 P99 < 200ms |

## 5 个必踩的坑

1. **新销售无成交 → 行业画像为空 → 40 分恒 0**  
   Fallback 到 部门 TOP3 行业 或 公司 TOP3 行业，避免新人体感"全是低分"。

2. **公海认领后必须立即从结果剔除**  
   状态字段 `pool_status` 走 CDC 实时同步；查询用 `filter` 而非 `must`（filter 走缓存）。

3. **GPS 实时 → 列表结果不能跨请求缓存**  
   但翻页时 GPS 应"冻结"——前端在第一页固定 GPS 参数透传后续页，否则翻页时分数会跳变、出现重复/遗漏。配合 `search_after` 而不是 `from/size`。

4. **score_mode / boost_mode 必须显式设置**  
   默认是 multiply，会把各项分相乘成 0；改成 `sum` + `replace`。

5. **script_score 是 ES 的热点**  
   - 必须用 `doc['location'].arcDistance`（不要 `getDistance` 已废弃）
   - 脚本编译会缓存，但**参数变化不会重编译**——把 GPS 用 params 传，别拼到 source 里
   - 压测目标：单分片 < 50ms

## 行业画像数据结构建议

```sql
account_industry_profile (
  account_id   BIGINT,
  industry_code VARCHAR,
  customer_cnt  INT,        -- 该销售在此行业的成交数
  weight        DECIMAL,    -- 归一化权重
  rank          TINYINT,    -- 1=TOP1, 2=TOP2...
  updated_at    DATETIME,
  PRIMARY KEY (account_id, industry_code)
)
```

查询时取 `rank <= 3` 的行业塞进 ES `terms` filter，"完全匹配 40 分"可放宽为"TOP1 给 40、TOP2 给 30、TOP3 给 20"，按需调。

# 你说得对，MySQL 就够，我之前过度设计了

ES 解决的是"**全量 100万 上的多维 ranking**"。一旦你限定**先 50KM、不够再 200KM**，geo 先把 99% 数据砍掉，剩下的几万条 MySQL 单机内存里转转就出结果。

## 数据量重估

| 半径 | 典型命中数（密集城市） | 排序计算耗时 |
|---|---|---|
| 50KM | 5K – 50K | < 100ms |
| 200KM | 50K – 300K | 200ms – 1s |
| 全量 1M | 100万 | 这才需要 ES |

ES 的 function_score 在 50KM 工作集上跟 MySQL 性能差不多（都 < 100ms），但你要多扛一个 ES 集群 + CDC 同步链路 + 数据一致性问题 — 完全不值。

## MySQL 方案

### 表设计

```sql
CREATE TABLE customer_pool (
  customer_id     BIGINT PRIMARY KEY,
  industry_code   VARCHAR(16),
  size_score      TINYINT,   -- 物化：0 或 10
  tag_score       TINYINT,   -- 物化：0 或 10
  behavior_score  TINYINT,   -- 物化：0 或 10
  lat             DECIMAL(9,6),
  lon             DECIMAL(9,6),
  location        POINT NOT NULL SRID 4326,
  pool_status     VARCHAR(16),
  SPATIAL INDEX idx_loc (location),
  INDEX idx_status_lat_lon (pool_status, lat, lon)
) ENGINE=InnoDB;
```

> 同时存 `lat/lon` 标量 + `location` POINT：前者给 B-tree 做 bounding box 粗筛（实测比 spatial index 更稳），后者用于 `ST_Distance_Sphere` 精算。

### 查询：bounding box 粗筛 + 精算分

```sql
SET @lat = 31.23, @lon = 121.47;
SET @r_km = 50;                        -- 阶梯：50 不够换 200
SET @dlat = @r_km / 111.0;             -- 1° 纬度 ≈ 111km
SET @dlon = @r_km / (111.0 * COS(RADIANS(@lat)));

SELECT
  customer_id,
  ROUND(dist_km, 2) AS dist_km,
  size_score + tag_score + behavior_score
    + CASE WHEN industry_code IN ('IT','金融','制造') THEN 40 ELSE 0 END
    + CASE
        WHEN dist_km <= 20 THEN 30
        WHEN dist_km <= 50 THEN 30 - (dist_km - 20)
        ELSE 0
      END AS ai_score
FROM (
  SELECT
    customer_id, industry_code,
    size_score, tag_score, behavior_score,
    ST_Distance_Sphere(POINT(@lon,@lat), location)/1000 AS dist_km
  FROM customer_pool
  WHERE pool_status = 'in_pool'
    AND lat BETWEEN @lat - @dlat AND @lat + @dlat   -- 走联合索引
    AND lon BETWEEN @lon - @dlon AND @lon + @dlon
) t
WHERE dist_km <= @r_km                              -- 精筛（矩形 → 圆）
ORDER BY ai_score DESC, dist_km ASC
LIMIT 20;
```

### 阶梯扩半径逻辑（业务层）

```python
for radius in [50, 200]:
    results = query(radius)
    if len(results) >= 20:    # 满足分页需求
        return results
return results                # 200KM 也凑不齐就这么多了
```

## 改两个原方案细节

1. **`customer_score` 不用拆表了** — 三个物化分直接做成 `customer_pool` 的列，事件触发原地 UPDATE。原本拆表是为了 ES 同步方便，MySQL 里反而绕路。

2. **`account_industry_profile` 保留** — T+1 离线算成交客户行业 TOP3，查询时 `IN (?, ?, ?)` 直接传进 SQL。这块跟存储引擎无关。

## 仅一个需要警惕的风险

**密集城市 + 200KM 扩半径**：上海/北京中心点 200KM 内可能有 30 万客户，排序到秒级。两种兜底，由轻到重：

1. **限 TOP N**：`ORDER BY ai_score DESC LIMIT 200`（其实业务上销售翻不到 200 条以后）
2. **加二级过滤**：200KM 时强制要求"行业匹配 OR 实缴 500万+"，先过滤再算分
3. **真撑不住再上 ES**：到了那一步再说，别预先做

---

**总结**：你的"先 50 后 200"是 geo-first 策略，跟 ES 的 ranking-first 哲学不匹配。MySQL spatial + bounding box 才是这个场景的标准答案，我之前推 ES 是没把 geo 当主轴看，反思一下。

# 半对 — LIMIT 救不了"算分"，但能救你免于过度优化

`LIMIT` 在 MySQL 里只在**排序键是索引列**时才能提前停。这里 `ai_score` 是查询时现算的（距离部分依赖运行时 GPS），所以：

## LIMIT 改变不了的事

```
执行计划（不管 LIMIT 是 20 还是 2000）：
├─ 1. bbox 过滤 1M → 30万         ← 走索引，便宜
├─ 2. 给 30万 行每行算 ST_Distance_Sphere + CASE  ← 必做
├─ 3. 给 30万 行排序                          ← 必做（哪怕用优先队列堆）
└─ 4. 取前 20 行                              ← LIMIT 唯一省的就这步
```

MySQL 有 `ORDER BY ... LIMIT N` 的 priority-queue 优化，把步骤 3 从 O(n log n) 降到 O(n log k)。但**步骤 2 是 O(n) 死成本**，每行都得算一遍距离才能知道分数。30万 × ST_Distance_Sphere 大概 300-500ms，加排序总共 0.5-1s。

## 但你的直觉本质上是对的

我之前那条"风险"其实是**纸面焦虑**：

- 0.5-1s 对销售刷公海列表来说**完全可接受**（用户预期就是"AI 在算"）
- 200KM 扩半径本身就是**降级路径**，少数情况触发
- 大多数城市 200KM 没那么密，30万是夸张数字，实际可能 5-10万

**所以先按"LIMIT 20 + 不优化"上，压测真出问题再补**，不要预先做。

## 真正会爆的只有一个场景：深翻页

```sql
ORDER BY ai_score DESC LIMIT 20 OFFSET 1000   -- 第 51 页
```

每翻一页都要重新扫 30万 + 算分 + 排序 + 丢掉前 1000 — **每页都是全成本**。

兜底也简单：**不让翻太深**。产品上：
- 最多翻 10-20 页（公海推荐没人翻 100 页）
- 或者前端限制"查看更多"按钮上限

实际销售用这种 AI 列表，看头部 50-100 个客户就够了，不是搜索引擎。

---

**结论修正**：你说得对，LIMIT + 不优化就够了。我那条风险删掉。真要补一句的话：**前端限制最大页数**，比后端做二级过滤实在得多。
