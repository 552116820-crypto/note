  # 附近客户与相似客户数据库选型建议

  ## Summary

  如果只看这个项目当前架构，MySQL 更容易落地；如果从“附近客户 + 相似客户 + 百万级推荐”的长期能力看，PostgreSQL 更合
  适。
  我的建议是：第一版如果你们现有 CRM/公海数据已经在 MySQL，就先用 MySQL 8 做；如果可以为推荐系统单独建一套推荐库，优先
  选 PostgreSQL + PostGIS + pgvector。

  ## Recommendation

  - 短期落地优先：MySQL 8
      - 当前项目已经有 MySQL 仓储、MySQL 迁移目录和 SQLAlchemy 接入。
      - 附近客户可以用 MySQL 8 的空间字段、空间索引、ST_Distance_Sphere 实现。
      - 适合第一版：公海客户筛选、附近查询、星级评分、规则匹配度排序。
  - 长期推荐能力优先：PostgreSQL
      - PostGIS 做地理位置检索比 MySQL 更强。
      - pgvector 可以做相似客户向量检索，适合把经营范围、主营标签、历史成交画像转成向量后找相似客户。
      - PostgreSQL 能同时承担：地理查询、规则画像、向量相似度、推荐候选池。

  ## Key Tradeoffs
  - MySQL 短板：复杂地理分析、向量相似度、全文搜索和推荐扩展能力不如 PostgreSQL/ES。

  ## Suggested Architecture

  - 第一阶段：继续用 MySQL 8。
      - 公海客户表增加 latitude、longitude、geo_status、geo_confidence。
      - 建空间索引或经纬度/geohash 索引。
      - 先实现附近客户 + 星级分 + 规则相似度。
  - 第二阶段：如果推荐效果和搜索复杂度上来，再引入 PostgreSQL。
      - 用 PostgreSQL/PostGIS 存推荐画像和地理坐标。
      - 用 pgvector 存客户向量和成交客户画像向量。
      - MySQL 继续作为业务主库，PostgreSQL 作为推荐检索库。
  - 第三阶段：如果还需要复杂搜索，再考虑 Elasticsearch/OpenSearch。
      - 用于企业名称、地址、经营范围、主营产品标签的模糊搜索和组合筛选。

  ## Assumptions

  - 当前项目主数据库倾向 MySQL，已有 MySQL 相关代码和迁移。
  - 公海客户规模约 100 万。
  - 推荐第一版目标是可用、可解释、维护成本低。
  - 相似客户第一版先用规则相似度；AI/向量检索可以作为增强能力逐步引入。

• 可以，而且这是比较合理的架构：MySQL 做业务主库，PostgreSQL 做推荐检索库。

  不要把 PostgreSQL 当成替代 MySQL，而是当成一个“推荐系统读模型”：

  MySQL
  - 客户基础信息
  - 公海归属状态
  - 领取/分配/成交等业务状态
  - 业务主数据源

  PostgreSQL
  - 经纬度
  - 地理索引
  - 客户画像
  - 客户向量
  - 相似度检索
  - 推荐候选池

  MySQL 会遇到的瓶颈

  1. 向量相似度不是 MySQL 8 的强项
     你现在如果是 MySQL 8，基本不适合直接做客户信息向量化后的相似检索。MySQL 新版本有 VECTOR 类型，但官方文档里也明确 VECTOR 列不能作为主键、外键、唯一键或分区键；整体生态和索引能力还不如 PostgreSQL + pgvector
     成熟。
  2. 地理能力够用但不强
     MySQL 8 可以做 SPATIAL INDEX 和 ST_Distance_Sphere，用于“30 公里内客户”这种第一版功能没问题。但如果后面要做复杂区域、路线附近、地图围栏、区域多边形、地理聚合，PostGIS 更强。
  3. 混合检索会变吃力
     你的推荐不是单一条件，而是：

     公海状态 + 销售权限 + 附近距离 + 客户评级 + 相似度 + 行业标签 + 风险过滤

     MySQL 做结构化过滤没问题，但把“空间查询 + 向量相似度 + 多条件排序”放一起，会比较别扭。
  4. 推荐系统会影响业务主库
     一百万公海客户不算特别大，但如果销售端频繁请求附近推荐，实时排序、距离计算、匹配度计算都压在 MySQL 主库上，会影响 CRM 正常业务查询。

  PostgreSQL 更适合的地方

  1. PostGIS 做地理查询成熟
     用 geography(Point, 4326) 存客户坐标，建 GiST 索引，用 ST_DWithin 查半径内客户。PostGIS 文档也说明 ST_DWithin 会利用可用索引做 bounding box 过滤，适合半径查询。
  2. pgvector 做相似客户检索更自然
     客户画像、经营范围、主营标签、历史成交客户画像都可以转成 embedding，存到 pgvector。pgvector 支持精确和近似最近邻检索，也支持 HNSW、IVFFlat 这类向量索引。
  3. 推荐读模型可以独立优化
     PostgreSQL 里可以专门建这些表：

     public_customer_profile
     - customer_id
     - public_pool_status
     - province/city/district
     - industry
     - product_tags
     - score/star
     - risk_level
     - location geography(Point, 4326)
     - embedding vector(1536)
     - updated_at

     MySQL 只负责业务事实，PostgreSQL 负责推荐查询和排序。

  性能上是不是 PostgreSQL 更好？
  不能简单说 PostgreSQL 全面比 MySQL 快。

  更准确地说：

  普通业务 CRUD、主键查询、订单客户基础表：
  MySQL 很好，不一定比 PostgreSQL 差。

  附近客户 + 地理查询：
  PostgreSQL + PostGIS 通常更专业、更好扩展。

  客户向量相似度：
  PostgreSQL + pgvector 明显比 MySQL 8 合适。

  附近 + 相似 + 评分 + 多条件排序：
  PostgreSQL 更适合作为推荐检索库。

  所以对你的场景，PostgreSQL 不是因为所有性能都更强，而是因为它更适合这个推荐检索工作负载。

  我建议的落地结构：

  4. MySQL 保持客户主数据
  5. 定时或实时同步公海客户到 PostgreSQL
  6. PostgreSQL 补充经纬度、画像字段、embedding
  7. 推荐接口只查 PostgreSQL
  8. 客户领取/分配/成交仍回写 MySQL
  9. PostgreSQL 接收状态同步，确保已领取客户不再推荐

  同步方式第一版可以简单点：

  每 5-10 分钟按 updated_at 增量同步

  后面量大或实时性要求高，再上：

  binlog / Canal / Debezium / 消息队列

  结论：你的方案可以做，而且方向是对的。MySQL 保留业务主库，PostgreSQL 承担经纬度、向量化、相似度匹配和推荐排序，是比“全塞 MySQL”更稳的架构。

• 这几个都是“比附近多少公里更复杂一点”的地理能力。

  地图围栏
  地图围栏就是在地图上划一个范围，判断客户或销售是否在这个范围里。

  例子：

  销售只能领取自己负责片区内的公海客户

  比如你给销售 A 划定一个负责区域：

  亳州市利辛县城关镇附近

  系统判断：

  客户地址坐标是否在销售 A 的负责区域内

  如果在，能推荐；不在，不推荐。

  常见用途：

  - 销售片区管理
  - 禁止跨区领取客户
  - 到达某个工业园后自动推送园区客户
  - 客户进入/离开某个服务范围提醒

  区域多边形
  区域多边形就是用多个经纬度点围出一个不规则区域。

  不是所有业务区域都是圆形。比如：

  某工业园区
  某开发区
  某销售负责片区
  某城市行政区
  某条道路两侧范围

  它们通常不是标准圆，而是不规则形状。

  例如一个片区可以存成：

  点1：经度A，纬度A
  点2：经度B，纬度B
  点3：经度C，纬度C
  点4：经度D，纬度D

  这些点连起来就是一个多边形。系统可以判断：

  客户坐标是否落在这个多边形里

  地理聚合
  地理聚合就是按地理范围统计客户分布。

  例子：

  每个城市有多少公海客户？
  每个区县有多少高星客户？
  哪个工业园附近客户最多？
  销售当前位置 10 公里内有多少 3 星以上客户？
  安徽省内客户按市分布如何？

  这些不是返回单个客户列表，而是做统计分析。

  常见结果：

  合肥市：12000 家
  芜湖市：8500 家
  亳州市：4300 家
  利辛县：620 家

  或者地图热力图：

  哪里客户密集，哪里颜色更深

  和你现在功能的关系
  你第一版只需要：

  销售当前位置 + 半径，比如 30 公里
  查附近公海客户

  这叫“半径查询”，最简单。

  后续如果你要做：

  - 销售负责片区
  - 限制跨区领取
  - 工业园客户推荐
  - 客户分布热力图
  - 区县/城市客户统计
  - 地图上框选一块区域找客户

  那就会用到地图围栏、区域多边形、地理聚合。

  简单说：

  附近客户 = 以销售为圆心画圆
  地图围栏/区域多边形 = 在地图上画不规则边界
  地理聚合 = 按地理区域做客户数量和质量统计

› 这句话是什么意思：MySQL 做结构化过滤没问题，但把“空间查询 + 向量相似度 + 多条件排序”放一起，会比较别扭。


• 这句话的意思是：

  MySQL 很适合处理这种明确字段条件：

  WHERE public_pool_status = '未分配'
    AND province = '安徽省'
    AND star >= 3
    AND risk_level = 'low'

  这叫“结构化过滤”，MySQL 很擅长。

  但你的推荐功能不只是过滤，而是同时要做三类计算：

  1. 空间查询
  查销售附近 30 公里内的客户：

  客户经纬度距离销售经纬度 <= 30km

  2. 向量相似度
  查这个公海客户像不像历史成交客户：

  客户A的向量 和 成交客户画像向量 的相似度

  比如经营范围、主营产品、行业标签转成向量后算相似。

  3. 多条件排序
  最后还要综合排序：

  推荐分 = 距离分 * 25% + 匹配度 * 40% + 客户评级分 * 35%

  也就是说，一条推荐查询可能是：

  先找附近客户
  再算相似度
  再结合客户评分
  再过滤风险
  最后排序取前30个

  这套事情在 MySQL 里不是完全不能做，但会比较麻烦：

  - 空间查询能力有，但不如 PostGIS 丰富
  - 向量相似度能力不是 MySQL 8 的强项
  - 相似度排序和距离排序叠加后，索引很难都用上
  - 查询 SQL 会复杂，性能调优困难
  - 大量推荐请求可能影响 MySQL 主业务库

  所以“别扭”的意思不是 MySQL 做不了，而是：

  MySQL 更适合做业务数据存储和普通条件查询；
  PostgreSQL + PostGIS + pgvector 更适合做附近客户 + 相似客户这种推荐检索。

  举个简单对比：

  MySQL 做普通筛选很顺：

  找安徽省、3星以上、未分配的公海客户

  PostgreSQL 更适合做这种推荐检索：

  找销售30公里内、和历史成交客户最像、客户评分较高、风险较低的前30家公海客户

• PostgreSQL 性能很好，尤其适合你现在说的这类场景：附近客户 + 相似客户 + 多条件推荐排序。

  但要说清楚：不是 PostgreSQL 在所有场景都一定比 MySQL 快，而是它在你这个推荐检索场景里更合适。

  对你的一百万公海客户来说
  一百万客户对 PostgreSQL 不算大。只要表结构和索引设计正确，完全可以支撑。

  你的核心查询大概是：

  1. 只查公海客户
  2. 查销售当前位置 30 公里内
  3. 过滤风险、星级、行业
  4. 计算相似度
  5. 综合排序取前 30 个

  PostgreSQL 可以比较自然地处理这类查询。

  附近客户性能
  用 PostGIS 建地理索引后，查附近客户很快。

  比如客户表有：

  location geography(Point, 4326)

  建索引：

  CREATE INDEX idx_customer_location
  ON public_customer_profile
  USING GIST (location);

  查询附近 30 公里：

  WHERE ST_DWithin(
    location,
    ST_SetSRID(ST_MakePoint(:lng, :lat), 4326)::geography,
    30000
  )

  这类查询不会全表扫 100 万客户，而是先用空间索引缩小范围。

  相似客户性能
  如果用 pgvector 存客户向量，例如：

  embedding vector(1536)

  再建 HNSW 或 IVFFlat 向量索引，就可以快速找相似客户：

  ORDER BY embedding <=> :target_embedding
  LIMIT 100

  这比 MySQL 8 直接做向量相似度更成熟。

  推荐排序性能
  推荐不要一条 SQL 对 100 万客户全部算分。正确做法是两阶段：

  第一阶段：用索引筛候选
  - 公海状态
  - 地理半径
  - 行业/星级/风险
  - 向量相似 Top N

  第二阶段：只对几百到几千个候选算综合推荐分

  例如：

  100万客户
  -> 地理半径筛到 3000 个
  -> 风险/星级筛到 1500 个
  -> 计算匹配度和推荐分
  -> 返回前 30 个

  这样性能会稳定很多。

  可能的瓶颈
  PostgreSQL 也不是随便写就快，主要瓶颈会在：

  - 没有给 location 建 GiST 索引
  - 向量字段没有建 HNSW/IVFFlat 索引
  - 每次推荐都全表计算距离或相似度
  - 把大段经营范围文本实时解析
  - 客户状态同步不及时，导致推荐已领取客户
  - 高频请求都打主库，没有缓存或连接池

  建议配置
  你这个场景建议：

  PostgreSQL + PostGIS + pgvector

  再配：

  PgBouncer 连接池
  定时增量同步 MySQL 数据
  推荐结果短缓存 5-15 分钟
  EXPLAIN ANALYZE 定期看慢查询

  结论：PostgreSQL 性能足够支撑你的一百万公海客户推荐场景。MySQL 做客户主数据没问题，但“地理位置 + 向量相似 + 推荐排序”这部分放 PostgreSQL，会更好扩展，也更容易调优。
