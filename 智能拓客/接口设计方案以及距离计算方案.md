**User:**

"code": 0是什么意思

**Assistant:**

`code` 是接口返回里的**业务状态码**——告诉前端"这次请求成功了还是失败了，失败是什么原因"。`"code": 0` 通常约定为 **成功**。

## 为什么需要它

HTTP 本身有状态码（200/404/500），但它管的是"网络/服务器层面"通不通。`code` 管的是"**业务层面**"成没成功。两者是两回事，举例：

> HTTP 返回 200（请求顺利到达、服务器正常响应），但业务上"员工编号不存在"——这时就靠 `code` 来表达失败。

如果只看 HTTP 200，前端会以为成功；有了 `code`，前端先看 `code` 再决定怎么处理。

## 常见约定

```json
{ "code": 0,    "message": "ok",        "data": {...} }   // 成功
{ "code": 1001, "message": "员工编号不存在", "data": null }   // 失败
{ "code": 1002, "message": "经纬度不合法",   "data": null }   // 失败
```

- **`0` = 成功**是最流行的一种约定（"零错误"）。
- 也有团队用 **`200` = 成功**（直接套用 HTTP 的习惯），失败用 `400`/`500` 之类。
- 非 0（或非 200）就是各种失败，每个数字对应一种错误原因，配合 `message` 给人看。

## 前端怎么用它

```js
if (res.code === 0) {
  // 成功，读 res.data 渲染列表
} else {
  // 失败，弹 res.message 给销售看（比如"经纬度不合法"）
}
```

## 给你的建议

这个不是强制的，是**团队约定**。两件事确认一下就行：

1. **跟你后端现有接口对齐**——如果你们其它接口都用 `0=成功`，这个就跟着用 `0`；如果用 `200`，就改成 `200`。别一个项目里两种混着来。
2. 如果你们目前**根本没有这层 code**（直接返回 data），那也可以不要，看团队习惯。我加上是因为大多数 B2B 后台都有这层，方便统一处理错误。

一句话：`code` 是给程序判断成败用的数字，`message` 是给人看的说明，`data` 才是真正的数据。

**User:**

**Assistant:**

好消息：**这个问题不用空间索引也能完美解决**,而且是业界标准做法。先说清楚 5.6 的两个坑,你就明白为什么不该折腾空间索引:

## MySQL 5.6 的两个限制

1. **`ST_Distance_Sphere()` 不存在**——它是 5.7.6 才加的。5.6 里算球面距离只能自己写 Haversine 公式。
2. **空间索引只支持 MyISAM 表**——InnoDB 的空间索引是 5.7.5 才有的。你的客户表大概率是 InnoDB,为了空间索引改成 MyISAM 会丢事务/外键,**不值得**。

所以结论:**别用空间索引,用"矩形预筛 + Haversine 精算",配普通 B-tree 索引就行**。这正好对应你那三步里的第 1、2 步。

## 解法:Bounding Box 矩形预筛 + Haversine 精算

**核心思路**:圆形范围不好用索引,但圆的外接**矩形**可以用普通索引!先用 `lat/lng BETWEEN` 卡一个矩形(走 B-tree 索引,秒筛掉 90%+),剩下几百几千条再用 Haversine 算精确距离、削掉矩形四角多出来的部分。

### 第 1 步:建普通索引(不是空间索引)

```sql
ALTER TABLE sea_customer ADD INDEX idx_lat_lng (lat, lng);
```

⚠️ 前提:`lat`、`lng` 字段必须是**数字类型**(`DECIMAL(10,6)` 或 `DOUBLE`)。如果现在存的是 `VARCHAR` 字符串,`BETWEEN` 会按字符串比大小,结果全错——这也是我之前一直说 lng/lat 别用字符串的原因。先确认字段类型。

### 第 2 步:在应用层算出矩形边界

```
latDelta = radius_km / 111.0                          // 1 纬度 ≈ 111km
lngDelta = radius_km / (111.0 * cos(radians(lat)))    // 经度随纬度收缩,要乘 cos
latMin = lat - latDelta;  latMax = lat + latDelta
lngMin = lng - lngDelta;  lngMax = lng + lngDelta
```

### 第 3 步:SQL 矩形预筛 + Haversine 精算

```sql
SELECT
  customer_id, customer_name, lat, lng, address, ...,
  6371 * ACOS( LEAST(1.0,
      COS(RADIANS(:lat)) * COS(RADIANS(lat)) * COS(RADIANS(lng) - RADIANS(:lng))
    + SIN(RADIANS(:lat)) * SIN(RADIANS(lat))
  )) AS distance_km
FROM sea_customer
WHERE lat BETWEEN :latMin AND :latMax     -- ← 走 idx_lat_lng,秒筛
  AND lng BETWEEN :lngMin AND :lngMax
HAVING distance_km <= :radius_km          -- ← 削掉矩形四角,得到真正的圆
ORDER BY distance_km;
```

- `6371` = 地球半径(km),所以结果单位是 km。
- `WHERE` 里的矩形条件走索引,把全表筛到候选集;`HAVING` 对这个小候选集再精确过滤(因为 `distance_km` 是计算出来的别名,只能放 HAVING 不能放 WHERE)。
- 这一步出来的就是你"step 2 剩下几千条",`distance_km` 直接带出来,内存里算总分时距离这一项都不用再算了。

## 两个必处理的坑

**1. `ACOS` 越界返回 NULL** —— 浮点误差可能让括号里的值算出 `1.0000001`,`ACOS` 超过 1 就返回 NULL,距离变空。所以套了 `LEAST(1.0, ...)` 把它钳在 1 以内。这个一定要加,否则正好在原点附近的客户距离会变 NULL。

**2. 矩形比圆大** —— 矩形四个角超出 60km,所以 `HAVING distance_km <= 60` 这步不能省,它负责把方角削成圆。

> 你做的是中越润滑油,纬度都在北半球中低纬,不涉及南北极和 180° 经线那些边界情况,公式直接用就行,不用考虑跨日期线。
