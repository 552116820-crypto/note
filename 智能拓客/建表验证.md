```
  ├─ registered_capital
  └─ social_insurance_count
```

```sql

FROM customer c
JOIN customer_detail d ON c.customer_id = d.id
WHERE ST_Distance(c.location, ?) < 60000
```

- 工商字段单一数据源，不用同步
- `customer` 表只放"智能拓客特有的扩展字段"，职责清晰
- JOIN 一张表性能可接受

**适合**：上面 Q1 的情况 B（两类不同实体）+ Q2 你能控制 customer_detail 或它结构稳定

## 但有几件事要落实，否则后面会踩坑

### 1. 经纬度上有空间索引吗？

如果两张表里只是普通的 `longitude DECIMAL`, `latitude DECIMAL` 两列，**没有空间索引**，那筛"60KM 内的客户"会全表扫，几万条以上就开始卡。

需要做的：
- MySQL 8：加一个 `POINT` 字段 + `SPATIAL INDEX`
- PostgreSQL：加 `GEOGRAPHY(POINT)` + `GIST` 索引
- 如果现有表不能改结构 → 单独建个轻量"客户位置索引表"`customer_location`（只放 customer_id + location + 空间索引），算分时先用这张表筛 ID

### 2. 行业字段是标准化代码，还是自由文本？

这是行业匹配 40 分能不能算准的关键：

- 如果是标准化代码（比如国标 GB/T 4754 的"I65 软件和信息技术服务业"） → 直接用
- 如果是自由文本（"互联网"、"软件开发"、"IT 服务"乱七八糟） → **必须先做标准化映射**，不然匹配率会低到没法看

你之前在 `export_industry_tag/` 目录里做过 industry_mapping —— 那个产物现在应该派上用场了，确认下落地到表里了没有。

### 3. 客户标签字段是什么结构？

"有标签 10 分，无标签 0 分" 看起来很简单，但要确认：

- 是 JSON 数组？逗号分隔字符串？还是关联另一张标签表？
- 空值是 NULL、空字符串、还是空数组 `[]`？
- 三种情况判空 SQL 写法完全不同，提前对齐

### 4. 两张现有表的关系

- 是 1:1 还是 1:N？
- 关联键是 `customer_id` 还是别的？

## 落地建议

给你两个减少踩坑的小做法：

**做法 1：建一个数据库视图聚合算分字段**

```sql
CREATE VIEW v_customer_score_input AS
SELECT 
  a.customer_id,
  a.customer_name,
  a.industry,
  a.customer_tag,
  a.longitude,
  a.latitude,
  b.paid_in_capital,
  b.registered_capital,
  b.social_insurance_count,
  b.is_qizhidao
FROM customer_table_a a
LEFT JOIN customer_table_b b ON a.customer_id = b.customer_id;
```

算分接口直接 `SELECT * FROM v_customer_score_input WHERE ...`，业务代码不感知底层是几张表，未来重构方便。

**做法 2：算分逻辑写成 SQL 一把梭**

四个维度的得分完全可以在一条 SQL 里算出来，不用拉到应用层循环：

```sql
SELECT 
  customer_id,
  customer_name,
  -- 行业匹配 40 分
  CASE WHEN industry IN (SELECT industry FROM sales_profile_industry WHERE sales_account_id = ?) 
       THEN 40 ELSE 0 END AS industry_score,
  -- 规模 10 分
  CASE WHEN paid_in_capital >= 5000000 THEN 10 ELSE 0 END AS scale_score,
  -- 标签 10 分
  CASE WHEN customer_tag IS NOT NULL AND customer_tag != '' THEN 10 ELSE 0 END AS tag_score,
  -- 距离 40 分（用空间函数算距离 km）
  CASE 
    WHEN ST_Distance_Sphere(POINT(longitude, latitude), POINT(?, ?))/1000 <= 20 THEN 40
    WHEN ST_Distance_Sphere(POINT(longitude, latitude), POINT(?, ?))/1000 <= 60 
         THEN 40 - (ST_Distance_Sphere(POINT(longitude, latitude), POINT(?, ?))/1000 - 20)
    ELSE 0
  END AS distance_score
FROM v_customer_score_input
WHERE ST_Distance_Sphere(POINT(longitude, latitude), POINT(?, ?))/1000 <= 60  -- 先卡 60km 内
ORDER BY (industry_score + scale_score + tag_score + distance_score) DESC
LIMIT 20 OFFSET ?;
```

SQL 算分的好处：数据库自己用索引筛，几万条数据毫秒级；坏处：调试不如代码方便。可以接受就用，不接受就拉到 Python/Java 里算。

## MySQL 5.6 的两个坑

### 坑 1：`ST_Distance_Sphere` 在 5.6 不存在

这个球面距离函数是 5.7.6 才加的。5.6 算距离的两个选项：

**A. 用 Haversine 公式手写 SQL**（推荐）
```sql
6371 * 2 * ASIN(SQRT(
  POWER(SIN((RADIANS(latitude) - RADIANS(?))/2), 2) +
  COS(RADIANS(?)) * COS(RADIANS(latitude)) *
  POWER(SIN((RADIANS(longitude) - RADIANS(?))/2), 2)
))
```
结果就是 km。三个 `?` 都是销售当前的纬度/纬度/经度。

**B. 拉到 Python 层算**（备选）
用 geopy 之类库算。但这就意味着要把范围内候选客户先拉到内存，再算距离，再排序。**只有数据量小（< 几千）时可行**。

### 坑 2：InnoDB 在 5.6 不支持空间索引

5.6 + InnoDB → 不能在经纬度上建 SPATIAL INDEX。

但有个替代方案叫 **"边界框预筛"**，性能完全够用：

1. 在 `latitude` 和 `longitude` 上各建一个**普通 BTREE 索引**（这个 5.6 InnoDB 完全支持）
2. 查询时先用经纬度范围做粗筛（走 BTREE 索引很快）
3. 粗筛剩下的几百条再用 Haversine 精筛

比如要找销售当前位置 60km 内的客户，60km ≈ 纬度 0.6 度、经度 0.8 度（中国大陆纬度范围内），SQL 这样写：

```sql
WHERE latitude  BETWEEN (?-0.6) AND (?+0.6)
  AND longitude BETWEEN (?-0.8) AND (?+0.8)
  -- 索引能用上，先把全表几万条卡到几百条
HAVING distance_km <= 60   -- Haversine 算出来再精筛
```

要给现有两张表加索引（注意：表大的话加索引会锁表，建议低峰期跑）：
```sql
ALTER TABLE 客户表 ADD INDEX idx_lat (latitude);
ALTER TABLE 客户表 ADD INDEX idx_lng (longitude);
-- 或者复合索引（latitude 区分度通常更高，放前面）
ALTER TABLE 客户表 ADD INDEX idx_geo (latitude, longitude);
```

```sql
-- 销售画像表：销售账号 → 历史成单行业
CREATE TABLE `sales_profile_industry` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `sales_account_id` BIGINT UNSIGNED NOT NULL COMMENT '销售账号 ID',
  `amer_industry_id` INT(6) NOT NULL COMMENT '安美行业 ID（对应 amer_industry 表）',
  `order_count` INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '该行业成单数',
  `last_order_at` DATETIME DEFAULT NULL COMMENT '该行业最近一次成单时间',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_sales_industry` (`sales_account_id`, `amer_industry_id`),
  KEY `idx_sales` (`sales_account_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='销售历史成单行业画像';
```

```sql
-- 客户行为时间轴
CREATE TABLE `customer_activity` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `customer_id` BIGINT UNSIGNED NOT NULL COMMENT '客户 ID',
  `sales_account_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '操作的销售账号',
  `activity_type` VARCHAR(32) NOT NULL COMMENT '行为类型：visit/call/wechat/email/...',
  `activity_content` VARCHAR(1000) DEFAULT NULL COMMENT '内容描述',
  `activity_at` DATETIME NOT NULL COMMENT '行为发生时间',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  KEY `idx_customer_time` (`customer_id`, `activity_at`),
  KEY `idx_sales_time` (`sales_account_id`, `activity_at`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='客户行为时间轴';
```

几个说明：
- `customer_id` 和 `sales_account_id` 的类型要和你现有表里的对齐，我猜的 `BIGINT UNSIGNED`，如果你现有是 `INT` 或 `VARCHAR` 记得改
- `sales_profile_industry` 的 `UNIQUE KEY` 防止同一销售同一行业重复
- `customer_activity` 的复合索引 `(customer_id, activity_at)` 是为了"按客户拉时间轴"走索引
- 没加外键约束 —— InnoDB 外键虽然支持，但生产环境一般业务层维护引用完整性，外键会拖性能
