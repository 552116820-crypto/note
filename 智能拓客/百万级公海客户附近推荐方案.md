  # 百万级公海客户附近推荐方案

  ## Summary

  对 100 万公海客户，推荐系统要分成“离线预处理 + 实时轻量查询”。
  实时请求不能现算地址解析、文本相似度、全量客户距离；必须提前把客户地址转经纬度、画像标准化、评分算好，查询时先用地理
  范围快速缩小候选集，再做匹配度和排序。

  ## Key Changes

  - 公海客户主表：保存客户基础信息、公海状态、地址、省市区、经纬度、评级总分、星级、风险状态、更新时间。
  - 客户画像表：把行业、主营标签、经营范围关键词、注册资本档位、参保人数档位、成立年限档位、风险等级、创新资质等级等预
    处理成可比较字段。
  - 地理索引：客户经纬度必须建空间索引或 geohash/H3 网格索引；实时查询先按销售定位和半径筛附近客户。
  - 成交画像表：把历史成交客户按销售、行业、区域、产品标签聚合成成交画像，用于计算候选客户匹配度。
  - 推荐结果缓存：对每个销售可预生成常驻区域的推荐候选池；实时定位时再按距离重排。

  ## Implementation Details

  - 地址解析：
      - 批量异步调用高德/腾讯/百度地图 API，把文字地址转成经纬度。
      - 保存 latitude、longitude、geo_provider、geo_confidence、geo_status、geo_updated_at。
      - 只有 geo_status=success 且 geo_confidence=high/medium 的客户进入附近推荐。
  - 地理检索：
      - MySQL 8 可用 POINT(longitude, latitude) + SPATIAL INDEX + ST_Distance_Sphere。
      - PostgreSQL 推荐 PostGIS。
      - Elasticsearch/OpenSearch 可用 geo_point。
      - 如果先做轻量版，可用 geohash/H3 网格：先查销售附近网格，再用 Haversine 精算距离。
  - 实时推荐流程：
      1. 接收销售 sales_id + latitude + longitude + radius_km + mode。
      2. 通过地理索引筛出半径内公海客户，最多取前 2000 个候选。
      3. 过滤已分配、已成交、冻结、黑名单、无效电话、低可信坐标客户。
      4. 对候选客户计算 distance_score、customer_score、match_score。
      5. 按模式合成推荐分并返回 Top 20/30。
  - 推荐分模式：
      - balanced：匹配度 40% + 客户评分 35% + 距离 25%
      - distance_first：距离 50% + 匹配度 30% + 客户评分 20%
      - match_first：匹配度 50% + 客户评分 30% + 距离 20%
      - score_first：客户评分 50% + 匹配度 30% + 距离 20%
  - 性能目标：
      - 单次查询 P95 小于 500ms。
      - 地理候选召回小于 100ms。
      - 候选重排控制在 2000 条以内。
      - Top N 推荐结果可缓存 5-15 分钟。

  ## Data Pipeline

  - 每日任务：
      - 同步公海客户增量变更。
      - 新增/地址变更客户进入地理编码队列。
      - 更新客户评级、画像字段、风险等级。
      - 更新历史成交客户画像。
  - 实时/准实时任务：
      - 客户领取后立即从推荐池移除。
      - 销售忽略、无效、已拜访、已领取反馈写入推荐日志。
      - 对重复曝光客户做降权，避免反复推送。
  - 推荐日志：
      - 记录 sales_id、customer_id、recommend_score、distance_km、mode、exposed_at、action。
      - action 包括曝光、点击、领取、忽略、无效、拜访、成交。
      - 后续可用这些反馈优化权重。

  - AI 大模型适合做三件事：
      - 生成推荐理由，例如“距离近、主营面料纺织、与该销售过往成交客户相似”。
  - 不建议让大模型实时参与百万客户排序；成本高、延迟高、不可控。最终排序应由规则分数或机器学习模型完成，大模型只做画像
    增强和解释生成。

  ## Test Plan

  - 用 100 万模拟客户压测半径查询，验证 P95 延迟和候选数量。
  - 验证半径边界，半径外客户不返回。
  - 验证公海状态变更，客户被领取后立即不再推荐。
  - 验证不同模式下排序符合权重预期。
  - 验证无坐标、低可信坐标、黑名单、冻结客户被过滤。
  - 验证推荐日志能正确记录曝光、点击、领取、忽略和成交。

  ## Assumptions

  - 销售端能提供实时经纬度。
  - 公海客户至少有省市区和详细地址。
  - 历史成交客户能通过客户 ID 或企业名称关联到同一套客户画像字段。
  - 第一版推荐默认半径 30km，默认返回 30 条，最大候选重排 2000 条。
  - 数据库优先选 MySQL 8 空间索引、PostGIS 或 Elasticsearch/OpenSearch 三者之一；百万级不建议用纯 Python 全表扫描。
