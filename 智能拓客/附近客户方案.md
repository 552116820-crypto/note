  # 附近公海客户推送方案

  ## Summary

  实现一个“附近客户”功能：销售打开页面或到达某地时，前端上传销售当前位置，经后端查询“已录入公海、可领取、带地址/经纬
  度”的客户，按距离和客户质量排序返回。
  核心不是用文字地址直接比距离，而是先把客户地址转成经纬度，再用经纬度计算半径内客户。

  ## Key Changes

  - 数据准备：公海客户必须有 客户ID、客户名称、公海状态、客户地址、省市区、经纬度、行业/主营标签、星级/评分、联系人/电
    话、更新时间。
  - 地址标准化与地理编码：客户入公海时，用高德/百度/腾讯地图 API 把客户地址转成 latitude、longitude；失败的保留地址但
    不进入附近推荐，进入“待补坐标”队列。
  - 附近客户接口：新增接口 POST /api/nearby-public-customers/search，请求包含销售ID、当前位置经纬度、半径、分页、可选
    行业/星级筛选；返回客户、距离、地址、星级、联系方式、推荐原因。
  - 查询逻辑：只查 public_pool_status = 可领取/未分配 的客户，排除已分配、已成交、冻结、黑名单、无效地址客户；按半径过
    滤后排序。
  - 排序规则：默认 距离优先 + 客户质量加权，推荐分可为：距离分 60% + 星级/评分 25% + 信息完整度 10% + 最近更新时间
    5%。
  - 权限控制：销售只能看到允许区域或组织范围内的公海客户；如果公司允许全国公海，则只做半径限制。
  - 防重复推送：记录销售查看/推送日志，同一客户短时间内不要反复推给同一销售；客户被领取后立即从公海结果中消失。

  ## Implementation Details

  - 推荐数据表：如果 CRM 已有公海表，就扩展经纬度字段；如果没有，新增一张同步表，例如 public_pool_customers，专门存推
    荐所需字段和坐标。
  - 地理距离计算：MySQL 8 优先用空间索引或 ST_Distance_Sphere；没有空间能力时，用经纬度范围粗筛，再用 Haversine 公式精
    算距离。
  - 接口请求示例：

    {
      "sales_id": "000455",
      "latitude": 31.2304,
      "longitude": 121.4737,
      "radius_km": 10,
      "limit": 20,
      "min_star": 2,
      "industry_tags": ["纺织", "面料"]
    }
  - 接口响应示例：

    {
      "items": [
        {
          "customer_id": 12345,
          "customer_name": "某某纺织有限公司",
          "score": 68.5,
          "address": "江苏省苏州市吴江区...",
          "contact_phone": "138****0000",
      ],
      "total": 18
    }

  ## Test Plan

  - 验证只返回公海客户，不返回已分配、已成交、冻结或黑名单客户。
  - 验证半径边界：5km 内返回，5km 外不返回。
  - 验证排序：近距离客户优先；距离相近时，高星级/高评分客户优先。
  - 验证无坐标客户不会进入附近推荐，但会进入待补坐标列表。
  - 验证权限：销售不能看到无权限区域或无权限组织的客户。
  - 验证客户被领取后，下一次查询不再返回。

  ## Assumptions

  - 销售定位由前端或企业微信/CRM 移动端提供经纬度，不建议只传文字地址。
  - 公海客户至少有详细地址；经纬度可以通过批量地理编码补齐。
  - 默认半径为 10km，默认返回 20 条，最多 100 条。
  - 推荐目标是“方便销售顺路拜访/开发”，不是严格的成交概率预测；如果后续有历史成交数据，可再叠加“相似成交客户匹配度”排
    序。



• 需要。只靠“安徽省亳州市利辛县城关镇向阳中路128号”这种文字地址，不能稳定计算“附近客户”。要做附近推荐，最好先把地址转成
  经纬度。

  关键点：

  1. 附近客户必须靠经纬度算距离
      - 销售定位通常是 latitude + longitude
      - 客户如果只有文字地址，系统无法直接判断距离
      - 所以要先做“地址解析/地理编码”：地址 → 经纬度
  2. 推荐用高德或腾讯，百度也可以但坐标系要注意
      - 高德、腾讯地图常用 GCJ-02 坐标
      - 百度地图用 BD-09 坐标
      - GPS 原始定位常见是 WGS-84
      - 前端、客户坐标、距离计算必须统一坐标系，否则距离会偏几百米到上千米
  3. 不要每次查询附近客户时实时调地图 API
      - 把解析出的 latitude、longitude 存到数据库
      - 销售查询附近客户时，直接用数据库里的经纬度算距离
      - 解析失败的客户进入“地址待清洗/待补坐标”列表
  4. 地址要尽量拼完整
     例如：
     安徽省亳州市利辛县城关镇向阳中路128号
     这种已经比较完整，可以直接解析。
     如果只有“向阳中路128号”，准确率会低很多，必须补省市区。
  5. 需要保存解析质量
     建议存这些字段：
      - address
      - province
      - city
      - district
      - latitude
      - longitude
      - geo_provider：amap / baidu / tencent
      - geo_confidence 或 geo_level：门牌号/道路/区县
      - geo_status：success / failed / ambiguous
      - geo_updated_at

  结论：如果你要做“附近客户列表”，就应该把公海客户地址批量转成经纬度。高德/腾讯更适合直接和国内移动定位配合；选百度也可
  以，但要统一坐标系。



• geo_provider 和 geo_confidence 是建议你自己存的两个字段，用来记录“这个经纬度是怎么来的、可信不可信”。

  geo_provider：坐标来源
  表示这个客户地址是用哪个地图服务解析出来的。

  例如：

  amap      高德地图
  baidu     百度地图
  tencent   腾讯地图
  manual    人工录入/人工修正

  为什么要存？
  因为不同地图服务使用的坐标系可能不同。比如高德/腾讯常用 GCJ-02，百度是 BD-09。以后排查距离偏差、切换地图服务、重新解析
  地址时会用到。

  geo_confidence：坐标准确度/可信度
  表示这个地址解析结果有多准。

  可以简单设计成：

  high     精确到门牌号/公司地址
  medium   精确到道路、小区、园区
  low      只精确到区县/乡镇
  failed   解析失败

  举例：

  地址是：

  安徽省亳州市利辛县城关镇向阳中路128号

  如果地图 API 返回结果能匹配到 向阳中路128号，就可以认为：

  geo_provider = amap
  geo_confidence = high

  如果只识别到：

  安徽省亳州市利辛县城关镇向阳中路

  没有精确到 128 号，可以认为：

  geo_confidence = medium

  如果只识别到：

  安徽省亳州市利辛县

  那就是：

  geo_confidence = low

  实际数据库里可以这样存：

  latitude          DECIMAL(10, 7)
  longitude         DECIMAL(10, 7)
  geo_provider      VARCHAR(20)
  geo_confidence    VARCHAR(20)
  geo_status        VARCHAR(20)
  geo_updated_at    DATETIME

  最实用的规则是：附近客户推荐只使用 geo_status = success 且 geo_confidence in ('high', 'medium') 的客户。low 可以展示但
  要降权，failed 不进入附近推荐。
