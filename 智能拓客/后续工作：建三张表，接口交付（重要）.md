**Assistant:**

这句话其实包含两个独立判断，分开讲：

## 1. "附近的客户推送" 是什么场景

有两种产品形态，对位置数据的处理完全不一样：

**A. 被动查询（你当前的需求）**
销售打开 App → 前端拿到 GPS → 请求里带上经纬度 → 后端算附近客户返回。
→ **销售位置根本不用存**，每次请求带过来就行，用完即弃。

**B. 主动推送（更进阶的玩法）**
销售开着 App 在外面跑业务 → 系统在后台**持续追踪**他的位置 → 一旦发现他离某个高分潜在客户只有 500 米，主动弹通知："你附近有个 XX 公司，匹配度 85 分，要不要顺便拜访？"
→ 销售位置必须**实时存起来**，因为推送是系统主动触发的，得知道每个销售当前在哪。

你目前的得分页面只需要 A，**B 是未来才会考虑的**。我那句话是顺手提醒一下，不是说现在就要做。

## 2. 为什么实时位置不放 MySQL 要放 Redis

如果真要做 B 这种推送，销售位置数据有三个特点，MySQL 完全不适合：

| 特点 | MySQL 的问题 | Redis 的优势 |
|------|------|------|
| **写得极频繁** | 销售每 10 秒上报一次位置，100 个销售一天写几十万次 UPDATE，MySQL 会被拖垮，主从同步也会延迟 | Redis 是内存数据库，写入快几个数量级 |
| **只关心"现在在哪"** | MySQL 的强项是持久化历史数据，但旧位置（5 分钟前在哪）根本没用，存了浪费 | Redis 可以设过期时间，旧数据自动清理 |
| **要算"附近 X 米内的人"** | MySQL 算空间距离要 SPATIAL INDEX，但对高频写入场景不友好 | Redis 有专门的 `GEOADD` / `GEOSEARCH` 命令，天生就是干这个的，一条命令搞定"找出 500 米内的所有销售" |

打个比方：MySQL 像档案柜（适合长期存档），Redis 像桌上的便签（适合随手记随手扔）。销售实时位置就是张便签 —— 只关心"此刻"，五分钟前的位置就该撕掉。

**User:**

给前端接口需要走nginx反代吗？

**Assistant:**

分环境答，不一样：

## 开发环境：不用

本地调试时，前端用 Vite / webpack-dev-server 自带的 `proxy` 配置直接转发到后端就行，简单：

```js
// vite.config.js 例子
server: {
  proxy: {
    '/api': 'http://localhost:8000'
  }
}
```

后端直接监听 localhost 端口，开发者各自一套，省事。

## 测试 / 生产环境：建议走

不是"必须"，但**大部分场景都该走**。原因：

| 痛点 | 不走 Nginx | 走 Nginx |
|------|------|------|
| **跨域** | 前端域名 ≠ 后端域名，要在后端配 CORS，容易出问题 | 同域访问，零跨域配置 |
| **HTTPS** | 后端自己处理 SSL，每个服务都要装证书 | Nginx 一处终结 SSL，后端裸 HTTP 即可 |
| **静态文件** | 让后端框架（Flask/Django/Spring）serve 前端打包产物，效率差 | Nginx 直出，性能高几个数量级 |
| **端口暴露** | 后端 8000 端口直接暴露公网，不安全也不优雅 | 对外只开 80/443，后端藏在内网 |
| **多实例** | 没法做负载均衡 | 一行 upstream 配置搞定 |
| **限流/防护** | 要在应用代码里写 | Nginx 层挡掉异常流量 |
| **gzip / 缓存** | 自己实现 | Nginx 一行 `gzip on;` |

典型的生产架构：

```
浏览器 ─HTTPS→ Nginx (80/443) ─┬─ /          → 前端 dist/ 静态文件
                               ├─ /api/      → 后端服务 :8000
                               └─ /ws/       → WebSocket :8001
```

## 什么情况下可以不走 Nginx

- **K8s 部署**：用 Ingress Controller（Nginx Ingress / Traefik）替代，本质还是 Nginx，只是 K8s 帮你管
- **云厂商场景**：直接用阿里云 SLB / AWS ALB / 腾讯云 CLB 做反代和 SSL 终结，不自己装 Nginx
- **纯内网工具**：员工就十几个、内网访问、HTTP 也无所谓，可以直接暴露后端端口（但其实加个 Nginx 也就 10 分钟）

## 给你的建议

智能拓客这种业务系统，建议：

- **开发期**：前端 dev server proxy，不碰 Nginx
- **联调/测试环境**：搭一台 Nginx，前后端域名/路径统一
- **生产环境**：必走 Nginx（或云厂商 LB），HTTPS、限流、日志都要

配置不复杂，给你贴一个最小可用版本：

```nginx
server {
    listen 80;
    server_name your-domain.com;

    # 前端静态文件
    location / {
        root /var/www/smart-lead/dist;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API
    location /api/ {
        proxy_pass http://127.0.0.1:8000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

后续要 HTTPS 用 certbot 一行命令加证书，要负载均衡加个 `upstream` 块就行，先这样起步够用。

一个项目对应一个数据库吗？

**Assistant:**

不强制，但**默认推荐一项目一库**。展开讲就是几个判断点：

## 默认建议：一个项目一个数据库

好处：

- **隔离**：A 项目跑挂了不会拖垮 B 项目（连接数、慢查询互不影响）
- **权限干净**：每个项目一个 DB 账号，只能访问自己的库，安全边界清晰
- **备份/迁移独立**：备份不用纠结哪些表是这个项目的
- **DDL 不打架**：你改表结构不会动到别的项目
- **命名空间清爽**：不用给表名加项目前缀（`smart_lead_customer` vs `crm_customer`）

## 但有几种典型的例外

### 1. 多项目共用一个库（业务紧密耦合时）

比如公司已经有 CRM 系统，里面有 `customer`、`sales_account`、`order` 等表。你新做的"智能拓客"严重依赖这些数据，**不应该新建库**，而是在 CRM 库里加你的新表（`customer_activity`、`sales_profile_industry` 等）。

否则会出现一个尴尬场景：你要算销售画像，得跨库查 CRM 的订单表 → 跨库 JOIN 性能差、事务不可控、CRM 改表你就崩。

### 2. 一个项目多个库（项目复杂时）

- **微服务架构**：用户服务一个库、订单服务一个库、商品服务一个库（DDD 思想）
- **读写分离**：主库写，多个从库读（但本质还是同一个逻辑库）
- **分库分表**：单表数据量过亿，按用户 ID 哈希分到多个物理库
- **冷热分离**：热数据 MySQL，历史数据归档到另一个库

这些都是项目大了才考虑，初期不用想。

## 关于"数据库"这个词的歧义

不同 DB 概念不一样，容易聊岔：

| DB | "database" 是什么 |
|----|---|
| **MySQL** | 等同于 schema，一个 MySQL 实例下可以有多个 database，平级 |
| **PostgreSQL** | 真正的隔离单位，跨 database 查询要专门处理（dblink/FDW），不方便 |
| **SQL Server** | 类似 MySQL，一个实例多个 database |
| **Oracle** | 一个实例通常一个 database，用 schema(user) 来分隔项目 |

所以"一项目一库"在 MySQL 里很轻量（就是 `CREATE DATABASE` 一下），在 PostgreSQL 里成本稍高（要单独建用户、连接池等）。
