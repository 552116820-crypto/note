```text
企微智能机器人消息

  -> 提取文本 prompt
  -> 调滴普 POST /chat/chat-messages
```

### Turn 2  *(2026-04-28 00:19)*

**User:**

我是小白，请你详细讲一下桥接

**Assistant:**

“桥接”可以理解成：这个项目站在 **企业微信机器人** 和 **滴普智能体** 中间，帮它们互相“翻译”。

企业微信不会直接懂滴普的接口，滴普也不会直接懂企业微信机器人的消息格式。所以这个项目做了一个中间服务：

```text
用户在企微群里 @机器人
        ↓
企业微信把消息推给本项目
        ↓
本项目把企微消息整理成滴普需要的提问格式
        ↓
本项目调用滴普智能体接口
        ↓
滴普返回答案
        ↓
本项目把答案整理成企微机器人能发送的格式
        ↓
机器人在群里回复用户
```

**为什么需要桥接**
企业微信和滴普是两个不同系统，它们的“说话方式”不一样。

企业微信这边关心的是：

```text
机器人是谁？
消息来自哪个群？
谁发的？
消息内容是什么？
要用什么企微协议回复？
```

滴普这边关心的是：

```text
app_id 是什么？
signature 签名对不对？
agent_id 是哪个智能体？
query 用户问了什么？
conversation_id 是否要继续上一轮对话？
```

桥接服务的工作，就是把这两套东西对上。

**这项目的主桥接方式**
当前项目推荐的是 **WebSocket 长连接模式**。

也就是说，这个服务启动后，会主动连到企业微信：

```text
本项目  --->  企业微信 WebSocket 网关
```

连接地址在 [wecom_ws.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/wecom_ws.py:17)：

```python
DEFAULT_WS_URL = "wss://openws.work.weixin.qq.com"
```

启动后，它会用企微后台给你的：

```env
WECOM_BOT_ID=...
WECOM_BOT_SECRET=...
```

去做机器人鉴权。对应代码在 [wecom_ws.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/wecom_ws.py:128)，发送的是 `aibot_subscribe`。

鉴权成功后，企业微信有消息就会通过这个长连接发过来。

**收到企微消息后做了什么**
核心处理在 [__main__.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/__main__.py:170)。

它会做几件事：

1. 取出消息 ID，避免重复处理。
2. 判断是群聊还是单聊。
3. 取出发送人 `userid`。
4. 从企微消息里提取文本内容。
5. 如果是群里 `@机器人 你好`，会把前面的 `@机器人` 去掉，只保留真正的问题。
6. 如果用户引用了别人的消息，会把引用内容也拼进问题里。
7. 判断是否是 `/reset`、`重置会话` 这种清空上下文命令。

比如用户在群里发：

```text
@机器人 帮我总结一下这段内容
```

项目会提取成：

```text
帮我总结一下这段内容
```

然后再发给滴普。

**怎么调用滴普智能体**
调用滴普的代码在 [agent_chat.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/agent_chat.py:106)。

它请求的是：

```text
POST {DEEPEXI_BASE_URL}/chat/chat-messages
```

请求体大概长这样：

```json
{
  "inputs": {},
  "agent_id": "你的滴普智能体ID",
  "query": "用户的问题",
  "response_mode": "blocking"
}
```

如果是多轮对话，还会多带一个：

```json
{
  "conversation_id": "上一轮滴普返回的会话ID"
}
```

滴普接口还需要鉴权。项目会自动生成请求头：

```text
app_id
timestamp
signature
app_user
```

其中 `signature` 是用 `DEEPEXI_SECRET_KEY` 算出来的签名，代码在 [agent_chat.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/agent_chat.py:511)。

所以你只需要在 `.env` 里配置：

```env
DEEPEXI_BASE_URL=...
DEEPEXI_APP_ID=...
DEEPEXI_SECRET_KEY=...
DEEPEXI_AGENT_ID=...
```

项目会负责把这些配置变成滴普 API 请求。

**滴普返回后怎么回企微**
滴普一般会返回类似：

```json
{
  "answer": "这是智能体的回答",
  "conversation_id": "xxx"
}
```

项目会取出 `answer`，然后通过企业微信 WebSocket 回给用户。

回消息代码在 [wecom_ws.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/wecom_ws.py:239)，发送的是企微要求的 `aibot_respond_msg`。

如果回答很短，就直接发文本。

如果回答太长，项目会把内容转成 `.md` 文件，通过企微文件接口上传后再发给用户。这个逻辑在 [__main__.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/__main__.py:258)。

**多轮对话怎么实现**
滴普返回 `conversation_id` 后，项目会缓存起来。

缓存代码在 [conversation_store.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/conversation_store.py:15)。

它的规则是：

```text
单聊：按 userid 记住上下文
群聊：按 chatid + userid 记住上下文
```

也就是说，同一个人在同一个群里连续问问题，会继续同一个滴普会话。

但不同人在同一个群里问问题，不会混在一起。

默认缓存 1800 秒，也就是 30 分钟。可以通过：

```env
CONVERSATION_TTL_SECONDS=1800
```

调整。

**按群切换不同智能体**
这个项目还支持一个实用功能：不同企微群接不同滴普智能体。

比如：

```env
DEEPEXI_GROUP_ROUTES_JSON={"销售群chatid":"sales-agent-id","运维群chatid":"ops-agent-id"}
```

项目收到群消息后，会看 `chatid`，然后决定调用哪个 `agent_id`。

对应逻辑在 [config.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/config.py:475)。

**一句话总结**
这个项目不是“把企微和滴普直接连起来”，而是自己做了一个中间服务：

```text
企微协议进来
  -> 转成滴普 Agent Chat 请求
  -> 拿到滴普答案
  -> 转成企微机器人回复
```

所以它的核心价值是：  
**接企微消息、调滴普智能体、把答案发回企微，并顺手处理鉴权、去重、上下文、群路由和长回复文件化。**

### Turn 3  *(2026-04-28 00:28)*

**User:**

chatid是什么，怎么知道群的chatid

**Assistant:**

`chatid` 就是 **企业微信群聊的会话 ID**。

它不是群名称，也不是群成员 ID，而是企业微信在机器人消息里带过来的一个唯一标识。这个项目用它来判断“这条消息来自哪个群”，从而决定要不要切换到不同的滴普智能体。

比如配置里这个：

```env
DEEPEXI_GROUP_ROUTES_JSON={"CHATID_A":"agent-sales"}
```

这里的 `CHATID_A` 不能写“销售群”这种群名，必须写企业微信消息里的真实 `chatid`。

怎么知道群的 `chatid`：

1. 先启动服务：

```bash
WECOM_MODE=websocket python3 -m wecom_dify_aibot
```

2. 在目标企业微信群里 `@机器人` 随便发一句，比如：

```text
@机器人 你好
```

3. 看服务日志，项目会打印类似：

```text
WS message: msgtype=text msgid=xxx chattype=group chat=xxxxxx
```

这里 `chat=xxxxxx` 后面的值，就是这个群的 `chatid`。

对应代码在 [__main__.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/__main__.py:174)，它从企微消息里取：

```python
chat_id = (body.get("chatid") or "").strip()
```

然后日志在 [__main__.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/__main__.py:177) 打出来。

拿到之后就可以这样配：

```env
DEEPEXI_GROUP_ROUTES_JSON={"真实chatid":"滴普agent_id"}
```

如果你还想让日志里显示可读群名，可以再配：

```env
DEEPEXI_GROUP_ALIASES_JSON={"真实chatid":"销售群"}
```

注意：单聊没有群 `chatid`，日志会显示 `single-chat`。只有群聊消息才适合用 `chatid` 做路由。

```env
DEEPEXI_GROUP_ROUTES_JSON={
```

### Turn 6  *(2026-04-28 00:48)*

**User:**

一个企微机器人在一个群可配置多个agent_id吗，我希望能路由到不同的agent

**Assistant:**

可以，但要分清两层：

**企微机器人本身**不配置多个 `agent_id`。  
企微只负责把消息发给你的服务，它不知道滴普的 `agent_id` 是什么。

**是否路由到多个 agent_id，是这个项目后端决定的。**

当前项目已经支持：

```text
不同群 -> 不同 agent_id
```

也就是：

```env
DEEPEXI_GROUP_ROUTES_JSON={
  "群A的chatid":"agent-a",
  "群B的chatid":"agent-b"
}
```

但当前代码默认是：

```text
同一个群 -> 只能命中一个 agent_id
```

因为代码里是按 `chatid` 选路由：[config.py](/mnt/e/wecom-dify-aibot/wecom_dify_aibot/config.py:475)

```python
def resolve_agent_route(self, *, chat_type: str, chat_id: str) -> AgentRoute:
    if chat_type == "group" and chat_id:
        return self.agent_group_routes.get(chat_id, self.agent_default_route)
    return self.agent_default_route
```

所以现在的逻辑是：看到这个群的 `chatid`，就选一个固定 agent。

如果你希望 **同一个群里路由到不同 agent**，需要再加一层规则。常见做法有三种：

**方案一：按命令前缀路由，最推荐**

比如同一个群里这样问：

```text
@机器人 /日报 今天销售情况
@机器人 /法务 帮我看合同条款
@机器人 /运维 查一下服务异常
```

配置类似：

```env
DEEPEXI_COMMAND_ROUTES_JSON={
  "/日报":"agent-sales",
  "/法务":"agent-legal",
  "/运维":"agent-ops"
}
```

项目收到消息后，先看问题开头有没有 `/日报`、`/法务`、`/运维`，然后选择对应 `agent_id`。

优点是稳定、好理解、不容易误判。

**方案二：按关键词路由**

比如：

```text
包含“合同”“条款” -> 法务 agent
包含“报表”“销售” -> 销售 agent
包含“故障”“告警” -> 运维 agent
```

这个对用户更自然，但容易误判。小白阶段不建议一开始就做这个，除非规则很简单。

**方案三：让一个总控 agent 先判断该转哪个 agent**

流程变成：

```text
用户问题
  -> 总控/路由 agent 判断类别
  -> 后端再调用真正的业务 agent
```

这个更智能，但实现复杂，也会多一次接口调用，延迟和成本都更高。

`DEEPEXI_APP_USER=004105` 是传给滴普接口请求头里的 `app_user`。

这个项目调滴普时会带这些请求头：

```text
app_id
signature

app_user
```

其中 `app_user` 表示“当前是谁在使用这个应用/智能体”。它不是企业微信机器人配置，也不是企微 `userid` 本身，而是传给滴普用来识别用户身份的字段。

代码逻辑是这样的：

如果你配置了：

```env
DEEPEXI_APP_USER=004105
```

那所有企微用户问问题时，都会固定用：

```text
app_user: 004105
```

如果你不配置：

```env
DEEPEXI_APP_USER=
DEEPEXI_APP_USER_PREFIX=wecom:
```

那项目会按企微消息里的用户 ID 自动生成，比如：

```text
app_user: wecom:zhangsan
app_user: wecom:lisi
```

所以：

```env
DEEPEXI_APP_USER=004105
```

企业微信全局错误码里，`6000` 对应“数据版本冲突”，排查建议是“可能有多个调用端同时修改数据，稍后重试”。参考：

- 某一次流式刷新可能失败

3. 对 `errcode=6000` 做短延迟重试，比如等 `0.5s` 后重发当前内容。
4. 最后一帧 `finish=true` 最好也做重试，避免消息一直处在未完成状态。
