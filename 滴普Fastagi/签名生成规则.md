# 签名生成规则 QA

## Q1：签名生成规则的代码逐行解释

### 1. 生成时间戳

```python
timestamp = str(int(time.time() * 1000))
```

- `time.time()` → 返回秒级浮点数，如 `1711526400.123`
- `* 1000` → 转成毫秒，`1711526400123.0`
- `int()` → 去掉小数，`1711526400123`
- `str()` → 转字符串，因为后面要拼接

### 2. 拼接待签名字符串

```python
message = f"{app_id}{timestamp}"
```

假设 `app_id = "abc123"`，时间戳 = `"1711526400123"`，拼出来就是：

```
"abc1231711526400123"
```

直接拼在一起，没有分隔符。

### 3. HMAC-SHA256 签名

```python
signature = hmac.new(secret_key.encode(), message.encode(), hashlib.sha256).hexdigest()
```

| 步骤  | 代码                         | 作用                               |
| --- | -------------------------- | -------------------------------- |
| 1   | `.encode()`                | 字符串转 bytes（HMAC 要求 bytes 输入）     |
| 2   | `hmac.new(密钥, 消息, sha256)` | 用 SECRET_KEY 对消息做 HMAC-SHA256 运算 |
| 3   | `.hexdigest()`             | 输出结果转成 64 位十六进制字符串               |

### HMAC-SHA256 是什么

可以理解为一个**带密码的哈希函数**：

```
普通哈希：  SHA256(消息)        → 任何人都能算出来
HMAC：     SHA256(消息 + 密钥)  → 只有知道密钥的人才能算出正确结果
```

服务端收到请求后，用同样的 `app_id`、`timestamp` 和存在数据库里的 `SECRET_KEY`，重新算一遍签名。算出来一样 = 请求合法，不一样 = 拒绝。

### 整个流程

```
客户端                                    服务端
──────                                   ──────
app_id = "abc123"                        数据库里存着:
secret_key = "my_secret"                 abc123 → my_secret

拼接: "abc123" + "1711526400123"
     = "abc1231711526400123"

HMAC-SHA256("abc1231711526400123",       收到请求头:
            "my_secret")                   app_id = "abc123"
     = "a1b2c3..."                         timestamp = "1711526400123"
                                           signature = "a1b2c3..."
发送请求 ──────────────────────────→
  header: app_id, timestamp, signature   1. 检查 timestamp 是否在 ±5分钟内
                                         2. 用 abc123 查出 secret_key
  ← 密钥本身从未传输 →                    3. 重算 HMAC → 得到 "a1b2c3..."
                                         4. 对比签名 → 一致 → 放行
```

**关键点：SECRET_KEY 始终不出现在网络请求中，只用来本地计算签名。**

## Q2：time.time() 是什么

`time.time()` 是 Python `time` 模块的函数，返回**当前时间的 Unix 时间戳**——从 **1970-01-01 00:00:00 UTC** 到现在经过的**秒数**：

```python
import time
time.time()  # → 1711526400.123456（秒级，带小数）
```

全世界不管什么时区，同一时刻调用 `time.time()` 得到的值一样，服务端可以直接比较，不用处理时区问题。

代码中的转换链：

```python
time.time()                 # 1711526400.123456  秒，浮点数
time.time() * 1000          # 1711526400123.456  毫秒，浮点数
int(time.time() * 1000)     # 1711526400123      毫秒，整数
str(int(time.time()*1000))  # "1711526400123"    毫秒，字符串
```

乘 1000 是因为这个 API 要求毫秒级时间戳。

## Q3：为什么是 1970-01-01

这是个历史约定，没有深层技术原因。

1971 年 Unix 操作系统的开发者需要定一个时间起点来计数。最初用的是 1971-01-01，后来改成了 1970-01-01 00:00:00 UTC，因为：

- 1970 是个整数年份，干净好记
- 刚好在 Unix 诞生（1969-1971）前后，时间戳不会出现负数
- 用 32 位整数存储，能覆盖到 2038 年，当时觉得够用了

类比：就像公历选耶稣出生那年当"公元1年"，纯粹是选了一个大家都认的参考点。

## Q4：2038 年之后呢（Y2K38 问题）

32 位有符号整数最大值是 2,147,483,647，对应 **2038-01-19 03:14:07 UTC**。过了这一秒，数值溢出，时间会回滚到 1901 年。

**已经解决了**：现代系统基本都改用 64 位整数存时间戳（能表示到约 2920 亿年后）。Linux 内核 5.6（2020 年）起全面支持，Windows、macOS 早已切换，Python 的 `time.time()` 返回浮点数不受 32 位限制。

还有风险的是一些老旧嵌入式设备（路由器、工控系统、老ATM机），仍在用 32 位时间且难以升级。

## Q5：Y2K38 和千年虫（Y2K）是什么

### 千年虫（Y2K）

上世纪六七十年代内存很贵，程序员用两位数存年份：

```
1998 → 98
1999 → 99
2000 → 00  ← 00 是 2000 还是 1900？
```

当时担心 2000 年 1 月 1 日大量系统把 00 当成 1900，导致银行利息计算错乱、航班系统崩溃等。全球花了几千亿美元排查修复，最终平稳过渡。

### 对比

| | Y2K（千年虫） | Y2K38 |
|--|--|--|
| 原因 | 年份只存 2 位数字 | 时间戳只用 32 位整数 |
| 触发时间 | 2000-01-01 | 2038-01-19 |
| 后果 | 年份回到 1900 | 时间回到 1901 |
| 解法 | 改成 4 位年份 | 改成 64 位整数 |

**一句话总结：都是早期为了省资源做的妥协，当时觉得够用，没想到系统能活这么久。**

## Q6：HMAC-SHA256 运算是什么

拆成两部分理解：

### SHA256 — 哈希函数

把**任意长度**的数据变成**固定 64 个十六进制字符**（256 位）：

```
SHA256("hello")     → 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
SHA256("hello!")    → ce06092fb948d9ffac7d1a376e404b26b7575bcc11ee05a4615fef4fec3a308b
```

特点：
- **不可逆** — 无法从结果反推出原始数据
- **雪崩效应** — 输入改一个字符，输出完全不同
- **确定性** — 同样的输入永远得到同样的输出

### HMAC — 加了密钥的哈希

HMAC = **H**ash-based **M**essage **A**uthentication **C**ode（基于哈希的消息认证码）

普通 SHA256 谁都能算，加上密钥后就变成了身份验证：

```
SHA256("hello")                → 任何人都能算出来
HMAC-SHA256("hello", 密钥)     → 只有知道密钥的人才能算出一样的结果
```

### 内部过程（简化）

```
输入：消息 = "abc1231711526400123"，密钥 = "my_secret"

第一步：密钥和一个固定值异或，得到 key1
第二步：SHA256(key1 + 消息)  → 中间结果
第三步：密钥和另一个固定值异或，得到 key2
第四步：SHA256(key2 + 中间结果) → 最终签名
```

本质上就是**做了两次 SHA256**，密钥参与了两次运算，这样比简单地 `SHA256(密钥+消息)` 更安全，能防御一些已知的攻击手段。

### 在签名场景中的作用

```
客户端：HMAC-SHA256("abc1231711526400123", "my_secret") → "a1b2c3..."
                                                          ↑ 发给服务端

服务端：HMAC-SHA256("abc1231711526400123", "my_secret") → "a1b2c3..."
                                                          ↑ 自己算一遍

两边结果一样 → 证明客户端确实知道 secret_key → 请求合法
```

**一句话：HMAC-SHA256 就是用密钥给消息"盖章"，只有持有同一把密钥的人才能盖出一样的章。**
