**Assistant:**

好问题——这其实是"把用户状态叠加到推荐结果上"的经典设计。先把这个项目里最关键的约束摆出来，它决定了整个做法：

**业务库 `amer_crm_v2` 是只读的**，所以"看过/跳过"这种会变的用户状态**绝对不能写进它**。而需求里那个第二库 `churn_db`（root 可写、`.env` 里已有 `LOG_DB_*` 但本期没连）正好是干这个的。状态按 `(employee_code, customer_id)` 存。

## 1. 机制：软分层，别硬删

两条路：
- **硬过滤**（跳过的永不再现）：简单，但手滑跳错就永久丢一个高价值客户，`total` 还会缩水。
- **软分层（推荐）**：未看过的排前、跳过的沉到末尾，两层各自按分数排。好处是跳过的不会永久消失（未看过的翻完后还能兜底出现），下次进来天然优先未看过。

## 2. 关键架构点：分层"烤进缓存"，让翻页稳定、跳过下次生效

你现在的链路是 `(员工+定位+半径) → 算分排序 → 缓存有序列表 → 翻页切片`。跳过状态是 per 业务员、跟定位无关的。我建议：

> **在缓存"未命中、重算"那一刻**，顺手读一次该业务员的 skip 集合，把候选列表切成 `[未看过, 跳过]`（各自按分数排），把这个顺序存进缓存 blob。翻页就切这个冻结的顺序。

这样有三个好处，而且正好命中你说的"**下次**进来优先未看过"：
- **翻页稳定**：一次浏览会话内顺序不变，不会因为你边翻边跳导致列表抖动、漏看（这是"读取时实时重排"会踩的坑）。
- **跳过下次生效**：skip 写入是即时的，但重排发生在缓存过期/下次进来重算时（≤5–10 分钟，或加个"刷新"按钮强制重算）。"跳过"本来就该影响下一次，不该在你正滚动时把脚下的客户抽走。
- **算分不重算**：贵的那步还在缓存里，skip 只是廉价地查一下集合。

## 3. 写入：一个小接口 + 一张表（churn_db）

```
POST /api/v1/leads/skip   { "employee_code": "000068", "customer_id": "198" }
```
留个 `action` 字段，以后能扩展 收藏 / 已联系 / 不感兴趣原因，不只跳过。

```sql
CREATE TABLE lead_skip (
  id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  employee_code VARCHAR(16)     NOT NULL,
  customer_id   BIGINT UNSIGNED NOT NULL,
  action        VARCHAR(16)     NOT NULL DEFAULT 'skip',
  created_at    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_emp_cust (employee_code, customer_id),   -- 幂等 upsert，跳两次不重复
  KEY idx_emp (employee_code)                            -- 读 skip 集合走索引，很快
);
```
读取：`SELECT customer_id FROM lead_skip WHERE employee_code=?` → 一个集合，分层时用。

顺手在 `pagination` 里加 `unseen_total` / `seen_total`，前端能显示"还有 N 个新客户"。

**服务器对，库不对。**
