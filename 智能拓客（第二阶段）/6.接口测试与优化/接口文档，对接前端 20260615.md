# 智能拓客 · 接口文档（对接前端）

> 版本：v1 ・ 更新日期：2026-06-11（本次变更：成功响应 `message` 由 `"ok"` 改为空串 `""`（代码已对齐）；score-and-info 请求示例补充行业筛选 `industry_ids` / `industry_names`）
> 共 **4 个业务接口** + 1 个健康检查。后端为 FastAPI，统一三段式响应。
> 本文档每个接口给出：请求体、返回体（逐字段注释）、测试用例、对接注意。

---

## 一、全局约定

### 1. Base URL
```
http(s)://<host>:<port>
```
（联调时由后端提供，例：`http://127.0.0.1:8000`。上线域名/网关另行通知。）

### 2. 请求与响应格式
- 所有业务接口：**方法 `POST`**，请求头 `Content-Type: application/json`，body 为 JSON 对象。
- 所有响应统一三段式 envelope：

```json
{ "code": 0, "message": "", "data": { } }
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | int | 业务码。**`0` 表示成功**，非 0 见「错误码表」 |
| `message` | string | 提示信息（成功为 `""`，失败为可读原因） |
| `data` | object \| null | 业务数据；失败时为 `null` |

> **前端判定成功只看 `code === 0`，不要只看 HTTP 状态码。**

### 3. 鉴权
当前接口**无鉴权**（无 token / 无自定义请求头）。上线如经网关加鉴权，另行通知。

### 4. HTTP 状态码规则
- **业务结果一律 HTTP 200**（成功与大多数业务错误都返回 200，靠 `code` 区分）。
- **请求体超过 1MB 返回 HTTP 413**（`code 1002`），用于防止异常大请求占用服务内存。
- **仅两类「库不可用」返回 HTTP 503**：业务库不可用（`code 1503`）、状态库不可用（`code 1505`）。
- **未预期的服务端异常返回 HTTP 500**（`code 1500`，兜底，正常不应出现）。

### 5. 全局错误码表

| code | HTTP | message（示例） | 触发场景 | 前端建议 |
|---|---|---|---|---|
| `0` | 200 | 空字符串 `""` | 成功 | 正常渲染 |
| `1002` | 200 | 参数校验失败: … | 请求体字段缺失/越界/类型错/格式错 | 提示参数错误，通常是前端传参问题，排查后重试 |
| `1002` | 413 | 请求体过大，最大 1048576 字节 | 请求体超过 1MB | 检查是否误传整页列表、文件/base64 或超大 `score_context` |
| `1003` | 200 | 候选客户过多，请缩小搜索半径 | 搜索半径内候选超过上限 | 提示用户缩小 `radius_km` 后重试 |
| `1500` | 500 | 服务内部错误，请稍后重试 | 未预期的服务端异常（兜底） | 提示稍后重试；上报后端排查 |
| `1503` | 503 | 数据库暂不可用，请稍后重试 | 业务库（amer_crm_v2）连不上 | 提示稍后重试 |
| `1505` | 503 | 状态库暂不可用，请稍后重试 | 状态库（amer_sales_assist）连不上（仅 `/skip`） | 提示稍后重试；不影响看分数/详情 |

> 说明：源码中另保留 `1001`/`1004` 等码，但**当前版本不会返回**，前端无需处理。

### 6. 降级约定（接口三、接口四）
行为时间轴与 AI 推荐理由依赖外部 CRM / 大模型，**全程降级、绝不报错**：任一外部依赖失败，接口仍返回 `code:0` + 可用数据，失败情况由 `data` 内的标记字段表达：
- `degraded`：`true` 表示部分 CRM 数据源缺失（数据不完整但可用）。
- `sources`：三个 CRM 源各自的 `"ok"`/`"error"` 状态（仅时间轴）。
- `reason_source`：推荐理由来自 `"llm"`（大模型）还是 `"template"`（确定性兜底模板）。

### 7. 字段展示约定（重要）
- 字符串字段缺失/为空 → 返回 **`"-"`**（连字符），前端可直接展示。
- 列表字段无内容 → 返回 **`[]`**。
- 数值字段（如 `star`、`social_insurance_count`）：**有值返回 int（含合法的 `0`）；缺失/不可解析返回 `"-"`**。即类型为 `number | "-"` 联合，前端需判类型后再渲染。

## 二、接口一：客户分数与基本信息

按业务员定位，对公海客户打分并按总分倒序分页返回。

- **URL**：`POST /api/v1/leads/score-and-info`
- **依赖**：业务库（地理/企业/星级）+ 状态库（跳过沉底）+ 缓存
- **典型耗时**：首个未缓存请求约 **7–10s**（地理候选查询为主）；同条件翻页命中缓存约 **~2ms**

### 请求体

| 字段 | 类型 | 必填 | 默认 | 取值/校验 | 说明 |
|---|---|---|---|---|---|
| `employee_code` | string | 是 | — | 非空（前后空格自动去除） | 业务员编号 |
| `lng` | number | 是 | — | [-180, 180] | 业务员定位经度 |
| `lat` | number | 是 | — | [-90, 90] | 业务员定位纬度 |
| `radius_km` | number | 否 | 60 | >0 且 ≤100000；**超过上限 200 会被裁剪** | 搜索半径(km) |
| `page` | int | 否 | 1 | ≥1 | 页码 |
| `page_size` | int | 否 | 20 | ≥1；**超过 100 自动截断为 100（不报错）** | 每页条数 |
| `industry_ids` | int[] | 否 | — | 可多选；**优先级高于 industry_names** | 按行业 id 筛选 |
| `industry_names` | string[] | 否 | — | 可多选；id 缺省时生效；名应取自 `GET …/industries`（逐字） | 按行业名筛选（后端反查成 id） |

请求示例（不筛行业，全量）：
```json
{ "employee_code": "000068", "lng": 117.860197, "lat": 39.966853, "radius_km": 60, "page": 1, "page_size": 20 }
```

按行业 id 筛选（**推荐**：id 取自 `GET …/industries` 下拉，可多选）：
```json
{ "employee_code": "000068", "lng": 117.860197, "lat": 39.966853, "radius_km": 60, "page": 1, "page_size": 20, "industry_ids": [43] }
```

按行业名筛选（仅当 `industry_ids` 缺省时生效；名须与 `GET …/industries` 返回**逐字一致**）：
```json
{ "employee_code": "000068", "lng": 117.860197, "lat": 39.966853, "radius_km": 60, "page": 1, "page_size": 20, "industry_names": ["数控机床", "工业自动化"] }
```

### 返回体 `data`

| 字段 | 类型 | 说明 |
|---|---|---|
| `employee_context` | object | 业务员上下文（见下） |
| `pagination` | object | 分页信息（见下） |
| `list` | array | 客户列表，按 `total_score` 倒序（同分按距离近优先） |

`employee_context`：

| 字段 | 类型 | 说明 |
|---|---|---|
| `employee_code` | string | 业务员编号 |
| `deal_customer_industries` | string[] | 该业务员成交过的行业名（去重排序）；未知业务员为 `[]` |
| `industry_top_customers` | object | `{行业名: 头部客户名}`，每个成交行业的代表客户 |

`pagination`：

| 字段 | 类型 | 说明 |
|---|---|---|
| `page` | int | 当前页 |
| `page_size` | int | 每页条数（已截断后的实际值） |
| `total` | int | 候选总数 |
| `total_pages` | int | 总页数 |
| `has_more` | bool | 是否还有下一页 |
| `unseen_total` | int | 未跳过（“新”）客户数（用于“还有 N 个新客户”） |
| `skipped_total` | int | 已跳过、沉底的客户数 |
| `requested_radius_km` | number? | **仅当半径被裁剪时出现**：前端请求的原始半径 |
| `actual_radius_km` | number? | **仅当半径被裁剪时出现**：实际生效半径（≤200） |

`list[]` 每个客户：

| 字段 | 类型 | 说明 |
|---|---|---|
| `customer_id` | string | 客户 id |
| `customer_name` | string | 客户名；缺失为 `"-"` |
| `total_score` | int | 总分（0–100）= 各明细之和 |
| `score_detail` | object | `{industry_match, scale, tags, distance}`，见下 |
| `amer_industry_id` | int \| "-" | 行业 id；无为 `"-"` |
| `amer_industry` | string | 行业名；无为 `"-"` |
| `distance_km` | int | 距业务员距离(km，四舍五入) |
| `customer_lng` | number \| null | 客户经度 |
| `customer_lat` | number \| null | 客户纬度 |
| `address` | string | 地址；缺失为 `"-"` |
| `actual_capi` | string | 实缴资本（原文）；缺失为 `"-"` |
| `tag_name` | string[] | 最近 2 个去重标签；无为 `[]` |
| `star` | int \| "-" | 星级（`0` 合法）；缺失为 `"-"` |
| `social_insurance_count` | int \| "-" | 参保人数（`0` 合法）；缺失为 `"-"` |
| `regist_capi` | string | 注册资本（原文）；缺失为 `"-"` |

`score_detail`（满分 100）：

| 字段 | 取值 | 规则 |
|---|---|---|
| `industry_match` | 0 或 40 | 客户行业∈业务员成交行业 → 40 |
| `scale` | 0 或 10 | 实缴资本 ≥ 500 万人民币 → 10（外币不计） |
| `tags` | 0 或 10 | 客户有标签 → 10 |
| `distance` | 0–40 | ≤20km=40；20–60km 线性衰减；>60km=0 |

返回示例：
```json
{
  "code": 0,
  "message": "",
  "data": {
    "employee_context": {
      "employee_code": "000068",
      "deal_customer_industries": ["数控机床", "工业自动化"],
      "industry_top_customers": { "数控机床": "某某精密机械有限公司" }
    },
    "pagination": {
      "page": 1, "page_size": 20, "total": 137, "total_pages": 7,
      "has_more": true, "unseen_total": 137, "skipped_total": 0
    },
    "list": [
      {
        "customer_id": "3599", "customer_name": "某某精密机械有限公司",
        "total_score": 90,
        "score_detail": { "industry_match": 40, "scale": 10, "tags": 0, "distance": 40 },
        "amer_industry_id": 43, "amer_industry": "数控机床",
        "distance_km": 8, "customer_lng": 117.85, "customer_lat": 39.97,
        "address": "天津市…", "actual_capi": "5000万人民币",
        "tag_name": ["意向客户"], "star": 3, "social_insurance_count": 120, "regist_capi": "5000万人民币"
      }
    ]
  }
}
```

### 错误返回示例
```json
{ "code": 1002, "message": "参数校验失败: lat lat 必须在 [-90, 90]", "data": null }
{ "code": 1003, "message": "候选客户过多，请缩小搜索半径", "data": null }
{ "code": 1503, "message": "数据库暂不可用，请稍后重试", "data": null }
```

### 测试用例

| 用例 | 关键入参 | 预期 code | 预期 data 要点 |
|---|---|---|---|
| 正常查询 | 合法 employee/坐标/半径 | 0 | `list` 按 total_score 倒序、`total_score == 各明细之和` |
| 翻页命中缓存 | 同条件 `page=2` | 0 | `total` 与首页一致、响应 ~ms 级 |
| 纬度越界 | `lat=999` | 1002 | data=null，message 含“参数校验” |
| 经度越界 | `lng=181` | 1002 | data=null |
| 半径为 0 | `radius_km=0` | 1002 | data=null |
| 半径超上限 | `radius_km=500` | 0 | `pagination.actual_radius_km=200`、`requested_radius_km=500` |
| page_size 超上限 | `page_size=500` | 0 | `pagination.page_size=100`（截断不报错） |
| page=0 | `page=0` | 1002 | data=null |
| employee_code 为空 | `""` 或 `"   "` | 1002 | data=null |
| 业务员前后带空格 | `" 000068 "` | 0 | 与 `"000068"` 等价（自动去空格） |
| 缺必填字段 | 不传 `lat` | 1002 | data=null |
| 候选过多 | 半径过大命中上限 | 1003 | data=null |
| 按行业 id 筛 | `industry_ids=[43]` | 0 | `list` 仅该行业、分页计数随之变 |
| 按行业名筛 | `industry_names=["数控机床"]` | 0 | 等价于对应 id 的结果 |
| 行业筛选无匹配 | `industry_names=["不存在"]` | 0 | `list` 空、`total=0` |
| industry_ids 类型错 | `industry_ids=["x"]` | 1002 | data=null |

### 对接注意
- `employee_code` 会自动去前后空格再处理。
- `page_size` 超 100 自动截断为 100，**不报错**；前端可放心传，但建议本就 ≤100。
- 半径超 200 会被裁剪，注意读取 `actual_radius_km` 给用户提示。
- 部分客户经纬度为 `(0,0)`（未地理编码）属历史数据，永远不会命中，无需特殊处理。
- 列表字段的 `"-"` / `[]` / `number|"-"` 约定见「全局约定 §7」。
- **行业筛选**：`industry_ids`/`industry_names` 都可选、可多选；**id 优先**（两者都传只用 id）。name 由后端反查成 id 再筛——**请用 `GET …/industries` 返回的名**（逐字）以保证命中；无行业 id 的候选会被排除；**不传或传 `[]` = 不筛选（全量）**；传了但匹配不到任何行业 → 返回空 `list`。筛选只收窄 `list` 与分页计数，不影响打分与业务员上下文。

### 配套：行业字典 `GET /api/v1/leads/industries`

供前端"按行业筛选"的下拉：返回全部行业 id+name。**无请求参数**。返回 `data`：

| 字段 | 类型 | 说明 |
|---|---|---|
| `industries` | array | `[{ "id": int, "name": string }, ...]`，按 name 排序 |
| `source` | string | 行业字典来源 `db`/`snapshot`/`none`（`none`=暂不可用、返回空列表） |

示例：`{ "code":0, "message":"", "data": { "industries": [ {"id":43,"name":"数控机床"} ], "source":"db" } }`

> 前端：下拉显示 `name`、把选中的 `id` 放进 score-and-info 的 `industry_ids`（推荐）；或直接传 `industry_names`（用这里返回的名）。

## 三、接口二：跳过客户

把某客户标记为“跳过”，下次重算时沉底。幂等。

- **URL**：`POST /api/v1/leads/skip`
- **依赖**：状态库 `amer_sales_assist.lead_skip`

### 请求体

| 字段 | 类型 | 必填 | 取值/校验 | 说明 |
|---|---|---|---|---|
| `employee_code` | string | 是 | 非空（前后空格自动去除） | 业务员编号 |
| `customer_id` | string | 是 | **必须为纯数字**（数字或数字字符串均可） | 要跳过的客户 id |

请求示例：
```json
{ "employee_code": "000068", "customer_id": "3599" }
```

### 返回体
成功固定返回：
```json
{ "code": 0, "message": "", "data": null }
```

### 错误返回

| 场景 | 返回 |
|---|---|
| 参数校验失败（customer_id 非数字、employee_code 空等） | `{ "code": 1002, "message": "参数校验失败: …", "data": null }` |
| 状态库连不上 | `{ "code": 1505, "message": "状态库暂不可用，请稍后重试", "data": null }`（HTTP 503） |

### 测试用例

| 用例 | 关键入参 | 预期 code |
|---|---|---|
| 正常跳过 | 合法 employee/customer | 0 |
| 重复跳过（幂等） | 同一对再跳一次 | 0（不报错、不重复行） |
| customer_id 传数字 | `customer_id: 3599`（int） | 0（自动转字符串） |
| customer_id 非数字 | `"abc"` / `"1.5"` / `""` | 1002 |
| employee_code 为空 | `""` / `"   "` | 1002 |
| 缺字段 | 不传 customer_id | 1002 |
| 状态库不可用 | （后端故障） | 1505（HTTP 503） |

### 对接注意
- **幂等**：同一 `(employee_code, customer_id)` 跳多次只一行，前端可放心重试。
- 跳过在**下次缓存重算**时才沉底；本次会话由前端**乐观隐藏**该卡片即可。
- 状态库不可用（1505）只影响“跳过”，不影响看分数/详情。

## 四、接口三：行为时间轴

客户的订单/拜访/服务统一时间轴，纯 CRM、无大模型。

- **URL**：`POST /api/v1/leads/timeline`
- **依赖**：CRM 三接口（订单/拜访/服务）+ 缓存
- **典型耗时**：命中缓存 **毫秒级**；**冷路径需串行调用 3 个 CRM 接口、每个 5s 超时（最坏数十秒）**

### 请求体

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `customer_id` | string | 是 | 客户 id（必须为纯数字） |
| `employee_code` | string | 否 | **可不填**；时间轴是客户完整历史，与业务员无关 |

请求示例：
```json
{ "customer_id": "3599" }
```

### 返回体 `data`

| 字段 | 类型 | 说明 |
|---|---|---|
| `customer_id` | string | 客户 id |
| `timeline` | array | 最近 **≤5 条**事件，按时间倒序（见下） |
| `sources` | object | `{orders, visits, services}` 三源状态：`"ok"`/`"error"` |
| `degraded` | bool | 是否有源失败（`sources` 含 `"error"` 时为 `true`） |

`timeline[]` 每个事件：

| 字段 | 类型 | 说明 |
|---|---|---|
| `type` | string | `"order"` \| `"visit"` \| `"service"` |
| `time` | string | 事件时间（CRM 原始时间字符串） |
| `title` | string | 如 `"订单 SO-123"` / `"拜访"` / `"服务单 SVC-1"` |
| `detail` | string | 摘要（≤200 字）；订单形如 `"￥金额｜产品"` |
| `status` | string | 订单恒为 `"-"`（CRM 无该字段）；拜访为拜访方式；服务为服务状态 |

返回示例：
```json
{
  "code": 0, "message": "",
  "data": {
    "customer_id": "3599",
    "timeline": [
      { "type": "visit",  "time": "2026-05-26 09:57:19", "title": "拜访",       "detail": "客户对数控机床有意向", "status": "上门拜访" },
      { "type": "order",  "time": "2026-05-01 10:00:00", "title": "订单 SO-123", "detail": "￥120000｜立式加工中心", "status": "-" }
    ],
    "sources": { "orders": "ok", "visits": "ok", "services": "ok" },
    "degraded": false
  }
}
```

降级示例（某源失败，仍 `code:0`）：
```json
{ "code": 0, "message": "",
  "data": { "customer_id": "3599", "timeline": [ /* 可用部分 */ ],
            "sources": { "orders": "error", "visits": "ok", "services": "ok" }, "degraded": true } }
```

### 测试用例

| 用例 | 关键入参 | 预期 code | 预期 data 要点 |
|---|---|---|---|
| 正常 | 合法 customer | 0 | `timeline` 倒序、≤5 条、`degraded=false` |
| 不传 employee_code | 仅 customer | 0 | 正常返回（与业务员无关） |
| employee_code 纯空白 | `"   "` | 0 | 等价于不传 |
| 某 CRM 源失败 | （后端故障） | 0 | `degraded=true`、对应 `sources` 为 `error` |
| customer_id 非数字 | `"abc"` / `""` | 1002 | data=null |
| 缺 customer_id | — | 1002 | data=null |

### 对接注意
- 与业务员无关，`employee_code` 可不传；时间轴按客户缓存、全员共享。
- 冷路径较慢（串行 3 次 CRM），前端建议加载态 + 与分数列表并行/懒加载。
- 永远不报错；用 `degraded` / `sources` 判断数据是否完整。

## 五、接口四：AI 推荐理由

结合评分上下文 + 行为时间轴，用大模型生成一句推荐理由；失败回退确定性模板。

- **URL**：`POST /api/v1/leads/recommend-reason`
- **依赖**：DeepSeek 大模型 + 复用时间轴（CRM）+ 缓存
- **典型耗时**：命中缓存毫秒级；走大模型约数秒（超时 20s，失败即回退模板）

### 请求体

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `employee_code` | string | 是 | 业务员编号（非空，自动去空格） |
| `customer_id` | string | 是 | 客户 id（必须为纯数字） |
| `score_context` | object | 否 | 该客户在**接口一**里的分数/行业/距离/标签 + 业务员上下文，原样回传 |

`score_context` 建议字段（均可选，允许多带）：`total_score` / `score_detail` / `customer_name` / `amer_industry` / `actual_capi` / `distance_km` / `tag_name` / `star` / `employee_context`。

请求示例：
```json
{
  "employee_code": "000068",
  "customer_id": "3599",
  "score_context": {
    "total_score": 90,
    "score_detail": { "industry_match": 40, "scale": 10, "tags": 0, "distance": 40 },
    "customer_name": "某某精密机械有限公司",
    "amer_industry": "数控机床",
    "distance_km": 8,
    "tag_name": ["意向客户"],
    "employee_context": { "deal_customer_industries": ["数控机床"], "industry_top_customers": {} }
  }
}
```

### 返回体 `data`

| 字段 | 类型 | 说明 |
|---|---|---|
| `customer_id` | string | 客户 id |
| `recommend_reason` | string | 一句话推荐理由（非空，≤300 字） |
| `reason_source` | string | `"llm"`（大模型）或 `"template"`（确定性兜底模板） |
| `degraded` | bool | CRM 数据源是否有缺失 |

返回示例：
```json
{ "code": 0, "message": "",
  "data": { "customer_id": "3599",
            "recommend_reason": "该客户所属「数控机床」与您成交行业匹配、规模达标且距您约 8 公里，近期有订单与拜访记录，建议优先跟进。",
            "reason_source": "llm", "degraded": false } }
```

### 测试用例

| 用例 | 关键入参 | 预期 code | 预期 data 要点 |
|---|---|---|---|
| 正常（有上下文） | 合法 + score_context | 0 | `recommend_reason` 非空；`reason_source` 为 llm 或 template |
| 不传 score_context | 仅 employee/customer | 0 | 仍返回理由（信息更少） |
| 大模型不可用 | （key 未配/超时） | 0 | `reason_source="template"`，理由仍非空 |
| CRM 全失败 | （后端故障） | 0 | `degraded=true`，走模板 |
| 多带未知字段 | score_context 含额外字段 | 0 | 正常（后端忽略未知字段） |
| employee_code 空 | `"   "` | 1002 | data=null |
| customer_id 非数字 | `"abc"` | 1002 | data=null |
| score_context 非对象 | `"x"` / `123` | 1002 | data=null |

### 对接注意
- **强烈建议**：`score_context` **只回传接口一返回的稳定字段**，**切勿带时间戳、随机数等易变字段**。
  缓存按 `score_context` 的内容指纹归一；带易变字段会导致每次都缓存未命中、每次都真调大模型，**显著增加延迟与费用**。
- **大模型失败必回退模板**：DeepSeek 网络/超时/非 JSON/格式不符等任何失败，自动回退确定性模板（`reason_source="template"`），接口仍 `code:0`；仅极端未预期异常才返 `code:1500`（仍是 envelope，非裸错）。前端正常展示即可。
- 该接口较慢，建议与时间轴并行调用、渐进式渲染。

### D. 承接能力（压测）结论摘要

> 实测：打真实后端（业务库/状态库/CRM/DeepSeek），**缓存后端 = Redis**（与生产一致）、批量日志已启用。`python3 scripts/load_test.py`（2026-06-09）。各阶段 0 错误。

| 阶段 | 并发 | 请求 | QPS | p50 | p95 | p99 | 说明 |
|---|---|---|---|---|---|---|---|
| 热路径（缓存命中） | 20 | 400 | 317 | 60 | 76 | 82 | Redis 后端：每次读多一跳网络（进程内缓存可达 ~690 QPS/p95 30ms） |
| 击穿（全新键并发） | 8 | 8 | 0.9 | 7530 | 9035 | 9089 | **Redis 无跨进程 singleflight → 8 并发 = 8 次冷算**（进程内则恒 1 次） |
| 冷路径（distinct 键） | 4 | 5 | 0.7 | 6219 | 6926 | 7059 | 真打地理库+打分，**系统承载瓶颈** |
| timeline 冷 | 3 | 3 | 1.2 | 1851 | 2371 | 2417 | 串行 3 次 CRM |
| timeline 热 | 10 | 40 | 893 | 10 | 13 | 15 | 命中缓存 |
| recommend | 6 | 6 | — | 4 | — | — | CRM 当前故障 → 时间轴空 → 走模板（见下） |
| skip 并发幂等 | 10 | 10 | — | 1971 | 2016 | 2018 | 10 并发跳同一客户 → DB **仅 1 行**（幂等通过） |
| 日志关 | 15 | 500 | 335 | 43 | 56 | 62 | — |
| 日志开（批量） | 15 | 500 | 319 | 45 | 59 | 64 | 吞吐 -5%；**落库 500/500、丢弃 0**（有界队列+攒批，~7s 异步写完） |
| 混合负载 | 20 | 200 | 419 | 46 | 59 | 60 | 四接口约 6:2:1:1 |

（延迟单位 ms。）

**关键结论：**
- **缓存命中路径吞吐高、延迟稳**；Redis 后端每次读多一跳网络，热路径 ~317 QPS / p95 76ms（进程内缓存约 690 QPS / p95 30ms）。
- **防击穿（已加 Redis 跨进程 singleflight）**：早期实测 Redis 后端无跨进程锁，8 并发打同一全新键 → 8 次冷算（每次 ~7s 地理查询），会击穿到只读业务库。**现已用 Redis SETNX 锁修复**（`app/cache/store.py`）：同 key 冷算全集群只跑一次，其余等结果、超时自算兜底；复测 10 并发新键 → **冷算 1 次**。锁带 TTL（`CACHE_LOCK_TTL_SECONDS`），Redis 异常自动退化为直接算、绝不阻塞。
- **冷路径是承载瓶颈**：未命中缓存的地理查询 p95 ~6.5s、单并发 ~0.7 QPS，承接能力高度依赖缓存命中。
- **接口日志已做到无丢失**：改为"有界队列 + 后台 flusher 攒批多行 INSERT"后，500 条全部落库、0 丢弃、~7s 异步写完，吞吐仅 -5%（旧逐条方案 200 上限下丢 196/200、-10%）。可调 `API_LOG_MAX_PENDING`(10000)/`API_LOG_BATCH_SIZE`(200)/`API_LOG_WRITERS`(1)。
- **幂等/降级稳健**：skip 并发幂等仅 1 行；全程 0 错误、无 5xx。
- **⚠️ CRM 当前故障**：实测 CRM 三接口均返回 PHP 致命错误 `Cannot redeclare ...getSalesmanCustomerServiceApply()`（HTTP 200 + `code:0`），导致**所有客户**时间轴为空、推荐理由全部走模板。这是 **CRM 服务端 bug，需 CRM 侧修**。客户端侧已加固：这种"`code:0` 但无 `data` 段且 message 是报错"的伪成功现在会被识别为该源失败 → `sources` 标 `error`、`degraded=true`（不再静默当"无记录"）。DeepSeek 本身在线可用（直连实测 ~1.5s 返回合法 JSON）。CRM 修复后复测大模型路径。

> 复现：到业务库的隧道网卡 MTU=1400（脚本自动探测打印命令）+ Redis 已起（`docker compose up -d redis`），再 `python3 scripts/load_test.py`；`--customer id1,id2` 指定有 CRM 活动的客户测大模型、`--no-llm` 跳过大模型阶段。
