`grep LEAD_PREFIX` 看着干净（都从一个常量派生），但**前缀驱动的行为**散在这几处，泛化要逐一覆盖：

## 二、先选定位：两条岔路，决定泛化的轻重

第二个面要的契约和 lead **一样**还是**不一样**，决定工作量差一个数量级：

- **路 A（轻）— 所有面共用同一套契约**（都要 envelope + 同样的 body limit）。泛化很轻：把 `_is_lead_path` 改成认一个**前缀列表**的 `_is_enveloped_path`，第 2/3 项各加一行。半天活。
- **路 B（重）— 每面契约可不同**（新面可能要标准 HTTP 状态码 / problem+json / 不同鉴权 / 不同 body 上限 / 不记日志）。需要把"面"抽象成对象 + 注册表。

我的判断：**按 B 设计**。因为"第二个 HTTP 面"之所以成立、而不是"lead 的更多 endpoint"，通常正是因为它**不一样**——对外开放 API、CRM 回调 webhook、内部 admin 面。如果它跟 lead 完全同契约，那它就该是 lead 里多挂一个 router，压根不是"第二个面"。所以"扩大共享谓词"多半撑不住，真正要的是"每面自带契约"。

## 三、目标形态：把"HTTP 面"抽象成一等公民 + 注册表

核心是把"路径前缀"这个**隐式 join key**，显式化成一个 `HttpFace` 对象，集齐它的全部契约：

```python
# app/shared/http_face.py  —— 新平台件，只依赖 shared，不碰业务模块
from dataclasses import dataclass, field
from collections.abc import Callable
from fastapi import APIRouter

@dataclass(frozen=True)
class HttpFace:
    name: str                       # "lead" —— 给日志/可观测打标
    prefix: str                     # "/api/v1/leads"
    routers: tuple[APIRouter, ...]  # 这个面的所有 router
    auth: Callable | None = None    # 鉴权依赖；None=放行
    body_limit_bytes: int = 0       # 0=不限
    body_limit_code: int = 1002     # 413 envelope 的业务码
    enveloped: bool = True          # True=4 个 handler 转 envelope；False=走默认契约
    logged_paths: frozenset[str] = field(default_factory=frozenset)

    def owns(self, path: str) -> bool:
        # ★ 边界感知，避免 /leads 误圈 /leads-admin（见坑位一）
        return path == self.prefix or path.startswith(self.prefix + "/")
```

每个模块导出自己的 face（lead 的放 `app/lead/__init__.py` 或新建 `app/lead/http_face.py`）：

```python
LEAD_FACE = HttpFace(
    name="lead",
    prefix=LEAD_PREFIX,
    routers=(leads.router, customer_detail.router),
    auth=require_api_key,
    body_limit_bytes=lead_settings.max_request_body_bytes,
    body_limit_code=1002,
    enveloped=True,
    logged_paths=frozenset(p for _, p in _LOGGED_PATHS),
)
```

`main.py` 从"硬编码 lead"变成"遍历注册表"：

```python
FACES: tuple[HttpFace, ...] = (LEAD_FACE,)   # 第二个面来了在这里加一项

def _face_for(path: str) -> HttpFace | None:
    # 最长前缀优先（见坑位二）
    return max((f for f in FACES if f.owns(path)), key=lambda f: len(f.prefix), default=None)

# 装配：一个循环替代两行 include_router
for face in FACES:
    deps = [Depends(face.auth)] if face.auth else []
    for r in face.routers:
        app.include_router(r, dependencies=deps)

# 异常门控：_is_lead_path → 按面查表
@app.exception_handler(RequestValidationError)
async def _validation_handler(request, exc):
    face = _face_for(request.scope["path"])
    if face is None or not face.enveloped:
        return await request_validation_exception_handler(request, exc)
    ...  # 走 envelope

# body-limit：中间件内部按面查 max_bytes/code，而不是二元 startswith
```

注意：`/healthz`、`/docs`、`/openapi.json` 不属于任何 face，`_face_for` 返回 `None` → 自动走默认契约，**正是现在的行为**。这是这套抽象正确的标志——重构后 lead 行为逐字节不变。

## 六、另一条路：物理隔离（`app.mount` 子应用）

如果第二个面契约差异**极大**（不同中间件栈、不同 docs、不要 envelope、甚至不同鉴权框架），比起"注册表让两个面共存于一个 app"，可以用 `app.mount("/api/ext", sub_app)` 挂一个独立的 FastAPI/Starlette 子应用——中间件、异常 handler、docs 全部物理隔离，互不污染。

- **注册表**（第三节）= 轻武器：两个面**共享**中间件栈与异常 handler，只是按面查表分流。适合"契约相似、只是前缀不同"。
- **mount 子应用** = 重武器：两个面**各自一套** ASGI 栈。代价是 lifespan / 依赖预热 / 健康检查要各自管，`main.py` 的统一异常契约不再覆盖子应用。适合"对外开放 API"这种和内部面世界观完全不同的场景。

二选一的判据：**新面和 lead 共享多少横切件**。共享多 → 注册表；几乎不共享 → mount。

## 一、泛化到底在做什么？

别被"前缀"这个词带偏——重点不是字符串前缀，是**横切行为的判定方式**。

现在 `main.py` 里，body-limit 和 4 个异常 handler 都靠这种判断：

```
if path 以 /api/v1/leads 开头:   # _is_lead_path / _LeadScopedBodyLimit
    走 lead 的契约（envelope / body 上限）
else:
    走默认（裸 500 / 422）
```

这是**把"lead 这一个面"硬编码进判定**。所谓泛化，就是把它换成：

```
face = 查表(path)        # 这条路径属于哪个面？
if face and face.要envelope:
    走这个面声明的契约
else:
    走默认
```

一句话：**把"if 是不是 lead"换成"查表得到是哪个面，再用那个面自己声明的契约"。** 硬编码的 if 变成数据驱动的注册表。

## 二、期待的效果是什么？

注意一个反直觉的点：**泛化对现在的功能零改善**。lead 的行为一字节不变。它的全部价值在**未来加面那一刻**：

所以泛化是一笔**纯未来投资**：收益对象是"第二个面"，它能省的是"加面时的改动量"和"防漏"。
