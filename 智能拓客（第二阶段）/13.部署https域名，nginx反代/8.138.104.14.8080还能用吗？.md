## resolver 是什么

它是 nginx 的一个指令，作用是**告诉 nginx 用哪个 DNS 服务器去解析域名**，并且（配合变量写法）**让 nginx 在运行时反复重新解析**，而不是只在启动时解析一次就钉死。

要理解它先得知道 **nginx 解析上游域名的时机有两种**，取决于你怎么写 `proxy_pass`：

**① 字面量写法（你现在的）**
```nginx
proxy_pass http://api:8000;   # api 是写死的字面量
```
nginx 只在**启动/reload 那一刻**解析一次 `api`，把得到的 IP **缓存到 worker 进程的整个生命周期**，之后**永不再查**。

**② 变量写法 + resolver**
```nginx
proxy_pass http://$upstream:8000;   # 上游是变量
```
nginx 改成**每次请求时**去问 `resolver` 指定的 DNS，按 TTL 动态拿最新 IP。没有 `resolver` 时 nginx 根本没法做运行时解析（会直接报 "no resolver defined"）。

## 为什么会 502 / 为什么必须 restart nginx

Docker 里每个容器在网络上有个 IP（比如 `172.18.0.5`），服务名 `api` → 这个 IP 是靠 **Docker 内置 DNS（`127.0.0.11`）** 映射的。串起来就是：

1. nginx 启动 → 解析 `api` → `172.18.0.5` → **钉死缓存**
2. `docker compose up -d` 把 api 容器**重建** → 新容器**通常拿到一个新 IP**，比如 `172.18.0.8`
3. nginx 还在往老地址 `172.18.0.5` 打 → 那儿没人了 → **502 Bad Gateway**
4. `restart nginx` → 重新解析 → 拿到 `172.18.0.8` → 好了

所以那条命令把 `up -d` 和 `restart nginx` 绑一起，就是因为**重建 api 必然换 IP，而你的 nginx 不会自动跟上，只能靠重启逼它重新解析**。

## 一劳永逸的改法（以后不用再记得 restart nginx）

给 `nginx/default.conf` 加 resolver + 把上游改成变量：
```nginx
server {
    listen 80;
    server_name _;
    resolver 127.0.0.11 valid=10s ipv6=off;   # Docker 内置 DNS
    location / {
        set $upstream api;                      # 关键：变量化
        proxy_pass http://$upstream:8000;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Request-ID      $request_id;
    }
}
```
这样 api 换 IP 后 nginx 最多 10s 自动跟上，`docker compose up -d` 重建 api 也不会 502 了。

- 如果 `up -d` **重建了 api**（拉了新代码/镜像/`.env` 变了）→ api **掉出了 `docker_default`** → **`aisales.amer.cn` 现在已经连不上了**，得重新 `docker network connect`。
