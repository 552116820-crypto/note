## 一、"小程序要登录才能进" ≠ 你的接口安全了

这两件事是**完全不同的两层**：

- **企业微信登录** = 客户端门禁，控制"谁能打开这个小程序界面"。
- **你的接口鉴权** = 服务端门禁，控制"谁能调用 `/api/v1/leads/*`"。

你的接口跑在公网（nginx 后面），**任何人拿到 URL 就能直接 curl**，根本不经过企业微信那道门。小程序前端的请求可以被抓包、被读包体看到 URL 和参数，然后任意重放。

而且你现在的情况比"裸奔"更糟一点——看你的 schema，**`employee_code` 是请求体里的普通字段**（`LeadQueryRequest` / `LeadSkipRequest` / `RecommendReasonRequest` 都有）。这意味着：

```bash
# 任何人，不需要任何凭证，就能拿到任意业务员的全部数据：
curl -X POST https://你的域名/api/v1/leads/score-and-info \
  -d '{"employee_code":"随便填别人的编号","lng":121,"lat":31}'
# 客户名单、定位、评分、AI 推荐理由全出来；
# 还能 POST /skip 往别人的跳过列表里塞脏数据
```

这是典型的**越权漏洞（IDOR）**：身份标识（employee_code）既是身份、又能被调用方自己伪造。**这才是当前最该堵的洞**——不只是"没鉴权"，而是"身份可伪造"。

## 二、标准做法：企业微信小程序免登 + 后端签发会话令牌

这是教科书方案，分前后端两半职责：

```
┌─ 前端（小程序）────────────────┐      ┌─ 后端（你要建的）──────────────────┐
│ 1. wx.qy.login() → 临时 code   │      │                                    │
│ 2. POST /auth/session {code} ──┼─────▶│ 3. gettoken(corpid,secret)         │
│                                │      │    → access_token（缓存到 Redis）   │
│                                │      │ 4. jscode2session(token, code)     │
│                                │      │    → userid（企业成员，可信身份）    │
│                                │      │ 5. userid → employee_code（映射）   │
│ 7. 存 token，之后每个请求带：   │◀─────┼─ 6. 签发 JWT(employee_code, exp)   │
│    Authorization: Bearer <jwt> ┼─────▶│ 8. 依赖校验 JWT → 注入 employee_code│
└────────────────────────────────┘      └────────────────────────────────────┘
```

关键点：

1. **可信身份的源头是 `jscode2session` 返回的 `userid`**——这一步后端持有 `corpsecret`，前端伪造不了。`code` 是一次性、5 分钟过期。
2. **签发你自己的 JWT**（HMAC 签名），后续请求只验你自己的令牌，不用每次都打企微 API。
3. **`employee_code` 从此只从令牌里取，不再信任请求体**——这一步才真正堵住 IDOR。否则就算加了登录，业务员 A 仍能在请求体里填 B 的编号。

涉及企微接口（我确认过的）：
- `GET /cgi-bin/gettoken?corpid=&corpsecret=` → `access_token`（7200s，**必须缓存**，有频率限制）
- `GET /cgi-bin/miniprogram/jscode2session?access_token=&js_code=&grant_type=authorization_code` → `userid`

## 1. 限流

### 算法：固定窗口计数（够用且最省）

你就 ~50 业务员灰度，不需要滑动窗口/令牌桶。Redis `INCR + PEXPIRE` 一个 key 一次 RTT，用 Lua 原子化（避免"INCR 成功但进程在 EXPIRE 前挂掉 → key 永不过期"）：

```python
# app/shared/ratelimit.py
"""平台限流 + nonce 防重放：Redis 固定窗口（INCR+PEXPIRE 原子 Lua）。
故障即放行（fail-open），与 cache.py 一致。键写 REDIS_URL 指向库，前缀 rl:/nonce:。"""
from __future__ import annotations
import logging
from app.shared.errors import DomainError

logger = logging.getLogger(__name__)

_HIT_LUA = (
    "local n = redis.call('INCR', KEYS[1]) "
    "if n == 1 then redis.call('PEXPIRE', KEYS[1], ARGV[1]) end "
    "return n"
)

class RateLimited(DomainError):
    code = 1429
    message = "请求过于频繁，请稍后再试"
    http_status = 429

class GuardRedis:
    """懒连接、容错。连不上则 hit()→放行、seen_nonce()→视作首次。"""
    def __init__(self, url: str) -> None:
        self._url, self._client = url, None

    @property
    def client(self):
        if self._client is None and self._url:
            import redis
            self._client = redis.Redis.from_url(self._url, socket_connect_timeout=1, socket_timeout=1)
        return self._client

    def hit(self, key: str, limit: int, window_s: int) -> bool:
        c = self.client
        if c is None:
            return True
        try:
            return int(c.eval(_HIT_LUA, 1, key, window_s * 1000)) <= limit
        except Exception as exc:  # noqa: BLE001
            logger.warning("ratelimit redis error (%s); fail-open", exc)
            return True

    def seen_nonce(self, nonce: str, ttl_s: int) -> bool:
        c = self.client
        if c is None:
            return False
        try:
            return not c.set(f"nonce:{nonce}", 1, nx=True, ex=ttl_s)  # NX 返回 None=已存在
        except Exception as exc:  # noqa: BLE001
            logger.warning("nonce redis error (%s); fail-open", exc)
            return False
```

### 维度：三把 key，贵接口单独收紧

每请求最多查 2~3 个计数器：

| key | 防什么 | 备注 |
|-----|--------|------|
| `rl:g:{path}` | 全局总量（保后端 + DeepSeek 账单） | 最该卡 `/recommend-reason` |
| `rl:ip:{path}:{client_ip}` | 单 IP 高频 | **client_ip 必须从 XFF 取，见下** |
| `rl:emp:{path}:{code}` | 单账号高频（次要，code 可伪造） | 可选，放 handler 里 |

`/recommend-reason`（打大模型）和 `/score-and-info`（冷算 7–10s + 吃内存）给最严的 limit。

### client_ip：你这套拓扑的坑（重点）

`request.client.host` = nginx 容器 IP，**没用**。而且你 nginx 在 edge 443 代理后面，所以到达 app 的 `X-Forwarded-For` 是 `用户IP, edge代理IP`（nginx 把 edge 的 IP append 上来）。真实用户 = **XFF 右起第 2 个**：

```python
def client_ip(request, trusted_hops: int = 2) -> str:
    """trusted_hops = 你与公网客户端之间、会往 XFF 追加 IP 的可信代理跳数。
    本部署 = edge 443 代理 + 本 nginx = 2 → 取 XFF[-2]。
    再往左的值客户端可伪造、不可信；hops 必须等于真实跳数，宁可小不可大。"""
    parts = [p.strip() for p in request.headers.get("x-forwarded-for", "").split(",") if p.strip()]
    if len(parts) >= trusted_hops:
        return parts[-trusted_hops]
    return request.client.host if request.client else "unknown"
```

- 为什么 `[-trusted_hops]` 抗伪造：客户端就算自己塞 `X-Forwarded-For`，也只会出现在更左边，被我们忽略；只要 `trusted_hops` 数准了，`[-2]` 永远是 edge 实际看到的对端=真用户。**前提**：确认 edge 代理确实转发了 XFF（没转发的话真实 IP 在它那一跳就丢了，得先修 edge）。
- **别在这台 nginx 上用 `limit_req_zone $binary_remote_addr`** —— 这里 remote_addr 是 edge 单个 IP，会把全体用户当一个 IP 限掉。要做 nginx 层 IP 限流，得放在 **edge 代理**（它才看得到真实客户端），或就用上面 app 层的 XFF 方案。这台 nginx 顶多加个 `limit_conn` 做粗粒度连接数兜底。

## 2. 签名 + 防重放

### canonical 串（前后端必须逐字节一致）

```
base = METHOD \n PATH \n X-Timestamp \n X-Nonce \n sha256_hex(raw_body)
X-Sign = hmac_sha256_hex(SECRET, base)
```

用 `sha256(body)` 而不是原始 body 进 HMAC，空 body（GET /industries）和大 body 都统一处理。头三件套：`X-Timestamp`（unix 秒）、`X-Nonce`（随机 ≥16 位）、`X-Sign`。

### 纯函数校验（无 Redis、可确定性单测）

```python
# app/shared/signature.py
import hashlib, hmac, time
from app.shared.errors import DomainError

class BadSignature(DomainError):
    code = 1401
    message = "签名校验失败"
    http_status = 401

def expected_sign(secret, method, path, ts, nonce, body: bytes) -> str:
    base = "\n".join([method.upper(), path, ts, nonce, hashlib.sha256(body).hexdigest()])
    return hmac.new(secret.encode(), base.encode(), hashlib.sha256).hexdigest()

def verify(secret, method, path, body, *, ts, nonce, sign, now=None, skew_s=300):
    if not (ts and nonce and sign):
        raise BadSignature("缺少签名头")
    try:
        ts_i = int(ts)
    except ValueError as e:
        raise BadSignature("时间戳非法") from e
    now = int(now if now is not None else time.time())
    if abs(now - ts_i) > skew_s:               # 时钟偏移容忍窗（5min）
        raise BadSignature("请求已过期")
    if not hmac.compare_digest(expected_sign(secret, method, path, ts, nonce, body), sign):
        raise BadSignature()                    # 常量时间比较，防时序侧信道
```

防重放（nonce 一次性）单独放 guard 里——它要碰 Redis，从纯函数里拆出去，保证 `verify()` 可注入 `now` 做确定性测试。

### 为什么能当 FastAPI 依赖（关键技巧）

在依赖里 `raw = await request.body()` 会把 body **缓存进 Request**，下游 handler 的 `req: LeadQueryRequest` 仍能正常解析同一份 body（Starlette `Request.body()` 缓存 `_body`，二次读走缓存）。所以不用搞 ASGI body 重放那套，直接做依赖最干净。

### secret 管理（诚实标注）

`API_SIGN_SECRET` 新增 env，带 dev 默认值、生产必填（铁律 #3 + 仿你 main.py 的 `_validate_startup_config` 加一句"生产用默认值就报错"）。**密钥随小程序包下发、可被解包，不是真秘密**——它把门槛从"看一眼 URL"抬到"逆向+实现签名"，配合 timestamp+nonce 让泄露的密钥也没法重放。要更强就上 `X-App-Id` 多密钥、服务端按 id 取，便于轮换/吊销。

## 4. 给前端的契约 + 最容易炸的坑

```js
import CryptoJS from 'crypto-js'      // 打包进小程序
const SECRET = 'xxx'                  // == 后端 API_SIGN_SECRET
function signedRequest(method, path, bodyObj) {
  const ts = String(Math.floor(Date.now() / 1000))
  const nonce = (Math.random().toString(36) + Date.now().toString(36)).slice(2, 20)
  const bodyStr = bodyObj ? JSON.stringify(bodyObj) : ''        // ★ 只 stringify 一次
  const base = [method.toUpperCase(), path, ts, nonce,
                CryptoJS.SHA256(bodyStr).toString()].join('\n')
  const sign = CryptoJS.HmacSHA256(base, SECRET).toString()
  return wx.request({
    url: 'https://你的域名' + path, method,
    data: bodyStr,                                              // ★ 发这个字符串，别发对象
    header: { 'Content-Type': 'application/json',
              'X-Timestamp': ts, 'X-Nonce': nonce, 'X-Sign': sign },
  })
}
```

**★ 头号坑**：HMAC 签的是 body 字节，后端哈希的是**收到的原始字节**。如果前端 `data` 传对象、让 `wx.request` 自己序列化，它的 JSON（键序、空格、unicode 转义）未必等于你 `JSON.stringify` 出来签名的那串 → 签名必挂。**解法**：前端 `JSON.stringify` 一次，签它、也发它（`data: bodyStr`）。后端哈希原始 body、不重新序列化，两边自然逐字节相等，连 unicode 转义策略都不用统一。次坑：小程序设备时钟不准 → 5min 窗口能兜大多数，频繁 401 再放宽到 600s 或让前端从服务端校一次时间。

## 三种东西的关系，一句话讲清

- **静态 API-Key** = 把"秘密"明文塞进每个请求 → **抓一次就废**。
- **签名（HMAC+时间戳+nonce）** = **同一个前端密钥，但密钥本身不上网线**（只送算出来的 HMAC），且抓到的包换个时间/nonce 就失效。它就是静态 key 的**打补丁版**——前端只多一个 `sign()` 函数，后端挂载/降级/错误码跟静态 key 一模一样。
- **限流** = 跟你用哪种 header **无关**，是唯一能在 key/secret 泄露之后还兜住"刷爆 DeepSeek / 刷库"的东西。

3. 不管选哪个，**天花板都是"前端持密钥"**：决心大的人解包总能拿到。要再往上只有服务端自己定身份（企微免登），你已决定缓——那就别纠结前端这层做到 100%，**做到 1 或 2 再加限流，挡掉 95% 的随手滥用，目标就达成了**。

## 1. `TRUSTED_PROXY_HOPS = 2` 是什么意思

要限流"单 IP 每分钟 N 次"，得先知道**请求的真实用户 IP**。但你的请求不是用户直连 api，中间隔了两层代理，所以 api 拿到的"对端 IP"是代理的、不是用户的。

用具体 IP 走一遍（用户手机公网 `1.2.3.4`、edge 网关 `10.0.0.9`、本 nginx `172.20.0.3`）：

```
用户手机 (1.2.3.4)
   │ HTTPS
   ▼
edge 网关 (:443)        它看到的对端 = 1.2.3.4（真用户）
   │ 转发，写 X-Forwarded-For: 1.2.3.4
   ▼
本 nginx (:80)          它看到的对端 = edge (10.0.0.9)
   │ $proxy_add_x_forwarded_for = 收到的XFF + ",对端"
   │ 转发，X-Forwarded-For: 1.2.3.4, 10.0.0.9
   ▼
api (uvicorn)
   request.client.host = 172.20.0.3 (nginx) ← 没用，永远是 nginx
   X-Forwarded-For     = "1.2.3.4, 10.0.0.9" → 拆成 [1.2.3.4, 10.0.0.9]
```

每经过一层代理，就往 `X-Forwarded-For`（简称 XFF）**右边追加一个"它看到的对端 IP"**。所以到 api 时这串里：

- 右起第 1 个 `10.0.0.9` = nginx 填的（nginx 的对端 = edge）
- 右起第 2 个 `1.2.3.4` = edge 填的（edge 的对端 = **真用户**）← 我们要的

中间隔了 **2 层可信代理（edge + nginx）→ 真用户就在右起第 2 个**。`TRUSTED_PROXY_HOPS=2` 就是告诉代码"从右边数 2 个"，即取 `XFF[-2]`。

**为什么从右数、而且必须数准：**

- **从右数**才防伪造：用户自己可以乱塞 `X-Forwarded-For: 假IP`，但那个假值只会出现在更左边，被我们忽略。只要 hops 数对，`[-2]` 永远是 edge 实际看到的真用户。
- 数**大了**（写 3）→ 读到用户能伪造的那一段 → 他随便变"IP"就能绕过单 IP 限流。数**小了**（写 1）→ 读到 edge 的 IP，所有用户共用一个 IP → 全体被一起限。所以必须**正好等于真实代理层数**，你这儿就是 2。

**⚠️ 一个要先确认的前提**：上面成立的前提是 **edge 真的把用户 IP 写进了 XFF**。如果 edge 没配转发，那 api 只会看到 `[10.0.0.9]`（只有 nginx 填的 edge IP），真用户 IP 在 edge 那一跳就丢了——得先让运维在 edge 上开 XFF 转发。
顺带：**别想用 `X-Real-IP` 省事**——你 nginx 里 `proxy_set_header X-Real-IP $remote_addr` 会把它覆盖成 edge 的 IP，照样没用；XFF 是 append 不是覆盖，才靠谱。
验证方法：拿手机连 4G（知道自己公网 IP）发一个请求，在 api 日志里看 `X-Forwarded-For` 长啥样、数几个、最左是不是你手机 IP。

- `30` = 次数

## 1. "老客户端升完"是什么意思

"客户端" = 装在每个业务员手机上、**正在运行的那份小程序代码**。小程序是会发版的：

- 你把新 key 打进**新版本小程序**发布出去，但用户不是同一秒全拿到新版——企微/微信小程序是用户**下次冷启动**时异步拉新版、再下次生效（或你用 `wx.getUpdateManager` 弹"有新版本，请重启"）。通常很快，但**有个尾巴、不是瞬时**。
- 轮换窗口期：有人已更到新版（发**新 key**），有人还没重开、跑旧版（发**旧 key**）。这段时间服务端 `API_KEYS=新,旧` 两把都收，谁都不断。
- **"老客户端升完"** = 还在跑旧版（带旧 key）的用户都更新到新版了、旧 key 没人用了 → 这时才把旧 key 从 `API_KEYS` 删掉，只留新的。

一句话：客户端=用户设备上那份小程序；升完=全部更新到带新 key 的新版本。

## 2. key 什么时候要换

**平时不用换。** 轮换是"保险"不是定期作业。真正的触发点：

1. **泄露/疑似泄露**（最主要）：key 出现在不该出现的地方——贴进公开仓库/聊天/截图、被日志打出来、开发机或手机丢了/被入侵。
2. **发现被刷**（等你上了限流后）：日志里同 key 来自一堆陌生 IP、量暴增、非业务时段高频。
3. **掌握 key 的人离职**：能碰 prod `.env` 的开发走了，稳妥起见换。
4. **（可选）一年一次卫生轮换**：对你这种"挡裸调"的低敏 key 非必须，看团队规范。

关键认知：key 随小程序包下发、**注定能被解包**，所以你**防不住靠"勤换 key"**——那是限流的活。换 key 主要是"已经泄了/在被刷"时的**止血**动作。我把轮换做进去，是为了那天来时你能**无缝换、不停服**，不是让你定期换着玩。**别过度轮换。**

## 3. `secrets.token_urlsafe(32)` 的强度 vs 备份项目

备份项目 `deploy.sh` 第 25 行用的是：`openssl rand -hex 20`。对比：

| | 备份 `openssl rand -hex 20` | 我建议 `secrets.token_urlsafe(32)` |
|---|---|---|
| 随机源 | OpenSSL CSPRNG ✅ | Python `secrets`／`os.urandom` CSPRNG ✅ |
| 熵 | **160 bit**（20 字节） | **256 bit**（32 字节） |
| 编码 | hex，40 字符 | base64url，~43 字符 |
| 强度结论 | 强、远超暴力破解 | 更强（裕量更大） |

**结论：两个都是密码学强随机，实际安全性等价、都完全够用。** 最关键的那一点——**用的是 CSPRNG（密码学安全随机源）而不是弱随机**——两者都满足。熵 160 vs 256 的差别在现实里没意义：行业"永远安全"的门槛是 128 bit，两者都远远越过了，160 已经不可能被猜/爆破，256 只是裕量更大而已。编码（hex vs base64url）纯外观、不影响安全。

什么样才算**弱** key（两者都避开了的坑）：
- `date | md5sum`、`$RANDOM`、`python -c "import random"` —— 可预测，**不是 CSPRNG**。
- 人手敲的 `amer2026` 之类 —— 熵极低，能直接猜。
- `uuidgen` —— 122 bit，能用但偏少。

⚠️ 但记住那个天花板：**再强的生成也只防"猜/爆破"，不防"解包提取"**。你这把 key 在小程序包里，威胁是被扒出来、不是被猜出来——所以强生成是必要的（堵住"瞎猜也能蒙对"），但堵不住决心大的人解包，那部分交给限流。

实操上：你随手用哪个都行。`secrets.token_urlsafe(32)` 顺手（项目本就有 python3、不依赖 openssl）；想跟备份一致就 `openssl rand -hex 32`（把 20 提到 32）。两者等价。

## ❌ 一步换的问题

`API_KEYS=旧` 直接改成 `API_KEYS=新` → 所有**还没更新到新版小程序**的用户立刻 401，直到他们重开小程序拿到新版。这正是要避免的。

## ✅ 无缝轮换 = 两步（加 → 等 → 删）

```
第 0 步  生成新 key
  python3 -c "import secrets; print(secrets.token_urlsafe(32))"

第 1 步  服务端先并存（新旧都认）
  .env:  API_KEYS=<旧key>,<新key>
  docker compose up -d api     # 让新 env 生效（注意：restart 不一定重读 .env，要 up -d）

第 2 步  前端发新版小程序（打进新 key）、发布，等用户陆续更新
  —— 此刻：新版用户发新 key ✅，老版用户发旧 key ✅，两边都通、零 401

第 3 步  确认老客户端都升完（旧 key 没流量了），服务端删掉旧 key
  .env:  API_KEYS=<新key>
  docker compose up -d api
```

**顺序铁律**：先让服务端认新 key（第 1 步）→ 再让前端开始发新 key（第 2 步）→ 旧 key 留到老客户端都没了才删（第 3 步）。任何一步顺序反了，就有人会 401。

**怎么判断"老客户端升完"（第 3 步时机）**：等你上了限流 / 打开 `ENABLE_API_LOGGING` 后，盯旧 key 的请求量——降到 0 且持续几天不回升，就能删。保守点多留一两周无妨。

**补充**：只动 `api` 这一个服务就行（nginx/redis/celery 不读 `API_KEYS`）。如果你用户量很小、又能接受"没更新的人短暂 401 到重开为止"，一步替换也能用——但既然功能就是为无缝做的，两步更稳。

一句话：**新 key 用那条命令生成没错；"换上"是「先加并存→发前端→等升完→再删旧」四个动作，不是一步覆盖。**

那现实里怎么知道，分两种：

## A. 不改代码，用间接信号（通常够了）

1. **小程序后台的版本分布（最靠谱）**：企微/微信小程序管理后台能看各版本的活跃用户/访问占比。**旧版本占比 → 0，就等价于旧 key 没人用了**。这是不动后端就能拿到的最直接信号——因为 key 是跟着小程序版本走的。
2. **时间兜底**：小程序自动更新（下次冷启动就升级），绝大多数用户几天内更新完。保守**等 2–4 周**，长尾用户也基本重开过了。
3. **你范围可控**：~50 个业务员、不是百万匿名用户。群里喊一句"重启下小程序"，就能基本确认全员更新。

## B. 想要真·按 key 观测，得给 key 加标签（一点代码）

把 `API_KEYS` 从"纯 key"改成"**标签:key**"：

```
API_KEYS=v1:旧key,v2:新key
```

- auth 命中时**记标签**（v1/v2）到 Redis 计数器或日志行，**绝不记原始 key**（标签不是秘密）。
- 这样"旧 key 没流量" = **v1 的计数不再增长**，可观测、可判定。
- **前端零感知**：它发的还是冒号后面那串纯 key（`X-API-Key: 旧key`），标签纯服务端记账用（token_urlsafe 是 base64url、不含冒号，按第一个冒号切就行）。
- 这个可以在做限流时**顺路加**（limiter 本就要碰 Redis）。

## 最关键的一点：删早了能自愈

就算你判断早了、旧 key 删早了，**还在用旧版的零星用户会 401 → 他们重开小程序 → 自动更新到新版 → 自己就好了**。代价小、自我修复。所以你**不需要 100% 确定**——"版本分布接近 0 + 等两周"就足够，删早了也不会酿成事故。

一句话：**现在靠"小程序版本分布 + 等两三周 + 自愈兜底"来判断；想要精确到 key 的可观测，就给 key 加个标签计数，我做限流时一起带上。**
