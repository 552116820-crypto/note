## 生成机制

关键就在 `deploy.sh:25-26` 这两行：

```bash
API_KEY=$(openssl rand -hex 20)
sed -i "s/^# API_KEY=.*/API_KEY=${API_KEY}/" .env
```

- `openssl rand -hex 20`：生成 **20 字节随机数**，再用十六进制编码 → **40 个 hex 字符**的字符串（你 `.env` 里那串 `b94590b8ff87fc722d6778492475cc32bc2895d1` 正好 40 位，就是这么来的）。
- `sed`：把 `.env.example` 拷过来的 `.env` 里那行注释 `# API_KEY=...` 替换成真正的 `API_KEY=<生成值>`，同时把 `APP_ENV` 改成 `production`。

## 背后的原理

**1. 熵来源——CSPRNG，不是普通随机**
`openssl rand` 用的是 OpenSSL 的**密码学安全伪随机数生成器**，底层从操作系统熵池（Linux 上是 `/dev/urandom`）取种子。它和 `$RANDOM`、时间戳、UUID 这类**可预测**的方式有本质区别——攻击者无法根据已知信息推断出 key。

**2. 长度——160 位熵**
20 字节 = 160 bit。这个强度远超暴力穷举的可行边界（对比一下：AES-128 才 128 bit）。hex 编码只是为了让它变成纯 ASCII、能安全塞进 `.env` 和 HTTP header，不影响熵值。

**3. 认证模型——共享密钥 / Bearer Token，不是签名**
看 `app/shared/auth.py`，逻辑很简单：

```python
expected = os.getenv("API_KEY", "")
if not expected:            # 没配 → 认证关闭（开发模式）
    return None
if not header_value or header_value != expected:
    raise HTTPException(401, "Invalid or missing API key")
```

- 同一个随机串，**服务端存在 `.env`**（作为环境变量读取），**客户端每次请求放在 `X-API-Key` 头**里发过来，服务端做字符串比对。
- 这是"对称共享秘密"模型——不是 HMAC 签名、也不是 JWT，没有过期、没有载荷，就是一把对得上就放行的钥匙。
- `API_KEY` 为空时认证直接**关闭**，方便本地开发 / 跑测试。
