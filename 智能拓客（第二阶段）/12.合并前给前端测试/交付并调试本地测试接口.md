## 根因

`/api/v1/leads/timeline` 要调 CRM 的三个 HTTP 接口（orders / visits / services，`https://crm.amer.cn/...`）。容器里**解析 `crm.amer.cn` 这个域名时好时坏**：

- 实测 `/timeline` 连打 5 次：第 1 次三源全 `error`、`degraded:true`；第 2–5 次全 `ok`。
- 容器内 `socket.getaddrinfo("crm.amer.cn")` 一会儿 20/20 全过，一会儿直接抛 `[Errno -5]` / `[Errno -3]`。
- 解析器是 Docker 内嵌 DNS `127.0.0.11` → 转发 `192.168.65.7`（**Docker Desktop**），这条转发链在冷查询/并发时会抽风。

关键点：timeline 是**3 个 CRM 调用并发**触发，所以一旦撞上 DNS 抽风的瞬间，三个 `getaddrinfo` 一起失败 → 三源同时 `error` → `degraded:true`、`timeline:[]`。而 `code:0` 是**故意的**——CRM 取数失败按设计降级，不把整个请求判失败（见 `timeline_service` 的各源 try/except）。

补充几点：
- DNS 正常时，直连 `crm.amer.cn`（47.114.77.122，跟 `DB_HOST` 同一台 Aliyun 机器）是通的（实测 200）。所以**不是网络不通，纯粹是域名解析不稳**。
- 现有重试 `CRM_HTTP_RETRIES=1`（共 2 次）救不了：两次都在 ~1.7s 内打完、无退避，DNS 抽风往往比这长，两次一起挂。
- `.env` 里 `CRM_HTTP_TRUST_ENV=1` + 注释里的 `127.0.0.1:7890` 代理是**给"宿主机裸跑"写的**；在容器里 `127.0.0.1` 是容器自己、代理根本不可达。容器内没有任何 proxy 环境变量，实际走的是直连，所以代理这条与本问题无关。

## 修法

最稳的是**绕开 DNS**：在 `docker-compose.yml` 的 `app` 服务加一行 `extra_hosts`，把 `crm.amer.cn` 钉死到 IP（不改 SNI/证书，TLS 仍按域名校验，安全）：

```yaml
    extra_hosts:
      - "crm.amer.cn:47.114.77.122"
```

只需 `docker compose up -d`（重建容器、无需 `--build`）。之后解析永远命中、不再受 Docker Desktop DNS 抖动影响。

另外两种可选：给容器指定可靠 DNS（`dns: [223.5.5.5, 8.8.8.8]`，更通用但 Docker Desktop 有时会忽略）；以及给 CRM 客户端的重试加个 0.3–0.5s 退避（纵深防御，但救不了多秒级抽风）。
