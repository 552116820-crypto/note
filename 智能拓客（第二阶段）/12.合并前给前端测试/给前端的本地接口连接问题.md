## 先理解 mirrored 模式在做什么

WSL2 本质是台 Hyper-V 轻量虚拟机，有自己独立的 Linux 网络栈。传统 NAT 模式下它有自己的私网 IP（172.x），和 Windows 是两个世界。**mirrored（镜像）模式**则是把 Windows 的网卡和 IP「镜像」进 WSL——所以你在 WSL 里 `ip addr` 看到的 eth0 也是 `192.168.40.252`，**同一个 IP，两套网络栈共用**。

为了让这个共用不穿帮，微软专门修了两条通道：

1. **回环互通**：`127.0.0.1`/`localhost` 双向打通——Windows 连 localhost 能到 WSL 里的监听，反之亦然。这是显式实现的特性。
2. **入站镜像**：从**物理网卡**进来的外部流量（局域网别的机器发来的包），镜像层会检查 WSL 里有没有进程监听该端口，有就把包导进 WSL（过 Hyper-V 防火墙，所以你之前要跑那条 Allow 命令）。

## 三条路径逐条对照（都有我们的实测佐证）

**① 局域网别的机器 → 192.168.40.252:8000 ✅**
包从网线/Wi-Fi 真实到达 Windows 物理网卡 → 走「入站镜像」通道 → 导入 WSL → 容器。这是设计目标场景。

**② Windows 本机 → localhost:8000 ✅**
走第 1 条专修的回环通道。实测通（curl.exe 返回 healthz）。

**③ Windows 本机 → 192.168.40.252:8000 ❌（你问的这个）**
Windows 的 TCP/IP 栈发现目标 IP **是本机自己的地址**，于是做了所有操作系统都会做的优化：**这个连接不出网卡**——不会真的把包发到交换机再绕回来，而是在自己栈内部按"本地投递"处理，直接查**Windows 自己的监听表**找 8000 的收件人。

问题就出在这：8000 的监听在 **WSL 那套栈**里。还记得我们用 `netstat.exe` 查过吗——Windows 的端口表里**根本没有 8000**。而这条"栈内部捷径"既不经过物理网卡（所以踩不到入站镜像的钩子），目标又不是 127.0.0.1（所以也不走回环互通通道）。两条修好的桥都绕过了，捷径尽头查无此人——死路。

打个比方：公司大楼（你这台机器），前台（物理网卡）收外部快递会帮忙转给楼里的 WSL 工位（通道②）；内线电话 127 有专线（通道①）。但你自己在楼里往「公司大门的街道地址」寄快递，邮件室一看地址是自家楼，根本不发出去，直接翻**本楼通讯录**找收件人——而 8000 这位登记在 WSL 的册子上，本楼通讯录没有，包裹无人签收。

**④ 顺带：为什么 WSL 里访问 192.168.40.252:8000 反而通？**
同样的"本机 IP 内部投递"逻辑，但发生在 **WSL 这套栈**里（那个 IP 在 WSL 里也是本机地址）——而 8000 的监听恰好就在这一侧，所以内部投递一查就命中。

## 为什么表现是「超时」而不是「拒绝连接」

正常情况下连一个没人监听的端口，系统会立刻回 RST（拒绝连接，秒失败）。但这条死路上，SYN 包是被 Hyper-V 防火墙/镜像过滤层**静默丢弃**的——没有任何一方回包；如果再叠加 Postman 走系统代理，请求还会先绕进 7890 那个代理挂起。没有回应 = 客户端一直等到超时。这也是当初 Postman 给你看「Timeout」而不是「Connection refused」的原因，迷惑性就在这。

## 实务结论

| 测试者 | 该用的地址 | 原因 |
|---|---|---|
| 你本机（Postman/浏览器） | `localhost:8000` | 走专修的回环通道 |
| WSL 终端 | `localhost` 或 IP 均可 | 监听就在这一侧 |
| 局域网其他设备（前端同事、你手机） | `192.168.40.252:8000` | 走入站镜像通道 |

所以「本机用 IP 测不通」**不携带任何关于别人能不能连上的信息**——它走的根本不是同一条路。要验证外部可达性，必须用真正的外部设备（这就是我建议拿手机试的原因）。

## 1. 自动加载：靠文件名约定，不靠任何参数

你在项目目录跑**不带 `-f`** 的任何 `docker compose` 命令（up/down/logs/config…）时，它按约定找两个文件：

```
docker-compose.yml            ← 基础（主文件）
docker-compose.override.yml   ← 存在就自动叠上去，不存在就跳过
```

不需要注册、不需要开关——**文件在，就生效；删掉/改名，就还原**。这就是为什么我说"什么都不用额外做"。

反过来要注意：一旦你显式写了 `-f docker-compose.yml`，compose 就**只用你指定的文件**，override 不会自动叠加（我之前查日志时写 `-f` 就是这样，对 logs 无所谓，但 `up` 时别带 `-f`，否则端口又变回仅回环）。

## 2. 合并规则：不同类型的字段，叠法不一样

两个文件按「主文件打底、override 盖上去」合并，但"盖"的方式分三种：

| 字段类型 | 例子 | 合并行为 |
|---|---|---|
| 单值 | `image`、`restart` | override **整个替换** |
| 字典 | `environment` | 逐 key 合并，同名 key override 赢 |
| **列表** | **`ports`**、`volumes` | **追加**（两边都保留！）|

注意最后一行——这正是坑所在：如果 override 里只写

```yaml
    ports:
      - "8000:8000"
```

合并结果会是**两条**映射：`127.0.0.1:8000:8000` **加** `8000:8000`，同一宿主端口两条规则，轻则冗余、重则绑定冲突。

## 3. `!override` 标签：把"追加"改成"整体替换"

所以我写的是：

```yaml
    ports: !override
      - "8000:8000"
```

`!override` 是 Compose 规范的 YAML 标签（较新版本支持，你的环境实测可用），意思是"**这个列表别跟主文件合并，整个换成我这份**"。于是最终只剩一条 `8000:8000`——上面 `docker compose config` 的输出验证了这一点。还有个兄弟标签 `!reset`，作用是把主文件里的某字段整个删掉。

## 4. 为什么用这个模式（而不是直接改主文件）

| | 主文件 `docker-compose.yml` | `docker-compose.override.yml` |
|---|---|---|
| 进 git | ✅ 是，团队/服务器共用 | ❌ 已 gitignore，只在你机器存在 |
| 端口策略 | `127.0.0.1`（生产安全：只许 nginx 反代访问）| `8000:8000`（本机联调：开给局域网）|

服务器上部署时拉的代码里**没有** override 文件 → 自动用主文件的安全配置；你本机有 → 自动用开放配置。**同一套命令，两种环境各得其所**，且永远不会把"端口大开"误提交带上生产——这比直接改主文件再"记得别提交"可靠得多。

## 5. 怎么核实当前生效的配置

任何时候想知道"合并完到底是什么"，跑：

```bash
docker compose config          # 输出最终生效的完整配置
docker compose config | grep -A4 ports    # 只看端口部分

排查"为什么端口/环境变量不对"这类问题时，第一步永远是看这个合并结果，而不是分别盯两个文件猜。

## 为什么会这样

`docker compose` 启动时的内部逻辑大致是：

```
1. 在当前目录找主文件，按这个写死的名单顺序：
   compose.yaml → compose.yml → docker-compose.yml → docker-compose.yaml
2. 找到主文件后，再看有没有对应的 override 名字：
   compose.override.yaml / docker-compose.override.yml ...

## 你其实天天在用这种机制

| 文件名 | 谁在认它 | 你注册过吗 |
|---|---|---|
| `.gitignore` | git 自动读 | 没有，名字对就行 |
| `README.md` | GitHub 自动渲染到仓库首页 | 没有 |
| `index.html` | Web 服务器访问目录时自动返回它 | 没有 |
| `Dockerfile` | `docker build` 默认找它 | 没有 |
| `.env` | docker compose 自动读做变量替换 | 没有 |

`docker-compose.override.yml` 和它们同理——这种设计风格就叫「**约定优于配置**」（convention over configuration）：用一个大家公认的默认名，省掉每次手动指定的麻烦。

## 对应的"显式配置"方式是什么

如果不想靠名字，也可以手动指定文件，这就是 `-f`：

```bash

```

但代价是**每条命令都得带全这串 `-f`**，忘一次就少一层配置。而靠约定名，你和同事只要 `docker compose up -d` 这一条朴素命令，各自机器上有什么 override 就自动叠什么——这正是当初选它的原因。

**User:**

DHCP是什么

**Assistant:**

DHCP（Dynamic Host Configuration Protocol，动态主机配置协议）是局域网里**自动分配 IP 地址**的机制——你电脑的 `192.168.40.252` 不是它天生就有的，而是连上网络那一刻，路由器临时"租"给它的。

## 工作过程（简化版）

1. 你的电脑连上 Wi-Fi/网线，对着网络喊一嗓子："谁能给我个 IP？"
2. 路由器（局域网里的 DHCP 服务器）从自己管理的地址池（比如 `192.168.40.2` ~ `192.168.40.254`）里挑一个没人用的，回复："给你 `192.168.40.252`，**租期** 24 小时，顺便告诉你网关和 DNS 是什么。"
3. 你的电脑拿着这个地址用，租期快到时自动续租。

好处显而易见：办公室几十台设备，没人需要挨个手动配 IP、还得保证不重复——全自动。你家里的手机、电视连 Wi-Fi 拿到 IP 也是同一套机制。

## 为什么我提醒你"IP 可能会变"

关键就在**"租"**字上。续租时路由器通常会续给你同一个地址，但**没有任何保证**：

- 电脑关机久了、租约过期，地址被回收分给别人；
- 你再开机时，路由器从池子里随手拿一个空闲的给你——可能就成了 `192.168.40.87`；
- 路由器重启、网管调整地址池，也可能洗牌。

类比：共享办公室的工位。前台每天给你安排一个空位，多数时候你坐的还是昨天那个，但哪天来晚了位置被占，你就换地方了——而前端同事手里记的是"找 252 号工位那个人"，自然就找不到你了。

## 对你的实际影响和对策

**影响**：发给前端的 `http://192.168.40.252:8000` 里这个 IP 一变，链接全失效，而且症状就是"昨天还好好的今天突然超时"，容易让人往代码上怀疑。

**对策**：

1. **症状识别**：前端突然连不上 → 你先跑 `ip -br addr`（或 Windows `ipconfig`）看 IP 变没变，变了就把新地址告诉前端，完事。
2. **一劳永逸（推荐，如果要长期联调）**：找网管在路由器上给你的电脑做 **DHCP 静态保留**（也叫 IP 绑定）——按你网卡的 MAC 地址登记"这台机器永远分 192.168.40.252"。机制还是 DHCP，但结果固定了，路由器界面上一般叫"静态租约 / 地址保留 / IP-MAC 绑定"。

顺带串一下你已有的知识：这也是为什么 `localhost`（127.0.0.1）永远不用担心——它不经过 DHCP，是每台机器写死指向自己的固定地址；会漂的只有这种从路由器租来的局域网地址。
