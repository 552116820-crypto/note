本机到远程 MySQL（8.138.104.14）的流量被代理路由到 us-west-2 出口，隧道丢弃 >1.5KB 的 TCP 段——小查询正常、大响应永久挂起。这解释了 pytest 里 2 个 auth测试超时，也意味着今后在这台机器上任何直连该库的工（DBeaver、mysql cli 等）都可能出现"小查询正常、大查询卡死"的现象；代理侧把 8.138.104.14（阿里云 IP）改成直连即可根治。


不是代码或机器性能的问题，是**网络路径**的差距——这次冒烟过程中已经实测定位了：

**服务器上快**：引擎跑在数据库旁边（Docker 部署，同机或同内网），每次 MySQL 往返延迟亚毫秒级。

**本地慢，有三层原因叠加：**

1. **绕地球半圈的路由**：本机到 MySQL（8.138.104.14，阿里云）的流量被代理路由到了 **us-west-2（美国）出口**——数据库端看到的连接来源是一台 EC2 机器。也就是 本机 → 美国 → 阿里云 的大迂回，光 TCP 握手就要约 4 秒，每次查询往返都背着几百毫秒延迟。

2. **隧道 MTU 黑洞**：这条代理链路会丢弃 >1.5KB 的 TCP 段（实测 1KB 响应 0.3 秒返回，2KB 响应永久挂死直到超时）。前两次冒烟就是死在这上面。为绕过它，我把冒烟进程的 MSS 钳到了 1000 字节——能跑通了，但吞吐量只有约 87KB/s（1MB 数据要 12 秒）。

3. **检测的写库阶段是"高频小往返"型负载**：1039 个销售分 21 批，每批要读活跃预警（含 JSON 大字段）+ 逐条写入/更新预警记录。这种模式对延迟极其敏感——服务器上每个往返 1ms，本地每个往返 300ms+，差 300 倍就体现在这了。

顺带说明：CRM API 那部分本地其实不算慢（代理对国内域名走了直连，全量 13.6 万订单约 13 分钟拉完），慢的瓶颈集中在 MySQL 持久化阶段。

**根治办法**：在代理客户端里给 `8.138.104.14`（或整个阿里云网段）加直连规则，不走美国出口——延迟和 MTU 问题会一起消失，本地冒烟速度会接近服务器水平。或者以后全量冒烟直接在服务器上跑（记得带 `CHURN_WARNING_EXPORT_ENABLED=false`）。
