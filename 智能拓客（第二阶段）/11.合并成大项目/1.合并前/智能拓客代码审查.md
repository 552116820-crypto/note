1. **鉴权里“从 token/session 推导 employee_code”是什么意思**

请求头加 token 是手段，但关键是：后端不能再完全相信请求体里的 `employee_code`。

比如现在请求体可以传：

```json
{ "employee_code": "000068" }
```

如果只是“请求头有 token 就放行”，那一个合法用户仍然可以把 body 改成别人的 `employee_code`，查询或写入别人的业务数据。

正确做法是：

- 前端请求头带 `Authorization: Bearer <token>`；
- 后端校验 token 是否有效；
- 后端从 token 或 session 里拿到当前登录人的身份，例如 `employee_code=000068`；
- 普通业务接口使用后端推导出的 `employee_code`，而不是相信 body 里传来的值；
- 如果允许管理员代查，再额外做权限判断。

2. **CRM passport salt 硬编码是什么意思**

MD5 只是算法，不是秘密。`passport = md5(日期 + salt)` 里真正需要保密的是 `salt`。

如果 `salt` 被写死在代码或公开配置样例里，别人知道规则后就能自己算每天的 `passport`，相当于这个 CRM 接口的“口令”泄露了。

不是说 CRM 不能用 MD5，而是说：参与 MD5 的那个 salt 应该只放在服务器 `.env` 或密钥管理里，代码里不应该有真实默认值。现在代码默认值和 `.env.example` 都是 `amer_fl_output`，如果这是真实生产 salt，就需要换掉并移出默认值。

3. **`/skip` 写入后缓存 TTL 内保持旧状态是什么意思**

推荐列表会被缓存，默认 TTL 是 300 秒。

流程是：

- 第一次请求 `/score-and-info`，后端算好列表，并把“哪些客户被 skip 后沉底”的结果一起缓存；
- 用户调用 `/skip` 写入数据库；
- 但是已经缓存好的列表没有被清掉；
- 所以在缓存过期前，再请求同样位置/半径的列表，可能还是旧排序、旧 `unseen_total/skipped_total`。

README 里也写了“下次缓存重算时沉底”。这不是数据没写进去，而是缓存里还保留着旧计算结果。

4. **API 日志的内存 DoS 风险是什么意思**

DoS 是 Denial of Service，拒绝服务攻击。意思是攻击者不一定要破解系统，只要让服务资源耗尽，就能让正常用户访问不了。

这里的问题是：开启 API 日志后，中间件会先把整个请求体读进内存。如果有人发一个非常大的 body，或者并发发很多大 body，服务内存会被占满，进程可能变慢、被系统杀掉，或者影响其他请求。

解决方式一般是：

- nginx/网关限制请求体大小；
- FastAPI 中间件读取 body 时也设置最大字节数；
- 超限直接返回 413；
- 日志只记录截断后的内容。

5. **数据库错误码怎么不统一**

现在连接数据库失败会被映射成约定错误码：

- 业务库不可用：`1503`
- 状态库不可用：`1505`

但这个映射主要覆盖“建立连接失败”。如果连接成功了，后面执行 SQL 或读取结果时超时、断连、连接被重置，这些异常目前可能直接穿透到全局兜底，返回：

```json
{ "code": 1500, "message": "服务内部错误，请稍后重试" }
```

这就和文档里的“数据库不可用返回 1503/1505”不一致。用户看到的是内部错误，但实际是数据库读超时或连接中断。

后续应该在 `fetch_all/fetch_one/execute` 这一层统一捕获 PyMySQL 的读写异常，再转换成 `DBUnavailable` 或 `LogDBUnavailable`。
