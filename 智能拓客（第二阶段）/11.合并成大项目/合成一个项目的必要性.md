
## "给不给前端提供接口"——这不是该不该分开的依据

一个服务里**同时有**对外 API 和后台批处理，太正常了——你的 churn-warning-engine **本身就是这样**：既有 FastAPI 接口，又有 Celery 定时批处理。所以"churn 不对前端、lead-gen 对前端"只说明**它们暴露的入口不同**，不代表要两个代码库。决定"一个项目还是两个"的是别的因素，而这些因素**几乎全指向放一个项目**。

## 建议：合成一个项目（模块化单体），理由很硬

1. **churn 本来就是按这个架构建的**：它已经有 `app/churn/`（业务模块）+ `app/shared/`（db / config / logging 公共层）——这就是"一个库、多模块"的模板，`app/shared/` 就是留给第二个模块的。lead-gen 进来就是 `app/lead/`。
2. **同领域、同库、同一拨人**：都是"AI 销售助手"，都连 `amer_sales_assist` + 只读的 `amer_crm_v2`，都你维护。
3. **公共设施复用不重复**：CRM 只读访问、MySQL 连接/配置/日志、CRM API 客户端、Docker/nginx/Celery/deploy——分两个库这些得复制一遍。
4. **运行形态不同 ≠ 要分库**：批处理 vs 请求响应，用"同一镜像跑不同进程"解决——你 compose 里已经是 `api` + `celery` 同一套代码两个进程。lead-gen 路由挂 `api`，churn 批处理留 `celery`，互不影响；nginx 上只把 `/api/v1/leads/*` 这种该公开的暴露出去就行。

## 什么时候才该拆两个库（对照，一条不占就别拆）

- 不同团队各自 own；
- 有**硬性**的独立伸缩 / 独立发布要求（lead-gen 要弹性扩很多实例、churn 只是每天一个定时批）；
- 必须强故障隔离；
- 技术栈不同。

你这边一条都不占。

## 落地两条路（smart_lead_generation 我已搭成独立服务）

- **(A) 现在并进去**：挪成 churn 仓库的 `app/lead/`，公共部分并进 `app/shared/`，统一镜像。是**追加一个模块**、不动 churn 现有模块，风险低；趁早合最省事。
- **(B) 先独立上线、以后再合**：快，但两套基础设施会慢慢长歪、越来越难合。

我选 **(A)**。顺手把仓库名从 `churn-warning-engine` 改成平台名（如 `amer-sales-assistant`），churn / lead 都是它下面的模块——和你这次把 `churn_db` 升级成 `amer_sales_assist` 是**同一个动作：从"单功能"升级成"平台"**。

## 文档：要，但别单独建"文档仓库"

合成一个项目后：
- 仓库根 `README` + `docs/` 放**平台总览**（有哪些模块、共用哪个库、怎么部署）；
- churn 现有那套 `docs/`（technical-design / migration-guide / runbook / api-guide）变成 **churn 模块文档**，lead-gen 的放一起。

一个新人 clone 一个库就看懂整个"销售助手"——正好是你要的"方便团队接管"。

## 顺带澄清

服务器上 `churn-warning-engine`、`dify`、`wecom-deepexi-aibot` 这些**目录是"部署落地"，跟"代码仓库怎么组织"是两回事**。就算合成一个仓库，部署时它仍可以是服务器上一个目录、一套 compose。所以服务器现在长什么样，不影响你这个"合不合"的决定。

---

一句话总结：**按平台合成一个模块化单体仓库（churn + lead 两个模块、共用 `app/shared/` 和 `amer_sales_assist` 库），而不是按"前不前端"拆。** 要的话,下一步我可以帮你把 smart_lead_generation 重构成 `app/lead/` 模块并入那套结构。

**User:**

问一下，以后开发新的功能，和客户流失预警，智能拓客并行的业务，是在合并成一个项目的文档下开发，还是另开一个文档，独立开发完之后再合并进去

**Assistant:**

直接答：**在平台仓库里、开个功能分支、作为新模块 `app/<新模块>/` 开发**——不要"另起一个独立项目、做完再合并进来"。

## 为什么不是"独立开发完再合并"

对**同领域、同栈**的新业务，"先在外面单独搞、再合回来"是个反模式：

1. **开发期就用不上 `app/shared/`**——而它正是合成一个项目的全部意义。你要么把 db/config/日志/CRM 客户端**复制一份**到独立项目（合并时再去重，痛苦），要么对着一份会过时的拷贝写。
2. **会漂移**：你在外面单独做几周，`app/shared/` 在主干又改了，两边越走越远，合并时一堆冲突。
3. **CI/测试/部署都要重搭一遍**，合并时再拆掉。

而你想要的"独立、互不干扰"，在 monorepo 里**靠模块边界 + 分支就已经有了**——新模块在 `app/<新>/`，根本不碰 `app/churn/`、`app/lead/`；在 `feat/<新模块>` 分支上做，做完测好再合主干。**这里的"合并"是一次 git 分支合并，不是合并两个仓库。** 这才是你要的工作流。

## 保持干净的纪律（也是"以后想拆也能拆"的前提）

1. 每个模块**自包含**在 `app/<模块>/`（自己的 routers/services/repositories/schemas）；
2. **只通过 `app/shared/` 共享**——别让 `app/lead/` 直接 import `app/churn/`（跨模块直连会把模块化的好处毁掉）；要共享就提升到 `app/shared/`；
3. 每个模块**自己的表前缀**（`churn_*`/`lead_*`/`<新>_*`）和 `tests/<模块>/`；
4. 批处理类的挂 `celery` 进程，对外 API 类的挂 `api` 进程，同一个镜像。

做到这几条有个额外好处：**模块边界清晰，将来某个模块真的要独立成单独服务（满足上次那个"该拆"清单时），抽出去很容易。** 所以"现在放一起开发"不是单向门、不会把你锁死。

## 什么时候才"先独立、后并入"（少数例外）

- **扔掉式原型**：只是快速验证想法、反正要重写——拿个 scratch 目录随便试，验证完在仓库里正经写。
- **技术栈不同**：比如一个 Go 服务、或重型 ML 训练管线，塞不进这个 Python/FastAPI 单体——那它本就是独立服务、独立仓库，不是"以后合"。
- **一开始就决定它是永久独立部署**（对照上次"该拆两个库"的清单）——那就一直独立，也不存在"合"。

这几种不沾，就**直接在平台仓库开分支做**。

---

一句话：**新业务 = 平台仓库里 `feat/x` 分支上的一个新 `app/<模块>/`**；分支期间常 rebase 主干防漂移，做完测好合主干。"独立做完再合"只留给原型、异栈、或注定要独立部署的东西。
