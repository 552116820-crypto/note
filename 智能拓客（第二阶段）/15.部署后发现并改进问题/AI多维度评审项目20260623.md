# AMER Sales Assist 技术评审报告

项目版本：v2.4.2 | 评审日期：2026-06-23 | 测试基线：538 passed

## 1. 架构设计

### 现状观察

项目是一个规范的模块化单体：`app/shared/`（平台件）+ `app/churn/`（流失预警）+ `app/lead/`（智能拓客），两个装配点 `app/main.py` 与 `app/celery_app.py` 是唯一允许跨模块 import 的地方。

验证隔离契约：
```
grep -rn "from app.lead|from app.churn" app/shared/  →  空
grep -rn "from app.churn" app/lead/                  →  空
grep -rn "from app.lead" app/churn/                  →  空
```
**三条铁律全部干净，无反向依赖，无横向依赖。**

### 优点

- **依赖方向完全落实**：shared → 无业务 import；churn/lead → shared；两者互不感知。与 `docs/architecture.md` 描述完全吻合。
- **构造注入彻底**：`app/shared/db_mysql.py`（`MySQLConfig`）、`app/shared/cache.py`（`build_cache()`）、`app/shared/llm.py`（`DeepSeekClient`）均为全参构造，不读任何 settings；业务装配在 `app/lead/db/pool.py`、`app/lead/cache/store.py` 等薄层完成——比多数 FastAPI 项目干净。
- **churn 纯任务驱动设计合理**：Celery beat 三任务（`app/celery_app.py`）→ `tasks.py` → `factory.get_cached_service()` → `service.run_detection()`，无 HTTP 面、无外来依赖，闭环清晰。
- **双轨错误契约有边界**：`app/main.py:142-185` 按路径前缀隔离，lead 路径走 envelope，非 lead 路径维持 FastAPI 默认，设计有意识、有记录。
- **`_LeadScopedBodyLimit`（main.py:107-118）** 显式按路径限制中间件作用域，防止污染其他模块的契约，设计精细。

### 问题/风险

**P1 — `app/main.py` 的 lifespan 对 churn 强耦合**：`main.py:62-66` 在 lifespan 里调 `from app.churn.factory import get_cached_service as churn_service` 并 `churn_service()` 阻断启动。lead 也有类似初始化但走 try-except 不阻断（main.py:68-73）。问题在于：main.py 作为 lead+churn 的装配点，churn 初始化失败会导致 lead API 完全不可用——两个不相关的业务模块在生命周期层面有隐性耦合，与"模块解耦"的声称有出入。

**P2 — churn `factory.py` 大量 `os.getenv` 直调**：`factory.py:24-100` 中 `_build_crm_repository()` 和 `_build_churn_repository()` 直接读 `os.getenv("CRM_API_URL")` 等十余个 env，没有经过 pydantic 的 settings 对象——与 lead 模块的规范做法（settings 对象统一读取）不一致，也无法在 mypy 下获得类型检查保护。`lead/config/settings.py` 把所有 env 收进 pydantic `Settings` 对象，`churn/config.py` 和 `factory.py` 则混用。

**P3 — celery worker 和 api 共享同一镜像但职责不同**：`docker-compose.yml:39` celery 容器用 `command` 覆盖，而 Dockerfile `CMD` 是 uvicorn——这意味着 celery 进程也预装了 web 依赖，并不是真的隔离，虽然这在单进程单体中是合理的权衡，但文档没有明确说明这一取舍。

### 改进建议

1. 将 churn 的 lifespan 初始化从阻断式改为警告式（参照 lead 预热），在 `/healthz` 里暴露 churn 服务状态，让两个模块在生命周期层面真正解耦。
2. 把 `app/churn/factory.py` 中的直接 `os.getenv` 调用收进 `app/churn/config.py`（`ChurnSettings` 或类似的 pydantic 模型），统一到与 lead 相同的 settings 风格。

## 2. 代码质量

### 现状观察

代码整体质量高于一般 FastAPI 项目。ruff 配置（`pyproject.toml:65-76`）已覆盖 E/F/W/I/UP/B/C4/SIM/PERF/BLE，mypy 覆盖全 `app/`（含 `check_untyped_defs=true`）。

### 优点

- **纯函数优先**：`app/lead/services/scoring.py` 全文无 I/O，全单元可测；`app/churn/anomaly_strategies/*.py` 同样（最长文件 `volume.py` 167 行）。
- **命名清晰**：`_dash`/`_num_or_dash` 语义自描述（`lead_service.py:24-35`）；churn 的 `WarningService`/`WarningPersister`/`WarningSettingsResolver` 职责划分明确。
- **错误处理防御充分**：`shared/cache.py` 的 Redis 后端所有操作都 `except Exception as exc`（BLE001）+ 降级，`shared/db_mysql.py` 的 `ping()` 吞一切异常恒返 bool——这些位置的宽 catch 是有意识的防御性设计，不是懒惰。
- **类型注解覆盖合理**：`from __future__ import annotations` 全覆盖，核心数据结构（`MySQLConfig`、`WarningRecord`、`Customer`、`Order` 等）均为 dataclass 或 pydantic。

### 问题/风险

**P1 — `app/lead/config/settings.py` Settings 初始化方式存在设计问题**：`settings.py:17-113` 在 `class Settings(BaseModel)` 的 field 默认值处直接调 `os.getenv()`，绕过 pydantic-settings 机制。这意味着 Settings 字段在类定义时就被固化（模块 import 时）——`monkeypatch` 环境变量对已实例化的 `settings` 对象无效，测试必须 `monkeypatch.setattr(settings, ...)` 而非 `monkeypatch.setenv()`。虽然已知问题，但这是一个隐患：如果有测试改了 env 变量期望 settings 跟着变，会静默失效。

**P2 — `app/churn/service.py:_frequency_warnings/_volume_warnings/_product_warnings`（第43-107行）三个私有方法签名几乎一致**（都接 `customer, orders, biz_date, *, settings_by_key, customer_configs, global_configs`），有 DRY 改进空间，但目前的 `_warning_detector()` 派发表（第127-133行）已经做了合理的多态封装，可接受。

**P3 — `crm_api_parsing.py` 中的多 key 解析（第13-33行）**：`_CUSTOMER_API_ID_KEYS`/`_CUSTOMER_API_NAME_KEYS`/`_CUSTOMER_API_SALES_NAME_KEYS` 是硬编码的 CRM 字段别名列表（6/5/6 个别名），这是针对真实 CRM API 字段不统一的防御，但没有注释说明各别名对应哪个 CRM 接口版本，未来维护者无法判断哪些别名还在生产中被使用。

**P4 — `engine.py`（第1-27行）是纯转发层**：导出 `_ewma`/`_prediction_confidence`/`_remove_iqr_outliers` 等下划线开头的"私有"函数。这些函数在 `__all__` 里出现，意味着它们成了公有 API——但名字仍有下划线前缀（暗示私有），命名与 re-export 语义矛盾。测试直接从 `engine` import 这些函数（测试合理），但函数名本身应去掉下划线或改注释澄清。

### 改进建议

1. 把 `Settings` 类改用 `pydantic_settings.BaseSettings` 或在 `__init__` 里显式读取（目前的 field-at-class-level-os.getenv 反模式在重构时会产生意外）。
2. 在 `crm_api_parsing.py` 的各别名列表上加注释，说明哪个别名对应哪个 CRM 接口/版本，便于日后裁剪。
3. 将 `engine.py` 导出的私有辅助函数（`_ewma`/`_remove_iqr_outliers` 等）去掉下划线前缀或在 `__all__` 注释中说明它们实际上是测试可见的公有辅助函数。

**C2 — `mysql_repo.py` 的真实 SQL 逻辑**：`test_mysql_repo.py` 用 SQLite 内存库跑 ORM，而生产用 MySQL。几处关键 SQL 差异：`mysql_insert(...).on_duplicate_key_update()`（`mysql_repo.py` 使用 MySQL 专属方言）在 SQLite 下不可执行，相关测试用了 `monkeypatch` 替换方言检查（`test_mysql_repository_validates_schema_instead_of_creating_tables`）。实际的 MySQL dedup 逻辑（`ON DUPLICATE KEY UPDATE`）在测试中未被端到端验证。

**C3 — lead 的真实数据库查询**：`app/lead/repositories/geo_repo.py` 的 Haversine SQL（含 `HAVING distance_km &lt;= %s`）、`employee_repo.py` 的 `COUNT(DISTINCT vbillcode)` GROUP BY——这些在测试中通过 `monkeypatch lead_service.query_leads` 桩掉，从未对真实 MySQL 执行。

**C4 — Celery 任务端到端**：`app/churn/tasks.py` 的 retry 逻辑（`max_retries=3, default_retry_delay=60`）没有测试；beat_schedule 时间正确性没有测试。

### 脆弱测试风险

- `tests/churn/test_mysql_repo.py:4 warnings`（SQLite DeprecationWarning）：`sqlalchemy.engine.default.py:952` 警告 Python 3.12 下 SQLite 日期适配器已弃用。当 Python 升级到 3.12+ 后，SQLite 测试可能开始静默报错，需要关注。
- `tests/lead/test_industries_fallback_http.py`：假设行业字典 CSV 快照存在于 `app/lead/data/`，这个文件的存在性在测试中是隐式依赖。

### 改进建议

1. 在 CI 中加 `test_crm_api_http.py`，用 `respx`（httpx mock）模拟 CRM 返回 HTML/302/gzip 等异常响应，覆盖 `_json_or_error` 的真实路径。
2. 对 `mysql_repo.py` 的 MySQL 专属 SQL（`ON DUPLICATE KEY UPDATE`、JSON 函数等）加集成测试层（可用 testcontainers 或 CI MySQL 服务），即使只覆盖 happy path 也比 SQLite 桩更真实。
3. 给 Celery `run_detection` 任务加至少一个"服务层抛出异常 → task retry 被触发"的单元测试。

## 5. 工程化

### 现状观察

Python 3.10-3.12（`pyproject.toml:10`），FastAPI/Starlette/Pydantic 全部带主版本 pin（`fastapi&gt;=0.116,&lt;0.117`）。Docker 多阶段构建，非 root 运行。

### 优点

- **依赖 pin 粒度恰当**：核心 web 栈 pin 到 minor 版本（`&gt;=0.116,&lt;0.117`），防隐式升级引起契约变化；数据库/Redis 客户端 pin 到 major。
- **Docker 多阶段构建合理**：builder/runtime 分离，runtime 镜像只有 site-packages + app/，非 root 运行（`USER app`），dev 工具不进镜像。
- **日志结构化处理到位**：`logging_config.py` 生产用 `JsonFormatter`，`json.dumps(ensure_ascii=False)` 正确转义，traceback 进 `traceback` 字段而非裸行追加。`X-Request-ID` 贯穿 nginx→api→日志（`middleware.py:32-43`）。
- **docker-compose.yml 的日志轮转配置**：`x-logging` anchor 统一 `max-size=10m/max-file=3`，防止磁盘打满，覆盖全部 4 个服务。
- **compose 层钉死 Redis 分库**：`api` 和 `celery` 服务的 `environment` 块里硬写 Redis URL（覆盖 .env），防止生产 .env 历史残留值 `/0` 造成分库混用。

### 问题/风险

**P1 — 没有 CI/CD 配置**：项目根目录无 `.github/workflows/`、`.gitlab-ci.yml` 等 CI 配置。所有质量门禁（ruff、mypy、pytest）依赖手动执行 `make` 命令，没有 PR 门禁。这是最高优先级的工程化缺失。

**P2 — `Makefile` 存在但未验证**：`CLAUDE.md` 说"或 `make format`/`make lint`"，但 Makefile 内容未读——这是测试/工具链的重要入口，如果 Makefile targets 与 pyproject.toml 脚本不对齐会造成混乱。

**P3 — `celery` 进程同时承担 worker 和 beat**：`docker-compose.yml:56` `celery -A app.celery_app worker --loglevel=info --beat`——worker 和 beat 合进一个进程，如果 worker 崩溃重启，beat 也会丢失调度状态（尽管 `-s /tmp/celerybeat-schedule` 做了持久化，但进程内的运行时状态仍会重置）。对于三个低频定时任务（流失预警）当前是可接受的，但扩容时需要拆分。

**P4 — `sqlalchemy_engine.py`（churn 用）的连接池参数来自 `os.getenv`**：未读完整内容，但根据 factory.py 的模式，churn 的 SQLAlchemy engine 可能也直接读 env，与 lead 的 settings 模式不一致（属 P2 的延伸）。

**P5 — 无可观测性端点（metric/trace）**：只有 `/healthz`（`main.py:129-137`），无 Prometheus metrics、无 Jaeger/OTEL 追踪。生产上排查"这次请求为什么慢"只能看 `lead_api_log` 表的 `cost_ms` 字段（写库本身 best-effort 可能丢），缺乏系统级可观测性。

### 改进建议

1. **立即**：加 GitHub Actions / GitLab CI，在 PR 上跑 `ruff check &amp;&amp; mypy &amp;&amp; pytest`，把现有质量门禁从手动变自动。
2. 把 celery beat 拆出独立容器（`command: celery beat`），worker 单独扩容不影响调度。
3. 接入 Prometheus + uvicorn metrics（`prometheus-fastapi-instrumentator` 或类似），补全系统级可观测性。

## 6. 安全性

### 现状观察

### 优点

- **SQL 参数化彻底**：所有用户输入都经参数化查询。`geo_repo.py:_GEO_SQL` 用 `%(lat0)s` 占位；`employee_repo.py:52` 的 IN 子句用 `",".join(["%s"] * len(ncc_codes))` + 值列表分离；`enrichment_repo.py:29-30` 同理——`ids = [int(c) for c in customer_ids]` 先强转 int 再用占位符，双重防御。lead_skip_repo 的 `SELECT`/`INSERT` 均参数化（`lead_skip_repo.py:14,21`）。
- **API log allowlist（fail-closed）**：`api_log.py:_BODY_ALLOW_KEYS` 是白名单而非黑名单，未来新增字段默认不落库，防止意外 PII 入库。
- **CORS 默认不挂**：`main.py:148-160`，`CORS_ORIGINS` 留空则完全不挂 CORSMiddleware，不存在"开着*"的风险。
- **非 root 容器运行**：`Dockerfile:22-23` 创建 `app` 用户并 `USER app`，减少容器逃逸面。
- **凭证不进镜像**：`.dockerignore`（根据 Dockerfile 设计推断）排除 .env；compose 通过 `env_file: .env` 在运行时注入。

### 问题/风险

**P1 — 接口完全无鉴权，且这是"钉死的契约"**：`test_security_surface.py:23-27` 显式把"无 Authorization / api-key 头也应正常服务"锁成测试契约。`employee_code` 只是业务员 ID，不是凭证——任何知道接口 URL 的人可以传任意 `employee_code` 查任意业务员的推荐列表、提交跳过记录。接口文档（`docs/lead/接口文档-对接前端.md`）可能描述了这是企微小程序内部调用（无浏览器、受企微鉴权保护），但 API 层面本身无防护。如果 URL 泄露，数据泄露风险为高。

**P2 — DeepSeek API Key 在 settings 中明文存储于内存**：`settings.deepseek_api_key: str = os.getenv("DEEPSEEK_API_KEY", "")` （`settings.py:95`）。这是常见做法，但 `/healthz` 接口暴露了 `industries`/`industries_source`，如果 settings 被意外打印（如日志记录整个 settings 对象），key 会泄露。需确认日志处理中 settings 对象不被序列化（当前日志配置未见此风险，但需持续注意）。

**P3 — `--forwarded-allow-ips *`（Dockerfile CMD）**：`uvicorn --forwarded-allow-ips *` 信任所有来源的 `X-Forwarded-For` 头，这意味着攻击者可以伪造 IP。当前项目没有基于 IP 的速率限制或访问控制，所以此风险影响面有限，但一旦将来基于 `request.client.host` 做判断，会存在 IP 欺骗风险。

**P4 — Redis 在 compose 网络内无认证**：`docker-compose.yml` 的 redis 服务无密码（无 `requirepass`）、仅 compose 内网访问。如果宿主机有其他容器或进程能访问 compose 网络，可以无密码读写 Redis（包括 Celery 任务队列）。对于小规模部署可接受，但需确认宿主机网络边界。

**P5 — GPS 坐标落库仍发生（历史数据）**：`CHANGELOG.md 2.4.2` 提到"本次起停记 lng/lat"，但 `_BODY_ALLOW_KEYS` 在 2.4.2 的当前代码里**包含** `lng` 和 `lat`（`api_log.py:51`）：

```python
_BODY_ALLOW_KEYS: frozenset[str] = frozenset(
    {"employee_code", "customer_id", "lng", "lat", "radius_km", ...}
)
```

而 CHANGELOG 写"刻意排除 `lng`/`lat`（业务员实时 GPS = 位置 PII，调用量分析用不上）"。**代码与 CHANGELOG 不一致**：当前代码 GPS 仍会落库，CHANGELOG 说已停记。这是一个潜在的隐私合规 bug。（注：如果 CHANGELOG 是对 2.4.1→2.4.2 的描述，而 2.4.2 主分支代码是最终态，那么 CHANGELOG 是错的，GPS 事实上仍在落库。）

### 改进建议

1. **中等优先级**：在 nginx 层添加基础速率限制（`limit_req_zone`），对 lead 接口按 IP 或企微 OpenID 限流，降低无鉴权带来的滥用风险。如果客户端携带企微 `openid`，可在中间件层做轻量 token 校验。
2. **立即核查** `api_log.py:_BODY_ALLOW_KEYS` 中 `lng`/`lat` 是否应该在 2.4.2 中被移除（CHANGELOG 和代码不一致），明确业务决策后修正其中一方。
3. 给 Redis 加密码（compose `requirepass` + 客户端连接 URL 带认证），即使仅在内网，防止意外暴露。

## (b) 全项目最关键改进项（按性价比排序）

### 改进项 1：加 CI/CD，把质量门禁从手动变自动
**性价比：极高。投入约 2 小时，永久收益。**

当前 ruff/mypy/pytest 全靠手动 `make`，没有 PR 门禁。一旦有人忘记跑，隐患直达生产。

GitHub Actions 示例（`.github/workflows/ci.yml`）：
```yaml
on: [push, pull_request]
jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -e .[all]
      - run: python -m ruff check app tests scripts
      - run: python -m mypy
      - run: python -m pytest -q
```

### 改进项 2：核查并修正 GPS 落库的 CHANGELOG vs 代码不一致

**性价比：高。隐私合规 bug，修正成本极低（删一行或改注释）。**

`app/lead/middleware/api_log.py:51` 的 `_BODY_ALLOW_KEYS` 包含 `lng` 和 `lat`，但 CHANGELOG 2.4.2 声称"刻意排除 lng/lat（业务员实时 GPS = 位置 PII）"。两者必有一错：
- 如果决策是"不落库 GPS"，则从 `_BODY_ALLOW_KEYS` 移除 `lng`/`lat`，并更新相关测试；
- 如果决策已反转（最终决定落库 GPS 供复盘），则修正 CHANGELOG。

这是一个潜在的隐私合规问题，需要在 2.4.3 前明确。

### 改进项 3：给 churn CRM HTTP 调用补 httpx mock 测试

**性价比：高。`respx` 库已在 httpx 生态，约半天工作量，覆盖最大测试空白。**

## 一、最关键:一条必须否决的 codex 建议(GPS 落库)

Codex 把 "api_log 落 `lng`/`lat`" 列为**安全 P5 / 改进项 2「隐私合规 bug,2.4.3 前必修」,建议从 allowlist 移除 lng/lat**。
