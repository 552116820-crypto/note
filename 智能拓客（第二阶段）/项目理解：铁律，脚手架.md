## 3. "shared 物理彻底干净"是什么意思

这是你自己的说法（docs 里没这四个字），对应的是**铁律#1 + 2026-06-13 结构 review 的成果**。拆开三个词：

- **干净** = `app/shared/` 只剩**平台件**：`cache / config / crm_passport / date_utils / db_mysql / errors / llm / logging_config / middleware`（9 个实体 + `__init__.py`= ls 出来 10 个）。每个都满足"≥2 模块共用，或本就是平台契约"。

- **物理**（关键词）= 不是"逻辑上/import 方向上干净"，而是**文件本身搬走了**。重构前，churn 专属的 CRM/持久层代码（`crm_*`、`crm_models`、`sqlalchemy_engine`、`business_calendar`）**人就坐在 `app/shared/` 目录里**（逻辑上属于 churn，物理上挤在 shared）。现在它们**搬回了 `app/churn/`**——我刚 `ls app/churn/` 证实：`crm_api*`、`crm_mysql`、`crm_memory`、`crm_models`、`sqlalchemy_engine`、`business_calendar`、`mysql_repo*` 全在 churn 自家。

- **彻底** = shared 17→10，churn 专属件**一个不留**；连"只被 churn 用"的也不许放 shared。

对照理解：如果只是"逻辑干净"，可能 shared 里还塞着 churn 专属代码、靠注释说"这其实是 churn 的"。**物理干净 = 眼见为实，`ls app/shared/` 里就是没有任何业务专属文件**，grep 业务 import 也为空。

## 4. "项目脚手架"是什么意思，你这项目的脚手架是哪些

**脚手架（scaffolding）** = 盖楼时的支撑骨架；软件里指"让你能快速、规范搭出功能的骨架结构 + 样板"。docs 里用的词是"**骨架**"（`module-guide.md:6` 目录骨架）。你这项目的脚手架有**两层**，分清很重要：

### A. 平台脚手架（底盘——已搭好、供模块插入）

新模块**不动它**，只往上插：

- **`app/shared/` 9 个平台件**：errors / middleware / cache / db_mysql / llm / config / crm_passport / date_utils / logging_config
- **两个装配点**：`app/main.py`（唯一 FastAPI 入口：路由 / 中间件栈 / 异常契约 / lifespan / `/healthz`）+ `app/celery_app.py`（唯一 Celery 入口：broker/result/beat_schedule）
- **平台契约**：200-envelope 错误契约 + 错误码分段、Redis 分库 db0/db1/db2、数据面两库（`amer_crm_v2` 只读 / `amer_sales_assist` 可写）

### B. 模块脚手架（模板——新建模块时照抄的骨架）

定义在 `docs/module-guide.md`，**参照实现就是 `app/lead/`**：

- **目录骨架**（`:6`，按需建、空目录不占位）：`config/settings.py`、`errors.py`、`schemas/`、`routers/`、`services/`、`repositories/`、`db|cache|clients/`、`middleware/`、`data/`、`tasks.py`
- **装配模式**（`:25`）：薄装配文件把 settings 翻译成 shared 件构造参数（照抄 `app/lead/db/pool.py`），**不留转发 shim**

> 注意：churn 是**特例**，不是脚手架样板——它扁平结构、无 HTTP 面、自带一套 CRM 抽象。`module-guide.md:107` 明确"新模块勿照搬 churn"。**要照抄就照 `app/lead/`**。
